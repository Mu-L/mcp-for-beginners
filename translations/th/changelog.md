### การปรับความสอดคล้องกับข้อกำหนด MCP & การตรวจสอบตัวอย่าง

ตรวจสอบความถูกต้องของหลักสูตรกับ **ข้อกำหนด MCP 2025-11-25** ปัจจุบันและ SDK อย่างเป็นทางการล่าสุด จากนั้นแก้ไขการอ้างอิงข้อกำหนดที่ล้าสมัยที่เหลืออยู่และยืนยันว่ายังสามารถสร้างและรันตัวอย่างหลักได้

#### การแก้ไขเวอร์ชันข้อกำหนด (2025-06-18 / 2025-03-26 → 2025-11-25)

อัปเดตเนื้อหาภาษาอังกฤษที่ยังอ้างว่าฉบับแก้ไขข้อกำหนดเก่าเป็นมาตรฐาน *ปัจจุบัน/ล่าสุด* และเปลี่ยนลิงก์ไปยังเส้นทางข้อกำหนดแบบแคนอนิเคิลของ `modelcontextprotocol.io`:
- **05-AdvancedTopics/mcp-security/README.md**: อัปเดตแบนเนอร์ "Current Standard", บทนำ, หัวข้อหลักการรักษาความปลอดภัย, หัวข้อข้อกำหนดบังคับ, ส่วน Microsoft Entra ID, ลิงก์อ้างอิง & ทรัพยากร, และประกาศความปลอดภัยปิดท้าย (8 อ้างอิง) เป็น 2025-11-25
- **05-AdvancedTopics/mcp-transport/README.md**: อัปเดตลิงก์ข้อกำหนดทรัพยากรเพิ่มเติมและแบนเนอร์ "Current Standard" เป็น 2025-11-25
- **05-AdvancedTopics/mcp-realtimesearch/README.md**: แทนที่ลิงก์ `2025-03-26` เก่าเรื่อง security-and-trust ด้วยหน้าการปฏิบัติความปลอดภัยที่ดีที่สุด ของ 2025-11-25
- **03-GettingStarted/14-sampling/README.md**: อัปเดตลิงก์เอกสารการสุ่มตัวอย่างอย่างเป็นทางการเป็น 2025-11-25
- **03-GettingStarted/05-stdio-server/README.md**: อัปเดตอ้างอิง "ข้อกำหนด MCP ปัจจุบัน" ในรูปแบบปัจจุบันและลิงก์ทรัพยากรเพิ่มเติมเป็น 2025-11-25 (บันทึกการเลิกใช้ SSE ที่เป็นประวัติศาสตร์ถูกเก็บไว้เพื่อความถูกต้อง)

### การเสริมความปลอดภัย & การแก้ไขช่องโหว่ใน Dependency

ดำเนินการตรวจสอบความปลอดภัยเต็มรูปแบบใน manifest ทุกตัวและซอร์สโค้ดตัวอย่าง จากนั้นแก้ไข npm advisory ตามรายงานทั้งหมดและการตรวจพบที่ระดับโค้ด หลังแก้ไข `npm audit` รายงานว่า **0 ช่องโหว่** ในทุกไดเรกทอรีที่ตรวจสอบ

#### ช่องโหว่ npm Dependency (แบบแฝง) — แก้ไขแล้ว

ตรวจสอบไฟล์ `package-lock.json` ที่คอมมิตทั้งหมด 15 ไฟล์ ช่องโหว่จำกัดเฉพาะ dependency แบบแฝงที่ดึงเข้ามาโดย MCP Inspector dev tool, ลูกค้า OpenAI, และ MCP SDK; ทั้งหมดแก้ไขแล้วโดยไม่ส่งผลกระทบต่อตัวอย่าง:
- **10-StreamliningAIWorkflows.../lab4/code/github_mcp_server/inspector** และ **lab3/code/weather_mcp/inspector**: ปรับเวอร์ชัน `@modelcontextprotocol/inspector` (`0.16.6` / `0.14.1` → `0.22.0`), ซึ่งล้าง advisory ของ `ajv`, `brace-expansion`, `diff`, `path-to-regexp` และ `ws` ที่รวมมาในแพ็กเกจแล้ว เพิ่มรายการ `overrides` ใน npm บังคับใช้แพตช์ `shell-quote@1.8.4` เพื่อกำจัด advisory ร้ายแรงสุดท้ายที่มาพร้อม `concurrently`; สร้างไฟล์ล็อกใหม่สองฉบับ (ตอนนี้ไม่มีช่องโหว่)
- **03-GettingStarted/samples/typescript**: `npm audit fix` อัปเดต package transitive `qs` (ระดับกลาง) เป็นเวอร์ชันที่แพตช์แล้ว
- **03-GettingStarted/samples/javascript**: `npm audit fix` อัปเดต package transitive `hono` (ระดับกลาง) เป็นเวอร์ชันที่แพตช์แล้ว
- **03-GettingStarted/03-llm-client/solution/typescript**: `npm audit fix` อัปเดต package transitive `form-data` (ระดับสูง) เป็นเวอร์ชันที่แพตช์แล้ว
- **03-GettingStarted/11-simple-auth/solution/typescript**: สร้างไฟล์ `package-lock.json` ที่ขาดหายเพื่อให้โครงการสร้างและตรวจสอบได้ (0 ช่องโหว่)

#### การแก้ไขในระดับโค้ดเพื่อความปลอดภัย (OWASP A03: Injection)

- **10-StreamliningAIWorkflows.../lab4/code/github_mcp_server/src/server.py**: เอา `shell=True` ออกจากเครื่องมือ `open_in_vscode` เดิมเรียกใช้ `subprocess.run(["start", "", vscode_path, folder_path], shell=True)` ทำให้ shell metacharacters ในพาธโฟลเดอร์สามารถถูกตีความโดย `cmd.exe` (ช่องโหว่การฉีดคำสั่ง) เปลี่ยนมาใช้การรัน `Code.exe` ที่แก้ไขแล้วโดยตรงพร้อมส่งอาร์กิวเมนต์โฟลเดอร์ — ไม่มี shell — ซึ่งทำงานเหมือนกันและปลอดภัย

#### การตรวจสอบ Dependency ของ Python

- ตรวจสอบทุกไฟล์ requirements ของ Python ด้วย `pip-audit` พบว่า `05-AdvancedTopics` และ `03-GettingStarted/samples/python` ไม่มี **ช่องโหว่ที่รู้จัก** (ช่วงเวอร์ชัน `mcp` / `httpx` / `pydantic` / `python-dotenv` ติดตั้งเป็นเวอร์ชันที่แพตช์แล้ว)
- **09-CaseStudy/docs-mcp/solution/python/requirements.txt**: `pip-audit` รายงาน dependency แบบแฝง **`werkzeug` 3.1.1** มี advisory ช่องโหว่ DoS ของชื่ออุปกรณ์ Windows ด้วย 3 กรณี `safe_join` — `CVE-2025-66221`, `CVE-2026-21860`, และ `CVE-2026-27199` (แก้ไขเรียบร้อยใน 3.1.6) เพิ่มข้อจำกัดความปลอดภัยแบบพินเฉพาะ `werkzeug>=3.1.6` เพื่อใช้เวอร์ชันที่แพตช์แล้ว; ยืนยันว่าข้อจำกัดนี้ติดตั้งได้ไม่มีปัญหากับสแต็ค `chainlit` / `mcp` / `semantic-kernel`

#### การเสริมความเข้าใจหลัก (01-CoreConcepts/)  
- **Primitive Client ใหม่ - Roots**: เพิ่มเอกสารครบถ้วนเกี่ยวกับ primitive client Roots ให้เซิร์ฟเวอร์เข้าใจขอบเขตไฟล์และสิทธิ์การเข้าถึง  
- **คำอธิบายเครื่องมือ (Tool Annotations)**: เพิ่มเอกสารเกี่ยวกับคำอธิบายพฤติกรรมเครื่องมือ (`readOnlyHint`, `destructiveHint`) เพื่อการตัดสินใจใช้งานเครื่องมือที่ดีขึ้น  
- **การเรียกใช้เครื่องมือในการ Sampling**: ปรับปรุงเอกสาร Sampling ให้รวมพารามิเตอร์ `tools` และ `toolChoice` สำหรับการเรียกใช้เครื่องมือโดยโมเดลขณะ sampling  
- **การเรียกใช้ในโหมด URL**: เพิ่มเอกสารเกี่ยวกับการเรียกใช้เว็บภายนอกจากเซิร์ฟเวอร์ผ่าน URL  
- **งาน (Tasks) (ทดลอง)**: เพิ่มหัวข้อใหม่เกี่ยวกับฟีเจอร์ Tasks สำหรับการห่อหุ้มการดำเนินการที่ทนทานและดึงผลลัพธ์ล่าช้า  
- **รองรับไอคอน**: ระบุว่าสามารถเพิ่มไอคอนเป็นเมตาดาต้าเพิ่มเติมในเครื่องมือ, ทรัพยากร, เทมเพลตทรัพยากร, และ prompt

  - **โซลูชันความปลอดภัย Microsoft**: ยืนยันคำศัพท์และลิงก์ปัจจุบันสำหรับ Prompt Shields (เดิม "การตรวจจับความเสี่ยง jailbreak"), Azure Content Safety, Microsoft Entra ID, Azure Key Vault

  - **มาตรฐาน OWASP**: ตรวจสอบ OWASP Top 10 สำหรับ LLMs ยังอัปเดต

- **ความสอดคล้องมาตรฐาน**: ยืนยันมาตรฐานความปลอดภัยทั้งหมดในเอกสารยังอัปเดต

  - ISO 27001:2022

  - กรอบงานความปลอดภัยและการปฏิบัติตาม Azure  
- **ทรัพยากรการใช้งาน**: ตรวจสอบลิงก์คู่มือและทรัพยากรการปฏิบัติทั้งหมด

  - โซลูชัน DevSecOps pipeline และการตรวจสอบ

### การประกันคุณภาพเอกสาร

- **ทรัพยากรเป็นปัจจุบัน**: ตรวจสอบว่าลิงก์ภายนอกทั้งหมดเชื่อมต่อกับเอกสาร Microsoft, มาตรฐานความปลอดภัย และคู่มือการใช้งานอยู่เสมอ  
- **ความครอบคลุมแนวทางปฏิบัติที่ดีที่สุด**: ยืนยันว่าครอบคลุมการพิสูจน์ตัวตน, การอนุญาต, ภัยคุกคาม AI เฉพาะ, ความปลอดภัยซัพพลายเชน และรูปแบบองค์กรอย่างครบถ้วน

## 6 ตุลาคม 2025

### ขยายส่วนเริ่มต้นใช้งาน – การใช้งานเซิร์ฟเวอร์ขั้นสูง & การพิสูจน์ตัวตนอย่างง่าย

#### การใช้งานเซิร์ฟเวอร์ขั้นสูง (03-GettingStarted/10-advanced)  
- **เพิ่มบทใหม่**: แนะนำคู่มือฉบับสมบูรณ์สำหรับการใช้งาน MCP server ระดับสูง ครอบคลุมสถาปัตยกรรมเซิร์ฟเวอร์ทั้งแบบปกติและระดับต่ำ
  - **เซิร์ฟเวอร์ปกติกับเซิร์ฟเวอร์ระดับต่ำ**: การเปรียบเทียบโดยละเอียดและตัวอย่างโค้ดใน Python และ TypeScript สำหรับทั้งสองแนวทาง
  - **การออกแบบแบบใช้ตัวจัดการ**: อธิบายการจัดการเครื่องมือ/ทรัพยากร/พรอมต์แบบใช้ตัวจัดการสำหรับการสร้างเซิร์ฟเวอร์ที่ยืดหยุ่นและปรับขนาดได้
  - **รูปแบบที่ใช้ได้จริง**: สถานการณ์จริงที่รูปแบบเซิร์ฟเวอร์ระดับต่ำช่วยให้ได้ฟีเจอร์ขั้นสูงและสถาปัตยกรรมที่ดีขึ้น

#### การพิสูจน์ตัวตนง่ายๆ (03-GettingStarted/11-simple-auth)
- **เพิ่มบทใหม่**: คู่มือทีละขั้นตอนในการใช้งานการพิสูจน์ตัวตนง่ายๆ ในเซิร์ฟเวอร์ MCP
  - **แนวคิดการพิสูจน์ตัวตน**: อธิบายชัดเจนเกี่ยวกับการพิสูจน์ตัวตนกับการอนุญาต และการจัดการข้อมูลรับรอง
  - **การใช้งาน Basic Auth**: รูปแบบการพิสูจน์ตัวตนผ่านมิดเดิลแวร์ใน Python (Starlette) และ TypeScript (Express) พร้อมตัวอย่างโค้ด
  - **การพัฒนาไปสู่ความปลอดภัยขั้นสูง**: คำแนะนำในการเริ่มต้นกับการพิสูจน์ตัวตนง่ายๆ และก้าวไปสู่ OAuth 2.1 และ RBAC พร้อมแหล่งข้อมูลโมดูลความปลอดภัยขั้นสูง

การเพิ่มเนื้อหาเหล่านี้มอบคำแนะนำการใช้งานจริงสำหรับสร้างเซิร์ฟเวอร์ MCP ที่มั่นคง ปลอดภัย และยืดหยุ่นมากขึ้น เชื่อมต่อแนวคิดพื้นฐานกับรูปแบบการผลิตขั้นสูง

#### แนวคิดหลัก (01-CoreConcepts/) - การปรับปรุงครั้งใหญ่
- **การอัปเดตเวอร์ชันโปรโตคอล**: อัปเดตเพื่ออ้างอิง MCP Specification ปัจจุบัน 2025-06-18 ด้วยการตั้งชื่อเวอร์ชันตามวันที่ (รูปแบบ YYYY-MM-DD)
- **การปรับแต่งสถาปัตยกรรม**: ปรับปรุงคำอธิบายของ Hosts, Clients, และ Servers เพื่อสะท้อนรูปแบบสถาปัตยกรรม MCP ปัจจุบัน
  - Hosts ถูกกำหนดอย่างชัดเจนว่าเป็นแอปพลิเคชัน AI ที่ประสานงานการเชื่อมต่อ MCP client หลายรายการ
  - Clients ถูกอธิบายเป็นตัวเชื่อมต่อโปรโตคอลที่รักษาความสัมพันธ์แบบหนึ่งต่อหนึ่งกับเซิร์ฟเวอร์
  - Servers ปรับปรุงด้วยสถานการณ์การปรับใช้แบบท้องถิ่นและระยะไกล
- **การปรับโครงสร้าง Primitive**: การปรับโครงสร้างครั้งใหญ่ของ primitive ของเซิร์ฟเวอร์และไคลเอ็นต์
  - Server Primitives: ทรัพยากร (แหล่งข้อมูล), Prompt (แม่แบบ), เครื่องมือ (ฟังก์ชันที่สามารถรันได้) พร้อมคำอธิบายและตัวอย่างละเอียด
  - Client Primitives: การสุ่มตัวอย่าง (ผลลัพธ์ LLM), การสืบค้น (ข้อมูลผู้ใช้), การบันทึกข้อมูล (ดีบัก/ตรวจสอบ)
  - อัปเดตด้วยรูปแบบวิธีการค้นหา (`*/list`), ดึงข้อมูล (`*/get`), และเรียกใช้งาน (`*/call`) ตามปัจจุบัน
- **สถาปัตยกรรมโปรโตคอล**: แนะนำโมเดลสถาปัตยกรรมสองชั้น
  - ชั้นข้อมูล: พื้นฐาน JSON-RPC 2.0 พร้อมการจัดการวงจรชีวิตและ primitives
  - ชั้นการขนส่ง: STDIO (ในเครื่อง) และ HTTP แบบสตรีมมิ่งพร้อม SSE (การขนส่งระยะไกล)
- **กรอบความปลอดภัย**: หลักการความปลอดภัยแบบครบถ้วนรวมถึงความยินยอมผู้ใช้ที่ชัดเจน, การปกป้องข้อมูลส่วนตัว, ความปลอดภัยในการเรียกใช้เครื่องมือ และความปลอดภัยของชั้นการขนส่ง
- **รูปแบบการสื่อสาร**: อัปเดตข้อความโปรโตคอลเพื่อแสดงกระบวนการเริ่มต้น, การค้นหา, การดำเนินการ, และการแจ้งเตือน
- **ตัวอย่างโค้ด**: ปรับปรุงตัวอย่างหลายภาษา (.NET, Java, Python, JavaScript) ให้สอดคล้องกับรูปแบบ SDK MCP ปัจจุบัน

#### ความปลอดภัย (02-Security/) - ปรับปรุงความปลอดภัยอย่างครบถ้วน  
- **การสอดคล้องมาตรฐาน**: สอดคล้องเต็มที่กับข้อกำหนดความปลอดภัย MCP Specification 2025-06-18
- **วิวัฒนาการการพิสูจน์ตัวตน**: บันทึกวิวัฒนาการจาก OAuth เซิร์ฟเวอร์แบบกำหนดเองไปสู่การมอบหมายผู้ให้บริการตัวตนภายนอก (Microsoft Entra ID)
- **การวิเคราะห์ภัยคุกคามเฉพาะ AI**: ขยายการครอบคลุมเวกเตอร์การโจมตี AI สมัยใหม่
  - สถานการณ์โจมตี prompt injection โดยละเอียดพร้อมตัวอย่างจริง
  - กลไกการวางยาเครื่องมือและรูปแบบการโจมตี "rug pull"
  - การวางยา context window และโจมตีสร้างความสับสนโมเดล
- **โซลูชันความปลอดภัย AI ของ Microsoft**: ครอบคลุมระบบนิเวศความปลอดภัยของ Microsoft อย่างครบถ้วน
  - AI Prompt Shields พร้อมเทคนิคการตรวจจับขั้นสูง, การเปิดไฟส่อง, และ delimiter
  - รูปแบบการบูรณาการ Azure Content Safety
  - GitHub Advanced Security สำหรับการป้องกันซัพพลายเชน
- **การบรรเทาภัยคุกคามขั้นสูง**: ควบคุมความปลอดภัยโดยละเอียดสำหรับ
  - การแฮ็กเซสชันด้วยสถานการณ์โจมตีเฉพาะ MCP และข้อกำหนด session ID เชิงคริปโตกราฟิก
  - ปัญหา confused deputy ในกรณี MCP proxy พร้อมข้อกำหนดความยินยอมชัดเจน
  - ช่องโหว่ token passthrough พร้อมมาตรการตรวจสอบที่บังคับใช้
- **ความปลอดภัยซัพพลายเชน**: ขยายการครอบคลุมซัพพลายเชน AI ทั้งโมเดลพื้นฐาน, บริการ embeddings, ผู้ให้บริบท, และ API ของบุคคลที่สาม
- **ความปลอดภัยฐานราก**: ปรับปรุงการบูรณาการกับรูปแบบความปลอดภัยองค์กรรวมถึงสถาปัตยกรรม zero trust และระบบนิเวศความปลอดภัย Microsoft
- **การจัดหมวดหมู่ทรัพยากร**: แยกลิงก์ทรัพยากรทั้งแบบครอบคลุมตามประเภท (เอกสารทางการ, มาตรฐาน, งานวิจัย, โซลูชัน Microsoft, คู่มือการใช้งาน)

### การปรับปรุงคุณภาพเอกสาร
- **วัตถุประสงค์การเรียนรู้มีโครงสร้าง**: ขยายวัตถุประสงค์การเรียนรู้ด้วยผลลัพธ์เฉพาะและปฏิบัติได้
- **อ้างอิงข้ามหัวข้อ**: เพิ่มลิงก์ระหว่างหัวข้อที่เกี่ยวข้องระหว่างด้านความปลอดภัยและแนวคิดหลัก
- **ข้อมูลปัจจุบัน**: อัปเดตการอ้างอิงวันที่และลิงก์สเปคทั้งหมดให้เป็นมาตรฐานล่าสุด
- **คำแนะนำการใช้งาน**: เพิ่มคำแนะนำการใช้งานเฉพาะที่สามารถปฏิบัติได้ทั่วทั้งสองส่วน
