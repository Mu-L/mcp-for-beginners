# MCP Security: การปกป้องที่ครอบคลุมสำหรับระบบ AI

[![MCP Security Best Practices](../../../translated_images/th/03.175aed6dedae133f.webp)](https://youtu.be/88No8pw706o)

_(คลิกที่ภาพด้านบนเพื่อดูวิดีโอของบทเรียนนี้)_

ความปลอดภัยเป็นพื้นฐานสำคัญของการออกแบบระบบ AI ซึ่งเป็นเหตุผลที่เราจัดลำดับความสำคัญไว้ในบทที่สองของเรา สิ่งนี้สอดคล้องกับหลักการ **Secure by Design** ของ Microsoft จาก [โครงการ Secure Future Initiative](https://www.microsoft.com/security/blog/2025/04/17/microsofts-secure-by-design-journey-one-year-of-success/)

Model Context Protocol (MCP) มอบความสามารถใหม่ที่ทรงพลังให้กับแอปพลิเคชันที่ขับเคลื่อนด้วย AI ขณะเดียวกันก็แนะนำความท้าทายด้านความปลอดภัยเฉพาะทางที่เกินขอบเขตของความเสี่ยงซอฟต์แวร์แบบดั้งเดิม ระบบ MCP ต้องเผชิญกับทั้งประเด็นความปลอดภัยที่ทราบกันดี (การเขียนโค้ดที่ปลอดภัย, สิทธิ์การเข้าถึงน้อยที่สุด, ความปลอดภัยในห่วงโซ่อุปทาน) และภัยคุกคามเฉพาะ AI ใหม่ ๆ รวมถึงการฉีดคำสั่ง, การปนเปื้อนเครื่องมือ, การยึดเซสชัน, การโจมตี confused deputy, ช่องโหว่การส่งผ่านโทเค็น และการปรับเปลี่ยนความสามารถแบบไดนามิก

บทเรียนนี้จะสำรวจความเสี่ยงด้านความปลอดภัยที่สำคัญที่สุดในการใช้งาน MCP — ครอบคลุมการตรวจสอบตัวตน, การอนุญาต, สิทธิ์การเข้าถึงที่เกินควร, การฉีดคำสั่งโดยอ้อม, ความปลอดภัยของเซสชัน, ปัญหา confused deputy, การจัดการโทเค็น และช่องโหว่ในห่วงโซ่อุปทาน คุณจะได้เรียนรู้การควบคุมและแนวทางปฏิบัติที่ใช้ได้จริงเพื่อบรรเทาความเสี่ยงเหล่านี้ พร้อมทั้งใช้ประโยชน์จากโซลูชันของ Microsoft เช่น Prompt Shields, Azure Content Safety และ GitHub Advanced Security เพื่อเสริมความแข็งแกร่งในการปรับใช้ MCP ของคุณ

## วัตถุประสงค์การเรียนรู้

เมื่อจบบทเรียนนี้ คุณจะสามารถ:

- **ระบุภัยคุกคามเฉพาะ MCP**: รับรู้ความเสี่ยงด้านความปลอดภัยเฉพาะในระบบ MCP เช่น การฉีดคำสั่ง, การปนเปื้อนเครื่องมือ, สิทธิ์การเข้าถึงที่เกินควร, การยึดเซสชัน, ปัญหา confused deputy, ช่องโหว่การส่งผ่านโทเค็น และความเสี่ยงในห่วงโซ่อุปทาน
- **นำมาตรการควบคุมความปลอดภัยไปใช้**: ใช้มาตรการบรรเทาอย่างมีประสิทธิภาพ รวมถึงการตรวจสอบตัวตนอย่างเข้มงวด, การเข้าถึงตามหลักสิทธิ์น้อยที่สุด, การจัดการโทเค็นอย่างปลอดภัย, การควบคุมความปลอดภัยของเซสชัน และการตรวจสอบห่วงโซ่อุปทาน
- **ใช้โซลูชันความปลอดภัยของ Microsoft**: เข้าใจและนำ Microsoft Prompt Shields, Azure Content Safety และ GitHub Advanced Security มาใช้เพื่อปกป้องงาน MCP
- **ตรวจสอบความปลอดภัยของเครื่องมือ**: ตระหนักถึงความสำคัญของการตรวจสอบข้อมูลเมตาเครื่องมือ, การติดตามการเปลี่ยนแปลงแบบไดนามิก และการป้องกันการโจมตีฉีดคำสั่งโดยอ้อม
- **รวมแนวทางปฏิบัติที่ดีที่สุด**: ผสานพื้นฐานความปลอดภัยที่มั่นคง (การเขียนโค้ดที่ปลอดภัย, การปรับเซิร์ฟเวอร์ให้แข็งแกร่ง, zero trust) กับมาตรการควบคุมเฉพาะ MCP เพื่อการปกป้องที่ครอบคลุม

# สถาปัตยกรรมและมาตรการควบคุมความปลอดภัย MCP

การใช้งาน MCP ยุคใหม่ต้องการแนวทางความปลอดภัยหลายชั้นที่ครอบคลุมทั้งความปลอดภัยซอฟต์แวร์แบบดั้งเดิมและภัยคุกคามเฉพาะ AI ข้อกำหนด MCP ที่กำลังพัฒนาอย่างรวดเร็วยังคงปรับปรุงมาตรการควบคุมความปลอดภัยเพื่อให้สามารถรวมกับสถาปัตยกรรมความปลอดภัยขององค์กรและแนวทางปฏิบัติที่ตั้งอยู่

งานวิจัยจาก [รายงาน Microsoft Digital Defense Report](https://aka.ms/mddr) แสดงให้เห็นว่า **98% ของการละเมิดที่รายงานจะถูกป้องกันด้วยสุขอนามัยความปลอดภัยที่เข้มงวด** กลยุทธ์การปกป้องที่มีประสิทธิภาพที่สุดคือการรวมการปฏิบัติด้านความปลอดภัยพื้นฐานกับมาตรการควบคุมเฉพาะ MCP — มาตรการความปลอดภัยพื้นฐานที่ได้รับการพิสูจน์แล้วยังคงเป็นวิธีที่มีผลมากที่สุดในการลดความเสี่ยงด้านความปลอดภัยโดยรวม

## ภาพรวมสถานการณ์ความปลอดภัยปัจจุบัน

> **หมายเหตุ:** ข้อมูลนี้สะท้อนมาตรฐานความปลอดภัย MCP เมื่อวันที่ **5 กุมภาพันธ์ 2026** ซึ่งสอดคล้องกับ **ข้อกำหนด MCP 2025-11-25** โปรโตคอล MCP ยังคงพัฒนาอย่างรวดเร็ว และการใช้งานในอนาคตอาจเพิ่มรูปแบบการตรวจสอบตัวตนและมาตรการควบคุมที่ปรับปรุงเสริมขึ้น กรุณาอ้างอิงไปยัง [ข้อกำหนด MCP ปัจจุบัน](https://spec.modelcontextprotocol.io/), [ที่เก็บ GitHub MCP](https://github.com/modelcontextprotocol) และ [เอกสารแนวทางปฏิบัติความปลอดภัย](https://modelcontextprotocol.io/specification/2025-11-25/basic/security_best_practices) สำหรับคำแนะนำล่าสุด

## 🏔️ MCP Security Summit Workshop (Sherpa)

สำหรับ **การฝึกอบรมด้านความปลอดภัยภาคปฏิบัติ** เราขอแนะนำอย่างยิ่งให้เข้าร่วม **MCP Security Summit Workshop** (Sherpa) — การเดินทางนำทางอย่างละเอียดเพื่อรักษาความปลอดภัยเซิร์ฟเวอร์ MCP บน Microsoft Azure

### ภาพรวมเวิร์กช็อป

[เวิร์กช็อป MCP Security Summit](https://azure-samples.github.io/sherpa/) ให้การฝึกอบรมด้านความปลอดภัยที่ปฏิบัติได้จริงผ่านแนวทาง "ช่องโหว่ → การโจมตี → การแก้ไข → การตรวจสอบ" ที่พิสูจน์แล้ว คุณจะได้:

- **เรียนรู้โดยทำลายระบบ**: สัมผัสช่องโหว่โดยตรงผ่านการโจมตีเซิร์ฟเวอร์ที่ตั้งใจไม่ปลอดภัย
- **ใช้ความปลอดภัยเนทีฟ Azure**: ใช้ Azure Entra ID, Key Vault, API Management และ AI Content Safety
- **ปฏิบัติตาม Defense-in-Depth**: ก้าวผ่านฐานตั้งแคมป์ที่สร้างชั้นความปลอดภัยครอบคลุม
- **ใช้มาตรฐาน OWASP**: ทุกเทคนิคสอดคล้องกับ [OWASP MCP Azure Security Guide](https://microsoft.github.io/mcp-azure-security-guide/)
- **รับโค้ดสำหรับผลิตจริง**: ออกไปพร้อมกับการใช้งานจริงที่ทำงานได้และผ่านการทดสอบ

### การพัฒนาการตรวจสอบตัวตน MCP

ข้อกำหนด MCP มีการพัฒนาอย่างมากในแนวทางตรวจสอบตัวตนและการอนุญาต:

- **แนวทางเดิม**: ข้อกำหนดแรกเริ่มให้ผู้พัฒนาสร้างเซิร์ฟเวอร์ตรวจสอบตัวตนเอง เซิร์ฟเวอร์ MCP ทำหน้าที่เป็น OAuth 2.0 Authorization Servers ที่ตรวจสอบตัวตนผู้ใช้โดยตรง
- **มาตรฐานปัจจุบัน (2025-11-25)**: ข้อกำหนดที่อัปเดตให้เซิร์ฟเวอร์ MCP สามารถมอบหมายการตรวจสอบตัวตนให้ผู้ให้บริการตัวตนภายนอก (เช่น Microsoft Entra ID) ซึ่งช่วยปรับปรุงท่าทางความปลอดภัยและลดความซับซ้อนในการใช้งาน
- **การรักษาความปลอดภัยชั้นขนส่ง**: สนับสนุนกลไกการขนส่งที่ปลอดภัยมากขึ้น พร้อมรูปแบบการตรวจสอบตัวตนที่เหมาะสมสำหรับการเชื่อมต่อทั้งแบบท้องถิ่น (STDIO) และระยะไกล (Streamable HTTP)

## ความปลอดภัยด้านการตรวจสอบตัวตนและการอนุญาต

### ความท้าทายด้านความปลอดภัยปัจจุบัน

การใช้งาน MCP ยุคใหม่ต้องเผชิญกับความท้าทายหลายประการด้านการตรวจสอบตัวตนและการอนุญาต:

### ความเสี่ยงและช่องทางการโจมตี

- **ตรรกะการอนุญาตที่กำหนดค่าไม่ถูกต้อง**: การใช้งานอนุญาตที่ผิดพลาดในเซิร์ฟเวอร์ MCP อาจเปิดเผยข้อมูลสำคัญและใช้การควบคุมการเข้าถึงอย่างผิดวิธี
- **การถูกขโมยโทเค็น OAuth**: การขโมยโทเค็นจากเซิร์ฟเวอร์ MCP ท้องถิ่นช่วยให้ผู้โจมตีย้อมตัวเป็นเซิร์ฟเวอร์และเข้าถึงบริการต่อไปได้
- **ช่องโหว่การส่งผ่านโทเค็น**: การจัดการโทเค็นไม่ถูกต้องสร้างช่องทางเลี่ยงมาตรการควบคุมและช่องว่างความรับผิดชอบ
- **สิทธิ์การเข้าถึงที่เกินจำเป็น**: เซิร์ฟเวอร์ MCP ที่ได้รับสิทธิ์เกินจำเป็นละเมิดหลักสิทธิ์น้อยที่สุดและขยายพื้นผิวการโจมตี

#### การส่งผ่านโทเค็น: รูปแบบต่อต้านที่ร้ายแรง

**การส่งผ่านโทเค็นถูกห้ามอย่างชัดเจน** ในข้อกำหนดการอนุญาต MCP ปัจจุบันเนื่องจากผลกระทบร้ายแรงด้านความปลอดภัย:

##### การหลีกเลี่ยงมาตรการความปลอดภัย
- เซิร์ฟเวอร์ MCP และ API ปลายทางใช้มาตรการควบคุมความปลอดภัยสำคัญ (จำกัดอัตรา, ตรวจสอบคำขอ, เฝ้าระวังทราฟฟิก) ซึ่งขึ้นอยู่กับการตรวจสอบโทเค็นอย่างถูกต้อง
- การใช้โทเค็นโดยตรงจากไคลเอนต์ไปยัง API เลี่ยงมาตรการป้องกันเหล่านี้ ทำลายสถาปัตยกรรมความปลอดภัย

##### ความท้าทายด้านความรับผิดชอบและการตรวจสอบ  
- เซิร์ฟเวอร์ MCP ไม่สามารถแยกแยะว่าไคลเอนต์ใช้โทเค็นที่ออกโดยผู้ให้บริการต้นทางหรือไม่ ทำให้เส้นทางการตรวจสอบขาดความชัดเจน
- บันทึกจากเซิร์ฟเวอร์ทรัพยากรปลายทางอาจแสดงแหล่งที่มาของคำขอผิดพลาดแทนตัวเซิร์ฟเวอร์ MCP ที่จริง
- การสืบสวนเหตุการณ์และการตรวจสอบตามกฎหมายจึงยากลำบากขึ้นมาก

##### ความเสี่ยงการขโมยข้อมูล
- การอ้างสิทธิ์โทเค็นที่ไม่ได้รับการตรวจสอบเปิดโอกาสให้ผู้โจมตีที่ขโมยโทเค็นใช้เซิร์ฟเวอร์ MCP เป็นพร็อกซีในการขโมยข้อมูล
- การละเมิดเขตแดนความไว้วางใจทำให้เกิดรูปแบบการเข้าถึงที่ไม่ได้รับอนุญาตที่เลี่ยงมาตรการความปลอดภัยที่ตั้งไว้

##### ช่องทางการโจมตีบริการหลายตัว
- โทเค็นที่ถูกเจาะและยอมรับโดยหลายบริการช่วยให้เคลื่อนที่ในระบบที่เชื่อมต่อได้
- สมมติฐานความไว้วางใจระหว่างบริการอาจถูกละเมิดเมื่อไม่สามารถตรวจสอบต้นทางโทเค็นได้

### มาตรการควบคุมความปลอดภัยและการแก้ไข

**ข้อกำหนดความปลอดภัยสำคัญ:**

> **บังคับใช้**: เซิร์ฟเวอร์ MCP **ต้องไม่** รับโทเค็นที่ไม่ได้ออกให้กับเซิร์ฟเวอร์ MCP โดยชัดเจน

#### การควบคุมการตรวจสอบตัวตนและการอนุญาต

- **ทบทวนการอนุญาตอย่างรัดกุม**: ทำการตรวจสอบตรรกะการอนุญาตของเซิร์ฟเวอร์ MCP อย่างครบถ้วนเพื่อให้แน่ใจว่าผู้ใช้และไคลเอนต์ที่ตั้งใจเท่านั้นที่สามารถเข้าถึงทรัพยากรสำคัญได้
  - **คู่มือการใช้งาน**: [การใช้ Azure API Management เป็นเกตเวย์การตรวจสอบตัวตนสำหรับเซิร์ฟเวอร์ MCP](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690)
  - **การรวมตัวตน**: [การใช้ Microsoft Entra ID สำหรับการตรวจสอบตัวตนเซิร์ฟเวอร์ MCP](https://den.dev/blog/mcp-server-auth-entra-id-session/)

- **การจัดการโทเค็นอย่างปลอดภัย**: นำ [แนวทางปฏิบัติที่ดีที่สุดของ Microsoft สำหรับการตรวจสอบโทเค็นและวงจรชีวิต](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens) มาใช้
  - ตรวจสอบให้มั่นใจว่า claims audience ในโทเค็นตรงกับตัวตนของเซิร์ฟเวอร์ MCP
  - นำการหมุนเวียนและหมดอายุของโทเค็นมาใช้อย่างเหมาะสม
  - ป้องกันการโจมตีแบบ replay และการใช้งานโทเค็นที่ไม่ได้รับอนุญาต

- **การจัดเก็บโทเค็นที่ปลอดภัย**: เก็บโทเค็นด้วยการเข้ารหัสทั้งขณะเก็บและขณะส่ง
  - **แนวทางปฏิบัติที่ดีที่สุด**: [คำแนะนำจัดเก็บโทเค็นและการเข้ารหัส](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)

#### การใช้งานการควบคุมการเข้าถึง

- **หลักสิทธิ์น้อยที่สุด**: มอบสิทธิ์เพียงขั้นต่ำที่จำเป็นให้กับเซิร์ฟเวอร์ MCP เพื่อให้ทำงานได้ตามที่ต้องการ
  - ทบทวนสิทธิ์เป็นประจำและปรับปรุงเพื่อป้องกันการเพิ่มสิทธิ์อย่างไม่เหมาะสม
  - **เอกสารของ Microsoft**: [การเข้าถึงที่ปลอดภัยตามหลักสิทธิ์น้อยที่สุด](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)

- **การควบคุมสิทธิ์ตามบทบาท (RBAC)**: กำหนดบทบาทละเอียด
  - กำหนดขอบเขตบทบาทให้เฉพาะเจาะจงสำหรับทรัพยากรและการกระทำ
  - หลีกเลี่ยงสิทธิ์ทั่วไปหรือไม่จำเป็นที่ขยายพื้นที่โจมตี

- **การติดตามสิทธิ์อย่างต่อเนื่อง**: ใช้การตรวจสอบและวิเคราะห์การใช้งานสิทธิ์ตลอดเวลา
  - เฝ้าระวังรูปแบบการใช้สิทธิ์ที่ผิดปกติ
  - แก้ไขสิทธิ์ที่เกินหรือไม่ได้ใช้ทันที

## ภัยคุกคามความปลอดภัยเฉพาะ AI

### การโจมตีฉีดคำสั่งและการจัดการเครื่องมือ

การใช้งาน MCP ยุคใหม่เผชิญช่องทางการโจมตีเฉพาะ AI ที่ซับซ้อนซึ่งมาตรการความปลอดภัยแบบดั้งเดิมไม่สามารถจัดการได้อย่างเต็มที่:

#### **การฉีดคำสั่งโดยอ้อม (Cross-Domain Prompt Injection)**

**การฉีดคำสั่งโดยอ้อม** เป็นหนึ่งในช่องโหว่ที่สำคัญที่สุดในระบบ AI ที่ใช้ MCP ผู้โจมตีฝังคำสั่งอันตรายไว้ในเนื้อหาภายนอก — เอกสาร, เว็บเพจ, อีเมล หรือแหล่งข้อมูล — ซึ่งระบบ AI ประมวลผลต่อเป็นคำสั่งที่ถูกต้อง

**สถานการณ์โจมตี:**
- **การฉีดในเอกสาร**: คำสั่งอันตรายที่ซ่อนอยู่ในเอกสารที่ประมวลผลซึ่งกระตุ้นให้เกิดการกระทำ AI ที่ไม่ตั้งใจ
- **การแสวงหาประโยชน์จากเนื้อหาเว็บ**: เว็บเพจถูกแฮ็กที่ฝังคำสั่งฉีดควบคุมการทำงานของ AI เมื่อถูกดึงข้อมูล
- **การโจมตีผ่านอีเมล**: คำสั่งอันตรายในอีเมลที่ทำให้ผู้ช่วย AI รั่วข้อมูลหรือทำงานโดยไม่ได้รับอนุญาต
- **การปนเปื้อนแหล่งข้อมูล**: ฐานข้อมูลหรือ API ที่ถูกปนเปื้อนให้เนื้อหาที่ไม่ปลอดภัยแก่ระบบ AI

**ผลกระทบในโลกจริง**: การโจมตีเหล่านี้อาจทำให้ข้อมูลถูกขโมย, ความเป็นส่วนตัวถูกละเมิด, สร้างเนื้อหาที่เป็นอันตราย และบิดเบือนปฏิสัมพันธ์กับผู้ใช้ ดูการวิเคราะห์ละเอียดได้ที่ [Prompt Injection ใน MCP (Simon Willison)](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)

![Prompt Injection Attack Diagram](../../../translated_images/th/prompt-injection.ed9fbfde297ca877.webp)

#### **การโจมตีปนเปื้อนเครื่องมือ**

**การปนเปื้อนเครื่องมือ** มุ่งโจมตีข้อมูลเมตาที่กำหนดเครื่องมือ MCP โดยใช้ประโยชน์จากวิธีที่ LLM ตีความคำอธิบายและพารามิเตอร์ของเครื่องมือในการตัดสินใจดำเนินการ

**กลไกการโจมตี:**
- **การแก้ไขข้อมูลเมตา**: ผู้โจมตีฉีดคำสั่งอันตรายลงในคำอธิบายเครื่องมือ, คำจำกัดความพารามิเตอร์ หรือแบบอย่างการใช้งาน
- **คำสั่งที่มองไม่เห็น**: คำสั่งฉีดที่ซ่อนอยู่ในข้อมูลเมตาของเครื่องมือ ซึ่งถูกประมวลผลโดยโมเดล AI แต่ผู้ใช้มนุษย์ไม่เห็น
- **การแก้ไขเครื่องมือแบบไดนามิก (“Rug Pulls”)**: เครื่องมือที่ผู้ใช้อนุมัติถูกแก้ไขภายหลังให้ทำงานที่เป็นอันตรายโดยที่ผู้ใช้ไม่รู้ตัว
- **การฉีดพารามิเตอร์**: เนื้อหาอันตรายฝังในสคีมาพารามิเตอร์เครื่องมือที่ส่งผลต่อพฤติกรรมของโมเดล

**ความเสี่ยงของเซิร์ฟเวอร์โฮสต์**: เซิร์ฟเวอร์ MCP ระยะไกลมีความเสี่ยงสูงขึ้นเนื่องจากคำจำกัดความเครื่องมือสามารถอัปเดตได้หลังการอนุมัติจากผู้ใช้ สร้างสถานการณ์ที่เครื่องมือที่เคยปลอดภัยกลายเป็นอันตราย ดูการวิเคราะห์ครอบคลุมได้ที่ [การโจมตีปนเปื้อนเครื่องมือ (Invariant Labs)](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)

![Tool Injection Attack Diagram](../../../translated_images/th/tool-injection.3b0b4a6b24de6bef.webp)

#### **ช่องทางการโจมตี AI เพิ่มเติม**

- **การฉีดคำสั่งข้ามโดเมน (XPIA)**: การโจมตีที่ซับซ้อนที่ใช้เนื้อหาจากหลายโดเมนเพื่อเลี่ยงมาตรการควบคุมความปลอดภัย
- **การปรับเปลี่ยนความสามารถแบบไดนามิก**: การเปลี่ยนแปลงแบบเรียลไทม์ของความสามารถเครื่องมือที่หลุดพ้นจากการประเมินความปลอดภัยเบื้องต้น
- **การโจมตีป้อนข้อมูลบริบทที่กว้าง**: การโจมตีที่จัดการหน้าต่างบริบทขนาดใหญ่เพื่อซ่อนคำสั่งอันตราย
- **การโจมตีทำให้โมเดลสับสน**: การใช้ข้อจำกัดของโมเดลเพื่อสร้างพฤติกรรมที่ไม่คาดคิดหรือไม่ปลอดภัย


### ผลกระทบความเสี่ยงด้านความปลอดภัยของ AI

**ผลกระทบสูง:**
- **การขโมยข้อมูล**: การเข้าถึงและขโมยข้อมูลที่ละเอียดอ่อนขององค์กรหรือข้อมูลส่วนบุคคลโดยไม่ได้รับอนุญาต
- **การละเมิดความเป็นส่วนตัว**: การเปิดเผยข้อมูลส่วนบุคคลที่สามารถระบุตัวตนได้ (PII) และข้อมูลทางธุรกิจที่เป็นความลับ  
- **การปรับเปลี่ยนระบบ**: การแก้ไขระบบและเวิร์กโฟลว์ที่สำคัญโดยไม่ได้ตั้งใจ
- **การขโมยข้อมูลรับรอง**: การละเมิดโทเค็นการรับรองตัวตนและข้อมูลรับรองบริการ
- **การเคลื่อนที่ในแนวนอน**: การใช้ระบบ AI ที่ถูกเจาะระบบเพื่อเป็นจุดหมุนในการโจมตีเครือข่ายที่กว้างขึ้น

### โซลูชันความปลอดภัย AI ของ Microsoft

#### **AI Prompt Shields: การป้องกันขั้นสูงจากการโจมตีแบบฉีดคำสั่ง**

Microsoft **AI Prompt Shields** ให้การป้องกันอย่างครบถ้วนทั้งจากการโจมตีฉีดคำสั่งโดยตรงและทางอ้อมผ่านหลายชั้นของความปลอดภัย:

##### **กลไกป้องกันหลัก:**

1. **การตรวจจับและกรองขั้นสูง**
   - อัลกอริทึมการเรียนรู้ของเครื่องและเทคนิค NLP ตรวจจับคำสั่งอันตรายในเนื้อหาภายนอก
   - การวิเคราะห์แบบเรียลไทม์ของเอกสาร หน้าเว็บ อีเมล และแหล่งข้อมูลสำหรับภัยคุกคามฝังตัว
   - ความเข้าใจบริบทของแพทเทิร์นคำสั่งที่ถูกต้องเทียบกับแบบอันตราย

2. **เทคนิคการเน้นจุดสำคัญ**  
   - แยกความแตกต่างระหว่างคำสั่งระบบที่เชื่อถือได้กับข้อมูลภายนอกที่อาจถูกเจาะระบบ
   - วิธีการแปลงข้อความที่ช่วยเพิ่มความเกี่ยวข้องกับโมเดลในขณะเดียวกันก็แยกเนื้อหาอันตรายออก
   - ช่วยให้ระบบ AI รักษาลำดับชั้นคำสั่งอย่างถูกต้องและไม่สนใจคำสั่งที่ถูกฉีด

3. **ระบบตัวแบ่งและการทำเครื่องหมายข้อมูล**
   - นิยามขอบเขตที่ชัดเจนระหว่างข้อความระบบที่เชื่อถือได้กับข้อความข้อมูลภายนอก
   - ตัวทำเครื่องหมายพิเศษที่เน้นขอบเขตระหว่างแหล่งข้อมูลที่เชื่อถือและไม่เชื่อถือ
   - การแยกที่ชัดเจนป้องกันความสับสนของคำสั่งและการดำเนินการคำสั่งที่ไม่ได้รับอนุญาต

4. **ข่าวกรองภัยคุกคามอย่างต่อเนื่อง**
   - Microsoft ตรวจสอบรูปแบบการโจมตีที่เกิดขึ้นใหม่อย่างต่อเนื่องและอัปเดตระบบป้องกัน
   - การล่าภัยคุกคามเชิงรุกสำหรับเทคนิคฉีดคำสั่งและเวกเตอร์การโจมตีใหม่ๆ
   - การอัปเดตโมเดลความปลอดภัยอย่างสม่ำเสมอเพื่อรักษาประสิทธิภาพต่อภัยคุกคามที่พัฒนาไป

5. **การผสานรวม Azure Content Safety**
   - เป็นส่วนหนึ่งของชุด Azure AI Content Safety แบบครบวงจร
   - การตรวจจับเพิ่มเติมสำหรับความพยายาม jailbreak เนื้อหาอันตราย และการละเมิดนโยบายความปลอดภัย
   - ควบคุมความปลอดภัยแบบรวมศูนย์ในส่วนประกอบแอปพลิเคชัน AI

**ทรัพยากรการใช้งาน**: [Microsoft Prompt Shields Documentation](https://learn.microsoft.com/azure/ai-services/content-safety/concepts/jailbreak-detection)

## ภัยคุกคามความปลอดภัย MCP ขั้นสูง

### ช่องโหว่การแย่งชิงเซสชัน

**การแย่งชิงเซสชัน** เป็นเวกเตอร์การโจมตีที่สำคัญในระบบ MCP แบบมีสถานะ ซึ่งผู้ไม่หวังดีสามารถขโมยและใช้ประโยชน์จากตัวระบุเซสชันที่ถูกต้องเพื่อแอบอ้างเป็นไคลเอนต์และดำเนินการโดยไม่ได้รับอนุญาต

#### **สถานการณ์การโจมตีและความเสี่ยง**

- **การฉีดคำสั่งแย่งชิงเซสชัน**: ผู้โจมตีที่มี ID เซสชันที่ถูกขโมยฉีดเหตุการณ์อันตรายสู่เซิร์ฟเวอร์ที่แชร์สถานะเซสชัน อาจกระตุ้นการกระทำที่เป็นอันตรายหรือเข้าถึงข้อมูลสำคัญ
- **การแอบอ้างโดยตรง**: ID เซสชันที่ถูกขโมยช่วยให้เรียกเซิร์ฟเวอร์ MCP โดยตรงโดยไม่ต้องผ่านการตรวจสอบตัวตน ทำให้ผู้โจมตีถูกปฏิบัติราวกับผู้ใช้ที่ถูกต้อง
- **การละเมิดสตรีมแบบต่อเนื่อง**: ผู้โจมตีสามารถยุติคำขอล่วงหน้า ทำให้ไคลเอนต์ถูกต้องต้องดำเนินการต่อด้วยเนื้อหาอันตรายได้

#### **การควบคุมความปลอดภัยสำหรับการจัดการเซสชัน**

**ข้อกำหนดสำคัญ:**
- **การตรวจสอบสิทธิ์**: เซิร์ฟเวอร์ MCP ที่ใช้การอนุญาตต้องตรวจสอบคำขอทั้งหมดและไม่ควรพึ่งพาเซสชันสำหรับการรับรองตัวตน
- **การสร้างเซสชันที่ปลอดภัย**: ใช้ ID เซสชันที่สร้างอย่างปลอดภัยด้วยตัวสร้างเลขสุ่มเชิงลับที่ไม่สามารถทำนายได้
- **ผูกกับผู้ใช้เฉพาะ**: ผูก ID เซสชันกับข้อมูลผู้ใช้เฉพาะในรูปแบบเช่น `<user_id>:<session_id>` เพื่อป้องกันการแอบใช้ข้ามผู้ใช้
- **การจัดการวงจรชีวิตเซสชัน**: กำหนดวันหมดอายุ การหมุนเวียน และการเพิกถอนอย่างเหมาะสมเพื่อลดช่วงเวลาที่เสี่ยง
- **ความปลอดภัยในการส่งข้อมูล**: ใช้ HTTPS ทุกการสื่อสารเพื่อป้องกันการดักจับ ID เซสชัน

### ปัญหา Confused Deputy

**ปัญหา confused deputy** เกิดขึ้นเมื่อเซิร์ฟเวอร์ MCP ทำหน้าที่เป็นพร็อกซีการรับรองตัวตนระหว่างไคลเอนต์กับบริการของบุคคลที่สาม สร้างโอกาสให้มีการ bypass การอนุญาตโดยใช้ประโยชน์จาก ID ลูกค้าคงที่

#### **กลไกโจมตีและความเสี่ยง**

- **การหลีกเลี่ยงการยินยอมโดยใช้คุกกี้**: การรับรองตัวตนของผู้ใช้ก่อนหน้าสร้างคุกกี้การยินยอมที่ผู้โจมตีใช้ประโยชน์จากคำขออนุญาตที่ประสงค์ร้ายพร้อม URI เปลี่ยนทางที่ถูกสร้างขึ้น
- **การขโมยรหัสอนุญาต**: คุกกี้การยินยอมที่มีอยู่ทำให้เซิร์ฟเวอร์อนุญาตข้ามหน้าจอยินยอม ส่งรหัสไปยังปลายทางที่ผู้โจมตีควบคุม  
- **การเข้าถึง API โดยไม่ได้รับอนุญาต**: รหัสอนุญาตที่ถูกขโมยทำให้เกิดการแลกเปลี่ยนโทเค็นและแอบอ้างผู้ใช้โดยไม่ได้รับความเห็นชอบอย่างชัดเจน

#### **กลยุทธ์การบรรเทา**

**ข้อควบคุมที่จำเป็น:**
- **ข้อกำหนดการยินยอมชัดเจน**: เซิร์ฟเวอร์พร็อกซี MCP ที่ใช้ ID ลูกค้าคงที่ต้องได้รับความยินยอมจากผู้ใช้สำหรับไคลเอนต์ที่ลงทะเบียนแบบไดนามิกแต่ละครั้ง
- **การใช้งาน OAuth 2.1 อย่างปลอดภัย**: ปฏิบัติตามแนวทางปฏิบัติที่ดีที่สุดใน OAuth ปัจจุบัน รวมถึง PKCE สำหรับคำขออนุญาตทั้งหมด
- **การตรวจสอบไคลเอนต์อย่างเข้มงวด**: ดำเนินการตรวจสอบ URI เปลี่ยนทางและตัวระบุไคลเอนต์อย่างเข้มงวดเพื่อป้องกันการโจมตี

### ช่องโหว่ Token Passthrough  

**Token passthrough** เป็นแอนตี้แพทเทิร์นที่ชัดเจนซึ่งเซิร์ฟเวอร์ MCP รับโทเค็นจากไคลเอนต์โดยไม่ตรวจสอบและส่งต่อไปยัง API ปลายทาง ทำให้ละเมิดข้อกำหนดการอนุญาตของ MCP

#### **ผลกระทบด้านความปลอดภัย**

- **การเลี่ยงการควบคุม**: การใช้โทเค็นจากไคลเอนต์ถึง API โดยตรงข้ามข้อจำกัดด้านอัตรา รายการตรวจสอบ และการเฝ้าระวัง
- **การทำลายร่องรอยตรวจสอบ**: โทเค็นที่ออกจากต้นทางทำให้ไม่สามารถระบุไคลเอนต์ได้ เมื่อเกิดเหตุการณ์จำเป็นต้องสอบสวน
- **การขโมยข้อมูลผ่านพร็อกซี**: โทเค็นที่ไม่ได้รับการตรวจสอบเปิดทางให้ผู้ประสงค์ร้ายใช้เซิร์ฟเวอร์เป็นพร็อกซีเพื่อเข้าถึงข้อมูลโดยไม่ได้รับอนุญาต
- **การละเมิดเขตศรัทธา**: บริการปลายทางอาจถูกละเมิดสมมุติฐานความปลอดภัยเมื่อไม่สามารถตรวจสอบแหล่งที่มาโทเค็นได้
- **การขยายการโจมตีข้ามหลายบริการ**: โทเค็นที่ถูกเจาะและยอมรับในหลายบริการช่วยให้เกิดการโจมตีแบบแนวนอน

#### **ข้อควบคุมความปลอดภัยที่จำเป็น**

**ข้อกำหนดที่ไม่อาจต่อรองได้:**
- **การตรวจสอบโทเค็น**: เซิร์ฟเวอร์ MCP ห้ามรับโทเค็นที่ไม่ได้ออกให้สำหรับเซิร์ฟเวอร์ MCP โดยเฉพาะ
- **การตรวจสอบผู้ฟัง (Audience)**: ตรวจสอบเสมอว่าค่า audience ในโทเค็นตรงกับตัวตนของเซิร์ฟเวอร์ MCP
- **วงจรชีวิตโทเค็นที่เหมาะสม**: ใช้โทเค็นที่มีอายุสั้นพร้อมวิธีหมุนเวียนที่ปลอดภัย


## ความปลอดภัยสายโซ่อุปทานสำหรับระบบ AI

ความปลอดภัยสายโซ่อุปทานได้พัฒนาเกินจากการพึ่งพาซอฟต์แวร์แบบดั้งเดิมเพื่อครอบคลุมระบบนิเวศ AI ทั้งหมด การใช้งาน MCP สมัยใหม่ต้องตรวจสอบและเฝ้าระวังส่วนประกอบ AI ทั้งหมดอย่างเข้มงวด เนื่องจากแต่ละส่วนอาจนำช่องโหว่ที่อาจทำลายความสมบูรณ์ของระบบ

### ส่วนประกอบสายโซ่อุปทาน AI ที่ขยายขอบเขต

**การพึ่งพาซอฟต์แวร์แบบดั้งเดิม:**
- ไลบรารีและเฟรมเวิร์กโอเพนซอร์ส
- อิมเมจคอนเทนเนอร์และระบบฐาน  
- เครื่องมือพัฒนาและพนักงานสร้าง
- ส่วนประกอบโครงสร้างพื้นฐานและบริการ

**องค์ประกอบสายโซ่อุปทานเฉพาะ AI:**
- **โมเดลฐาน**: โมเดลที่ฝึกล่วงหน้าจากผู้ให้บริการต่างๆ ที่ต้องตรวจสอบแหล่งที่มา
- **บริการฝังตัว**: บริการเวกเตอร์และการค้นหาเชิงความหมายภายนอก
- **ผู้ให้บริการบริบท**: แหล่งข้อมูล ฐานความรู้ และคลังเอกสาร  
- **API ของบุคคลที่สาม**: บริการ AI ภายนอก ท่อ ML และจุดสิ้นสุดประมวลผลข้อมูล
- **ชิ้นส่วนโมเดล**: น้ำหนัก การตั้งค่า และรุ่นโมเดลที่ปรับแต่งแล้ว
- **แหล่งข้อมูลฝึกอบรม**: ชุดข้อมูลที่ใช้สำหรับการฝึกและปรับแต่งโมเดล

### กลยุทธ์ความปลอดภัยสายโซ่อุปทานอย่างครบถ้วน

#### **การตรวจสอบส่วนประกอบและศรัทธา**
- **การตรวจสอบแหล่งที่มา**: ตรวจสอบต้นกำเนิด สิทธิ์การใช้งาน และความสมบูรณ์ของส่วนประกอบ AI ทั้งหมดก่อนรวมเข้าใช้งาน
- **การประเมินความปลอดภัย**: ทำการสแกนช่องโหว่และทบทวนความปลอดภัยสำหรับโมเดล แหล่งข้อมูล และบริการ AI
- **การวิเคราะห์ชื่อเสียง**: ประเมินประวัติความปลอดภัยและแนวทางปฏิบัติของผู้ให้บริการ AI
- **การตรวจสอบการปฏิบัติตามกฎระเบียบ**: ให้แน่ใจว่าส่วนประกอบทั้งหมดตรงตามข้อกำหนดด้านความปลอดภัยและกฎระเบียบท้องถิ่น

#### **สายส่งการปรับใช้ที่ปลอดภัย**  
- **CI/CD ความปลอดภัยอัตโนมัติ**: รวมการสแกนความปลอดภัยในสายส่งการปรับใช้อัตโนมัติ
- **การตรวจสอบความสมบูรณ์ของชิ้นงาน**: ใช้การตรวจสอบเชิงลายเซ็นและเข้ารหัสสำหรับชิ้นงานที่ปรับใช้ทั้งหมด (โค้ด โมเดล การตั้งค่า)
- **การปรับใช้เป็นขั้นตอน**: ใช้กลยุทธ์การปรับใช้อย่างก้าวหน้าโดยมีการตรวจสอบความปลอดภัยในแต่ละขั้นตอน
- **ที่เก็บชิ้นงานที่เชื่อถือได้**: ปรับใช้จากที่เก็บชิ้นงานที่ได้รับการยืนยันและปลอดภัยเท่านั้น

#### **การเฝ้าระวังและตอบสนองอย่างต่อเนื่อง**
- **การสแกนการพึ่งพา**: การตรวจสอบความเปราะบางอย่างต่อเนื่องสำหรับซอฟต์แวร์และส่วนประกอบ AI ทั้งหมด
- **การเฝ้าระวังโมเดล**: ประเมินพฤติกรรมโมเดลอย่างต่อเนื่อง ตรวจจับความเบี่ยงเบนในการทำงานและสิ่งผิดปกติด้านความปลอดภัย
- **การติดตามสุขภาพบริการ**: เฝ้าระวังบริการ AI ภายนอกสำหรับความพร้อมใช้งาน เหตุการณ์ความปลอดภัย และการเปลี่ยนแปลงนโยบาย
- **การผสานรวมข่าวกรองภัยคุกคาม**: นำข้อมูลภัยคุกคามเฉพาะทาง AI และ ML มาใช้

#### **การควบคุมการเข้าถึงและการให้สิทธิ์ต่ำสุด**
- **สิทธิ์ระดับส่วนประกอบ**: จำกัดการเข้าถึงโมเดล ข้อมูล และบริการตามความจำเป็นทางธุรกิจ
- **การจัดการบัญชีบริการ**: ใช้บัญชีบริการเฉพาะที่มีสิทธิ์ขั้นต่ำที่ต้องการ
- **การแบ่งแยกเครือข่าย**: แยกส่วนประกอบ AI และจำกัดการเข้าถึงเครือข่ายระหว่างบริการ
- **การควบคุมเกตเวย์ API**: ใช้เกตเวย์ API แบบรวมศูนย์เพื่อควบคุมและตรวจสอบการเข้าถึงบริการ AI ภายนอก

#### **กระบวนการตอบสนองและการกู้คืนเหตุการณ์**
- **กระบวนการตอบโต้รวดเร็ว**: มีขั้นตอนที่กำหนดสำหรับการแพตช์หรือแทนที่ส่วนประกอบ AI ที่ถูกเจาะระบบ
- **การหมุนเวียนข้อมูลรับรอง**: ระบบอัตโนมัติสำหรับการหมุนเวียนความลับ คีย์ API และข้อมูลรับรองบริการ
- **ความสามารถในการย้อนกลับ**: สามารถย้อนเวอร์ชันส่วนประกอบ AI กลับสู่สถานะก่อนหน้าที่ปลอดภัยได้อย่างรวดเร็ว
- **การกู้คืนจากการละเมิดสายโซ่อุปทาน**: ขั้นตอนเฉพาะสำหรับตอบสนองต่อการละเมิดบริการ AI ที่ต้นน้ำ

## แนวปฏิบัติความปลอดภัยพื้นฐานสำหรับ Foundation

การใช้งาน MCP สืบทอดและเพิ่มความแข็งแกร่งให้กับฐานะความปลอดภัยที่มีอยู่ขององค์กรของคุณ การเสริมสร้างแนวปฏิบัติความปลอดภัยพื้นฐานช่วยเพิ่มความปลอดภัยโดยรวมของระบบ AI และการปรับใช้ MCP อย่างมาก

### หลักการพื้นฐานความปลอดภัยหลัก

#### **แนวปฏิบัติการพัฒนาที่ปลอดภัย**
- **ปฏิบัติตาม OWASP**: ปกป้องจากช่องโหว่เว็บแอปพลิเคชัน [OWASP Top 10](https://owasp.org/www-project-top-ten/)
- **การป้องกันเฉพาะ AI**: ดำเนินมาตรการป้องกันตาม [OWASP Top 10 สำหรับ LLMs](https://genai.owasp.org/download/43299/?tmstv=1731900559)
- **การจัดการความลับอย่างปลอดภัย**: ใช้คลังข้อมูลที่ทุ่มเทสำหรับโทเค็น คีย์ API และข้อมูลการตั้งค่าที่ละเอียดอ่อน
- **การเข้ารหัสแบบ end-to-end**: ดำเนินการสื่อสารที่ปลอดภัยผ่านส่วนประกอบและการไหลของข้อมูลทั้งหมด
- **การตรวจสอบค่าป้อนเข้า**: ตรวจสอบค่าป้อนเข้าผู้ใช้ พารามิเตอร์ API และแหล่งข้อมูลอย่างเข้มงวด

#### **การเสริมความแข็งแรงของโครงสร้างพื้นฐาน**
- **การพิสูจน์ตัวตนหลายปัจจัย**: MFA บังคับใช้กับบัญชีผู้ดูแลและบัญชีบริการทั้งหมด
- **การจัดการแพตช์**: แพตช์อัตโนมัติและทันเวลาสำหรับระบบปฏิบัติการ เฟรมเวิร์ก และการพึ่งพา  
- **การผสานรวมผู้ให้บริการตัวตน**: การจัดการตัวตนแบบรวมศูนย์ผ่านผู้ให้บริการตัวตนองค์กร (Microsoft Entra ID, Active Directory)
- **การแบ่งแยกเครือข่าย**: การแยกส่วนประกอบ MCP อย่างมีเหตุผลเพื่อจำกัดการเคลื่อนที่ในแนวนอน
- **หลักการให้สิทธิ์ต่ำสุด**: ให้สิทธิ์ขั้นต่ำที่จำเป็นสำหรับทุกส่วนประกอบและบัญชีระบบ

#### **การตรวจสอบและการตรวจจับความปลอดภัย**
- **การบันทึกข้อมูลอย่างครบถ้วน**: การบันทึกรายละเอียดกิจกรรมแอปพลิเคชัน AI รวมถึงปฏิสัมพันธ์ไคลเอนต์-เซิร์ฟเวอร์ MCP
- **การผสานรวม SIEM**: การจัดการข้อมูลความปลอดภัยและเหตุการณ์แบบรวมศูนย์เพื่อตรวจจับความผิดปกติ
- **การวิเคราะห์พฤติกรรม**: การเฝ้าระวังด้วย AI เพื่อตรวจจับรูปแบบที่ผิดปกติในระบบและพฤติกรรมผู้ใช้
- **ข่าวกรองภัยคุกคาม**: การผสานรวมฟีดข้อมูลภัยคุกคามภายนอกและตัวบ่งชี้การโจมตี (IOC)
- **การตอบสนองเหตุการณ์**: ขั้นตอนที่กำหนดไว้อย่างดีสำหรับการตรวจจับและตอบสนองต่อเหตุการณ์ความปลอดภัย

#### **สถาปัตยกรรมความเชื่อถือศูนย์ (Zero Trust)**
- **ไม่เชื่อถือใคร ตรวจสอบเสมอ**: การตรวจสอบผู้ใช้ อุปกรณ์ และการเชื่อมต่อเครือข่ายอย่างต่อเนื่อง
- **การแบ่งแยกขนาดเล็ก (Micro-Segmentation)**: การควบคุมเครือข่ายที่ละเอียดเพื่อแยกงานและบริการแต่ละส่วน
- **ความปลอดภัยรอบตัวตัวตน**: นโยบายความปลอดภัยที่อิงตามตัวตนที่ตรวจสอบได้มากกว่าตำแหน่งเครือข่าย
- **การประเมินความเสี่ยงอย่างต่อเนื่อง**: ประเมินสถานะความปลอดภัยแบบไดนามิกตามบริบทและพฤติกรรมปัจจุบัน
- **การเข้าถึงตามเงื่อนไข**: การควบคุมการเข้าถึงที่ปรับตามปัจจัยความเสี่ยง ตำแหน่ง และความเชื่อถืออุปกรณ์

### รูปแบบการผสานรวมในองค์กร

#### **การผสานรวมระบบนิเวศความปลอดภัยของ Microsoft**
- **Microsoft Defender for Cloud**: การจัดการสถานะความปลอดภัยบนคลาวด์แบบครบวงจร
- **Azure Sentinel**: SIEM และ SOAR บนคลาวด์สำหรับการปกป้องงาน AI
- **Microsoft Entra ID**: การจัดการตัวตนและการเข้าถึงองค์กรพร้อมนโยบายการเข้าถึงตามเงื่อนไข
- **Azure Key Vault**: การจัดการความลับแบบรวมศูนย์พร้อมระบบฮาร์ดแวร์ HSM รองรับ
- **Microsoft Purview**: การกำกับดูแลข้อมูลและการปฏิบัติตามข้อกำหนดสำหรับแหล่งข้อมูลและเวิร์กโฟลว์ AI

#### **การปฏิบัติตามกฎระเบียบและการกำกับดูแล**
- **การปฏิบัติตามข้อกำหนด**: ให้แน่ใจว่าการใช้งาน MCP สอดคล้องกับข้อกำหนดอุตสาหกรรมเฉพาะทาง (GDPR, HIPAA, SOC 2)
- **การจำแนกประเภทข้อมูล**: การจัดหมวดหมู่และการจัดการข้อมูลที่ละเอียดอ่อนในระบบ AI อย่างเหมาะสม
- **ร่องรอยตรวจสอบ**: การบันทึกข้อมูลอย่างครบถ้วนเพื่อตอบสนองข้อกำหนดและการสอบสวนทางนิติวิทยาศาสตร์
- **การควบคุมความเป็นส่วนตัว**: การใช้หลักการ privacy-by-design ในสถาปัตยกรรมระบบ AI
- **การจัดการการเปลี่ยนแปลง**: กระบวนการอย่างเป็นทางการสำหรับการทบทวนความปลอดภัยก่อนเปลี่ยนแปลงระบบ AI

แนวปฏิบัติพื้นฐานเหล่านี้สร้างฐานความปลอดภัยที่เข้มแข็ง ช่วยเพิ่มประสิทธิผลของการควบคุมความปลอดภัยเฉพาะ MCP และให้การป้องกันครบวงจรสำหรับแอปพลิเคชันที่ขับเคลื่อนด้วย AI

## ข้อสรุปด้านความปลอดภัยที่สำคัญ
- **แนวทางความปลอดภัยแบบชั้นซ้อน**: ผสมผสานแนวทางปฏิบัติความปลอดภัยพื้นฐาน (การเขียนโค้ดอย่างปลอดภัย การจำกัดสิทธิ์ต่ำสุด การตรวจสอบห่วงโซ่อุปทาน การตรวจสอบอย่างต่อเนื่อง) กับการควบคุมเฉพาะ AI เพื่อการป้องกันโดยรวม

- **ภูมิทัศน์ภัยคุกคามเฉพาะ AI**: ระบบ MCP เผชิญกับความเสี่ยงเฉพาะ เช่น การฉีดคำสั่ง (prompt injection), การพิษเครื่องมือ (tool poisoning), การแฮกช่วงเซสชัน, ปัญหาผู้ช่วยสับสน (confused deputy), ช่องโหว่การส่งต่อโทเค็น และสิทธิ์ที่มากเกินไป ซึ่งต้องการมาตรการแก้ไขเฉพาะ

- **ความเป็นเลิศด้านการพิสูจน์ตัวตนและการอนุญาต**: ใช้การพิสูจน์ตัวตนที่แข็งแกร่งโดยใช้ผู้ให้บริการตัวตนภายนอก (Microsoft Entra ID), บังคับใช้การตรวจสอบโทเค็นอย่างเหมาะสม และไม่รับโทเค็นที่ไม่ได้ออกให้แก่เซิร์ฟเวอร์ MCP ของคุณโดยชัดเจน

- **การป้องกันการโจมตี AI**: ใช้ Microsoft Prompt Shields และ Azure Content Safety เพื่อป้องกันการโจมตีแบบฉีดคำสั่งทางอ้อมและการพิษเครื่องมือ พร้อมทั้งตรวจสอบข้อมูลเมตาเครื่องมือและติดตามการเปลี่ยนแปลงแบบไดนามิก

- **ความปลอดภัยของเซสชันและการส่งผ่านข้อมูล**: ใช้รหัสเซสชันที่ปลอดภัยเชิงเข้ารหัสและไม่สามารถทำนายได้ ผูกกับตัวตนผู้ใช้, ดำเนินการจัดการวงจรชีวิตเซสชันอย่างเหมาะสม และไม่ใช้เซสชันสำหรับการพิสูจน์ตัวตน

- **แนวทางปฏิบัติที่ดีที่สุดสำหรับความปลอดภัย OAuth**: ป้องกันการโจมตี confused deputy ผ่านความยินยอมของผู้ใช้ที่ชัดเจนสำหรับไคลเอนต์ที่ลงทะเบียนแบบไดนามิก, ใช้งาน OAuth 2.1 อย่างถูกต้องด้วย PKCE, และตรวจสอบ URI การเปลี่ยนเส้นทางอย่างเข้มงวด

- **หลักการความปลอดภัยของโทเค็น**: หลีกเลี่ยงรูปแบบโทเค็นส่งต่อที่ผิดพลาด, ตรวจสอบคำเรียกร้องกลุ่มเป้าหมายของโทเค็น, ใช้โทเค็นระยะสั้นที่มีการหมุนเวียนอย่างปลอดภัย, และรักษาขอบเขตความน่าเชื่อถือที่ชัดเจน

- **ความปลอดภัยห่วงโซ่อุปทานโดยรวม**: ปฏิบัติต่อส่วนประกอบทั้งหมดของระบบนิเวศ AI (โมเดล, ตัวฝัง, ผู้ให้บริการบริบท, API ภายนอก) ด้วยความเข้มงวดของความปลอดภัยเช่นเดียวกับซอฟต์แวร์ทั่วไป

- **วิวัฒนาการอย่างต่อเนื่อง**: รักษาความทันสมัยกับข้อกำหนด MCP ที่พัฒนาอย่างรวดเร็ว, มีส่วนร่วมในมาตรฐานชุมชนความปลอดภัย, และรักษาท่าทีความปลอดภัยที่ยืดหยุ่นเมื่อโปรโตคอลเติบโต

- **การผนวกรวมความปลอดภัยของ Microsoft**: ใช้ประโยชน์จากระบบนิเวศความปลอดภัยครอบคลุมของ Microsoft (Prompt Shields, Azure Content Safety, GitHub Advanced Security, Entra ID) เพื่อการป้องกันการใช้งาน MCP ที่ดียิ่งขึ้น

- **[การใช้งาน Azure Content Safety](./azure-content-safety-implementation.md)** - ตัวอย่างการใช้งานจริงสำหรับการผนวกรวม Azure Content Safety
- **[มาตรการควบคุมความปลอดภัย MCP 2025](./mcp-security-controls-2025.md)** - มาตรการควบคุมและเทคนิคความปลอดภัยล่าสุดสำหรับการใช้งาน MCP
- **[คู่มืออ้างอิงด่วนแนวทางปฏิบัติ MCP](./mcp-best-practices.md)** - คู่มืออ้างอิงด่วนสำหรับแนวทางปฏิบัติความปลอดภัย MCP ที่สำคัญ
- **[BlueHat 2026: การรักษาความปลอดภัยอนาคตของ AI: การรักษาความปลอดภัย MCP ด้วยรูปแบบการป้องกันแบบลึก](https://www.youtube.com/watch?v=cVWB58kEt-Y)** - รูปแบบการป้องกันเชิงลึกจากศูนย์ตอบสนองความปลอดภัย Microsoft (MSRC)

### **การฝึกอบรมความปลอดภัยเชิงปฏิบัติ**

- **[เวิร์กช็อป MCP Security Summit (Sherpa)](https://azure-samples.github.io/sherpa/)** - เวิร์กช็อปเชิงปฏิบัติที่ครอบคลุมสำหรับการรักษาความปลอดภัยเซิร์ฟเวอร์ MCP บน Azure พร้อมค่ายฝึกขั้นสูงจาก Base Camp จนถึง Summit
- **[คู่มือความปลอดภัย Azure สำหรับ OWASP MCP](https://microsoft.github.io/mcp-azure-security-guide/)** - สถาปัตยกรรมอ้างอิงและคำแนะนำการใช้งานสำหรับความเสี่ยง OWASP MCP Top 10 ทุกด้าน
