# การสตรีม HTTPS ด้วยโปรโตคอลบริบทโมเดล (MCP)

บทนี้ให้คำแนะนำอย่างครบถ้วนในการใช้งานการสตรีมที่ปลอดภัย ขยายขนาดได้ และเรียลไทม์ผ่านโปรโตคอลบริบทโมเดล (MCP) โดยใช้ HTTPS ครอบคลุมทั้งแรงจูงใจสำหรับการสตรีม กลไกการขนส่งที่มีอยู่ วิธีการใช้งาน HTTP แบบสตรีมใน MCP แนวปฏิบัติด้านความปลอดภัย การย้ายจาก SSE และคำแนะนำในการสร้างแอปพลิเคชัน MCP แบบสตรีมของคุณเอง

## กลไกการขนส่งและการสตรีมใน MCP

ส่วนนี้อธิบายกลไกการขนส่งต่าง ๆ ที่มีอยู่ใน MCP และบทบาทของพวกมันในการเปิดใช้งานความสามารถสตรีมสำหรับการสื่อสารแบบเรียลไทม์ระหว่างไคลเอนต์กับเซิร์ฟเวอร์

### กลไกการขนส่งคืออะไร?

กลไกการขนส่งคือลักษณะที่ข้อมูลถูกแลกเปลี่ยนระหว่างไคลเอนต์กับเซิร์ฟเวอร์ MCP รองรับชนิดการขนส่งหลายรูปแบบเพื่อตอบสนองต่อสภาพแวดล้อมและความต้องการที่ต่างกัน:

- **stdio**: อินพุต/เอาต์พุตมาตรฐาน เหมาะสำหรับเครื่องมือท้องถิ่นและแบบ CLI ง่ายแต่ไม่เหมาะสำหรับบนเว็บหรือคลาวด์
- **SSE (Server-Sent Events)**: อนุญาตให้เซิร์ฟเวอร์ส่งอัปเดตแบบเรียลไทม์ไปยังไคลเอนต์ผ่าน HTTP ดีสำหรับ UI เว็บแต่จำกัดในเรื่องการขยายและความยืดหยุ่น ตั้งแต่ MCP Specification 2025-06-18 การขนส่ง SSE แบบแยกเดี่ยวได้ถูกเลิกใช้งานและถูกแทนที่ด้วยการขนส่ง "Streamable HTTP"
- **Streamable HTTP**: การขนส่งสตรีมใหม่บนพื้นฐาน HTTP รองรับการแจ้งเตือนและมีความสามารถในการขยายที่ดีกว่า แนะนำสำหรับสถานการณ์การผลิตและคลาวด์ส่วนใหญ่

### ตารางเปรียบเทียบ

ดูตารางเปรียบเทียบด้านล่างเพื่อเข้าใจความแตกต่างระหว่างกลไกการขนส่งเหล่านี้:

| กลไกการขนส่ง     | อัปเดตแบบเรียลไทม์ | สตรีม    | การขยายขนาด | กรณีการใช้งาน           |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | ไม่               | ไม่        | ต่ำ         | เครื่องมือ CLI ท้องถิ่น  |
| SSE               | ใช่              | ใช่       | ปานกลาง    | เว็บ, อัปเดตเรียลไทม์   |
| Streamable HTTP   | ใช่              | ใช่       | สูง         | คลาวด์, หลายไคลเอนต์    |

> **เคล็ดลับ:** การเลือกกลไกการขนส่งที่เหมาะสมมีผลต่อประสิทธิภาพ การขยาย และประสบการณ์ผู้ใช้ **Streamable HTTP** แนะนำสำหรับแอปพลิเคชันสมัยใหม่ที่ต้องการความสามารถในการขยายและรองรับคลาวด์

สังเกตถึงการขนส่ง stdio และ SSE ที่คุณได้เห็นในบทก่อนหน้า และ Streamable HTTP คือกลไกการขนส่งที่พูดถึงในบทนี้

## การสตรีม: แนวคิดและแรงจูงใจ

การเข้าใจแนวคิดพื้นฐานและแรงจูงใจเบื้องหลังการสตรีมเป็นสิ่งจำเป็นสำหรับการสร้างระบบสื่อสารเรียลไทม์ที่มีประสิทธิภาพ

**การสตรีม** คือเทคนิคในโปรแกรมเครือข่ายที่ช่วยให้ข้อมูลถูกส่งและรับทีละส่วนเล็ก ๆ หรือลำดับเหตุการณ์ แทนที่จะต้องรอการตอบสนองทั้งหมดพร้อมกัน ซึ่งเป็นประโยชน์อย่างยิ่งสำหรับ:

- ไฟล์หรือชุดข้อมูลขนาดใหญ่
- อัปเดตแบบเรียลไทม์ (เช่น แชท แถบความคืบหน้า)
- การคำนวณระยะยาวที่ต้องแจ้งผู้ใช้บ่อย ๆ

นี่คือสิ่งที่คุณควรรู้เกี่ยวกับการสตรีมในภาพรวม:

- ข้อมูลถูกส่งอย่างต่อเนื่อง ไม่ใช่ทั้งหมดพร้อมกัน
- ไคลเอนต์สามารถประมวลผลข้อมูลทันทีที่มาถึง
- ลดความล่าช้าที่รู้สึกและปรับปรุงประสบการณ์ผู้ใช้

### ทำไมต้องใช้การสตรีม?

เหตุผลในการใช้การสตรีมมีดังนี้:

- ผู้ใช้ได้รับผลตอบรับทันที ไม่ใช่แค่ตอนท้าย
- ช่วยให้แอปแบบเรียลไทม์และ UI ที่ตอบสนองได้
- ใช้เครือข่ายและทรัพยากรคอมพิวเตอร์ได้อย่างมีประสิทธิภาพมากขึ้น

### ตัวอย่างง่าย ๆ: เซิร์ฟเวอร์ & ไคลเอนต์สตรีม HTTP

นี่คือตัวอย่างง่าย ๆ ว่าการสตรีมทำงานอย่างไร:

#### Python

**เซิร์ฟเวอร์ (Python, ใช้ FastAPI และ StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ไคลเอนต์ (Python, ใช้ requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ตัวอย่างนี้แสดงให้เห็นว่าเซิร์ฟเวอร์ส่งชุดข้อความไปยังไคลเอนต์เมื่อพร้อม ไม่ต้องรอทั้งหมดพร้อมกัน

**วิธีการทำงาน:**

- เซิร์ฟเวอร์ส่งแต่ละข้อความทันทีที่พร้อม
- ไคลเอนต์รับและพิมพ์ข้อความแต่ละชิ้นเมื่อมาถึง

**ข้อกำหนด:**

- เซิร์ฟเวอร์ต้องใช้การตอบสนองแบบสตรีม (เช่น `StreamingResponse` ใน FastAPI)
- ไคลเอนต์ต้องประมวลผลการตอบสนองเป็นสตรีม (`stream=True` ใน requests)
- Content-Type ส่วนใหญ่เป็น `text/event-stream` หรือ `application/octet-stream`

#### Java

**เซิร์ฟเวอร์ (Java, ใช้ Spring Boot และ Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ไคลเอนต์ (Java, ใช้ Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**หมายเหตุการใช้งาน Java:**

- ใช้สแต็ครีแอกทีฟของ Spring Boot พร้อม `Flux` สำหรับการสตรีม
- `ServerSentEvent` ให้สตรีมเหตุการณ์ที่มีโครงสร้างพร้อมประเภทเหตุการณ์
- `WebClient` กับ `bodyToFlux()` เปิดใช้งานการบริโภคสตรีมแบบรีแอกทีฟ
- `delayElements()` จำลองเวลาการประมวลผลระหว่างเหตุการณ์
- เหตุการณ์อาจมีประเภท (`info`, `result`) เพื่อช่วยการจัดการของไคลเอนต์ให้ดีขึ้น

### เปรียบเทียบ: การสตรีมแบบคลาสสิก vs MCP Streaming

ความแตกต่างระหว่างการสตรีมแบบ "คลาสสิก" กับวิธีใน MCP อธิบายได้ดังนี้:

| คุณสมบัติ             | การสตรีม HTTP แบบคลาสสิก    | การสตรีม MCP (การแจ้งเตือน)       |
|------------------------|-------------------------------|-------------------------------------|
| การตอบกลับหลัก        | แบ่งเป็นชิ้น                   | ตอบกลับครั้งเดียวตอนท้าย             |
| การอัปเดตความคืบหน้า | ส่งเป็นข้อมูลชิ้น ๆ           | ส่งเป็นการแจ้งเตือน                 |
| ความต้องการของไคลเอนต์ | ต้องประมวลผลสตรีม            | ต้องมีตัวจัดการข้อความ                 |
| กรณีใช้งาน           | ไฟล์ใหญ่, สตรีมโทเค็น AI    | ความคืบหน้า, บันทึก, ข้อเสนอแนะเรียลไทม์   |

### ความแตกต่างหลักที่พบ

นอกจากนี้ยังมีความแตกต่างสำคัญอื่น ๆ:

- **รูปแบบการสื่อสาร:**
  - การสตรีม HTTP แบบคลาสสิก: ใช้การเข้ารหัสแบบแบ่งชิ้นง่าย ๆ ส่งข้อมูลเป็นส่วน ๆ
  - การสตรีม MCP: ใช้ระบบแจ้งเตือนที่มีโครงสร้างด้วยโปรโตคอล JSON-RPC

- **รูปแบบข้อความ:**
  - HTTP คลาสสิก: ข้อความธรรมดาแบ่งเป็นชิ้น ๆ พร้อมขึ้นบรรทัดใหม่
  - MCP: วัตถุ LoggingMessageNotification ที่มีโครงสร้างและเมตาดาต้า

- **การใช้งานไคลเอนต์:**
  - HTTP คลาสสิก: ไคลเอนต์ง่ายที่ประมวลผลการตอบสนองแบบสตรีม
  - MCP: ไคลเอนต์ซับซ้อนกว่าที่มีตัวจัดการข้อความสำหรับประมวลผลข้อความหลายประเภท

- **อัปเดตความคืบหน้า:**
  - HTTP คลาสสิก: ความคืบหน้าอยู่ในสตรีมตอบกลับหลัก
  - MCP: ความคืบหน้าส่งแยกเป็นข้อความแจ้งเตือน ส่วนผลลัพธ์หลักส่งตอนท้าย

### คำแนะนำ

มีข้อแนะนำบางประการเมื่อเลือกใช้การสตรีมในแบบคลาสสิก (เช่นจุดสิ้นสุด `/stream` ที่เคยแสดง) กับการสตรีมผ่าน MCP:

- **สำหรับความต้องการสตรีมง่าย ๆ:** การสตรีม HTTP แบบคลาสสิกง่ายกว่าและเหมาะสำหรับความต้องการพื้นฐาน

- **สำหรับแอปที่ซับซ้อนและมีปฏิสัมพันธ์:** การสตรีม MCP ให้โครงสร้างที่ชัดเจนขึ้น พร้อมเมตาดาต้าที่มากขึ้นและแยกแยะระหว่างการแจ้งเตือนกับผลลัพธ์สุดท้าย

- **สำหรับแอป AI:** ระบบแจ้งเตือนของ MCP มีประโยชน์มากสำหรับงาน AI ระยะยาวที่ต้องการแจ้งผู้ใช้ถึงความคืบหน้า

## การสตรีมใน MCP

ตกลง คุณเห็นคำแนะนำและการเปรียบเทียบเบื้องต้นแล้ว เกี่ยวกับความแตกต่างระหว่างการสตรีมแบบคลาสสิกและใน MCP ตอนนี้มาดูรายละเอียดว่าเราจะใช้การสตรีมใน MCP ได้อย่างไร

การเข้าใจว่าการสตรีมทำงานอย่างไรในกรอบงาน MCP เป็นสิ่งจำเป็นสำหรับการสร้างแอปที่ตอบสนองและให้ข้อมูลเรียลไทม์แก่ผู้ใช้ระหว่างการประมวลผลงานระยะยาว

ใน MCP การสตรีมไม่ได้เกี่ยวกับการส่งการตอบกลับหลักเป็นชิ้น ๆ แต่เป็นการส่ง **การแจ้งเตือน** ไปยังไคลเอนต์ในระหว่างที่เครื่องมือกำลังประมวลผลคำขอ ข้อความแจ้งเตือนเหล่านี้อาจรวมถึงอัปเดตความคืบหน้า บันทึก หรือเหตุการณ์อื่น ๆ

### วิธีการทำงาน

ผลลัพธ์หลักยังส่งเป็นการตอบกลับครั้งเดียว แต่อาจส่งข้อความแจ้งเตือนเป็นข้อความแยกระหว่างการประมวลผลเพื่ออัปเดตไคลเอนต์แบบเรียลไทม์ ไคลเอนต์ต้องสามารถจัดการและแสดงข้อความแจ้งเตือนเหล่านี้ได้

## การแจ้งเตือนคืออะไร?

เราพูดถึง "การแจ้งเตือน" นั่นหมายความว่าอะไรในบริบทของ MCP?

การแจ้งเตือนคือข้อความที่เซิร์ฟเวอร์ส่งไปยังไคลเอนต์เพื่อแจ้งเกี่ยวกับความคืบหน้า สถานะ หรือเหตุการณ์อื่น ๆ ในระหว่างการทำงานที่ใช้เวลานาน การแจ้งเตือนช่วยเพิ่มความโปร่งใสและประสบการณ์ผู้ใช้

ตัวอย่างเช่น ไคลเอนต์จะต้องส่งการแจ้งเตือนเมื่อการเชื่อมต่อเริ่มต้นกับเซิร์ฟเวอร์เสร็จสมบูรณ์

การแจ้งเตือนมีลักษณะเป็นข้อความ JSON ดังนี้:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

การแจ้งเตือนอยู่ภายใต้หัวข้อ MCP ที่เรียกว่า ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging)

เพื่อให้การบันทึกทำงาน เซิร์ฟเวอร์ต้องเปิดใช้งานเป็นฟีเจอร์/ความสามารถดังนี้:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ขึ้นอยู่กับ SDK ที่ใช้ การบันทึกอาจเปิดใช้งานโดยค่าเริ่มต้น หรือคุณอาจต้องเปิดใช้งานในคอนฟิกของเซิร์ฟเวอร์ด้วยตนเอง

มีประเภทของการแจ้งเตือนดังนี้:

| ระดับ     | คำอธิบาย                     | ตัวอย่างการใช้งาน            |
|-----------|------------------------------|------------------------------|
| debug     | ข้อมูลการดีบักอย่างละเอียด  | จุดเข้า/ออกฟังก์ชัน          |

| error     | สถานการณ์ผิดพลาด            | ความล้มเหลวในการทำงาน         |

## การใช้งานการแจ้งเตือนใน MCP

เพื่อใช้งานการแจ้งเตือนใน MCP คุณต้องตั้งค่าทั้งฝั่งเซิร์ฟเวอร์และไคลเอนต์ให้รองรับการอัปเดตเรียลไทม์ ซึ่งช่วยให้แอปของคุณสามารถให้ผลตอบรับทันทีแก่ผู้ใช้ระหว่างงานที่ใช้เวลานาน

### ฝั่งเซิร์ฟเวอร์: การส่งการแจ้งเตือน

เริ่มต้นที่ฝั่งเซิร์ฟเวอร์ ใน MCP คุณกำหนดเครื่องมือที่สามารถส่งการแจ้งเตือนได้ในระหว่างการประมวลผลคำขอ เซิร์ฟเวอร์ใช้อ็อบเจ็กต์ context (ปกติคือ `ctx`) เพื่อส่งข้อความไปยังไคลเอนต์

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

ในตัวอย่างข้างต้น เครื่องมือ `process_files` ส่งการแจ้งเตือนสามครั้งไปยังไคลเอนต์ขณะประมวลผลแต่ละไฟล์ เมธอด `ctx.info()` ใช้สำหรับส่งข้อความข้อมูล

นอกจากนี้ เพื่อเปิดใช้งานการแจ้งเตือน ตรวจสอบให้แน่ใจว่าเซิร์ฟเวอร์ใช้การขนส่งแบบสตรีม (เช่น `streamable-http`) และไคลเอนต์ใช้งานตัวจัดการข้อความเพื่อจัดการแจ้งเตือน ตัวอย่างการตั้งค่าเซิร์ฟเวอร์ให้ใช้การขนส่ง `streamable-http` มีดังนี้:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ในตัวอย่าง .NET นี้ เครื่องมือ `ProcessFiles` ถูกตกแต่งด้วยแอตทริบิวต์ `Tool` และส่งการแจ้งเตือนสามครั้งไปยังไคลเอนต์ในระหว่างประมวลผลแต่ละไฟล์ เมธอด `ctx.Info()` ใช้สำหรับส่งข้อความข้อมูล

เพื่อเปิดใช้งานการแจ้งเตือนในเซิร์ฟเวอร์ MCP .NET ของคุณ ให้ตรวจสอบว่ากำลังใช้การขนส่งแบบสตรีม:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### ฝั่งไคลเอนต์: การรับการแจ้งเตือน

ไคลเอนต์ต้องใช้งานตัวจัดการข้อความเพื่อประมวลผลและแสดงการแจ้งเตือนเมื่อมาถึง

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

ในโค้ดข้างต้น ฟังก์ชัน `message_handler` ตรวจสอบว่าเมสเสจที่เข้ามาเป็นการแจ้งเตือนหรือไม่ หากใช่จะพิมพ์การแจ้งเตือนนั้น มิฉะนั้นจะประมวลผลเป็นข้อความจากเซิร์ฟเวอร์ทั่วไป นอกจากนี้สังเกตว่าการสร้าง `ClientSession` มีการระบุตัวจัดการข้อความเพื่อจัดการการแจ้งเตือนที่เข้ามา

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ในตัวอย่าง .NET นี้ ฟังก์ชัน `MessageHandler` ตรวจสอบว่าเมสเสจที่เข้ามาเป็นการแจ้งเตือนหรือไม่ หากใช่จะพิมพ์การแจ้งเตือน มิฉะนั้นจะประมวลผลเป็นข้อความจากเซิร์ฟเวอร์ทั่วไป `ClientSession` ถูกสร้างโดยใช้ตัวจัดการข้อความผ่าน `ClientSessionOptions`

เพื่อเปิดใช้งานการแจ้งเตือน ตรวจสอบว่าเซิร์ฟเวอร์ใช้การขนส่งแบบสตรีม (เช่น `streamable-http`) และไคลเอนต์ใช้งานตัวจัดการข้อความเพื่อประมวลผลการแจ้งเตือน

## การแจ้งเตือนความคืบหน้า & สถานการณ์

ส่วนนี้อธิบายแนวคิดของการแจ้งเตือนความคืบหน้าใน MCP ทำไมจึงสำคัญ และวิธีการใช้งานผ่าน Streamable HTTP นอกจากนี้ยังมีแบบฝึกหัดเชิงปฏิบัติเพื่อเสริมความเข้าใจของคุณ

การแจ้งเตือนความคืบหน้าเป็นข้อความแบบเรียลไทม์ที่เซิร์ฟเวอร์ส่งไปยังไคลเอนต์ในระหว่างการทำงานที่ใช้เวลานาน แทนที่จะรอจนจบกระบวนการ เซิร์ฟเวอร์จะรายงานสภาพปัจจุบันให้ไคลเอนต์ทราบ ซึ่งช่วยเพิ่มความโปร่งใส ประสบการณ์ผู้ใช้ และทำให้การดีบักง่ายขึ้น

**ตัวอย่าง:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ทำไมต้องใช้การแจ้งเตือนความคืบหน้า?

การแจ้งเตือนความคืบหน้ามีความจำเป็นด้วยเหตุผลหลายประการ:

- **ประสบการณ์ผู้ใช้ที่ดีกว่า:** ผู้ใช้เห็นอัปเดตเมื่อการทำงานดำเนินไป ไม่ใช่แค่ตอนจบ
- **ผลตอบรับแบบเรียลไทม์:** ไคลเอนต์สามารถแสดงแถบความคืบหน้าหรือบันทึก ทำให้แอปตอบสนองเร็วขึ้น
- **การดีบักและตรวจสอบง่ายขึ้น:** นักพัฒนาและผู้ใช้เห็นว่ากระบวนการช่วงใดช้า หรือหยุดนิ่ง

### วิธีใช้งานการแจ้งเตือนความคืบหน้า

นี่คือวิธีการใช้งานการแจ้งเตือนความคืบหน้าใน MCP:

- **ฝั่งเซิร์ฟเวอร์:** ใช้ `ctx.info()` หรือ `ctx.log()` ส่งการแจ้งเตือนเมื่อแต่ละรายการถูกประมวลผล ส่งข้อความไปยังไคลเอนต์ก่อนผลลัพธ์หลักจะพร้อม
- **ฝั่งไคลเอนต์:** ใช้งานตัวจัดการข้อความที่ฟังและแสดงการแจ้งเตือนเมื่อมาถึง ตัวจัดการนี้จะจำแนกระหว่างการแจ้งเตือนกับผลลัพธ์สุดท้าย

**ตัวอย่างเซิร์ฟเวอร์:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**ตัวอย่างไคลเอนต์:**

#### Python

```python

    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## ข้อควรพิจารณาด้านความปลอดภัย

เมื่อใช้งานเซิร์ฟเวอร์ MCP ที่ใช้การขนส่งผ่าน HTTP ความปลอดภัยจะเป็นสิ่งที่สำคัญมากซึ่งต้องให้ความสนใจอย่างรอบคอบต่อช่องทางการโจมตีหลายรูปแบบและกลไกป้องกัน

### ภาพรวม

ความปลอดภัยเป็นสิ่งสำคัญเมื่อต้องเปิดเผยเซิร์ฟเวอร์ MCP ผ่าน HTTP การขนส่งแบบ Streamable HTTP สร้างช่องทางโจมตีใหม่ ๆ และต้องการการตั้งค่าที่รอบคอบ

### ประเด็นสำคัญ

- **การตรวจสอบค่า Origin Header**: ตรวจสอบค่า `Origin` เสมอเพื่อป้องกันการโจมตีแบบ DNS rebinding
- **การผูกไอพีแบบ Localhost**: สำหรับการพัฒนาท้องถิ่น ให้ผูกเซิร์ฟเวอร์ที่ `localhost` เพื่อไม่ให้เปิดเผยบนอินเทอร์เน็ตสาธารณะ
- **การตรวจสอบสิทธิ์**: ใช้วิธีการตรวจสอบสิทธิ์ (เช่น API keys, OAuth) สำหรับการใช้งานจริง
- **CORS**: กำหนดนโยบาย Cross-Origin Resource Sharing (CORS) เพื่อจำกัดการเข้าถึง
- **HTTPS**: ใช้ HTTPS ในการผลิตเพื่อเข้ารหัสการรับส่งข้อมูล

### แนวปฏิบัติที่ดีที่สุด

- อย่าเชื่อคำขอที่เข้ามาโดยไม่ตรวจสอบ
- บันทึกและตรวจสอบการเข้าถึงและข้อผิดพลาดทั้งหมด
- อัปเดตไลบรารีและแพ็กเกจอย่างสม่ำเสมอเพื่อติดตั้งแพตช์ความปลอดภัย

### ความท้าทาย
- การสร้างสมดุลระหว่างความปลอดภัยกับความง่ายในการพัฒนา
- การรับรองความเข้ากันได้กับสภาพแวดล้อมไคลเอ็นต์ต่าง ๆ

## การอัปเกรดจาก SSE เป็น Streamable HTTP

สำหรับแอปพลิเคชันที่ใช้ Server-Sent Events (SSE) อยู่ในปัจจุบัน การย้ายไปใช้ Streamable HTTP จะเพิ่มขีดความสามารถและความยั่งยืนในระยะยาวที่ดีกว่าสำหรับการใช้งาน MCP ของคุณ

### ทำไมต้องอัปเกรด?

มีสองเหตุผลที่น่าสนใจสำหรับการอัปเกรดจาก SSE ไปเป็น Streamable HTTP:

- Streamable HTTP มีความสามารถในการปรับขนาดได้ดีขึ้น ความเข้ากันได้ดีกว่า และรองรับการแจ้งเตือนที่หลากหลายกว่าการใช้ SSE
- เป็นการขนส่งที่แนะนำสำหรับแอป MCP ใหม่

### ขั้นตอนการย้ายระบบ

นี่คือวิธีการย้ายจาก SSE ไปยัง Streamable HTTP ในแอป MCP ของคุณ:

- **อัปเดตโค้ดเซิร์ฟเวอร์** ให้ใช้ `transport="streamable-http"` ใน `mcp.run()`
- **อัปเดตโค้ดไคลเอ็นต์** ให้ใช้ `streamablehttp_client` แทน SSE client
- **สร้างตัวจัดการข้อความ** ในไคลเอ็นต์เพื่อประมวลผลการแจ้งเตือน
- **ทดสอบความเข้ากันได้** กับเครื่องมือและเวิร์กโฟลว์ที่มีอยู่

### การรักษาความเข้ากันได้

แนะนำให้รักษาความเข้ากันได้กับไคลเอ็นต์ SSE ที่มีอยู่ในระหว่างกระบวนการย้ายระบบ ต่อไปนี้คือกลยุทธ์บางประการ:

- คุณสามารถรองรับทั้ง SSE และ Streamable HTTP โดยการรัน transport ทั้งสองบน endpoint ที่ต่างกัน
- ค่อย ๆ ย้ายไคลเอ็นต์ไปยัง transport ใหม่

### ความท้าทาย

ควรจัดการกับความท้าทายต่อไปนี้ในระหว่างการย้ายระบบ:

- การทำให้แน่ใจว่าไคลเอ็นต์ทุกเครื่องได้รับการอัปเดต
- การจัดการความแตกต่างในการส่งมอบการแจ้งเตือน

## ข้อพิจารณาด้านความปลอดภัย

ความปลอดภัยควรเป็นเรื่องสำคัญสูงสุดเมื่อดำเนินการเซิร์ฟเวอร์ใด ๆ โดยเฉพาะอย่างยิ่งเมื่อใช้ transport ที่ใช้ HTTP เช่น Streamable HTTP ใน MCP

เมื่อพัฒนาเซิร์ฟเวอร์ MCP ด้วย transport ที่ใช้ HTTP ความปลอดภัยจะเป็นเรื่องสำคัญสูงสุดที่ต้องใส่ใจอย่างระมัดระวังต่อช่องทางการโจมตีและกลไกป้องกันต่าง ๆ

### ภาพรวม

ความปลอดภัยเป็นสิ่งสำคัญเมื่อเปิดเผยเซิร์ฟเวอร์ MCP ผ่าน HTTP Streamable HTTP สร้างพื้นผิวช่องโหว่ใหม่ ๆ และต้องการการตั้งค่าที่ระมัดระวัง

นี่คือข้อควรพิจารณาด้านความปลอดภัยหลัก ๆ:

- **การตรวจสอบหัวข้อ Origin**: ควรตรวจสอบหัวข้อ `Origin` เสมอเพื่อป้องกันการโจมตี DNS rebinding
- **การผูกกับ localhost**: สำหรับการพัฒนาในเครื่อง ควรผูกเซิร์ฟเวอร์กับ `localhost` เพื่อไม่ให้เปิดเผยต่ออินเทอร์เน็ตสาธารณะ
- **การพิสูจน์ตัวตน**: ใช้มาตรการพิสูจน์ตัวตน (เช่น API keys, OAuth) สำหรับการใช้งานจริง
- **CORS**: ตั้งค่านโยบาย Cross-Origin Resource Sharing (CORS) เพื่อจำกัดการเข้าถึง
- **HTTPS**: ใช้ HTTPS ในการผลิตเพื่อเข้ารหัสข้อมูล

### แนวทางปฏิบัติที่ดีที่สุด

นอกจากนี้ นี่คือแนวทางปฏิบัติที่ดีที่สุดที่ควรทำเมื่อดำเนินการเรื่องความปลอดภัยในเซิร์ฟเวอร์สตรีม MCP ของคุณ:

- อย่าเชื่อถือคำขอที่เข้ามาโดยไม่มีการตรวจสอบ
- บันทึกและตรวจสอบการเข้าถึงและข้อผิดพลาดทั้งหมด
- อัปเดต dependencies อย่างสม่ำเสมอเพื่อแพตช์ช่องโหว่ด้านความปลอดภัย

### ความท้าทาย

คุณจะเผชิญกับความท้าทายบางประการเมื่อดำเนินการด้านความปลอดภัยในเซิร์ฟเวอร์สตรีม MCP:

- สร้างสมดุลระหว่างความปลอดภัยกับความง่ายในการพัฒนา
- รับรองความเข้ากันได้กับสภาพแวดล้อมไคลเอ็นต์ต่าง ๆ

### งานที่ได้รับมอบหมาย: สร้างแอป MCP แบบสตรีมของตัวเอง

**สถานการณ์:**
สร้างเซิร์ฟเวอร์และไคลเอ็นต์ MCP โดยที่เซิร์ฟเวอร์ประมวลผลรายการของไอเท็ม (เช่น ไฟล์หรือเอกสาร) และส่งการแจ้งเตือนสำหรับแต่ละไอเท็มที่ประมวลผล ไคลเอ็นต์ควรแสดงการแจ้งเตือนแต่ละรายการที่มาถึง

**ขั้นตอน:**

1. พัฒนาเครื่องมือเซิร์ฟเวอร์ที่ประมวลผลรายการและส่งการแจ้งเตือนไปยังแต่ละไอเท็ม
2. พัฒนาไคลเอ็นต์ที่มีตัวจัดการข้อความเพื่อแสดงการแจ้งเตือนแบบเรียลไทม์
3. ทดสอบการใช้งานของคุณโดยรันทั้งเซิร์ฟเวอร์และไคลเอ็นต์แล้วสังเกตการแจ้งเตือน
