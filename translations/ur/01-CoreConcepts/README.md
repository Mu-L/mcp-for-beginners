# MCP کور تصورات: AI انضمام کے لئے ماڈل کانٹیکسٹ پروٹوکول میں مہارت حاصل کرنا

[![MCP Core Concepts](../../../translated_images/ur/02.8203e26c6fb5a797.webp)](https://youtu.be/earDzWGtE84)

_(اس سبق کی ویڈیو دیکھنے کے لیے اوپر تصویر پر کلک کریں)_

[Model Context Protocol (MCP)](https://github.com/modelcontextprotocol) ایک طاقتور، معیاری فریم ورک ہے جو بڑے زبان ماڈلز (LLMs) اور بیرونی آلات، ایپلیکیشنز، اور ڈیٹا ذرائع کے درمیان مواصلات کو بہتر بناتا ہے۔  
یہ گائیڈ آپ کو MCP کے بنیادی تصورات سے روشناس کرائے گا۔ آپ اس کی کلائنٹ-سرور آرکیٹیکچر، اہم اجزاء، مواصلاتی طریقہ کار، اور نفاذ کی بہترین مشقوں کے بارے میں سیکھیں گے۔

- **واضح صارف کی رضا مندی**: تمام ڈیٹا رسائی اور عمل درآمد کے لیے صارف کی واضح منظوری ضروری ہے۔ صارفین کو بخوبی سمجھنا چاہیے کہ کون سا ڈیٹا استعمال ہوگا اور کون سے اقدامات کیے جائیں گے، اور اجازتوں اور اختیارات پر باریک بینی سے کنٹرول ہونا چاہیے۔

- **ڈیٹا کی رازداری کا تحفظ**: صارف کا ڈیٹا صرف واضح رضامندی سے ظاہر کیا جاتا ہے اور پورے تعامل کے دوران سخت رسائی کنٹرولز سے محفوظ رہنا چاہیے۔ نفاذات غیر مجاز ڈیٹا کی منتقلی کو روکنا اور سخت پرائیویسی کی حد بندی قائم رکھنا لازمی ہے۔

- **آلات کے عمل درآمد کی حفاظت**: ہر آلے کی کال کے لیے صارف کی واضح رضامندی درکار ہے جس میں آلے کی فعالیت، پیرا میٹرز، اور ممکنہ اثرات کی مکمل سمجھ ہو۔ مضبوط سیکیورٹی کی حد بندی غیر ارادی، غیر محفوظ، یا نقصان دہ آلات کے استعمال کو روکنی چاہیے۔

- **ٹرانسپورٹ لیئر سیکیورٹی**: تمام مواصلاتی چینلز میں مناسب انکرپشن اور تصدیقی طریقہ کار استعمال ہونے چاہئیں۔ ریموٹ کنکشنز میں محفوظ ٹرانسپورٹ پروٹوکولز اور درست اسناد کے انتظام کا نفاذ ہونا چاہیے۔

#### نفاذ کے رہنما اصول:

- **اجازت کا انتظام**: باریک بینی سے اجازتوں کے نظام نافذ کریں جو صارفین کو کنٹرول کی اجازت دیں کہ کون سے سرورز، آلات، اور وسائل دستیاب ہیں  
- **تصدیق اور اجازت**: محفوظ تصدیقی طریقے (OAuth، API کیز) استعمال کریں جن میں مناسب ٹوکن مینجمنٹ اور میعاد ختم ہونے کا انتظام ہو  
- **ان پٹ کی تصدیق**: تمام پیرا میٹرز اور ڈیٹا ان پٹس کو متعین اسکیموں کے مطابق جانچیں تاکہ انجیکشن حملوں سے بچا جا سکے  
- **آڈٹ لاگنگ**: تمام عملیات کے جامع لاگز رکھیں تاکہ سیکیورٹی مانیٹرنگ اور تعمیل ممکن ہو

## جائزہ

یہ سبق ماڈل کانٹیکسٹ پروٹوکول (MCP) کے ماحولیاتی نظام کی بنیادی آرکیٹیکچر اور عناصر کا جائزہ لیتا ہے۔ آپ کو کلائنٹ-سرور ساخت، اہم اجزاء، اور MCP تعاملات کو چلانے والے مواصلاتی طریقوں کے بارے میں سیکھنے کو ملے گا۔

## اہم تعلیمی مقاصد

اس سبق کے اختتام تک، آپ:  

- MCP کے کلائنٹ-سرور آرکیٹیکچر کو سمجھیں گے۔  
- ہوسٹ، کلائنٹ، اور سرور کے کردار اور ذمہ داریاں شناخت کریں گے۔  
- MCP کو ایک لچکدار انضمامی پرت بنانے والی بنیادی خصوصیات کا تجزیہ کریں گے۔  
- MCP ماحولیاتی نظام میں معلومات کے بہاؤ کو سمجھیں گے۔  
- .NET، جاوا، پائتھن، اور جاوا اسکرپٹ میں کوڈ کی مثالوں کے ذریعے عملی بصیرت حاصل کریں گے۔

## MCP آرکیٹیکچر: ایک گہرا جائزہ

MCP ماحولیاتی نظام ایک کلائنٹ-سرور ماڈل پر مبنی ہے۔ یہ ماڈیولر ساخت AI ایپلیکیشنز کو آلات، ڈیٹا بیس، APIs، اور کانٹیکسچول وسائل کے ساتھ مؤثر انداز میں بات چیت کرنے کی اجازت دیتی ہے۔ آئیے اس آرکیٹیکچر کو اس کے بنیادی اجزاء میں تفصیل سے دیکھتے ہیں۔

اپنے بنیادی طور پر، MCP ایک کلائنٹ-سرور آرکیٹیکچر کی پیروی کرتا ہے جہاں ایک ہوسٹ ایپلیکیشن متعدد سرورز سے کنیکٹ ہو سکتی ہے:

```mermaid
flowchart LR
    subgraph "آپ کا کمپیوٹر"
        Host["ایم سی پی کے ساتھ میزبان (ویژول اسٹوڈیو، وی ایس کوڈ، آئی ڈی ایز، ٹولز)"]
        S1["ایم سی پی سرور اے"]
        S2["ایم سی پی سرور بی"]
        S3["ایم سی پی سرور سی"]
        Host <-->|"ایم سی پی پروٹوکول"| S1
        Host <-->|"ایم سی پی پروٹوکول"| S2
        Host <-->|"ایم سی پی پروٹوکول"| S3
        S1 <--> D1[("مقامی\ڈیٹا ماخذ اے")]
        S2 <--> D2[("مقامی\ڈیٹا ماخذ بی")]
    end
    subgraph "انٹرنیٹ"
        S3 <-->|"ویب اے پی آئیز"| D3[("ریموٹ\سروسز")]
    end
```
- **MCP ہوسٹس**: پروگرام جیسے VSCode، Claude Desktop، IDEs، یا AI آلات جو MCP کے ذریعے ڈیٹا تک رسائی چاہتے ہیں  
- **MCP کلائنٹس**: پروٹوکول کلائنٹس جو سرورز کے ساتھ 1:1 کنکشن برقرار رکھتے ہیں  
- **MCP سرورز**: ہلکے پھلکے پروگرام جو ہر ایک معیاری Model Context Protocol کے ذریعہ مخصوص صلاحیتیں فراہم کرتے ہیں  
- **مقامی ڈیٹا ذرائع**: آپ کے کمپیوٹر کی فائلیں، ڈیٹا بیس، اور سروسز جن تک MCP سرورز محفوظ طریقے سے رسائی حاصل کر سکتے ہیں  
- **ریموٹ سروسز**: بیرونی نظام جو انٹرنیٹ پر دستیاب ہیں اور MCP سرورز APIs کے ذریعے ان سے جڑ سکتے ہیں۔

MCP پروٹوکول ایک ارتقاء پذیر معیار ہے جو تاریخ کی بنیاد پر ورژننگ (YYYY-MM-DD فارمیٹ) استعمال کرتا ہے۔ موجودہ پروٹوکول ورژن **2025-11-25** ہے۔ آپ تازہ ترین اپڈیٹس [پروٹوکول وضاحت](https://modelcontextprotocol.io/specification/2025-11-25/) پر دیکھ سکتے ہیں۔

### 1. ہوسٹ

Model Context Protocol (MCP) میں، **ہوسٹ** AI ایپلیکیشنز ہیں جو صارفین کے لیے پروٹوکول سے تعامل کا مرکزی انٹرفیس فراہم کرتی ہیں۔ ہوسٹ کئی MCP سرورز سے رابطے کا انتظام اور ہم آہنگی کرتی ہے، ہر سرور کنکشن کے لیے مخصوص MCP کلائنٹس تخلیق کر کے۔ ہوسٹ کی مثالوں میں شامل ہیں:

- **AI ایپلیکیشنز**: Claude Desktop, Visual Studio Code, Claude Code  
- **ڈیولپمنٹ ماحول**: IDEs اور کوڈ ایڈیٹرز جو MCP انضمام رکھتے ہیں  
- **حسب ضرورت ایپلیکیشنز**: مقصد کے لیے مخصوص AI ایجنٹس اور آلات

**ہوسٹ** وہ ایپلیکیشنز ہیں جو AI ماڈلز کے تعاملات کی ہم آہنگی کرتی ہیں۔ وہ:

- **AI ماڈلز کو منظم کرتی ہیں**: LLMs کو چلانا یا ان سے تعامل کرنا تاکہ جوابات تیار کیے جا سکیں اور AI ورک فلو کو مربوط کیا جا سکے  
- **کلائنٹ کنکشن کا انتظام کرتی ہیں**: ہر MCP سرور کنکشن کے لیے ایک کلائنٹ بناتی اور برقرار رکھتی ہیں  
- **صارف انٹرفیس کا کنٹرول کرتی ہیں**: بات چیت کے بہاؤ، صارف تعاملات، اور جوابات کی پیش کش کو سنبھالتی ہیں  
- **سیکیورٹی نافذ کرتی ہیں**: اجازتوں، سیکیورٹی پابندیوں، اور تصدیق کو کنٹرول کرتی ہیں  
- **صارف کی رضامندی کو سنبھالتی ہیں**: ڈیٹا شیئرنگ اور آلے کے عملدرآمد کے لیے صارف کی منظوری کی ذمہ داری لیتی ہیں

### 2. کلائنٹس

**کلائنٹس** ایسے اہم اجزاء ہیں جو ہوسٹ اور MCP سرورز کے درمیان مخصوص ایک سے ایک کنکشن برقرار رکھتے ہیں۔ ہر MCP کلائنٹ ہوسٹ کی جانب سے ایک مخصوص MCP سرور سے جڑتا ہے، تاکہ منظم اور محفوظ مواصلاتی چینلز ممکن ہو سکیں۔ متعدد کلائنٹس ہوسٹ کو بیک وقت کئی سرورز سے جڑنے کا اہل بناتے ہیں۔

**کلائنٹس** ہوسٹ ایپلیکیشن کے اندر کنیکٹر اجزاء ہیں۔ وہ:

- **پروٹوکول مواصلات**: JSON-RPC 2.0 درخواستیں سرورز کو بھیجتے ہیں جس میں پرامپٹس اور ہدایات شامل ہوتی ہیں  
- **صلاحیتوں پر گفت و شنید**: ابتداء میں سرورز کے ساتھ معاون فیچرز اور پروٹوکول ورژنز پر گفتگو کرتے ہیں  
- **آلے کے عملدرآمد کا انتظام**: ماڈلز کی جانب سے آلہ کے استعمال کی درخواستیں سنبھالتے اور جوابات کا عمل کرتے ہیں  
- **حالیہ اپ ڈیٹس**: سرورز سے اطلاعات اور حقیقی وقت کے اپ ڈیٹس کو ہینڈل کرتے ہیں  
- **جواب کی پروسیسنگ**: سرور جوابات کو صارفین کے لیے دکھانے کے لیے مربوط اور فارمیٹ کرتے ہیں  

### 3. سرورز

**سرورز** پروگرامز ہوتے ہیں جو MCP کلائنٹس کو کانٹیکسٹ، آلات، اور صلاحیتیں فراہم کرتے ہیں۔ یہ لوکل سطح پر (ہوسٹ کی مشین پر) یا ریموٹ (بیرونی پلیٹ فارمز پر) چل سکتے ہیں، اور کلائنٹ کی درخواستوں کو وصول کرنے اور ساختی جوابات فراہم کرنے کے ذمہ دار ہوتے ہیں۔ سرور معیاری Model Context Protocol کے تحت مخصوص فعالیت ظاہر کرتے ہیں۔

**سرورز** خدمات ہیں جو کانٹیکسٹ اور صلاحیتیں مہیا کرتی ہیں۔ وہ:

- **خصوصیات کی رجسٹریشن**: کلائنٹس کو دستیاب بنیادی عناصر (وسائل، پرامپٹس، آلات) مہیا کرتے ہیں  
- **درخواستوں کا عمل**: کلائنٹس کی جانب سے آلے کی کالز، وسائل کی درخواستیں، اور پرامپٹ کی درخواستیں وصول اور عمل کرتے ہیں  
- **کانٹیکسچول معلومات مہیا کرنا**: ماڈل کی جوابات کو بہتر بنانے کے لیے متعلقہ معلومات اور ڈیٹا فراہم کرتے ہیں  
- **حالت کا انتظام**: سیشن کی حالت برقرار رکھتے ہیں اور ضرورت پڑنے پر حالت پر مبنی تعاملات کو سنبھالتے ہیں  
- **حالیہ اطلاعات**: صلاحیتوں میں تبدیلیوں اور اپ ڈیٹس کے بارے میں منسلک کلائنٹس کو اطلاعات بھیجتے ہیں

سرورز کو کوئی بھی مخصوص فعالیت کے ساتھ ماڈل کی صلاحیتوں کو بڑھانے کے لیے تیار کر سکتا ہے، اور یہ مقامی و ریموٹ دونوں طرح کے نفاذ کی حمایت کرتے ہیں۔

### 4. سرور بنیادیات

Model Context Protocol (MCP) میں سرورز تین بنیادی **بنیادیات** فراہم کرتے ہیں جو کلائنٹس، ہوسٹس، اور زبان ماڈلز کے درمیان بھرپور تعاملات کے لیے بنیادی بلڈنگ بلاک کی وضاحت کرتے ہیں۔ یہ بنیادیات پروٹوکول کے ذریعے دستیاب کانٹیکسچول معلومات اور اقدامات کی اقسام کی وضاحت کرتی ہیں۔

MCP سرورز درج ذیل تین بنیادیات کے کسی بھی امتزاج کو ظاہر کر سکتے ہیں:

#### وسائل

**وسائل** وہ ڈیٹا ذرائع ہیں جو AI ایپلیکیشنز کو کانٹیکسچول معلومات فراہم کرتے ہیں۔ یہ جامد یا متحرک مواد کی نمائندگی کرتے ہیں جو ماڈل کی سمجھ اور فیصلہ سازی کو بہتر بناتے ہیں:

- **کانٹیکسچول ڈیٹا**: AI ماڈل کے استعمال کے لیے ساختی معلومات اور سیاق و سباق  
- **علمی بنیادیں**: دستاویزات کے ذخیرے، مضامین، دستی کتابیں، اور تحقیقی مقالے  
- **مقامی ڈیٹا ذرائع**: فائلیں، ڈیٹا بیس، اور لوکل سسٹم کی معلومات  
- **بیرونی ڈیٹا**: API جوابات، ویب سروسز، اور ریموٹ نظام کے ڈیٹا  
- **متحرک مواد**: حقیقی وقت کا ڈیٹا جو بیرونی حالات کے مطابق اپ ڈیٹ ہوتا ہے

وسائل URIs سے شناخت کیے جاتے ہیں اور `resources/list` کے ذریعے دریافت اور `resources/read` کے ذریعے حاصل کیے جا سکتے ہیں:

```text
file://documents/project-spec.md
database://production/users/schema
api://weather/current
```

#### پرامپٹس

**پرامپٹس** قابل استعمال ٹیمپلیٹس ہوتے ہیں جو زبان ماڈلز کے ساتھ تعامل کو منظم کرنے میں مدد دیتے ہیں۔ یہ معیاری تعامل کے نمونوں اور ٹیمپلیٹڈ ورک فلو فراہم کرتے ہیں:

- **ٹیمپلیٹ بیسڈ تعاملات**: پہلے سے ترتیب دی گئی پیغامات اور گفتگو کے آغاز  
- **ورک فلو ٹیمپلیٹس**: عام کاموں اور تعاملات کے معیاری سلسلے  
- **مثال کی بنیاد پر ٹیمپلیٹس**: ماڈل رہنمائی کے لیے مثال پر مبنی ٹیمپلیٹس  
- **سسٹم پرامپٹس**: بنیادی پرامپٹس جو ماڈل کے رویے اور سیاق و سباق کی تعریف کرتے ہیں  
- **متحرک ٹیمپلیٹس**: پیرامیٹرائزڈ پرامپٹس جو مخصوص سیاق و سباق کے مطابق ڈھلتے ہیں

پرامپٹس متغیر کے تبادلے کی اجازت دیتے ہیں اور `prompts/list` کے ذریعے دریافت اور `prompts/get` سے حاصل کیے جا سکتے ہیں:

```markdown
Generate a {{task_type}} for {{product}} targeting {{audience}} with the following requirements: {{requirements}}
```

#### آلات

**آلات** قابل عمل فنکشن ہوتے ہیں جنہیں AI ماڈلز مخصوص کارروائیاں کرنے کے لیے کال کر سکتے ہیں۔ یہ MCP ماحولیاتی نظام کے "افعال" کی نمائندگی کرتے ہیں، جو ماڈلز کو بیرونی نظاموں کے ساتھ تعامل کی سہولت دیتے ہیں:

- **قابل عمل فنکشنز**: منفرد عملیات جو ماڈلز مخصوص پیرا میٹرز کے ساتھ کال کر سکتے ہیں  
- **بیرونی نظام کے انضمام**: API کالز، ڈیٹا بیس پوچھ گچھ، فائلز کے آپریشنز، حسابات  
- **منفرد شناخت**: ہر آلے کا ایک خاص نام، تفصیل، اور پیرامیٹر اسکیمہ ہوتا ہے  
- **ساختی ان پٹ/آؤٹ پٹ**: آلات تصدیق شدہ پیرا میٹرز قبول کرتے ہیں اور ساختی، ٹائپڈ جوابات لوٹاتے ہیں  
- **عمل کی صلاحیتیں**: ماڈلز کو حقیقی دنیا کے اعمال انجام دینے اور لائیو ڈیٹا حاصل کرنے کی اجازت دیتے ہیں

آلات کو پیرامیٹر کی تصدیق کے لیے JSON اسکیمہ کے ساتھ بیان کیا جاتا ہے اور `tools/list` کے ذریعے دریافت کیا جاتا ہے اور `tools/call` کے ذریعے چلائے جاتے ہیں۔ آلات بہتر UI پیش کش کے لیے اضافی میٹا ڈیٹا کے طور پر **آئیکنز** بھی شامل کر سکتے ہیں۔

**آلہ نوٹیشنز**: آلات رویے کی وضاحت کے لیے نوٹیشنز (مثلاً `readOnlyHint`, `destructiveHint`) کی حمایت کرتے ہیں جو بتاتے ہیں کہ آیا آلہ صرف پڑھنے والا ہے یا نقصان دہ، تاکہ کلائنٹس کو آلے کی کارکردگی کے بارے میں باخبر فیصلے کرنے میں مدد ملے۔

آلہ کی مثال کی تعریف:

```typescript
server.tool(
  "search_products", 
  {
    query: z.string().describe("Search query for products"),
    category: z.string().optional().describe("Product category filter"),
    max_results: z.number().default(10).describe("Maximum results to return")
  }, 
  async (params) => {
    // تلاش کو انجام دیں اور منظم نتائج واپس کریں
    return await productService.search(params);
  }
);
```

## کلائنٹ بنیادیات

Model Context Protocol (MCP) میں، **کلائنٹس** وہ بنیادیات ظاہر کر سکتے ہیں جو سرورز کو ہوسٹ ایپلیکیشن سے اضافی صلاحیتیں مانگنے کی اجازت دیتی ہیں۔ یہ کلائنٹ-پہلو بنیادیات زیادہ بھرپور، زیادہ تعاملی سرور نفاذ کی اجازت دیتی ہیں جو AI ماڈل کی صلاحیتوں اور صارفین کے تعاملات تک رسائی حاصل کر سکتے ہیں۔

### سیمپلنگ

**سیمپلنگ** سرورز کو کلائنٹ کی AI ایپلیکیشن سے زبان ماڈل کی تکمیل کی درخواست کرنے کی اجازت دیتا ہے۔ یہ بنیادیات سرورز کو اپنی ماڈل انحصار کے بغیر LLM صلاحیتوں تک رسائی ممکن بناتی ہے:

- **ماڈل سے آزاد رسائی**: سرورز مکمل کرنے کی درخواستیں کر سکتے ہیں بغیر LLM SDKs شامل کیے یا ماڈل رسائی کا انتظام کیے  
- **سرور کی جانب سے AI انیشی ایشن**: سرورز کلائنٹ کے AI ماڈل کا استعمال کرتے ہوئے خود مختار طور پر مواد تخلیق کر سکتے ہیں  
- **ریکرسیو LLM تعاملات**: پیچیدہ حالات کی حمایت جہاں سرورز کو پروسیسنگ کے لیے AI مدد چاہیے ہوتی ہے  
- **متحرک مواد کی تخلیق**: ہوسٹ کے ماڈل کا استعمال کرتے ہوئے کانٹیکسچول جوابات بنانا  
- **آلے کی کالنگ کی حمایت**: سرورز `tools` اور `toolChoice` پیرامیٹرز شامل کر سکتے ہیں تاکہ کلائنٹ کا ماڈل سیمپلنگ کے دوران آلات کو کال کر سکے

سیمپلنگ `sampling/complete` طریقہ کار کے ذریعے شروع ہوتی ہے، جہاں سرورز مکمل کرنے کی درخواستیں کلائنٹس کو بھیجتے ہیں۔

### روٹس

**روٹس** سرورز کو فائل سسٹم کی حدود ظاہر کرنے کا معیاری طریقہ فراہم کرتے ہیں تاکہ سرورز سمجھ سکیں کہ کون سی ڈائریکٹریز اور فائلز تک ان کی رسائی ہے:

- **فائل سسٹم کی حدود**: وہ حدود متعین کرنا جہاں سرورز کام کر سکتے ہیں  
- **رسائی کنٹرول**: سرورز کو یہ سمجھنے میں مدد دینا کہ انہیں کن ڈائریکٹریز اور فائلز تک اجازت حاصل ہے  
- **متحرک اپ ڈیٹس**: کلائنٹس سرورز کو روٹس کی فہرست میں تبدیلیوں کے بارے میں مطلع کر سکتے ہیں  
- **URI کی بنیاد پر شناخت**: روٹس `file://` URIs استعمال کرتے ہوئے قابل رسائی ڈائریکٹریز اور فائلز کی شناخت کرتے ہیں

روٹس کو `roots/list` کے ذریعے دریافت کیا جاتا ہے، اور کلائنٹس `notifications/roots/list_changed` بھیج کر روٹس کی تبدیلی اطلاع دیتے ہیں۔

### ایلیسیٹیشن

**ایلیسیٹیشن** سرورز کو کلائنٹ انٹرفیس کے ذریعے صارفین سے اضافی معلومات یا تصدیق طلب کرنے کی اجازت دیتا ہے:

- **صارف ان پٹ کی درخواستیں**: ضرورت کے وقت آلے کے عمل کے لیے اضافی معلومات مانگنا  
- **تصدیقی ڈائیلاگز**: حساس یا اہم عملیات کے لیے صارف کی منظوری طلب کرنا  
- **تعاملی ورک فلو**: صارف کے ساتھ قدم بہ قدم تعاملات بنانا  
- **متحرک پیرامیٹر جمع کرنا**: آلے کے عمل میں گمشدہ یا اختیاری پیرامیٹرز جمع کرنا

ایلیسیٹیشن کی درخواستیں `elicitation/request` طریقہ کے ذریعے صارف کے انٹرفیس پر جمع کی جاتی ہیں۔

**URL موڈ ایلیسیٹیشن**: سرورز URL کی بنیاد پر صارف کے تعاملات کی بھی درخواست کر سکتے ہیں، جو صارفین کو بیرونی ویب صفحات کی طرف رجوع کرتے ہیں جیسے توثیق، تصدیق، یا ڈیٹا انٹری کے لیے۔

### لاگنگ

**لاگنگ** سرورز کو کلائنٹس کو ساختی لاگ پیغامات بھیجنے کی اجازت دیتی ہے تاکہ ڈیبگنگ، نگرانی، اور آپریشنل مرئیت ممکن ہو:

- **ڈیبگنگ کی حمایت**: سرورز تفصیلی عملدرآمد لاگز فراہم کر کے مسائل کی تشخیص ممکن بناتے ہیں  
- **آپریشنل نگرانی**: کلائنٹس کو اسٹیٹس اپ ڈیٹس اور کارکردگی کے میٹرکس بھیجنا  
- **خرابی رپورٹنگ**: تفصیلی خرابی کی سیاق و سباق اور تشخیصی معلومات مہیا کرنا  
- **آڈٹ ٹریلز**: سرور کے آپریشنز اور فیصلوں کے جامع لاگز تیار کرنا

لاگنگ پیغامات سرور آپریشنز میں شفافیت فراہم کرنے اور ڈیبگنگ کو آسان بنانے کے لیے کلائنٹس کو بھیجے جاتے ہیں۔

## MCP میں معلومات کا بہاؤ

Model Context Protocol (MCP) میزبان، کلائنٹس، سرورز، اور ماڈلز کے درمیان معلومات کے ساختی بہاؤ کی وضاحت کرتا ہے۔ اس بہاؤ کو سمجھنے سے یہ واضح ہوتا ہے کہ صارف کی درخواستیں کیسے پروسیس ہوتی ہیں اور ماڈل جوابوں میں بیرونی آلات اور ڈیٹا کس طرح شامل کیے جاتے ہیں۔
- **میزبان کنکشن شروع کرتا ہے**  
  میزبان ایپلیکیشن (جیسے IDE یا چیٹ انٹرفیس) MCP سرور سے کنکشن قائم کرتی ہے، عام طور پر STDIO، WebSocket، یا کوئی دوسرا قابلِ قبول ٹرانسپورٹ کے ذریعے۔

- **صلاحیت کی مذاکرات**  
  کلائنٹ (جو میزبان میں شامل ہوتا ہے) اور سرور ایک دوسرے کے سپورٹ کردہ فیچرز، آلات، وسائل، اور پروٹوکول ورژنز کی معلومات کا تبادلہ کرتے ہیں۔ اس سے یہ یقینی بنتا ہے کہ دونوں طرف سمجھتے ہیں کہ سیشن کے لیے کون سی صلاحیتیں دستیاب ہیں۔

- **صارف کی درخواست**  
  صارف میزبان کے ساتھ بات چیت کرتا ہے (مثلاً پرامپٹ یا کمانڈ داخل کرتا ہے)۔ میزبان یہ ان پٹ جمع کرتا ہے اور پراسیسنگ کے لیے کلائنٹ کو بھیجتا ہے۔

- **وسائل یا آلات کا استعمال**  
  - کلائنٹ سرور سے اضافی سیاق و سباق یا وسائل (جیسے فائلیں، ڈیٹا بیس کے اندراجات، یا علم کی بنیاد کے مضامین) طلب کر سکتا ہے تاکہ ماڈل کی سمجھ بوجھ کو فروغ دیا جا سکے۔  
  - اگر ماڈل سمجھتا ہے کہ کوئی آلہ استعمال کرنا ضروری ہے (مثلاً ڈیٹا حاصل کرنا، حساب لگانا، یا API کال کرنا)، تو کلائنٹ سرور کو آلے کی درخواست بھیجتا ہے، جس میں آلے کا نام اور پارامیٹرز شامل ہوتے ہیں۔

- **سرور کا عملدرآمد**  
  سرور وسائل یا آلے کی درخواست وصول کرتا ہے، ضروری کارروائیاں انجام دیتا ہے (جیسے فنکشن چلانا، ڈیٹا بیس سے استفسار کرنا، یا فائل بازیافت کرنا) اور نتائج ساختہ فارم میں کلائنٹ کو بھیجتا ہے۔

- **ردعمل کی تشکیل**  
  کلائنٹ سرور کے ردعمل (وسائل کے ڈیٹا، آلے کے آؤٹ پٹ وغیرہ) کو جاری ماڈل کی تعامل میں شامل کرتا ہے۔ ماڈل اس معلومات کو استعمال کر کے جامع اور سیاق و سباق کے لحاظ سے مناسب جواب تیار کرتا ہے۔

- **نتائج کی پیش کش**  
  میزبان کلائنٹ سے حتمی آؤٹ پٹ وصول کرتا ہے اور اسے صارف کو دکھاتا ہے، عموماً اس میں ماڈل کی تخلیق کردہ متن اور آلے کے عمل یا وسائل کی تلاش کے نتائج شامل ہوتے ہیں۔

یہ عمل MCP کو جدید، انٹرایکٹو، اور سیاق و سباق سے آگاہ AI ایپلیکیشنز کی حمایت کرنے کے قابل بناتا ہے، ماڈلز کو بیرونی آلات اور ڈیٹا ذرائع کے ساتھ بلا تعطل جوڑتا ہے۔

## پروٹوکول کی ساخت اور تہہ دار

MCP دو مختلف معماری تہوں پر مشتمل ہے جو مکمل مواصلاتی فریم ورک فراہم کرنے کے لیے ایک ساتھ کام کرتی ہیں:

### ڈیٹا کی تہہ

**ڈیٹا کی تہہ** MCP پروٹوکول کے بنیادی اصولوں کو **JSON-RPC 2.0** کے ذریعے نافذ کرتی ہے۔ یہ تہہ پیغام کی ساخت، معنی، اور تعامل کے نمونوں کی تعریف کرتی ہے:

#### بنیادی اجزاء:

- **JSON-RPC 2.0 پروٹوکول**: تمام مواصلات معیاری JSON-RPC 2.0 پیغام فارمیٹ استعمال کرتے ہیں جس میں طریقہ کالز، جوابات، اور نوٹیفیکیشنز شامل ہیں  
- **لائف سائیکل مینجمنٹ**: کلائنٹس اور سرورز کے درمیان کنکشن کا آغاز، صلاحیت کی مذاکرات، اور سیشن کا اختتام سنبھالتا ہے  
- **سرور بنیادیات**: سرورز کو آلات، وسائل، اور پرامپٹ کے ذریعے بنیادی فعالیت مہیا کرنے کے قابل بناتا ہے  
- **کلائنٹ بنیادیات**: سرورز کو LLMs سے سیمپل لینے، صارف کا ان پٹ حاصل کرنے، اور لاگ پیغام بھیجنے کے اہل بناتا ہے  
- **ریئل ٹائم نوٹیفیکیشنز**: پولنگ کے بغیر متحرک اپ ڈیٹ کے لیے اسینکرونس نوٹیفیکیشنز کی حمایت کرتا ہے  

#### کلیدی خصوصیات:

- **پروٹوکول ورژن مذاکرات**: مطابقت کو یقینی بنانے کے لیے تاریخ پر مبنی ورژنز (YYYY-MM-DD) استعمال کرتا ہے  
- **صلاحیت کی دریافت**: شروعات کے دوران کلائنٹس اور سرورز سپورٹ کردہ فیچرز کی معلومات کا تبادلہ کرتے ہیں  
- **ریاستی سیشنز**: متعدد تعاملات کے دوران کنکشن کی حالت برقرار رکھتا ہے تاکہ سیاق و سباق کی تسلسل ممکن ہو  

### ٹرانسپورٹ کی تہہ

**ٹرانسپورٹ کی تہہ** MCP کے شرکاء کے درمیان مواصلاتی چینلز، پیغام کی فریم بندی، اور توثیق کا انتظام کرتی ہے:

#### قابلِ قبول ٹرانسپورٹ میکانزم:

1. **STDIO ٹرانسپورٹ**:  
   - براہِ راست عمل کی مواصلات کے لیے معیاری ان پٹ/آؤٹ پٹ اسٹریمز استعمال کرتا ہے  
   - ایک ہی مشین پر مقامی عمل کے لیے موزوں، بغیر نیٹ ورک کے اضافی بوجھ کے  
   - عام طور پر مقامی MCP سرور کی تنصیبات میں استعمال ہوتا ہے  

2. **سٹریمیبل HTTP ٹرانسپورٹ**:  
   - کلائنٹ سے سرور کو پیغامات کے لیے HTTP POST استعمال کرتا ہے  
   - اختیاری سرور-سینٹ ایونٹس (SSE) سرور سے کلائنٹ کے لیے سٹریمنگ کے لیے  
   - نیٹ ورک کے ذریعے ریموٹ سرور سے مواصلات ممکن بناتا ہے  
   - معیاری HTTP توثیق (بیئرر ٹوکنز، API کیز، کسٹم ہیڈرز) کی حمایت کرتا ہے  
   - MCP محفوظ ٹوکن بیسڈ توثیق کے لیے OAuth کی سفارش کرتا ہے  

#### ٹرانسپورٹ کا تجریدی عمل:

ٹرانسپورٹ کی تہہ مواصلاتی تفصیلات کو ڈیٹا کی تہہ سے الگ کرتی ہے، جس سے تمام ٹرانسپورٹ میکانزمز پر ایک ہی JSON-RPC 2.0 پیغام فارمیٹ استعمال کرنا ممکن ہوتا ہے۔ یہ تجریدی عمل درخواستوں کو مقامی اور ریموٹ سرورز کے درمیان آسانی سے منتقل کرنے کی اجازت دیتا ہے۔

### سیکیورٹی کے پہلو

MCP کی تنصیبات کو کئی اہم حفاظتی اصولوں کی پابندی کرنی ہوگی تاکہ پروٹوکول کی تمام کارروائیوں کے دوران محفوظ، قابل اعتماد، اور محفوظ تعاملات کو یقینی بنایا جا سکے:

- **صارف کی رضامندی اور کنٹرول**: کسی بھی ڈیٹا تک رسائی یا کارروائیاں کرنے سے پہلے صارف کی واضح رضامندی ضروری ہے۔ انہیں یہ واضح کنٹرول حاصل ہونا چاہیے کہ کون سا ڈیٹا شیئر کیا جاتا ہے اور کون سی کارروائیاں مجاز ہیں، جس کی مدد کے لیے آسان یوزر انٹرفیس مہیا کیے جائیں تاکہ وہ سرگرمیوں کا جائزہ لے سکیں اور منظوری دے سکیں۔  

- **ڈیٹا کی رازداری**: صارف کا ڈیٹا صرف واضح رضامندی کے ساتھ ظاہر کیا جانا چاہیے اور مناسب رسائی کنٹرولز کے ذریعے محفوظ رکھا جانا چاہیے۔ MCP کی تنصیبات کو غیر مجاز ڈیٹا کی ترسیل کے خلاف تحفظ فراہم کرنا چاہیے اور تمام تعاملات کے دوران رازداری کو برقرار رکھنا چاہیے۔  

- **آلات کی حفاظت**: کسی بھی آلے کو کال کرنے سے پہلے واضح صارف کی رضامندی ضروری ہے۔ صارفین کو ہر ٹول کی فعالیت کی مکمل سمجھ ہونی چاہیے، اور مضبوط حفاظتی حدود نافذ کی جانی چاہئیں تاکہ ناخواستی یا غیر محفوظ آلہ کے عملدرآمد کو روکا جا سکے۔  

ان حفاظتی اصولوں پر عمل کر کے، MCP صارف کے اعتماد، رازداری، اور حفاظت کو پروٹوکول کی تمام بات چیت میں یقینی بناتا ہے، جبکہ طاقتور AI انضمام کو ممکن بناتا ہے۔

## کوڈ کی مثالیں: کلیدی اجزاء

نیچے چند مشہور پروگرامنگ زبانوں میں کوڈ کی مثالیں دی گئی ہیں جو MCP سرور کے کلیدی اجزاء اور آلات کو کیسے نافذ کیا جائے، اس کی وضاحت کرتی ہیں۔

### .NET مثال: سادہ MCP سرور بنانا آلات کے ساتھ

یہاں ایک عملی .NET کوڈ کی مثال ہے جو دکھاتی ہے کہ کس طرح ایک سادہ MCP سرور کو کسٹم آلات کے ساتھ نافذ کیا جا سکتا ہے۔ یہ مثال آلات کی تعریف، اندراج، درخواستوں کے ہینڈلنگ، اور ماڈل کانٹیکسٹ پروٹوکول کے ذریعے سرور کو کنیکٹ کرنا دکھاتی ہے۔

```csharp
using System;
using System.Threading.Tasks;
using ModelContextProtocol.Server;
using ModelContextProtocol.Server.Transport;
using ModelContextProtocol.Server.Tools;

public class WeatherServer
{
    public static async Task Main(string[] args)
    {
        // Create an MCP server
        var server = new McpServer(
            name: "Weather MCP Server",
            version: "1.0.0"
        );
        
        // Register our custom weather tool
        server.AddTool<string, WeatherData>("weatherTool", 
            description: "Gets current weather for a location",
            execute: async (location) => {
                // Call weather API (simplified)
                var weatherData = await GetWeatherDataAsync(location);
                return weatherData;
            });
        
        // Connect the server using stdio transport
        var transport = new StdioServerTransport();
        await server.ConnectAsync(transport);
        
        Console.WriteLine("Weather MCP Server started");
        
        // Keep the server running until process is terminated
        await Task.Delay(-1);
    }
    
    private static async Task<WeatherData> GetWeatherDataAsync(string location)
    {
        // This would normally call a weather API
        // Simplified for demonstration
        await Task.Delay(100); // Simulate API call
        return new WeatherData { 
            Temperature = 72.5,
            Conditions = "Sunny",
            Location = location
        };
    }
}

public class WeatherData
{
    public double Temperature { get; set; }
    public string Conditions { get; set; }
    public string Location { get; set; }
}
```

### جاوا مثال: MCP سرور کے اجزاء

یہ مثال اوپر دی گئی .NET مثال کی طرح MCP سرور اور آلے کی رجسٹریشن کو ظاہر کرتی ہے، لیکن جاوا میں نافذ کی گئی ہے۔

```java
import io.modelcontextprotocol.server.McpServer;
import io.modelcontextprotocol.server.McpToolDefinition;
import io.modelcontextprotocol.server.transport.StdioServerTransport;
import io.modelcontextprotocol.server.tool.ToolExecutionContext;
import io.modelcontextprotocol.server.tool.ToolResponse;

public class WeatherMcpServer {
    public static void main(String[] args) throws Exception {
        // ایک MCP سرور بنائیں
        McpServer server = McpServer.builder()
            .name("Weather MCP Server")
            .version("1.0.0")
            .build();
            
        // ایک موسمی آلہ رجسٹر کریں
        server.registerTool(McpToolDefinition.builder("weatherTool")
            .description("Gets current weather for a location")
            .parameter("location", String.class)
            .execute((ToolExecutionContext ctx) -> {
                String location = ctx.getParameter("location", String.class);
                
                // موسمی ڈیٹا حاصل کریں (سادہ کیا گیا)
                WeatherData data = getWeatherData(location);
                
                // فارمیٹ شدہ جواب واپس کریں
                return ToolResponse.content(
                    String.format("Temperature: %.1f°F, Conditions: %s, Location: %s", 
                    data.getTemperature(), 
                    data.getConditions(), 
                    data.getLocation())
                );
            })
            .build());
        
        // stdio ٹرانسپورٹ کا استعمال کرتے ہوئے سرور کو جوڑیں
        try (StdioServerTransport transport = new StdioServerTransport()) {
            server.connect(transport);
            System.out.println("Weather MCP Server started");
            // عمل ختم ہونے تک سرور کو چلتا رکھیں
            Thread.currentThread().join();
        }
    }
    
    private static WeatherData getWeatherData(String location) {
        // نفاذ موسمی API کو کال کرے گا
        // مثال کی غرض سے سادہ کیا گیا
        return new WeatherData(72.5, "Sunny", location);
    }
}

class WeatherData {
    private double temperature;
    private String conditions;
    private String location;
    
    public WeatherData(double temperature, String conditions, String location) {
        this.temperature = temperature;
        this.conditions = conditions;
        this.location = location;
    }
    
    public double getTemperature() {
        return temperature;
    }
    
    public String getConditions() {
        return conditions;
    }
    
    public String getLocation() {
        return location;
    }
}
```

### جاوا اسکرپٹ مثال: MCP سرور بنانا

یہ مثال دکھاتی ہے کہ جاوا اسکرپٹ میں MCP سرور کیسے بنایا جائے اور دو موسمی آلات کو کیسے رجسٹر کیا جائے۔

```javascript
// سرکاری ماڈل کانٹیکسٹ پروٹوکول ایس ڈی کے کا استعمال
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod"; // پیرامیٹر کی توثیق کے لیے

// ایک MCP سرور بنائیں
const server = new McpServer({
  name: "Weather MCP Server",
  version: "1.0.0"
});

// ایک موسمیاتی آلہ کی تعریف کریں
server.tool(
  "weatherTool",
  {
    location: z.string().describe("The location to get weather for")
  },
  async ({ location }) => {
    // عام طور پر یہ موسمی معلومات کے API کو کال کرے گا
    // مظاہرے کے لیے آسان بنایا گیا
    const weatherData = await getWeatherData(location);
    
    return {
      content: [
        { 
          type: "text", 
          text: `Temperature: ${weatherData.temperature}°F, Conditions: ${weatherData.conditions}, Location: ${weatherData.location}` 
        }
      ]
    };
  }
);

// ایک پیش گوئی کا آلہ کی تعریف کریں
server.tool(
  "forecastTool",
  {
    location: z.string(),
    days: z.number().default(3).describe("Number of days for forecast")
  },
  async ({ location, days }) => {
    // عام طور پر یہ موسمی معلومات کے API کو کال کرے گا
    // مظاہرے کے لیے آسان بنایا گیا
    const forecast = await getForecastData(location, days);
    
    return {
      content: [
        { 
          type: "text", 
          text: `${days}-day forecast for ${location}: ${JSON.stringify(forecast)}` 
        }
      ]
    };
  }
);

// مددگار افعال
async function getWeatherData(location) {
  // API کال کی نقل کریں
  return {
    temperature: 72.5,
    conditions: "Sunny",
    location: location
  };
}

async function getForecastData(location, days) {
  // API کال کی نقل کریں
  return Array.from({ length: days }, (_, i) => ({
    day: i + 1,
    temperature: 70 + Math.floor(Math.random() * 10),
    conditions: i % 2 === 0 ? "Sunny" : "Partly Cloudy"
  }));
}

// stdio ٹرانسپورٹ کے ذریعے سرور سے رابطہ کریں
const transport = new StdioServerTransport();
server.connect(transport).catch(console.error);

console.log("Weather MCP Server started");
```
  
یہ جاوا اسکرپٹ مثال MCP سرور بنانے کا طریقہ بتاتی ہے جو موسمی آلات کو رجسٹر کرتا ہے اور STDIO ٹرانسپورٹ استعمال کرتے ہوئے آنے والی کلائنٹ درخواستوں کو سنبھالتا ہے۔

## سیکیورٹی اور اجازت نامہ

MCP پروٹوکول میں کئی اندرونی تصورات اور میکنزم شامل ہیں جو سیکیورٹی اور اجازت نامہ کو پورے پروٹوکول میں منظم کرتے ہیں:

1. **آلہ کی اجازت کنٹرول**:  
   کلائنٹ مخصوص کر سکتے ہیں کہ ماڈل کو سیشن کے دوران کون سے آلات استعمال کرنے کی اجازت ہے۔ اس سے یقینی بنتا ہے کہ صرف واضح طور پر مجاز آلات قابل رسائی ہوں، جس سے غیر ارادی یا غیر محفوظ کارروائیوں کا خطرہ کم ہوتا ہے۔ اجازتیں صارف کی ترجیحات، تنظیمی پالیسیوں، یا تعامل کے سیاق و سباق کی بنیاد پر متحرک طور پر ترتیب دی جا سکتی ہیں۔

2. **توثیق**:  
   سرورز آلات، وسائل، یا حساس کارروائیوں تک رسائی دینے سے پہلے توثیق کا مطالبہ کر سکتے ہیں۔ اس میں API کیز، OAuth ٹوکنز، یا دیگر توثیقی طریقے شامل ہو سکتے ہیں۔ مناسب توثیق یقینی بناتی ہے کہ صرف معتبر کلائنٹس اور صارفین ہی سرور کی صلاحیتوں کو بلا سکیں۔

3. **توثیقِ معیار (ویلیڈیشن)**:  
   تمام آلہ کی کالز کے لیے پیرامیٹر ویلیڈیشن لازمی ہے۔ ہر آلہ اپنے متوقع اقسام، فارمیٹس، اور پابندیوں کی تعریف کرتا ہے، اور سرور آنے والی درخواستوں کی اس کے مطابق تصدیق کرتا ہے۔ یہ غیر معیاری یا نقصان دہ ان پٹ کو آلہ کی عملدرآمد تک پہنچنے سے روکتا ہے اور آپریشنز کی سالمیت کو برقرار رکھتا ہے۔

4. **ریٹ لمٹنگ**:  
   کنٹرول کیے بغیر زیادتی سے بچاؤ اور سرور کے وسائل کے منصفانہ استعمال کو یقینی بنانے کے لیے MCP سرورز آلہ کالز اور وسائل کی رسائی کے لیے ریٹ لمٹنگ نافذ کر سکتے ہیں۔ یہ ریٹ لمٹس ہر صارف، ہر سیشن، یا عالمی سطح پر نافذ کی جا سکتی ہیں، اور ڈینائل آف سروس حملوں یا وسائل کے حد سے زیادہ استعمال سے تحفظ فراہم کرتی ہیں۔

ان میکنزمز کو یکجا کر کے، MCP زبان کے ماڈلز کو بیرونی آلات اور ڈیٹا ذرائع کے ساتھ مربوط کرنے کے لیے محفوظ بنیاد فراہم کرتا ہے، اور صارفین و ڈویلپرز کو رسائی اور استعمال پر باریک بینی سے کنٹرول فراہم کرتا ہے۔

## پروٹوکول پیغامات اور مواصلاتی بہاؤ

MCP مواصلات منظم **JSON-RPC 2.0** پیغامات کا استعمال کرتے ہیں تاکہ میزبانوں، کلائنٹس، اور سرورز کے درمیان واضح اور معتبر تعاملات کو سہولت فراہم کی جا سکے۔ پروٹوکول مختلف قسم کی کارروائیوں کے لیے مخصوص پیغام کے نمونے تعریف کرتا ہے:

### بنیادی پیغام کی اقسام:

#### **ابتدائی پیغامات**  
- **`initialize` درخواست**: کنکشن قائم کرتا ہے اور پروٹوکول ورژن اور صلاحیتوں کی مذاکرات کرتا ہے  
- **`initialize` جواب**: سپورٹ کردہ فیچرز اور سرور کی معلومات کی تصدیق کرتا ہے  
- **`notifications/initialized`**: اشارہ کرتا ہے کہ ابتدائی مرحلہ مکمل ہو گیا ہے اور سیشن تیار ہے  

#### **دریافت کے پیغامات**  
- **`tools/list` درخواست**: سرور سے دستیاب آلات کی دریافت  
- **`resources/list` درخواست**: دستیاب وسائل (ڈیٹا ذرائع) کی فہرست  
- **`prompts/list` درخواست**: دستیاب پرامپٹ ٹیمپلیٹس حاصل کرنا  

#### **عملدرآمد کے پیغامات**  
- **`tools/call` درخواست**: مخصوص آلے کو دیے گئے پارامیٹرز کے ساتھ چلانا  
- **`resources/read` درخواست**: مخصوص وسائل سے مواد حاصل کرنا  
- **`prompts/get` درخواست**: اختیاری پارامیٹرز کے ساتھ پرامپٹ ٹیمپلیٹ لانا  

#### **کلائنٹ طرف کے پیغامات**  
- **`sampling/complete` درخواست**: سرور کلائنٹ سے LLM مکمل ہونے کی درخواست کرتا ہے  
- **`elicitation/request`**: سرور کلائنٹ انٹرفیس کے ذریعے صارف کا ان پٹ طلب کرتا ہے  
- **لاگنگ پیغامات**: سرور منظم لاگ پیغامات کلائنٹ کو بھیجتا ہے  

#### **نوٹیفیکیشن پیغامات**  
- **`notifications/tools/list_changed`**: سرور کلائنٹ کو آلے کی تبدیلیوں کی اطلاع دیتا ہے  
- **`notifications/resources/list_changed`**: سرور کلائنٹ کو وسائل کی تبدیلیوں کی اطلاع دیتا ہے  
- **`notifications/prompts/list_changed`**: سرور کلائنٹ کو پرامپٹ کی تبدیلیوں کی اطلاع دیتا ہے  

### پیغام کی ساخت:

تمام MCP پیغامات JSON-RPC 2.0 فارمیٹ پر عمل کرتے ہیں جس میں:  
- **درخواست پیغامات**: `id`, `method`, اور اختیاری `params` شامل ہوتے ہیں  
- **جواب پیغامات**: `id` اور `result` یا `error` شامل ہوتے ہیں  
- **نوٹیفیکیشن پیغامات**: `method` اور اختیاری `params` شامل ہوتے ہیں (کسی `id` یا جواب کی توقع نہیں ہوتی)  

یہ منظم مواصلات قابل اعتماد، قابلِ ٹریس، اور توسیع پذیر تعاملات کو یقینی بناتی ہے جو جدید مواقع جیسے ریئل ٹائم اپ ڈیٹس، آلہ سلسلہ بندی، اور مضبوط ایرر ہینڈلنگ کی حمایت کرتی ہے۔

### کام (تجربی)

**کام** ایک تجرباتی فیچر ہے جو پائیدار عملدرآمد ریپرز مہیا کرتا ہے، جو MCP درخواستوں کے مؤخرہ نتیجے کی بازیافت اور حالت کا پتہ لگانا ممکن بناتے ہیں:

- **طویل مدتی آپریشنز**: مہنگے حسابات، ورک فلو آٹومیشن، اور بیچ پروسیسنگ کو ٹریک کرتا ہے  
- **موخرہ نتائج**: کام کی حالت معلوم کرنے اور آپریشن مکمل ہونے پر نتائج حاصل کرنے کے لیے پولنگ کرتا ہے  
- **اسٹیٹس ٹریکنگ**: متعین زندگی کے مراحل کے ذریعے کام کی پیش رفت کی نگرانی کرتا ہے  
- **کئی مرحلے کے آپریشنز**: پیچیدہ ورک فلو کی حمایت کرتا ہے جو متعدد تعاملات پر محیط ہوتے ہیں  

کام معیاری MCP درخواستوں کو لپیٹتے ہیں تاکہ ان آپریشنز کے لیے اسینکرونس عملدرآمد کے نمونے ممکن ہوں جو فوری مکمل نہیں ہو سکتے۔

## کلیدی نکات

- **معماری**: MCP کلائنٹ-سرور معماری کا استعمال کرتا ہے جہاں میزبان متعدد کلائنٹ کنکشنز کو سرورز تک منظم کرتے ہیں  
- **شرکاء**: نظام میں میزبان (AI ایپلیکیشنز)، کلائنٹس (پروٹوکول کنیکٹرز)، اور سرورز (صلاحیت فراہم کنندگان) شامل ہیں  
- **ٹرانسپورٹ میکانزمز**: مواصلات STDIO (مقامی) اور سٹریمیبل HTTP کے ساتھ اختیاری SSE (ریموٹ) کی حمایت کرتی ہے  
- **بنیادی بنیادیات**: سرورز آلات (قابلِ عمل فنکشنز)، وسائل (ڈیٹا ذرائع)، اور پرامپٹس (ٹیمپلیٹس) پیش کرتے ہیں  
- **کلائنٹ بنیادیات**: سرورز سیمپلنگ (آلے کی کال سپورٹ کے ساتھ LLM مکمل)، ایلیسیٹیشن (صارف کا ان پٹ بشمول URL موڈ)، روٹس (فائل سسٹم کی حدود)، اور لاگنگ کلائنٹس سے طلب کر سکتے ہیں  
- **تجربی خصوصیات**: کام طویل مدتی آپریشنز کے لیے پائیدار عملدرآمد ریپرز فراہم کرتے ہیں  
- **پروٹوکول کی بنیاد**: JSON-RPC 2.0 اور تاریخ پر مبنی ورژننگ (موجودہ: 2025-11-25)  
- **ریئل ٹائم صلاحیتیں**: متحرک اپ ڈیٹس اور ریئل ٹائم ہم آہنگی کے لیے نوٹیفیکیشنز کی حمایت  
- **سیکیورٹی اولین ترجیح**: واضح صارف کی رضامندی، ڈیٹا کی رازداری کا تحفظ، اور محفوظ ٹرانسپورٹ بنیادی ضروریات ہیں  

## مشق

اپنے شعبے میں مفید ایک سادہ MCP آلہ ڈیزائن کریں۔ تعریف کریں:  
1. آلے کا نام کیا ہوگا  
2. کون سے پارامیٹرز قبول کرے گا  
3. کون سا آؤٹ پٹ فراہم کرے گا  
4. ماڈل اس آلے کو صارف کے مسائل حل کرنے کے لیے کیسے استعمال کر سکتا ہے
