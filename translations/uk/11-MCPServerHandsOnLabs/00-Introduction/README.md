# Вступ до інтеграції бази даних MCP

## 🎯 Що охоплює ця лабораторна робота

Ця вступна лабораторна робота надає комплексний огляд створення серверів Model Context Protocol (MCP) з інтеграцією бази даних. Ви зрозумієте бізнес-кейс, технічну архітектуру та реальні застосування на прикладі аналітики Zava Retail за адресою https://github.com/microsoft/MCP-Server-and-PostgreSQL-Sample-Retail.

## Огляд

**Model Context Protocol (MCP)** дозволяє AI-асистентам безпечно отримувати доступ та взаємодіяти з зовнішніми джерелами даних у режимі реального часу. У поєднанні з інтеграцією бази даних MCP відкриває потужні можливості для AI-застосунків, що працюють з даними.

Цей навчальний шлях навчає вас створювати сервери MCP, готові до виробництва, які підключають AI-асистентів до даних про роздрібні продажі через PostgreSQL, реалізуючи корпоративні патерни, такі як Row Level Security, семантичний пошук і багатокористувацький доступ до даних.

## Цілі навчання

Після завершення цієї лабораторної роботи ви зможете:

- **Визначити** Model Context Protocol та його основні переваги для інтеграції з базою даних
- **Ідентифікувати** ключові компоненти архітектури сервера MCP з базами даних
- **Зрозуміти** кейс використання Zava Retail та його бізнес-вимоги
- **Розпізнати** корпоративні патерни для безпечного та масштабованого доступу до бази даних
- **Перелічити** інструменти та технології, які використовуються протягом цього навчального шляху

## 🧭 Виклик: AI зустрічається з реальними даними

### Традиційні обмеження AI

Сучасні AI-асистенти неймовірно потужні, але мають суттєві обмеження при роботі з реальними бізнес-даними:

| **Виклик** | **Опис** | **Вплив на бізнес** |
|------------|----------|---------------------|
| **Статичні знання** | AI-моделі, навчені на фіксованих наборах даних, не можуть отримати доступ до актуальних бізнес-даних | Застаріла інформація, пропущені можливості |
| **Ізольовані дані** | Інформація заблокована в базах даних, API та системах, до яких AI не має доступу | Неповний аналіз, фрагментовані робочі процеси |
| **Обмеження безпеки** | Прямий доступ до бази даних викликає питання безпеки та відповідності | Обмежене впровадження, ручна підготовка даних |
| **Складні запити** | Бізнес-користувачам потрібні технічні знання для отримання аналітики | Зменшене впровадження, неефективні процеси |

### Рішення MCP

Model Context Protocol відповідає на ці виклики через:

- **Доступ до даних у реальному часі**: AI-асистенти роблять запити до живих баз даних і API
- **Безпечна інтеграція**: Контрольований доступ з автентифікацією та дозволами
- **Інтерфейс природною мовою**: Бізнес-користувачі задають питання звичайною англійською
- **Стандартизований протокол**: Працює в різних AI-платформах і інструментах

## 🏪 Познайомтеся з Zava Retail: Наш кейс для вивчення https://github.com/microsoft/MCP-Server-and-PostgreSQL-Sample-Retail

Протягом цього навчального шляху ми побудуємо сервер MCP для **Zava Retail**, вигаданої мережі магазинів для домашнього ремонту з кількома точками продажу. Цей реалістичний сценарій демонструє впровадження MCP корпоративного рівня.

### Бізнес-контекст

**Zava Retail** працює:

- **8 фізичних магазинів** по всьому штату Вашингтон (Сіетл, Белв’ю, Такома, Спокан, Еверетт, Редмонд, Кіркланд)
- **1 інтернет-магазин** для електронної комерції
- **Різноманітний каталог продукції** включно з інструментами, апаратними засобами, товарами для саду та будівельними матеріалами
- **Багаторівневе управління** з менеджерами магазинів, регіональними менеджерами та керівництвом

### Бізнес-вимоги

Менеджери магазинів і керівники потребують AI-аналітики, щоб:

1. **Аналізувати продажі** по магазинах і часових періодах
2. **Відслідковувати рівні запасів** і визначати потреби у поповненні
3. **Розуміти поведінку клієнтів** та їхні патерни покупок
4. **Виявляти інсайти про продукти** через семантичний пошук
5. **Генерувати звіти** за допомогою запитань природною мовою
6. **Забезпечувати безпеку даних** через контроль доступу за ролями

### Технічні вимоги

Сервер MCP повинен надавати:

- **Багатокористувацький доступ** де менеджери магазинів бачать лише дані свого магазину
- **Гнучкий запит** з підтримкою складних SQL-операцій
- **Семантичний пошук** для відкриття та рекомендацій продуктів
- **Актуальні дані** що відображають поточний бізнес-стан
- **Безпечну автентифікацію** з Row Level Security
- **Масштабовану архітектуру** для підтримки багатьох одночасних користувачів

│  │ • Query Tools   │ │ • RLS Context   │ │ • Environment │ │

### Ключові компоненти

#### **1. Шар сервера MCP**
- **FastMCP Framework**: Сучасна Python-реалізація сервера MCP
- **Реєстрація інструментів**: Декларативне визначення інструментів з типобезпекою
- **Контекст запиту**: Ідентифікація користувача і управління сесією
- **Обробка помилок**: Надійне керування помилками та логування

#### **2. Шар інтеграції з базою даних**
- **Пул підключень**: Ефективне управління підключеннями через asyncpg
- **Провайдер схеми**: Динамічне виявлення схеми таблиць
- **Виконавець запитів**: Безпечне виконання SQL з контекстом RLS
- **Управління транзакціями**: Відповідність ACID та обробка відкату

#### **3. Шар безпеки**
- **Row Level Security**: PostgreSQL RLS для ізоляції даних мультиорендарності
- **Ідентифікація користувача**: Автентифікація і авторизація менеджерів магазинів
- **Контроль доступу**: Тонкозернисті дозволи та аудит
- **Валідація введення**: Захист від SQL-ін’єкцій та перевірка запитів

#### **4. Шар покращення AI**
- **Семантичний пошук**: Векторні вбудовані представлення для відкриття продуктів
- **Інтеграція Azure OpenAI**: Генерація текстових вбудувань
- **Алгоритми подібності**: Пошук косинусної схожості через pgvector
- **Оптимізація пошуку**: Індексування та налаштування продуктивності

1. Запит обробляється інструментом семантичного пошуку

3. pgvector виконує пошук за подібністю
4. Схожі продукти ранжуються за релевантністю

## 🔒 Глибоке занурення в безпеку та мультиорендарність

Наша реалізація пріоритезує безпеку корпоративного рівня:

### Row Level Security (RLS)

PostgreSQL RLS забезпечує ізоляцію даних:

```sql
-- Store managers see only their store's data
CREATE POLICY store_manager_policy ON retail.orders
  FOR ALL TO store_managers
  USING (store_id = get_current_user_store());

-- Regional managers see multiple stores
CREATE POLICY regional_manager_policy ON retail.orders
  FOR ALL TO regional_managers
  USING (store_id = ANY(get_user_store_list()));
```

### Управління ідентифікацією користувача

Кожне підключення MCP включає:
- **ID менеджера магазину**: унікальний ідентифікатор для контексту RLS
- **Призначення ролі**: дозволи та рівні доступу
- **Управління сесією**: безпечні токени автентифікації
- **Логування аудиту**: повна історія доступу

### Захист даних

Декілька шарів безпеки:
- **Шифрування підключень**: TLS для усіх підключень до бази даних
- **Запобігання SQL-ін’єкціям**: Використання параметризованих запитів
- **Валідація введення**: Комплексна перевірка запитів
- **Обробка помилок**: Відсутність конфіденційної інформації в повідомленнях про помилки
