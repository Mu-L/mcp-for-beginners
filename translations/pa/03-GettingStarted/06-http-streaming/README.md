# HTTPS ਸਟ੍ਰੀਮਿੰਗ ਵਿਥ ਮਾਡਲ ਕਾਂਟੈਕਸਟ ਪ੍ਰੋਟੋਕਾਲ (MCP)

ਇਹ ਅਧਿਆਇ ਐਚਟੀਟੀਪੀਐਸ ਦੀ ਵਰਤੋਂ ਕਰਦਿਆਂ ਮਾਡਲ ਕਾਂਟੈਕਸਟ ਪ੍ਰੋਟੋਕਾਲ (MCP) ਨਾਲ ਸੁਰੱਖਿਅਤ, ਸਕੇਲਬਲ ਅਤੇ ਅਸਲੀ ਸਮੇਂ ਸਟ੍ਰੀਮਿੰਗ ਨੂੰ ਲਾਗੂ ਕਰਨ ਲਈ ਇੱਕ ਵਿਸ਼ਤृत ਮਾਰਗਦਰਸ਼ਨ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ। ਇਸ ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ ਦੀ ਪ੍ਰੇਰਣਾ, ਉਪਲਬਧ ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ, MCP ਵਿੱਚ ਸਟ੍ਰੀਮਏਬਲ HTTP ਲਾਗੂ ਕਰਨ ਦਾ ਤਰੀਕਾ, ਸੁਰੱਖਿਆ ਦੀਆਂ ਸਭ ਤੋਂ ਵਧੀਆਂ ਪ੍ਰਥਾਵਾਂ, SSE ਤੋਂ ਮਾਈਗ੍ਰੇਸ਼ਨ, ਅਤੇ ਆਪਣੇ ਆਪ ਦੇ ਸਟ੍ਰੀਮਿੰਗ MCP ਐਪਲੀਕੇਸ਼ਨ ਬਣਾਉਣ ਲਈ ਪ੍ਰਾਇਕਟਿਕਲ ਮਾਰਗਦਰਸ਼ਨ ਸ਼ਾਮਲ ਹਨ।

## MCP ਵਿੱਚ ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ ਅਤੇ ਸਟ੍ਰੀਮਿੰਗ

ਇਸ ਸੈਕਸ਼ਨ ਵਿੱਚ MCP ਵਿੱਚ ਉਪਲਬਧ ਵੱਖ-ਵੱਖ ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ ਦੀ ਜਾਂਚ ਕੀਤੀ ਜਾਂਦੀ ਹੈ ਅਤੇ ਇਹ ਕਿਵੇਂ ਗਾਹਕ-ਸਰਵਰ ਵਿਚਕਾਰ ਅਸਲੀ ਸਮੇਂ ਸੰਚਾਰ ਲਈ ਸਟ੍ਰੀਮਿੰਗ ਸਮਰੱਥਾਵਾਂ ਨੂੰ ਯੋਗ ਬਣਾਉਂਦੇ ਹਨ।

### ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ ਕੀ ਹੈ?

ਇੱਕ ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ ਵਿਆਖਿਆ ਕਰਦਾ ਹੈ ਕਿ ਕਿਵੇਂ ਡਾਟਾ ਗਾਹਕ ਅਤੇ ਸਰਵਰ ਵਿਚਕਾਰ ਅਦਾਨ-ਪ੍ਰਦਾਨ ਕੀਤਾ ਜਾਂਦਾ ਹੈ। MCP ਵੱਖ-ਵੱਖ ਪਾਰਿਵਾਰਕ ਵਾਤਾਵਰਨਾਂ ਅਤੇ ਜ਼ਰੂਰਤਾਂ ਲਈ ਬਹੁਤ ਸਾਰੇ ਟ੍ਰਾਂਸਪੋਰਟ ਕਿਸਮਾਂ ਨੂੰ ਸਹਾਇਤਾ ਦਿੰਦਾ ਹੈ:

- **stdio**: ਮਿਆਰੀ ਇਨਪੁੱਟ/ਆਉਟਪੁੱਟ, ਲੋਕਲ ਅਤੇ ਕਮਾਂਡ ਲਾਈਨ ਇੰਟਰਫੇਸ (CLI) ਅਧਾਰਿਤ ਟੂਲਾਂ ਲਈ ਉਚਿਤ। ਸਾਦਾ ਪਰ ਵੈੱਬ ਜਾਂ ਕੁਲਾਊਡ ਲਈ ਉਚਿਤ ਨਹੀਂ।
- **SSE (ਸਰਵਰ-ਸੈਂਟ ਇਵੈਂਟਸ)**: ਸਰਵਰ HTTP ਰਾਹੀਂ ਗਾਹਕਾਂ ਨੂੰ ਅਸਲੀ ਸਮੇ ਵਿੱਚ ਅੱਪਡੇਟ ਪ੍ਰੇਰਣਾ ਦੀ ਆਗਿਆ ਦਿੰਦਾ ਹੈ। ਵੈੱਬ UI ਲਈ ਉਚਿਤ, ਪਰ ਸਕੇਲਬਿਲਟੀ ਅਤੇ ਲਚੀਲਤਾ ਵਿੱਚ ਸੀਮਤ। MCP ਵਿਸ਼ੇਸ਼ਤਾ 2025-06-18 ਅਨੁਸਾਰ, ਸਟੈਂਡਅਲੋਨ SSE ਟ੍ਰਾਂਸਪੋਰਟ ਨੂੰ ਹਟਾ ਕੇ "ਸਟ੍ਰੀਮਏਬਲ HTTP" ਟ੍ਰਾਂਸਪੋਰਟ ਨਾਲ ਬਦਲ ਦਿੱਤਾ ਗਿਆ ਹੈ।
- **ਸਟ੍ਰੀਮਏਬਲ HTTP**: ਆਧੁਨਿਕ HTTP-ਆਧਾਰਿਤ ਸਟ੍ਰੀਮਿੰਗ ਟ੍ਰਾਂਸਪੋਰਟ, ਸੁਚੇਤਨਾਂ ਅਤੇ ਚੰਗੀ ਸਕੇਲਬਿਲਟੀ ਨੂੰ ਸਮਰਥਨ ਕਰਦਾ ਹੈ। ਜ਼ਿਆਦਾਤਰ ਪ੍ਰੋਡਕਸ਼ਨ ਅਤੇ ਕੁਲਾਊਡ ਸਿੰਨਾਰਿਓ ਲਈ ਸਿਫ਼ਾਰਸ਼ੀ।

### ਤੁਲਨਾਧਾਰਾ ਸਾਰਣੀ

ਹੇਠਾਂ ਦਿੱਤੀ ਸਾਰਣੀ ਦੇਖੋ ਤਾਂ ਜੋ ਇਹ ਸਮਝ ਸਕੋ ਕਿ ਇਹ ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ ਇੱਕ ਦੂਜੇ ਤੋਂ ਕਿਵੇਂ ਵੱਖਰੇ ਹਨ:

| ਟ੍ਰਾਂਸਪੋਰਟ           | ਅਸਲੀ ਸਮੇਂ ਅੱਪਡੇਟ | ਸਟ੍ਰੀਮਿੰਗ  | ਸਕੇਲਬਿਲਟੀ  | ਵਰਤੋਂ ਦਾ ਕੇਸ          |
|-----------------------|------------------|-----------|-------------|------------------------|
| stdio                 | ਨਹੀਂ              | ਨਹੀਂ       | ਘੱਟ        | ਲੋਕਲ CLI ਟੂਲ          |
| SSE                   | ਹਾਂ               | ਹਾਂ        | ਦਰਮਿਆਨਾ   | ਵੈੱਬ, ਅਸਲੀ ਸਮੇਂ ਅੱਪਡੇਟ  |
| ਸਟ੍ਰੀਮਏਬਲ HTTP       | ਹਾਂ               | ਹਾਂ        | ਉੱਚ        | ਕੁਲਾਊਡ, ਬਹੁ-ਗਾਹਕ      |

> **ਸੰਕੇਤ:** ਸਹੀ ਟ੍ਰਾਂਸਪੋਰਟ ਚੁਣਨਾ ਪ੍ਰਦਰਸ਼ਨ, ਸਕੇਲਬਿਲਟੀ ਅਤੇ ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਨੂੰ ਪ੍ਰਭਾਵਿਤ ਕਰਦਾ ਹੈ। **ਸਟ੍ਰੀਮਏਬਲ HTTP** ਆਧੁਨਿਕ, ਸਕੇਲਬਲ ਅਤੇ ਕੁਲਾਊਡ-ਤਿਆਰ ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ ਸਿਫ਼ਾਰਸ਼ੀ ਕੀਤਾ ਜਾਂਦਾ ਹੈ।

ਪਿਛਲੇ ਅਧਿਆਇਆਂ ਵਿੱਚ ਤੁਸੀਂ ਜੋ stdio ਅਤੇ SSE ਟ੍ਰਾਂਸਪੋਰਟ ਵੇਖੇ ਉਹ ਧਿਆਨ ਵਿੱਚ ਰੱਖੋ ਅਤੇ ਇਹ ਵੀ ਕਿ ਇਸ ਅਧਿਆਇ ਵਿੱਚ ਢੱਕਿਆ ਗਿਆ ਟ੍ਰਾਂਸਪੋਰਟ ਸਟ੍ਰੀਮਏਬਲ HTTP ਹੈ।

## ਸਟ੍ਰੀਮਿੰਗ: ਧਾਰਣਾਵਾਂ ਅਤੇ ਪ੍ਰੇਰਣਾ

ਸਟ੍ਰੀਮਿੰਗ ਦੇ ਮੂਲ ਭਾਵਨਾ ਅਤੇ ਪ੍ਰੇਰਣਾ ਨੂੰ ਸਮਝਣਾ ਪ੍ਰਭਾਵਸ਼ਾਲੀ ਅਸਲੀ ਸਮੇਂ ਸੰਚਾਰ ਸਿਸਟਮ ਬਣਾਉਣ ਲਈ ਬਹੁਤ ਜ਼ਰੂਰੀ ਹੈ।

**ਸਟ੍ਰੀਮਿੰਗ** ਨੈੱਟਵਰਕ ਪ੍ਰੋਗ੍ਰਾਮਿੰਗ ਵਿੱਚ ਉਹ ਤਕਨੀਕ ਹੈ ਜੋ ਡਾਟਾ ਨੂੰ ਛੋਟੇ, ਸੰਭਾਲਣਯੋਗ ਹਿੱਸਿਆਂ ਵਿੱਚ ਜਾਂ ਘਟਨਾਵਾਂ ਦੀ ਲੜੀ ਵਜੋਂ ਭੇਜਣ ਅਤੇ ਪ੍ਰਾਪਤ ਕਰਨ ਦੀ ਆਗਿਆ ਦਿੰਦੀ ਹੈ, ਬਜਾਇਆ ਕਿ ਪੂਰੇ ਜਵਾਬ ਦੇ ਤਿਆਰ ਹੋਣ ਦੀ ਉਡੀਕ ਕਰਨ ਦੇ। ਇਹ ਖਾਸ ਕਰਕੇ ਇਨ੍ਹਾਂ ਲਈ ਲਾਭਦਾਇਕ ਹੈ:

- ਵੱਡੀਆਂ ਫਾਈਲਾਂ ਜਾਂ ਡਾਟਾਸੈਟਸ।
- ਅਸਲੀ ਸਮੇਂ ਅੱਪਡੇਟ (ਜਿਵੇਂ ਕਿ ਗੱਲਬਾਤ, ਪ੍ਰਗਟਿਵ ਬਾਰ)।
- ਲੰਬੇ ਸਮੇਂ ਚੱਲਣ ਵਾਲੇ ਗਣਨਾ ਕੰਮ ਜਿੱਥੇ ਤੁਸੀਂ ਉਪਭੋਗਤਾ ਨੂੰ ਜਾਣੂ ਰੱਖਣਾ ਚਾਹੁੰਦੇ ਹੋ।

ਸਰਲ ਸਤਰ 'ਤੇ ਸਟ੍ਰੀਮਿੰਗ ਬਾਰੇ ਤੁਹਾਨੂੰ ਇਹ ਜਾਣਨਾ ਚਾਹੀਦਾ ਹੈ:

- ਡਾਟਾ ਧੀਰੇ-ਧੀਰੇ ਵੀਤਰੇ ਜਾਂਦਾ ਹੈ, ਇੱਕ ਵਾਰੀ ਵਿੱਚ ਨਹੀਂ।
- ਗਾਹਕ ਆਉਂਦੇ ਹੀ ਡਾਟਾ ਨੂੰ ਪ੍ਰਕਿਰਿਆ ਕਰ ਸਕਦਾ ਹੈ।
- ਪ੍ਰਤੀਤ ਹੋਣ ਵਾਲੀ ਦੇਰੀ ਨੂੰ ਘਟਾਉਂਦਾ ਹੈ ਅਤੇ ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਨੂੰ ਸੁਧਾਰਦਾ ਹੈ।

### ਸਟ੍ਰੀਮਿੰਗ ਦੀ ਵਰਤੋਂ ਕਿਉਂ ਕਰੀਏ?

ਸਟ੍ਰੀਮਿੰਗ ਦੀ ਵਰਤੋਂ ਕਰਨ ਦੇ ਕਾਰਨ ਇਹ ਹਨ:

- ਉਪਭੋਗਤਾ ਤੁਰੰਤ ਫੀਡਬੈਕ ਪਾਉਂਦੇ ਹਨ, ਸਿਰਫ ਅੰਤ ਵਿੱਚ ਨਹੀਂ।
- ਅਸਲੀ ਸਮੇਂ ਐਪਲੀਕੇਸ਼ਨਾਂ ਅਤੇ ਜਵਾਬਦਾਰ UI ਨੂੰ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ।
- ਨੈੱਟਵਰਕ ਅਤੇ ਕਮਪਿਊਟ ਸਰੋਤਾਂ ਦੀ ਜ਼ਿਆਦਾ ਪ੍ਰਭਾਵਸ਼ਾਲੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ।

### ਸਧਾਰਣ ਉਦਾਹਰਨ: HTTP ਸਟ੍ਰੀਮਿੰਗ ਸਰਵਰ ਅਤੇ ਗਾਹਕ

ਇਹ ਰਹੀ ਇੱਕ ਸਧਾਰਣ ਉਦਾਹਰਨ ਕਿ ਸਟ੍ਰੀਮਿੰਗ ਕਿਵੇਂ ਲਾਗੂ ਕੀਤੀ ਜਾ ਸਕਦੀ ਹੈ:

#### ਪਾਇਥਨ

**ਸਰਵਰ (ਪਾਇਥਨ, FastAPI ਅਤੇ StreamingResponse ਦੀ ਵਰਤੋਂ ਕਰਦਿਆਂ):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```
  
**ਗਾਹਕ (ਪਾਇਥਨ, requests ਦੀ ਵਰਤੋਂ ਕਰਦਿਆਂ):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```
  
ਇਹ ਉਦਾਹਰਨ ਦਰਸਾਉਂਦੀ ਹੈ ਕਿ ਸਰਵਰ ਕਿਵੇਂ ਗਾਹਕ ਨੂੰ ਸੰਦੇਸ਼ਾਂ ਦੀ ਲੜੀ ਭੇਜਦਾ ਹੈ ਜਿਵੇਂ ਉਹ ਤਿਆਰ ਹੁੰਦੇ ਹਨ, ਬਜਾਈ ਸਾਰੀਆਂ ਸੰਦੇਸ਼ਾਂ ਦੀ ਤਿਆਰੀ ਦੇ ਉਡੀਕ ਕਰਨ ਦੇ।

**ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ:**

- ਸਰਵਰ ਹਰ ਸੰਦੇਸ਼ ਨੂੰ ਜਿਵੇਂ ਤਿਆਰ ਹੁੰਦਾ ਹੈ ਤਿਵੇਂ ਸਪੁਰਦ ਕਰਦਾ ਹੈ।
- ਗਾਹਕ ਹਰ ਚੰਕ ਨੂੰ ਜਿਵੇਂ ਮਿਲਦਾ ਹੈ ਪ੍ਰਿੰਟ ਕਰਦਾ ਹੈ।

**ਲੋੜਾਂ:**

- ਸਰਵਰ ਨੂੰ ਸਟ੍ਰੀਮਿੰਗ ਜਵਾਬ ਦੀ ਵਰਤੋਂ ਕਰਨੀ ਚਾਹੀਦੀ ਹੈ (ਜਿਵੇਂ FastAPI ਵਿੱਚ `StreamingResponse`)।
- ਗਾਹਕ ਨੂੰ ਜਵਾਬ ਨੂੰ ਇੱਕ ਸਟ੍ਰੀਮ ਵਜੋਂ ਪ੍ਰਕਿਰਿਆ ਕਰਨਾ ਚਾਹੀਦਾ ਹੈ (`requests` ਵਿੱਚ `stream=True`)।
- Content-Type ਆਮ ਤੌਰ 'ਤੇ `text/event-stream` ਜਾਂ `application/octet-stream` ਹੁੰਦੀ ਹੈ।

#### ਜਾਵਾ

**ਸਰਵਰ (ਜਾਵਾ, Spring Boot ਅਤੇ Server-Sent Events ਦੀ ਵਰਤੋਂ ਨਾਲ):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```
  
**ਗਾਹਕ (ਜਾਵਾ, Spring WebFlux WebClient ਦੀ ਵਰਤੋਂ ਕਰਦਿਆਂ):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```
  
**ਜਾਵਾ ਲਾਗੂ ਕਰਨ ਦੇ ਨੋਟਸ:**

- Spring Boot ਦੀ ਪ੍ਰਤੀਕ੍ਰਿਆਮੂਲ ਸਟੈਕ ਤੋਂ `Flux` ਦੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ ਸਟ੍ਰੀਮਿੰਗ ਲਈ
- `ServerSentEvent` ਘਟਨਾਵਾਂ ਦੀ ਸਟ੍ਰਕਚਰਡ ਸਟ੍ਰੀਮਿੰਗ ਲਈ ਘਟਨਾ ਕਿਸਮਾਂ ਨਾਲ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ
- `WebClient` ਦਾ `bodyToFlux()` ਪ੍ਰਤੀਕ੍ਰਿਆਮੂਲ ਸਟ੍ਰੀਮਿੰਗ ਖਪਤ ਨੂੰ ਯੋਗ ਕਰਦਾ ਹੈ
- `delayElements()` ਘਟਨਾਵਾਂ ਵਿਚਕਾਰ ਪ੍ਰਕਿਰਿਆ ਸਮਾਂ ਦਾ ਅਨੁਕਰਨ ਕਰਦਾ ਹੈ
- ਘਟਨਾਵਾਂ ਦੇ ਟਾਈਪਾਂ (`info`, `result`) ਹੋ ਸਕਦੇ ਹਨ ਜੋ ਗਾਹਕ ਦੇਡਾਲਿੰਗ ਨੂੰ ਸੁਧਾਰਦੇ ਹਨ

### ਤੁਲਨਾ: ਕਲਾਸਿਕ ਸਟ੍ਰੀਮਿੰਗ ਵర్స MCP ਸਟ੍ਰੀਮਿੰਗ

ਇਸ ਤਰ੍ਹਾਂ ਕਲਾਸਿਕ ਸਟ੍ਰੀਮਿੰਗ ਅਤੇ MCP ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ ਵਿਚਲਾ ਫਰਕ ਦਰਸਾਇਆ ਜਾ ਸਕਦਾ ਹੈ:

| ਵਿਸ਼ੇਸ਼ਤਾ             | ਕਲਾਸਿਕ HTTP ਸਟ੍ਰੀਮਿੰਗ          | MCP ਸਟ੍ਰੀਮਿੰਗ (ਸੂਚਨਾਵਾਂ)      |
|------------------------|-------------------------------|-------------------------------|
| ਮੂਲ ਜਵਾਬ             | ਚੰਕਡ                         | ਇਕੱਲਾ, ਅੰਤ ਵਿੱਚ               |
| ਪ੍ਰਗਟਿਵ ਅੱਪਡੇਟ        | ਡਾਟਾ ਚੰਕ ਵਜੋਂ ਭੇਜੇ ਜਾਂਦੇ ਹਨ    | ਸੂਚਨਾਵਾਂ ਦੇ ਰੂਪ ਵਿੱਚ ਭੇਜੇ ਜਾਂਦੇ ਹਨ |
| ਗਾਹਕ ਦੀਆਂ ਲੋੜਾਂ       | ਸਟ੍ਰੀਮ ਨੂੰ ਪ੍ਰਕਿਰਿਆ ਕਰਨੀ ਪੈਂਦੀ ਹੈ | ਸੁਨੇਹਾ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰਨਾ ਪੈਂਦਾ ਹੈ |
| ਵਰਤੋਂ ਦਾ ਕੇਸ          | ਵੱਡੀਆਂ ਫਾਈਲਾਂ, AI ਟੋਕਨ ਸਟ੍ਰੀਮ | ਪ੍ਰਗਟਿਵ, ਲੌਗ, ਅਸਲੀ ਸਮੇਂ ਫੀਡਬੈਕ |

### ਮੁੱਖ ਫਰਕ ਜਿਹੜੇ ਵੇਖੇ ਗਏ

ਹੋਰ ਕੁਝ ਮੁੱਖ ਫਰਕ ਹਨ:

- **ਸੰਚਾਰ ਪੈਟਰਨ:**
  - ਕਲਾਸਿਕ HTTP ਸਟ੍ਰੀਮਿੰਗ: ਸਧਾਰਣ ਚੰਕਡ ਟ੍ਰਾਂਸਫਰ ਕੋਡਿੰਗ ਵਰਤਦਾ ਹੈ ਡਾਟਾ ਚੰਕ ਭੇਜਣ ਲਈ
  - MCP ਸਟ੍ਰੀਮਿੰਗ: JSON-RPC ਪ੍ਰੋਟੋਕਾਲ ਨਾਲ ਇੱਕ ਸਟ੍ਰਕਚਰਡ ਨੋਟੀਫਿਕੇਸ਼ਨ ਸਿਸਟਮ ਵਰਤਦਾ ਹੈ

- **ਮੇਸੇਜ ਫਾਰਮੈਟ:**
  - ਕਲਾਸਿਕ HTTP: ਨਿਵਾਸੀ ਟੈਕਸਟ ਚੰਕਾਂ ਨਾਲ ਨਿਊਲਾਈਨ ਹੋਈ
  - MCP: ਮੈਟਾ ਡਾਟਾ ਸਮੇਤ ਸਟ੍ਰਕਚਰਡ LoggingMessageNotification objects

- **ਗਾਹਕ ਲਾਗੂ ਕਰਨ:**
  - ਕਲਾਸਿਕ HTTP: ਸਧਾਰਣ ਗਾਹਕ ਜੋ ਸਟ੍ਰੀਮਿੰਗ ਜਵਾਬਾਂ ਨੂੰ ਪ੍ਰਕਿਰਿਆ ਕਰਦਾ ਹੈ
  - MCP: ਵੱਧ ਸੁਖਾਧਾਈ ਗਾਹਕ ਜਿਹੜਾ ਵੱਖ-ਵੱਖ ਕਿਸਮਾਂ ਦੇ ਸੂਨੇਹੇ ਸਮਭਾਲਣ ਵਾਲਾ ਹੈ

- **ਪ੍ਰਗਟਿਵ ਅੱਪਡੇਟ:**
  - ਕਲਾਸਿਕ HTTP: ਪ੍ਰਗਟਿਵ ਮੂਲ ਜਵਾਬ ਸਟ੍ਰੀਮ ਦਾ ਹਿੱਸਾ ਹੁੰਦਾ ਹੈ
  - MCP: ਪ੍ਰਗਟਿਵ ਵੱਖ-ਵੱਖ ਸੂਚਨਾ ਸੰਦੇਸ਼ਾਂ ਰਾਹੀਂ ਭੇਜੇ ਜਾਂਦੇ ਹਨ, ਜਦ ਕਿ ਮੂਲ ਜਵਾਬ ਅੰਤ ਵਿੱਚ ਹੁੰਦਾ ਹੈ

### ਸਿਫ਼ਾਰਸ਼ਾਂ

ਕਲਾਸਿਕ ਸਟ੍ਰੀਮਿੰਗ (ਜਿਵੇਂ ਕਿ `/stream` ਨੂੰ ਦਰਸਾਇਆ) ਅਤੇ MCP ਸਟ੍ਰੀਮਿੰਗ ਵਿੱਚੋਂ ਚੁਣਨ ਸੰਬੰਧੀ ਕੁਝ ਸਿਫ਼ਾਰਸ਼ਾਂ ਇਹ ਹਨ:

- **ਸਧਾਰਣ ਸਟ੍ਰੀਮਿੰਗ ਲੋੜਾਂ ਲਈ:** ਕਲਾਸਿਕ HTTP ਸਟ੍ਰੀਮਿੰਗ ਲਾਗੂ ਕਰਨ ਲਈ ਸਾਦਾ ਹੈ ਅਤੇ ਮੁੱਖ ਸਟ੍ਰੀਮਿੰਗ ਲੋੜਾਂ ਲਈ ਯਥੇਸ਼ਟ ਹੈ।
- **ਜਟਿਲ, ਇੰਟਰਐਕਟਿਵ ਐਪ ਲਈ:** MCP ਸਟ੍ਰੀਮਿੰਗ ਵਿਸ਼ਤ੍ਰਿਤ ਮੈਟਾ ਡਾਟਾ ਅਤੇ ਸੂਚਨਾਵਾਂ ਅਤੇ ਆਖਰੀ ਨਤੀਜਿਆਂ ਵਿਚਕਾਰ ਵੱਖਰੇਪਣ ਨਾਲ ਇੱਕ ਹੋਰ ਬੇਹਤਰੀਨ ਪਹੁੰਚ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ।
- **AI ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ:** MCP ਦਾ ਸੂਚਨਾ ਸਿਸਟਮ ਖਾਸ ਤੌਰ ਤੇ ਲੰਬੇ ਸਮੇਂ ਚੱਲ ਰਹੇ AI ਕਾਰਜਾਂ ਲਈ ਲਾਭਦਾਇਕ ਹੈ ਜਿੱਥੇ ਤੁਸੀਂ ਉਪਭੋਗਤਾਵਾਂ ਨੂੰ ਪ੍ਰਗਟਿਵ ਨਾਲ ਜਾਣੂ ਰੱਖਣਾ ਚਾਹੁੰਦੇ ਹੋ।

## MCP ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ

ਠੀਕ ਹੈ, ਤਾਂ ਤੁਸੀਂ ਹੁਣ ਤੱਕ ਕੁਝ ਸਿਫ਼ਾਰਸ਼ਾਂ ਅਤੇ ਤੁਲਨਾਵਾਂ ਵੇਖੀਆਂ ਜੋ ਕਲਾਸਿਕ ਸਟ੍ਰੀਮਿੰਗ ਅਤੇ MCP ਸਟ੍ਰੀਮਿੰਗ ਵਿੱਚ ਫਰਕ ਦੱਸਦੀਆਂ ਹਨ। ਚਲੋ ਹੁਣ ਵੇਖੀਏ ਕਿ MCP ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ ਨੂੰ ਅਸੀਂ ਕਿਵੇਂ ਲਾਗੂ ਕਰ ਸਕਦੇ ਹਾਂ।

MCP ਫਰੇਮਵਰਕ ਦੇ ਅੰਦਰ ਸਟ੍ਰੀਮਿੰਗ ਕਿਵੇਂ ਕੰਮ ਕਰਦੀ ਹੈ, ਇਹ ਜਾਣਨਾ ਬਹੁਤ ਜ਼ਰੂਰੀ ਹੈ ਤਾਂ ਜੋ ਤੁਸੀਂ ਜਵਾਬਦਾਰ ਐਪਲੀਕੇਸ਼ਨ ਬਣਾਈਆਂ ਜਿਨ੍ਹਾਂ ਨਾਲ ਉਪਭੋਗਤਾਵਾਂ ਨੂੰ ਲੰਬੇ ਸਮੇਂ ਚੱਲਣ ਵਾਲੀਆਂ ਕਾਰਵਾਈਆਂ ਦੌਰਾਨ ਅਸਲੀ ਸਮੇਂ ਫੀਡਬੈਕ ਮਿਲੇ।

MCP ਵਿੱਚ, ਸਟ੍ਰੀਮਿੰਗ ਦਾ ਮਤਲਬ ਮੁੱਖ ਜਵਾਬ ਨੂੰ ਟੁਕੜਿਆਂ ਵਿੱਚ ਭੇਜਣਾ ਨਹੀ, ਬਲਕਿ ਜਦੋਂ ਇੱਕ ਟੂਲ ਕਿਸੇ ਬੇਨਤੀ ਦੀ ਪ੍ਰਕਿਰਿਆ ਕਰ ਰਿਹਾ ਹੁੰਦਾ ਹੈ, ਉਸ ਦੌਰਾਨ ਗਾਹਕ ਨੂੰ **ਸੂਚਨਾਵਾਂ** ਭੇਜਣ ਦਾ ਹੁੰਦਾ ਹੈ। ਇਹ ਸੂਚਨਾਵਾਂ ਪ੍ਰਗਟਿਵ ਅੱਪਡੇਟ, ਲੌਗ ਜਾਂ ਹੋਰ ਇਵੈਂਟਸ ਹੋ ਸਕਦੇ ਹਨ।

### ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ

ਮੁੱਖ ਨਤੀਜਾ ਅਜੇ ਵੀ ਇੱਕੋ ਜਵਾਬ ਵਜੋਂ ਭੇਜਿਆ ਜਾਂਦਾ ਹੈ। ਪਰ ਸੂਚਨਾਵਾਂ ਨੂੰ ਪ੍ਰਕਿਰਿਆ ਦੌਰਾਨ ਵੱਖਰੇ ਸੰਦੇਸ਼ਾਂ ਵਜੋਂ ਭੇਜਿਆ ਜਾ ਸਕਦਾ ਹੈ ਜਿਸ ਨਾਲ ਗਾਹਕ ਨੂੰ ਅਸਲੀ ਸਮੇਂ ਅੱਪਡੇਟ ਮਿਲ ਸਕਦੇ ਹਨ। ਗਾਹਕ ਨੂੰ ਇਹਨਾਂ ਸੂਚਨਾਵਾਂ ਨੂੰ ਸੰਭਾਲਣ ਅਤੇ ਦਰਸਾਉਣ ਯੋਗ ਹੋਣਾ ਚਾਹੀਦਾ ਹੈ।

## ਸੂਚਨਾ ਕੀ ਹੈ?

ਅਸੀਂ ਕਿਹਾ "ਸੂਚਨਾ," MCP ਸੰਦਰਭ ਵਿੱਚ ਇਸ ਦਾ ਕੀ ਅਰਥ ਹੈ?

ਸੂਚਨਾ ਇੱਕ ਅਜਿਹਾ ਸੰਦੇਸ਼ ਹੈ ਜੋ ਸਰਵਰ ਵੱਲੋਂ ਗਾਹਕ ਨੂੰ ਭੇਜਿਆ ਜਾਂਦਾ ਹੈ ਤਾਂ ਜੋ ਲੰਬੇ ਸਮੇਂ ਚੱਲ ਰਹੀ ਕਾਰਵਾਈ ਦੌਰਾਨ ਪ੍ਰਗਟਿਵ, ਸਥਿਤੀ ਜਾਂ ਹੋਰ ਇਵੈਂਟਸ ਬਾਰੇ ਜਾਣੂ ਕਰਾਇਆ ਜਾ ਸਕੇ। ਸੂਚਨਾਵਾਂ ਪਾਰਦਰਸ਼ਤਾ ਅਤੇ ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਨੂੰ ਬਿਹਤਰ ਬਣਾਉਂਦੀਆਂ ਹਨ।

ਉਦਾਹਰਨ ਵਜੋਂ, ਗਾਹਕ ਨੂੰ ਇੱਕ ਸੂਚਨਾ ਭੇਜਣੀ ਚਾਹੀਦੀ ਹੈ ਜਦੋਂ ਪਹਿਲਾ ਹੈਂਡਸ਼ੇਕ ਸਰਵਰ ਨਾਲ ਮੁਕੰਮਲ ਹੋ ਜਾਵੇ।

ਇੱਕ ਸੂਚਨਾ JSON ਸੁਨੇਹੇ ਵਾਂਗ ਇਸ ਤਰ੍ਹਾਂ ਦੇਖਦੀ ਹੈ:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```
  
ਸੂਚਨਾਵਾਂ MCP ਵਿੱਚ ਉਸ ਵਿਸ਼ੇ ਨਾਲ ਜੁੜੀਆਂ ਹੁੰਦੀਆਂ ਹਨ ਜਿਸਨੂੰ ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) ਕਿਹਾ ਜਾਂਦਾ ਹੈ।

ਲੌਗਿੰਗ ਚਾਲੂ ਕਰਨ ਲਈ, ਸਰਵਰ ਨੂੰ ਇਸਨੂੰ ਵਿਸ਼ੇਸ਼ਤਾ / ਕੈਪੇਬਿਲਿਟੀ ਵਜੋਂ ਇੰਨੇਬਲ ਕਰਨਾ ਪੈਂਦਾ ਹੈ ਇਸ ਤਰ੍ਹਾਂ:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```
  
> [!NOTE]  
> ਵਰਤਿਆ ਗਿਆ SDK ਦੇ ਅਨੁਸਾਰ, ਲੌਗਿੰਗ ਆਮ ਤੌਰ 'ਤੇ ਡਿਫਾਲਟ ਰੂਪ ਵਿੱਚ ਚਾਲੂ ਹੋ ਸਕਦੀ ਹੈ, ਜਾਂ ਤੁਹਾਨੂੰ ਸਰਵਰ ਸੰਰਚਨਾ ਵਿੱਚ ਇਸਨੂੰ ਵਿਸ਼ੇਸ਼ ਤੌਰ ਤੇ ਚਾਲੂ ਕਰਨਾ ਪੈ ਸਕਦਾ ਹੈ।

ਸੂਚਨਾਵਾਂ ਦੀਆਂ ਵੱਖ-ਵੱਖ ਕਿਸਮਾਂ ਹਨ:

| ਪੱਧਰ     | ਵੇਰਵਾ                        | ਉਦਾਹਰਨ ਵਰਤੋਂ                   |
|-----------|------------------------------|---------------------------------|
| debug     | ਵਿਸਥਾਰਪੂਰਵਕ ਡੀਬੱਗਿੰਗ ਸੂਚਨਾ | ਫੰਕਸ਼ਨ ਦਾਖਲ/ਨਿਕਾਸ ਬਿੰਦੂ        |
| info      | ਆਮ ਜਾਣਕਾਰੀ ਸੰਦੇਸ਼             | ਕਾਰਵਾਈ ਪ੍ਰਗਟਿਵ ਅੱਪਡੇਟ          |
| notice    | ਸਧਾਰਣ ਪਰ ਮਹੱਤਵਪੂਰਨ ਘਟਨਾਵਾਂ | ਸੰਰਚਨਾ ਬਦਲਾਅ                   |
| warning   | ਚਿਤਾਵਨੀ ਸਥਿਤੀਆਂ              | ਪੁਰਾਣੀ ਹੋਈ ਵਿਸ਼ੇਸ਼ਤਾ ਵਰਤੋਂ     |
| error     | ਗਲਤੀ ਦੀਆਂ ਸਥਿਤੀਆਂ           | ਕਾਰਵਾਈ ਅਸਫਲਤਾਵਾਂ              |
| critical  | ਗੰਭੀਰ ਸਥਿਤੀਆਂ               | ਸਿਸਟਮ ਕੰਪੋਨੈਂਟ ਨੁਕਸਾਨ          |
| alert     | ਤੁਰੰਤ ਕਾਰਵਾਈ ਦੀ ਲੋੜ          | ਡਾਟਾ ਖਰਾਬੀ ਮਿਲੀ               |
| emergency | ਸਿਸਟਮ ਅਕਮਰਦ ਹੈ               | ਪੂਰਾ ਸਿਸਟਮ ਨੁਕਸਾਨ               |

## MCP ਵਿੱਚ ਸੂਚਨਾਵਾਂ ਲਾਗੂ ਕਰਨਾ

MCP ਵਿੱਚ ਸੂਚਨਾਵਾਂ ਲਾਗੂ ਕਰਨ ਲਈ, ਤੁਹਾਨੂੰ ਸਰਵਰ ਅਤੇ ਗਾਹਕ ਦੋਹਾਂ ਪਾਸੇ ਅਸਲੀ ਸਮੇਂ ਅੱਪਡੇਟ ਸੰਭਾਲਣ ਲਈ ਤੈਅ ਦਿੱਤਾ ਜਾਣਾ ਚਾਹੀਦਾ ਹੈ। ਇਸ ਨਾਲ ਤੁਹਾਡੀ ਐਪਲੀਕੇਸ਼ਨ ਉਪਭੋਗਤਾਵਾਂ ਨੂੰ ਲੰਬੇ ਸਮੇਂ ਚੱਲ ਰਹੀਆਂ ਕਾਰਵਾਈਆਂ ਦੌਰਾਨ ਤੁਰੰਤ ਫੀਡਬੈਕ ਦੇ ਸਕਦੀ ਹੈ।

### ਸਰਵਰ-ਪਾਸਾ: ਸੂਚਨਾਵਾਂ ਭੇਜਣਾ

ਚਲੋ ਸਰਵਰ ਪਾਸੇ ਤੋਂ ਸ਼ੁਰੂ ਕਰੀਏ। MCP ਵਿੱਚ ਤੁਸੀਂ ਟੂਲ ਤਈਂ ਕਰਦੇ ਹੋ ਜੋ ਬੇਨਤੀਆਂ ਪ੍ਰਕਿਰਿਆ ਕਰਦਿਆਂ ਸੂਚਨਾਵਾਂ ਭੇਜ ਸਕਦਾ ਹੈ। ਸਰਵਰ ਸੰਦਰਭ ਆਬਜੈਕਟ (ਆਮ ਤੌਰ 'ਤੇ `ctx`) ਦੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ ਗਾਹਕ ਨੂੰ ਸੁਨੇਹੇ ਭੇਜਣ ਲਈ।

#### ਪਾਇਥਨ

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```
  
ਉਪਰੋਕਤ ਉਦਾਹਰਨ ਵਿੱਚ, `process_files` ਟੂਲ ਹਰ ਫਾਈਲ ਪ੍ਰਕਿਰਿਆ ਕਰਦੇ ਸਮੇਂ ਗਾਹਕ ਨੂੰ ਤਿੰਨ ਸੂਚਨਾਵਾਂ ਭੇਜਦਾ ਹੈ। `ctx.info()` ਮੈਥਡ ਜਾਣੂਕਾਰੀਆਂ ਭੇਜਣ ਲਈ ਵਰਤੀ ਜਾਂਦੀ ਹੈ।

## ਪ੍ਰਗਟਿਵ ਸੂਚਨਾਵਾਂ ਅਤੇ ਸਿੰਨਾਰਿਓ

ਇਹ ਸੈਕਸ਼ਨ MCP ਵਿੱਚ ਪ੍ਰਗਟਿਵ ਸੂਚਨਾਵਾਂ ਦੀ ਧਾਰਣਾ ਨੂੰ ਸਮਝਾਉਂਦਾ ਹੈ, ਇਹ ਕਿਉਂ ਮਹੱਤਵਪੂਰਨ ਹਨ ਅਤੇ ਕਿਵੇਂ ਤੁਸੀਂ ਸਟ੍ਰੀਮਏਬਲ HTTP ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਇਨ੍ਹਾਂ ਨੂੰ ਲਾਗੂ ਕਰ ਸਕਦੇ ਹੋ। ਇਹਥੇ ਇੱਕ ਪ੍ਰਾਇਕਟਿਕਲ ਅਸਾਈਨਮੈਂਟ ਵੀ ਦਿੱਤਾ ਗਿਆ ਹੈ ਤਾਂ ਤੁਹਾਡੀ ਸਮਝ ਮਜਬੂਤ ਹੋਵੇ।

ਪ੍ਰਗਟਿਵ ਸੂਚਨਾਵਾਂ ਉਹ ਅਸਲੀ ਸਮੇਂ ਸੁਨੇਹੇ ਹੁੰਦੇ ਹਨ ਜੋ ਸਰਵਰ ਵੱਲੋਂ ਗਾਹਕ ਨੂੰ ਲੰਬੇ ਸਮੇਂ ਚੱਲ ਰਹੀ ਕਾਰਵਾਈ ਦੌਰਾਨ ਭੇਜੇ ਜਾਂਦੇ ਹਨ। ਪੂਰੀ ਕਾਰਵਾਈ ਦੇ ਖਤਮ ਹੋਣ ਦੇ ਉਡੀਕ ਕਰਨ ਦੀ ਬਜਾਏ, ਸਰਵਰ ਗਾਹਕ ਨੂੰ ਮੌਜੂਦਾ ਸਥਿਤੀ ਬਾਰੇ ਜਾਣੂ ਕਰਦਾ ਰਹਿੰਦਾ ਹੈ। ਇਸ ਨਾਲ ਪਾਰਦਰਸ਼ਤਾ, ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਸੋਧਦਾ ਹੈ ਅਤੇ ਡੀਬੱਗਿੰਗ ਆਸਾਨ ਹੁੰਦੀ ਹੈ।

### ਪ੍ਰਗਟਿਵ ਸੂਚਨਾਵਾਂ ਦੀ ਵਰਤੋਂ ਕਿਉਂ ਕਰੀਏ?

ਪ੍ਰਗਟਿਵ ਸੂਚਨਾਵਾਂ ਕਈ ਕਾਰਨਾਂ ਕਰਕੇ ਜ਼ਰੂਰੀ ਹਨ:

- **ਵਧੀਆ ਉਪਭੋਗਤਾ ਅਨੁਭਵ:** ਉਪਭੋਗਤਾ ਕੰਮ ਚੱਲਦੇ ਸਮੇਂ ਅੱਪਡੇਟ ਵੇਖਦੇ ਹਨ, ਸਿਰਫ ਅੰਤ ਵਿੱਚ ਨਹੀਂ।
- **ਅਸਲੀ ਸਮੇਂ ਫੀਡਬੈਕ:** ਗਾਹਕ ਪ੍ਰਗਟਿਵ ਬਾਰ ਜਾਂ ਲੌਗ ਦਰਸਾ ਸਕਦੇ ਹਨ, ਜਿਸ ਨਾਲ ਐਪ ਜਵਾਬਦਾਰ ਮਹਿਸੂਸ ਹੁੰਦੀ ਹੈ।
- **ਡਿਬੱਗਿੰਗ ਅਤੇ ਮਾਨੀਟਰਿੰਗ ਸੁਗਮ:** ਵਿਕਾਸਕਾਰ ਅਤੇ ਉਪਭੋਗਤਾ ਵੇਖ ਸਕਦੇ ਹਨ ਕਿ ਪ੍ਰਕਿਰਿਆ ਕਿੱਥੇ ਧੀਮੀ ਜਾਂ ਅਟਕੀ ਹੈ।

### ਪ੍ਰਗਟਿਵ ਸੂਚਨਾਵਾਂ ਲਾਗੂ ਕਰਨ ਦਾ ਤਰੀਕਾ

ਇਸ ਤਰ੍ਹਾਂ MCP ਵਿੱਚ ਪ੍ਰਗਟਿਵ ਸੂਚਨਾਵਾਂ ਲਾਗੂ ਕੀਤੀਆਂ ਜਾਂਦੀਆਂ ਹਨ:

- **ਸਰਵਰ 'ਤੇ:** `ctx.info()` ਜਾਂ `ctx.log()` ਦੀ ਵਰਤੋਂ ਵਰਕ ਆਈਟਮ ਪ੍ਰਕਿਰਿਆ ਕਰਦੇ ਸਮੇਂ ਸੂਚਨਾਵਾਂ ਭੇਜਣ ਲਈ। ਇਹ ਮੂਲ ਨਤੀਜੇ ਤੋਂ ਪਹਿਲਾਂ ਗਾਹਕ ਨੂੰ ਸੁਨੇਹਾ ਭੇਜਦਾ ਹੈ।
- **ਗਾਹਕ 'ਤੇ:** ਇੱਕ ਸੁਨੇਹਾ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰੋ ਜੋ ਆਉਂਦੀਆਂ ਸੂਚਨਾਵਾਂ ਨੂੰ ਸੁਣਦਾ ਅਤੇ ਦਰਸਾਉਂਦਾ ਹੈ। ਇਹ ਹੈਂਡਲਰ ਸੂਚਨਾਵਾਂ ਅਤੇ ਆਖਰੀ ਨਤੀਜੇ ਵਿੱਚ ਤਫ਼ਰੀਕ ਕਰਦਾ ਹੈ।

## ਸੁਰੱਖਿਆ ਸੰਬੰਧੀ ਵਿਚਾਰ

ਜਦੋਂ HTTP-ਆਧਾਰਿਤ ਟ੍ਰਾਂਸਪੋਰਟ ਨਾਲ MCP ਸਰਵਰ ਬਣਾਏ ਜਾਂਦੇ ਹਨ, ਤਦ ਸੁਰੱਖਿਆ ਇੱਕ ਬਹੁਤ ਮਹੱਤਵਪੂਰਨ ਮਸਲਾ ਬਣ ਜਾਂਦਾ ਹੈ ਜਿਸ ਲਈ ਕਈ ਤਰ੍ਹਾਂ ਦੇ ਹਮਲਿਆਂ ਦੇ ਪੇਚਾਂ ਅਤੇ ਰੱਖਿਆ ਦੇ ਤਰੀਕਿਆਂ ਨੂੰ ਧਿਆਨ ਨਾਲ ਵੇਖਣਾ ਲਾਜ਼ਮੀ ਹੈ।

### ਜਾਇਜ਼ਾ

HTTP ਰਾਹੀਂ MCP ਸਰਵਰ ਖੋਲ੍ਹਦੇ ਸਮੇਂ ਸੁਰੱਖਿਆ ਮੂਲਭੂਤ ਹੈ। ਸਟ੍ਰੀਮਏਬਲ HTTP ਨਵੇਂ ਹਮਲੇ ਦੇ ਸਤਹ ਲਿਆਉਂਦਾ ਹੈ ਜੋ ਧਿਆਨ ਨਾਲ ਸੰਰਚਨਾ ਦੀ ਲੋੜ ਹੈ।

### ਮੁੱਖ ਬਿੰਦੂ

- **Origin ਹੈਡਰ ਦੀ ਜਾਂਚ:** ਸਦਾ `Origin` ਹੈਡਰ ਨੂੰ ਜਾਂਚੋ ਤਾਂ ਜੋ DNS ਰੀਬਾਈਂਡਿੰਗ ਹਮਲੇ ਤੋਂ ਬਚਿਆ ਜਾ ਸਕੇ।
- **ਲੋਕਲਹੋਸਟ ਬਾਈਂਡਿੰਗ:** ਲੋਕਲ ਵਿਕਾਸ ਲਈ ਸਰਵਰਾਂ ਨੂੰ `localhost` 'ਤੇ ਬਾਈਂਡ ਕਰੋ ਤਾਂ ਕਿ ਓਹ ਜਨਤਾ ਇੰਟਰਨੈਟ ਉੱਤੇ ਖੁਲ੍ਹੇ ਨਾ ਹੋਣ।
- **ਪਹਿਚਾਣ ਪ੍ਰਮਾਣੀਕਰਨ:** ਉਤਪਾਦਨ ਵਾਤਾਵਰਨਾਂ ਲਈ ਪ੍ਰਮਾਣੀਕਰਨ (ਜਿਵੇਂ API ਕੁੰਜੀਆਂ, OAuth) ਲਾਗੂ ਕਰੋ।
- **CORS:** ਐਕਸੈਸ ਸੀਮਿਤ ਕਰਨ ਲਈ ਕ੍ਰਾਸ-ਓਰਿਜ਼ਨ ਰਿਸੋਰਸ ਸ਼ੇਅਰਿੰਗ ਨੀਤੀਆਂ ਸੰਰਚਿਤ ਕਰੋ।
- **HTTPS:** ਟ੍ਰੈਫਿਕ ਇੰਕ੍ਰਿਪਟ ਕਰਨ ਲਈ ਉਤਪਾਦਨ ਵਿੱਚ HTTPS ਦੀ ਵਰਤੋਂ ਕਰੋ।

### ਵਧੀਆ ਪ੍ਰਥਾਵਾਂ

- ਆਪੇਖਾ ਦੇ ਬਿਨਾ ਆਉਣ ਵਾਲੀ ਬੇਨਤੀਆਂ ਤੋ ਕਦੇ ਵੀ ਵਿਸ਼ਵਾਸ ਨਾ ਕਰੋ।
- ਸਾਰੇ ਪਹੁੰਚ ਅਤੇ ਗਲਤੀਆਂ ਨੂੰ ਲਾਗ ਅਤੇ ਗੜ੍ਹਦੇ ਰਹੋ।
- ਸੁਰੱਖਿਆ ਖਾਮੀਆਂ ਨੂੰ ਠੀਕ ਕਰਨ ਲਈ ਨਿਯਮਤ ਤੌਰ ਤੇ ਡਿਪੈਂਡੇੰਸੀਜ਼ ਅੱਪਡੇਟ ਕਰੋ।

### ਚੁਣੌਤੀਆਂ
- ਵਿਕਾਸ ਦੀ ਸੁਵਿਧਾ ਨਾਲ ਸੁਰੱਖਿਆ ਦਾ ਸੰਤੁਲਨ
- ਵੱਖ-ਵੱਖ ਕਲਾਇੰਟ ਵਾਤਾਵਰਣਾਂ ਨਾਲ ਅਨੁਕੂਲਤਾ ਨੂੰ ਯਕੀਨੀ ਬਣਾਉਣਾ

## SSE ਤੋਂ Streamable HTTP ਵਿੱਚ ਅਪਗਰੇਡ ਕਰਨਾ

ਜਿਹੜੇ ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿੱਚ ਇਸ ਸਮੇਂ Server-Sent Events (SSE) ਵਰਤੇ ਜਾ ਰਹੇ ਹਨ, ਉਨ੍ਹਾਂ ਲਈ Streamable HTTP ਨੂੰ ਅਪਗਰੇਡ ਕਰਨਾ MCP ਇੰਪਲੀਮੈਂਟੇਸ਼ਨਾਂ ਲਈ ਵਧੀਆ ਸਮਰਥਾ ਅਤੇ ਲੰਬੇ ਸਮੇਂ ਦੀ ਸਥਿਰਤਾ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ।

### ਅਪਗਰੇਡ ਕਰਨ ਦੇ ਕਾਰਨ

SSE ਤੋਂ Streamable HTTP ਵਿੱਚ ਅਪਗਰੇਡ ਕਰਨ ਲਈ ਦੋ ਮੁੱਖ ਕਾਰਣ ਹਨ:

- Streamable HTTP SSE ਨਾਲੋਂ ਵਧੀਆ ਸਕੇਲਬਿਲਟੀ, ਅਨੁਕੂਲਤਾ ਅਤੇ ਅਮਰੀ ਧੁਨੀ ਸਹਿਯੋਗ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ।
- ਇਹ ਨਵੀਂ MCP ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ ਸਿਫਾਰਸ਼ ਕੀਤੇ ਗਏ ਟ੍ਰਾਂਸਪੋਰਟ ਹੈ।

### ਮਾਈਗ੍ਰੇਸ਼ਨ ਕਦਮ

ਆਪਣੇ MCP ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿੱਚ SSE ਤੋਂ Streamable HTTP ਵਿੱਚ ਕਿਵੇਂ ਮਾਈਗ੍ਰੇਟ ਕਰਨਾ ਹੈ:

- **ਸਰਵਰ ਕੋਡ ਨੂੰ ਅਪਡੇਟ ਕਰੋ** ਤਾਂ ਜੋ `mcp.run()` ਵਿੱਚ `transport="streamable-http"` ਵਰਤਿਆ ਜਾਵੇ।
- **ਕਲਾਇੰਟ ਕੋਡ ਨੂੰ ਅਪਡੇਟ ਕਰੋ** ਤਾਂ ਜੋ SSE ਕਲਾਇੰਟ ਦੀ ਥਾਂ `streamablehttp_client` ਵਰਤਿਆ ਜਾਵੇ।
- **ਕਲਾਇੰਟ ਵਿੱਚ ਸਨੇਹਾ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰੋ** ਜੋ ਸੂਚਨਾਵਾਂ ਨੂੰ ਪ੍ਰੋਸੈਸ ਕਰੇ।
- **ਹਾਲ ਹੀ ਦੇ ਟੂਲਜ਼ ਅਤੇ ਵਰਕਫਲੋਜ਼ ਨਾਲ ਅਨੁਕੂਲਤਾ ਨੂੰ ਟੈਸਟ ਕਰੋ**।

### ਅਨੁਕੂਲਤਾ ਨੂੰ ਕਾਇਮ ਰੱਖਣਾ

ਮਾਈਗ੍ਰੇਸ਼ਨ ਦੇ ਦੌਰਾਨ ਮੌਜੂਦਾ SSE ਕਲਾਇੰਟਾਂ ਨਾਲ ਅਨੁਕੂਲਤਾ ਕਾਇਮ ਰੱਖਣ ਦੀ ਸਿਫਾਰਸ਼ ਕੀਤੀ ਜਾਂਦੀ ਹੈ। ਕੁਝ ਤਾਕਤਵਰ ਰਣਨੀਤੀਆਂ ਹਨ:

- ਤੁਸੀਂ ਦੋਹਾਂ SSE ਅਤੇ Streamable HTTP ਨੂੰ ਵੱਖਰੇ ਐਂਡਪੋਇੰਟਾਂ 'ਤੇ ਚਲਾ ਕੇ ਦੋਹਾਂ ਟ੍ਰਾਂਸਪੋਰਟ ਸਮਰਥਿਤ ਕਰ ਸਕਦੇ ਹੋ।
- ਕਲਾਇੰਟਾਂ ਨੂੰ ਤਰਤੀਬਵਾਰ ਨਵੀਂ ਟ੍ਰਾਂਸਪੋਰਟ ਵੱਲ ਮਾਈਗ੍ਰੇਟ ਕਰੋ।

### ਚੁਣੌਤੀਆਂ

ਮਾਈਗ੍ਰੇਸ਼ਨ ਦੌਰਾਨ ਇਹ ਚੁਣੌਤੀਆਂ ਧਿਆਨ ਵਿੱਚ ਰੱਖੋ:

- ਸਾਰੇ ਕਲਾਇੰਟ ਅਪਡੇਟ ਹੋਣ ਯਕੀਨੀ ਬਣਾਉਣਾ
- ਸੂਚਨਾ ਡਿਲਿਵਰੀ ਵਿੱਚ ਫਰਕਾਂ ਨੂੰ ਹਿੱਟਾਂਲ ਕਰਨਾ

## ਸੁਰੱਖਿਆ ਸੰਬੰਧੀ ਵਿਚਾਰ

ਸਰਵਰ ਇੰਪਲੀਮੈਂਟ ਕਰਦੇ ਸਮੇਂ, ਖਾਸਕਰ MCP ਵਿੱਚ Streamable HTTP ਵਰਗੀਆਂ HTTP-ਆਧਾਰਿਤ ਟ੍ਰਾਂਸਪੋਰਟਾਂ ਦੇ ਸਹਾਰੇ ਸੁਰੱਖਿਆ ਸਭ ਤੋਂ ਵਧੀਆ ਤਰਜੀਹ ਹੋਣੀ ਚਾਹੀਦੀ ਹੈ।

HTTP-ਆਧਾਰਿਤ ਟ੍ਰਾਂਸਪੋਰਟਾਂ ਨਾਲ MCP ਸਰਵਰ ਇੰਪਲੀਮੈਂਟ ਕਰਦੇ ਸਮੇਂ, ਸੁਰੱਖਿਆ ਅਨੇਕ ਕਿਸਮਾਂ ਦੇ ਅਟੈਕ ਵੈਕਟਰਾਂ ਅਤੇ ਰੱਖਿਆ ਤਰੀਕਿਆਂ ਦੀ ਵਿਆਖਿਆ ਕਰਦੀ ਹੈ, ਜੋ ਗੰਭੀਰ ਧਿਆਨ ਦੀ ਮੰਗ ਕਰਦੀ ਹੈ।

### ਜਾਇਜ਼ਾ

MCP ਸਰਵਰ ਨੂੰ HTTP ਰਾਹੀਂ ਖੋਲ੍ਹਦਿਆਂ ਸੁਰੱਖਿਆ ਬਹੁਤ ਜਰੂਰੀ ਹੁੰਦੀ ਹੈ। Streamable HTTP ਨਵੀਆਂ ਅਟੈਕ ਸਤਹਾਂ ਪੇਸ਼ ਕਰਦਾ ਹੈ ਅਤੇ ਇਸਦੇ ਲਈ ਧਿਆਨ ਨਾਲ ਸੈਟਿੰਗਾਂ ਕਰਨ ਦੀ ਲੋੜ ਹੁੰਦੀ ਹੈ।

ਇੱਥੇ ਕੁਝ ਮੁੱਖ ਸੁਰੱਖਿਆ ਵਿਚਾਰ ਦਿੱਤੇ ਗਏ ਹਨ:

- **Origin ਹੇਡਰ ਦੀ ਜਾਂਚ**: DNS ਰੀਬਾਈਂਡਿੰਗ ਅਟੈਕ ਨੂੰ ਰੋਕਣ ਲਈ ਹਮੇਸ਼ਾ `Origin` ਹੇਡਰ ਦੀ ਪੁਸ਼ਟੀ ਕਰੋ।
- **ਲੋਕਲਹੋਸਟ ਬਾਈਂਡਿੰਗ**: ਵਿਕਾਸ ਲਈ ਸਰਵਰ ਨੂੰ `localhost` ਨਾਲ ਬਾਈਂਡ ਕਰੋ ਤਾਂ ਜੋ ਉਹ ਜਨਤਕ ਇੰਟਰਨੈੱਟ 'ਤੇ ਨਾ ਖੁਲੇ।
- **ਪ੍ਰਮਾਣੀਕਰਨ**: ਪ੍ਰੋਡਕਸ਼ਨ ਲਈ ਪ੍ਰਮਾਣੀਕਰਨ (ਜਿਵੇਂ ਕਿ API ਕੁੰਜੀਆਂ, OAuth) ਲਾਗੂ ਕਰੋ।
- **CORS**: ਐਕਸੈਸ ਸੀਮਤ ਕਰਨ ਲਈ Cross-Origin Resource Sharing (CORS) ਨੀਤੀਆਂ ਬਣਾਓ।
- **HTTPS**: ਟ੍ਰੈਫਿਕ ਸੁਰੱਖਿਅਤ ਕਰਨ ਲਈ ਪ੍ਰੋਡਕਸ਼ਨ ਵਿੱਚ HTTPS ਵਰਤੋ।

### ਸਰਵੋਤਮ ਅਮਲ

ਤੁਹਾਡੇ MCP ਸਟ੍ਰੀਮਿੰਗ ਸਰਵਰ ਵਿੱਚ ਸੁਰੱਖਿਆ ਲਾਗੂ ਕਰਦੇ ਸਮੇਂ, ਇਹ ਕੁਝ ਸਰਵੋਤਮ ਅਮਲ ਹਨ:

- ਆਉਣ ਵਾਲੇ ਬਿਨਾਂ ਪੁਸ਼ਟੀ ਵਾਲੇ ਅਨੁਰੋਧਾਂ 'ਤੇ ਕਦੇ ਭਰੋਸਾ ਨਾ ਕਰੋ।
- ਸਾਰੇ ਪਹੁੰਚ ਅਤੇ ਗਲਤੀਆਂ ਦਾ ਲੌਗ ਅਤੇ ਨਿਗਰਾਨੀ ਕਰੋ।
- ਸੁਰੱਖਿਆ ਖਾਮੀਆਂ ਨੂੰ ਠੀਕ ਕਰਨ ਲਈ Dependencies ਨੂੰ ਨਿਯਮਤ ਅਪਡੇਟ ਕਰਦੇ ਰਹੋ।

### ਚੁਣੌਤੀਆਂ

MCP ਸਟ੍ਰੀਮਿੰਗ ਸਰਵਰ ਵਿੱਚ ਸੁਰੱਖਿਆ ਲਾਗੂ ਕਰਦੇ ਸਮੇਂ ਤੁਹਾਨੂੰ ਕੁਝ ਚੁਣੌਤੀਆਂ ਦਾ ਸਾਹਮਣਾ ਕਰਨਾ ਪਵੇਗਾ:

- ਵਿਕਾਸ ਦੀ ਸੁਵਿਧਾ ਨਾਲ ਸੁਰੱਖਿਆ ਦਾ ਸੰਤੁਲਨ
- ਵੱਖ-ਵੱਖ ਕਲਾਇੰਟ ਵਾਤਾਵਰਣਾਂ ਨਾਲ ਅਨੁਕੂਲਤਾ ਨੂੰ ਯਕੀਨੀ ਬਣਾਉਣਾ

### ਕਾਰਜ: ਆਪਣੀ ਮਾਈਪ MCP ਐਪ ਬਣਾਓ

**ਪਰਿਦ੍ਰਸ਼:**
ਇੱਕ MCP ਸਰਵਰ ਅਤੇ ਕਲਾਇੰਟ ਬਣਾਓ, ਜਿੱਥੇ ਸਰਵਰ ਆਈਟਮਾਂ ਦੀ ਸੂਚੀ (ਜਿਵੇਂ ਕਿ ਫਾਇਲਾਂ ਜਾਂ ਦਸਤਾਵੇਜ਼) ਪ੍ਰੋਸੈਸ ਕਰਦਾ ਹੈ ਅਤੇ ਹਰ ਪ੍ਰੋਸੈਸ ਕੀਤਾ ਆਈਟਮ ਲਈ ਇੱਕ ਸੂਚਨਾ ਭੇਜਦਾ ਹੈ। ਕਲਾਇੰਟ ਹਰ ਆਈਟਮ ਦੀ ਸੂਚਨਾ ਨੂੰ ਅਸਲ ਸਮੇਂ ਵਿੱਚ ਦਿਖਾਏ।

**ਕਦਮ:**

1. ਇੱਕ ਸਰਵਰ ਟੂਲ ਲਾਗੂ ਕਰੋ ਜੋ ਸੂਚੀ ਨੂੰ ਪ੍ਰੋਸੈਸ ਕਰੇ ਅਤੇ ਹਰ ਆਈਟਮ ਲਈ ਸੂਚਨਾ ਭੇਜੇ।
2. ਇੱਕ ਕਲਾਇੰਟ ਲਾਗੂ ਕਰੋ ਜਿਸ ਵਿੱਚ ਸਨੇਹਾ ਹੈਂਡਲਰ ਹੋਵੇ ਜੋ ਸੂਚਨਾਵਾਂ ਨੂੰ ਤੁਰੰਤ ਦਿਖਾਏ।
3. ਆਪਣੇ ਲਾਗੂਕਰਨ ਨੂੰ ਟੈਸਟ ਕਰਨ ਲਈ ਸਰਵਰ ਅਤੇ ਕਲਾਇੰਟ ਦੋਹਾਂ ਨੂੰ ਚਲਾਓ ਅਤੇ ਸੂਚਨਾਵਾਂ ਨੂੰ ਵੇਖੋ।
