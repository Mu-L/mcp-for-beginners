# HTTPS സ്റ്റ്രീമിംഗ് മോഡൽ കോൺടെക്സ്റ്റ് പ്രോട്ടോക്കോൾ (MCP) ഉപയോഗിച്ച്

ഈ അധ്യായം HTTPS ഉപയോഗിച്ച് മോഡൽ കോൺടെക്‌സ്‌റ് പ്രോട്ടോക്കോൾ (MCP) ഉപയോഗിച്ച് സുരക്ഷിതവും സ്കേലബിളും റിയൽ-ടൈംസ്റ്റ്രീമിംഗ് നടപ്പിലാക്കുന്നതിനുള്ള വിശദമായ മാർഗ്ഗനിർദ്ദേശം നൽകുന്നു. സ്റ്റ്രീമിംഗിന്റെ പ്രേരണ, ലഭ്യമായ ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങൾ, MCP-ൽ സ്റ്റ്രീമബിൾ HTTP നടപ്പാക്കുന്നത്, സുരക്ഷാ മികച്ച പ്രാക്ടിസുകൾ, SSE-യിൽ നിന്ന് മാറ്റം എന്നിവ ഉൾപ്പെടുന്നു, കൂടാതെ നിങ്ങളുടെ സ്വന്തം സ്റ്റ്രീമിംഗ് MCP ആപ്ലിക്കേഷനുകൾ നിർമ്മിക്കുന്നതിനുള്ള പ്രായോഗിക മാർഗ്ഗനിർദ്ദേശങ്ങളും നൽകിയിരിക്കുന്നു.

## ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങൾയും MCP-ൽ സ്റ്റ്രീമിംഗും

ഈ വിഭാഗം MCP-ൽ ലഭ്യമായ വ്യത്യസ്ത ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങൾ പരിശോധിക്കുന്നു, കൂടാതെ ക്ലയന്റുകളുടെയും സർവറുകളുടെയും ഇടയിൽ റിയൽ-ടൈം കമ്മ്യൂണിക്കേഷനു സ്റ്റ്രീമിംഗ് സവിശേഷതകൾ പൂർണ്ണമാക്കുന്നതിൽ അവയുടെ പങ്ക് വ്യക്തമാക്കുന്നു.

### ഒരു ട്രാൻസ്പോർട്ട് മെക്കാനിസം എന്താണ്?

ഒരു ട്രാൻസ്പോർട്ട് മെക്കാനിസം ക്ലയന്റും സർവറും തമ്മിൽ ഡാറ്റ എങ്ങനെ കൈമാറപ്പെടുന്നതെന്ന് നിർവചിക്കുന്നു. MCP വ്യത്യസ്ത അന്തരീക്ഷങ്ങൾക്കും ആവശ്യങ്ങൾക്കും അനുയോജ്യമായ നിരവധി ട്രാൻസ്പോർട്ട് തരം പിന്തുണക്കുന്നു:

- **stdio**: സ്റ്റാൻഡേർഡ് ഇൻപുട്ട്/ഔട്ട്പുട്ട്, പ്രാദേശികവും CLI ആധാരിത ഉപകരണങ്ങൾക്കും അനുയോജ്യം. ലളിതമായതു കൊണ്ട് വെബ്ബ് അല്ലെങ്കിൽ ക്ലൗഡിനായി ഉചിതമല്ല.
- **SSE (സെർവർ-സെന്റ് ഇവന്റുകൾ)**: HTTP മുഖേന സർവർകൾക്ക് ക്ലയന്റുകൾക്ക് റിയൽ-ടൈം അപ്ഡേറ്റുകൾ പുഷ് ചെയ്യാൻ അനുവദിക്കുന്നു. വെബ് UI-കൾക്കു നല്ലത്, പക്ഷേ സ്കേലബിലിറ്റിയിലും ലReusable-ലും പരിമിതമാണ്. MCP സ്പെസിഫിക്കേഷൻ 2025-06-18 മുതൽ സ്റ്റാൻഡ്ഐൻഡ് SSE ട്രാൻസ്പോർട്ട് ഒഴിവാക്കി "Streamable HTTP" ട്രാൻസ്പോർട്ടോടെ പകരം വച്ചു.
- **Streamable HTTP**: ആധുനിക HTTP അടിസ്ഥാനത്തിലുള്ള സ്റ്റ്രീമിംഗ് ട്രാൻസ്പോർട്ട്, അറിയിപ്പുകളും മെച്ചപ്പെട്ട സ്കേലബിലിറ്റിയും പിന്തുണയ്‌ക്കുന്നു. പലത്തരം പ്രൊഡക്ഷൻ ക്ലൗഡ് സാഹചര്യങ്ങൾക്കായി ശുപാർശ ചെയ്യപ്പെടുന്നത്.

### താരതമ്യ പട്ടിക

കൂടുതൽ മനസ്സിലാക്കുന്നതിനായി ഈ ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങളിലുണ്ടായ വ്യത്യാസങ്ങൾ താഴെയുള്ള പട്ടികയിൽ നോക്കുക:

| Transport         | റിയൽ-ടൈം അപ്ഡേറ്റുകൾ | സ്റ്റ്രീമിംഗ് | സ്കേലബിലിറ്റി | ഉപയോഗ കേസ്              |
|-------------------|---------------------|-----------|--------------|--------------------------|
| stdio             | ഇല്ല                   | ഇല്ല        | കുറഞ്ഞത്      | പ്രാദേശിക CLI ഉപകരണങ്ങൾ |
| SSE               | ഉണ്ട്                   | ഉണ്ട്       | മധ്യസ്ഥം     | വെബ്, റിയൽ-ടൈം അപ്ഡേറ്റുകൾ |
| Streamable HTTP   | ഉണ്ട്                   | ഉണ്ട്       | ഉയർന്നത്     | ക്ലൗഡ്, മൾട്ടി-ക്ലയന്റ്    |

> **ടിപ്പ്:** ശരിയായ ട്രാൻസ്പോർട്ട് തെരഞ്ഞെടുക്കുന്നത് പ്രകടനം, സ്കേലബിലിറ്റി, ഉപയോക്തൃ അനുഭവം എന്നിവയെ ബാധിക്കുന്നു. ആധുനികവും സ്കേലബിൾ ക്ലൗഡ്-സന്നദ്ധ ആപ്പ്‌സിനായി **Streamable HTTP** ശുപാർശ ചെയ്യപ്പെടുന്നു.

മുൻ അധ്യായങ്ങളിൽ നിങ്ങൾക്ക് കാണിച്ച stdio, SSE ട്രാൻസ്പോർട്ടുകളും ഈ അധ്യായത്തിൽ കവർ ചെയ്ത Streamable HTTP ട്രാൻസ്പോർട്ടും ശ്രദ്ധിക്കുക.

## സ്റ്റ്രീമിംഗ്: ആശയങ്ങളും പ്രേരണയും

സ്റ്റ്രീമിംഗ് പിന്തുടരുന്നതിലെ അടിസ്ഥാന ആശയങ്ങളും പ്രേരണകളും മനസ്സിലാക്കുക ഫലപ്രദമായ റിയൽ-ടൈം കമ്മ്യൂണിക്കേഷൻ സിസ്റ്റങ്ങൾ നടപ്പിലാക്കുന്നതിനും അത്യന്താപേക്ഷിതമാണ്.

**സ്റ്റ്രീമിംഗ്** എന്നത് നെറ്റ്‌വർക്ക് പ്രോഗ്രാമിങ്ങിൽ ഒരു സാങ്കേതിക വിദ്യയാണ്, ഇത് മുഴുവൻ പ്രതികരണം തയ്യാറാകുന്നതിന് പകരം ചെറിയ, നിയന്ത്രണയോഗ്യമായ കഷ്ണങ്ങളായി അല്ലെങ്കിൽ സംഭവങ്ങളുടെ തുടർച്ചയായി ഡാറ്റ അയയ്ക്കാനും സ്വീകരിക്കാനും അനുവദിക്കുന്നു. ഇത് പ്രത്യേകിച്ച് ഉപകാരപ്രദമാണ്:

- വലിയ ഫയലുകളും ഡാറ്റാസെറ്റുകളുമോൾക്കുള്ളപ്പോൾ.
- റിയൽ-ടൈം അപ്ഡേറ്റുകൾ (ഉദാ: ചാറ്റ്, പ്രോഗ്രസ് ബാറുകൾ).
- നീണ്ടകാലം നടത്തുന്ന കണക്കുകൾ, ഉപയോക്താവിനെ അറിയിക്കുന്നതിനായുള്ളത്.

ഉയർന്ന തലത്തിൽ സ്റ്റ്രീമിംഗിനെ കുറിച്ച് നിങ്ങൾ അറിയേണ്ടത്:

- ഡാറ്റ പരിണാമങ്ങളായി ഡെലിവർ ചെയ്യപ്പെടുന്നു, ഒരേസമയം എല്ലാവരും ലഭിക്കുന്നതല്ല.
- ക്ലയന്റ് ഡാറ്റ ലഭിച്ചതുപോലെയാണ് പ്രോസസ്സ് ചെയ്യുന്നത്.
- അനുഭവം മെച്ചപ്പെടുത്താനും ക്ഷാമം കുറയ്ക്കാനും സഹായിക്കുന്നു.

### 왜 스트리밍을 사용합니까?

스트리밍을 사용하는 이유는 다음과 같습니다:

- ഉപയോക്താക്കൾക്ക് ഉടനെ ഫീഡ്ബാക്ക് ലഭിക്കുന്നു, അവസാനത്തിലും അല്ല.
- റിയൽ-ടൈം ആപ്ലിക്കേഷനുകളും പ്രതികരിക്കുന്ന UI-കളും സാധ്യമാക്കുന്നു.
- നെറ്റ്‌വർക്ക് റീസോഴ്സുകളും കംപ്യൂട്ട് കരുത്തും കൂടുതൽ കാര്യക്ഷമമായി ഉപയോഗിക്കുന്നു.

### ലളിതമായ ഉദാഹരണം: HTTP സ്റ്റ്രീമിംഗ് സർവർ & ക്ലയന്റ്

ഇങ്ങനെ സ്റ്റ്രീമിംഗ് എങ്ങനെ നടപ്പിലാക്കാമെന്ന് ലളിതമായ ഒരു ഉദാഹരണം:

#### പൈതൺ

**സർവർ (Python, FastAPI, StreamingResponse ഉപയോഗിച്ച്):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ക്ലയന്റ് (Python, requests ഉപയോഗിച്ച്):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ഈ ഉദാഹരണം സർവർ ഒരുസംഖ്യ സന്ദേശങ്ങൾ ഒരുക്കുമ്പോൾ ക്ലയന്റിനെ അയയ്ക്കുന്നു, എല്ലാവരും ഒരേസമയം ലഭിക്കുകക്കായി കാത്തിരിക്കാതെ.

**എങ്ങനെ പ്രവർത്തിക്കുന്നു:**

- സർവർ ഓരോ സന്ദേശവും ദിവസങ്ങളായി നൽകുന്നു.
- ക്ലയന്റ് ഓരോ കഷ്ണവും എത്തിച്ചപ്പോൾ സ്വീകരിച്ച് പ്രിന്റ് ചെയ്യുന്നു.

**അവശ്യങ്ങൾ:**

- സർവർ streamable response ഉം (ഉദാ: FastAPI-യിലുള്ള `StreamingResponse`) ഉപയോഗിക്കണം.
- ക്ലയന്റ് stream ആയി (requests-ൽ `stream=True`) പ്രതികരണം പ്രോസസ്സ് ചെയ്യണം.
- Content-Type സാധാരണയായി `text/event-stream` അല്ലെങ്കിൽ `application/octet-stream` ആകുന്നു.

#### ജാവ

**സർവർ (Java, Spring Boot, Server-Sent Events ഉപയോഗിച്ച്):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ക്ലയന്റ് (Java, Spring WebFlux WebClient ഉപയോഗിച്ച്):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java നടപ്പാക്കൽ കുറിപ്പുകൾ:**

- Spring Boot ന്റെ reactive stack `Flux` ഉപയോഗിച്ച് സ്റ്റ്രീമിംഗിനായി
- `ServerSentEvent` പ്രോഗ്രാമറ്റിക് ഇവന്റ് ടൈപ്പോടുകൂടിയ ഘടനയുള്ള സ്റ്റ്രീമിംഗ് നൽകുന്നു
- `WebClient` ൽ `bodyToFlux()` ഉപയോഗിച്ച് reactive stream സ്വീകരിക്കുന്നു
- `delayElements()` ഇവന്റെ ഇടയിൽ പ്രോസസ്സിംഗ് സമയവും സിമുലേറ്റ് ചെയ്യുന്നു
- ഇവന്റുകൾക്ക് (ഉദാ: `info`, `result`) തരം നൽകാം ക്ലയന്റ് മെച്ചപ്പെടുത്തിയ കൈകാര്യംക്ക്

### താരതമ്യം: ക്ലാസിക് സ്റ്റ്രീമിംഗ് vs MCP സ്റ്റ്രീമിംഗ്

"ക്ലാസിക്" രീതിയിലെ സ്റ്റ്രീമിംഗ് എങ്ങനെ പ്രവർത്തിക്കുന്നുവെന്നും MCP-ൽ സ്റ്റ്രീമിംഗ് എങ്ങനെ പ്രവർത്തിക്കുന്നു എന്നതിന്റെ വ്യത്യാസം ഇങ്ങനെ കാണാം:

| സവിശേഷത            | ക്ലാസിക് HTTP സ്റ്റ്രീമിംഗ്        | MCP സ്റ്റ്രീമിംഗ് (അറിയിപ്പുകൾ)         |
|--------------------|-----------------------------|-------------------------------------|
| പ്രധാന പ്രതികരണം     | ചങ്കുചെയ്ത്                    | ഒറ്റത്തവണ, അവസാനം                  |
| പ്രോഗ്രസ് അപ്ഡേറ്റ്‌സ് | ഡാറ്റ കഷ്ണങ്ങളായി അയക്കുന്നു      | അറിയിപ്പുകളായി അയയ്ക്കുന്നു             |
| ക്ലയന്റ് ആവശ്യങ്ങൾ   | സ്റ്റ്രീം പ്രോസസ്സിംഗ് വേണം          | സന്ദേശ ഹാൻഡ്ലർ നടപ്പിലാക്കണം          |
| ഉപയോഗ കേസ്          | വലിയ ഫയലുകൾ, AI ടോകൺ സ്റ്റ്രീമുകൾ | പ്രോഗ്രസ്, ലോഗുകൾ, റിയൽ-ടൈം ഫീഡ്ബാക്ക്  |

### പ്രധാന വ്യത്യാസങ്ങൾ

കൂടാതെ ചില പ്രധാന വ്യത്യാസങ്ങൾ:

- **കമ്മ്യൂണിക്കൽ ഡിസൈൻ:**
  - ക്ലാസിക് HTTP സ്റ്റ്രീമിംഗ്: ലളിതമായ ചങ്കുചെയ്‌ത ട്രാൻസ്ഫർ എഞ്ചിനിയറിംഗ് ഉപയോക്ത ബാഹ്യ ഡാറ്റ അയയ്ക്കാൻ.
  - MCP സ്റ്റ്രീമിംഗ്: JSON-RPC പ്രോട്ടോക്കോളുമായി ഘടനാപരമായ അറിയിപ്പ് സിസ്റ്റം ഉപയോഗിക്കുന്നു.

- **സന്ദേശ ഫോർമാറ്റ്:**
  - ക്ലാസിക് HTTP: പ്ലെയ്ൻ ടെക്സ്റ്റ് കഷ്ണങ്ങൾ, പുതിയ വരികൾ.
  - MCP: ഘടനാപരമായ LoggingMessageNotification ഒബ്‌ജക്റ്റുകൾ മെറ്റഡേറ്റയോടെ.

- **ക്ലയന്റ് നടപ്പാക്കൽ:**
  - ക്ലാസിക് HTTP: ലളിതമായ stream process ചെയ്യുന്ന ক্লയന്റ്.
  - MCP: സന്ദേശ തരം തിരിച്ചറിയുന്ന കൂടുതൽ സങ്കീർണ്ണമായ സന്ദേശ ഹാൻഡ്ലർ.

- **പ്രോഗ്രസ് അപ്ഡേറ്റുകൾ:**
  - ക്ലാസിക് HTTP: പ്രധാന റസ്പോൺസ് സ്റ്റ്രീമിന്റെ ഭാഗമാണ്.
  - MCP: അറിയിപ്പ് സന്ദേശങ്ങളായി വ്യത്യസ്തമായി അയക്കുന്നു; പ്രധാന ഫലം അവസാനം.

### ശുപാർശകൾ

ക്ലാസിക് സ്റ്റ്രീമിങ്ങും MCP സ്റ്റ്രീമിങ്ങും നടപ്പിലാക്കലിൽ തിരഞ്ഞെടുക്കുന്നതിനെക്കുറിച്ച് ഞങ്ങൾ ശുപാർശ ചെയ്യുന്ന കാര്യങ്ങൾ:

- **ലളിതമായ സ്റ്റ്രീമിംഗ് ആവശ്യങ്ങൾക്ക്:** ക്ലാസിക് HTTP സ്റ്റ്രീമിംഗ് ലളിതമാണ്, അടിസ്ഥാന ആവശ്യങ്ങൾക്കു മതി.

- **സങ്കീർണ്ണവും ഇന്ററാക്റ്റീവ് ആപ്പുകൾക്കായി:** MCP സ്റ്റ്രീമിംഗ് കൂടുതൽ ഘടനാപരവും സമൃദ്ധ മെടാഡേറ്റയും അറിയിപ്പുകളും ഫൈനൽ ഫലങ്ങളും സ്പർശാതീതമാക്കുന്നു.

- **AI ആപ്ലിക്കേഷനുകൾക്ക്:** MCP-യുടെ അറിയിപ്പ് സിസ്റ്റം നീണ്ടകാലം റണ്ണിംഗ് AI ടാസ്കുകളിൽ ഉപയോക്താക്കളെ പ്രോഗ്രസ്സ് അറിയിക്കാനാകൂ.

## MCP-ൽ സ്റ്റ്രീമിംഗ്

നിങ്ങൾക്ക് ഇതുവരെ ക്ലാസിക്കൽ സ്റ്റ്രീമിംഗ്ും MCP സ്റ്റ്രീമിംഗ് വ്യത്യാസവും ശുപാർശകളും കണ്ടിട്ടുണ്ട്. ഇനി MCP-ൽ സ്റ്റ്രീമിംഗ് എങ്ങനെ ഉപയോഗിക്കാമെന്ന് വിശദീകരിക്കാം.

MCP ഫ്രെയിംവർക്കിനുള്ളിൽ സ്റ്റ്രീമിംഗ് എങ്ങനെ പ്രവർത്തിക്കുന്നു എന്ന് മനസ്സിലാക്കുക, നീണ്ടകാല പ്രവർത്തനങ്ങളിൽ ഉപയോക്താക്കൾക്ക് റിയൽ-ടൈം ഫീഡ്ബാക്ക് നൽകുന്ന പ്രതികരിക്കുന്ന ആപ്ലിക്കേഷൻസ് നിർമ്മിക്കാൻ അത്യാവശ്യമാണ്.

MCP-യിൽ, പ്രധാന ഫലം ചങ്കുകളായി അയയ്ക്കുക എന്നല്ല, ടൂൾ ഒരു അഭ്യർത്ഥന പ്രോസസ്സ് ചെയ്യുമ്പോൾ ക്ലയന്റിന് **അറിയിപ്പുകൾ** അയയ്ക്കുന്നതാണ് സ്റ്റ്രീമിംഗ്. അറിയിപ്പുകൾക്ക് പ്രോഗ്രസ് അപ്ഡേറ്റുകൾ, ലോഗുകൾ, മറ്റ് ഇവന്റുകൾ ഉൾപ്പെടാം.

### എങ്ങനെ പ്രവർത്തിക്കുന്നു

പ്രധാന ഫലം ഒറ്റത്തവണയായ അഭ്യർത്ഥനയായി അയയ്ക്കുന്നു. എന്നാൽ പ്രോസസ്സിംഗ് സമയത്ത് അറിയിപ്പുകൾ പ്രത്യേകം സന്ദേശങ്ങളായി അയച്ചുകൊണ്ട് ക്ലൈന്റ് പരിചരിക്കാനാവും. ക്ലയന്റ് ഈ അറിയിപ്പുകൾ കൈകാര്യം ചെയ്ത് പ്രദർശിപ്പിക്കാൻ കഴിവുള്ളതായിരിക്കണം.

## അറിയിപ്പ് എന്താണ്?

നാം "അറിയിപ്പ്" എന്ന് പറഞ്ഞു, MCP പരിപ്രശ്നത്തിൽ അതിന്റെ അർഥം എന്താണ്?

ഒരു അറിയിപ്പ് ഒരു നീണ്ട നിലം പ്രവർത്തനത്തിനിടയിൽ പുരോഗതി, സ്ഥിതി, അല്ലെങ്കിൽ മറ്റ് ഇവന്റുകൾ കൈമാറാൻ സർവറിൽ നിന്ന് ക്ലയന്റിലേക്ക് അയക്കുന്ന സന്ദേശമാണ്. അറിയിപ്പുകൾ സുതാര്യതയും ഉപയോക്തൃ അനുഭവവും മെച്ചപ്പെടുത്തുന്നു.

ഉദാഹരണം, ക്ലയന്റ് സർവറുമായി ആദ്യ ഹാൻഡ്‌ഷേക്ക് നടത്തിയപ്പോൾ അറിയിപ്പ് അയയ്ക്കണം.

ഒരു അറിയിപ്പ് JSON സന്ദേശമായി ഇങ്ങനെ കാണും:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

അറിയിപ്പുകൾ MCP-ൽ ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) എന്ന വിഷയത്തിൽ (ടോപ്പിക്കിൽ) അതിന്റെ ഭാഗമായി ആണ്.

ലോഗിംഗ് പ്രവർത്തിക്കുന്നതിന് സർവർ മികച്ച രീതിയിൽ ഫീച്ചർ / കഴിവായി ഇത് എനേബിൾ ചെയ്യണം:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> SDK അനുസരിച്ച് ലോഗിംഗ് സ്വയം പ്രവർത്തനക്ഷമമാകാം, അല്ലെങ്കിൽ സൈറ്റർവറിലെ കോൺഫിഗറേഷനിൽ ഇത് വ്യക്തമായി അനുവദിക്കണം.

വിവിധ തരത്തിലുള്ള അറിയിപ്പുകൾ ഉണ്ട്:

| നില        | വിവരണം                        | ഉദാഹരണ ഉപയോഗം               |
|------------|-------------------------------|------------------------------|
| debug      | വിശദമായ ഡീബഗ്ഗിംഗ് വിവരങ്ങൾ    | ഫംഗ്ഷൻ എൻട്രി/എക്സിറ്റ് പോയിന്റുകൾ |
| info       | സാധാരണ വിവരസൂചനകൾ          | പ്രവർത്തന പുരോഗതി അപ്ഡേറ്റുകൾ       |
| notice     | സാധാരണ പക്ഷെ ഗൗരവമുള്ള ഇവന്റ്‌സ് | കോൺഫിഗറേഷൻ മാറ്റങ്ങൾ           |
| warning    | മുന്നറിയിപ്പ് നിലകൾ               | പഴയ ഫീച്ചർ ഉപയോഗം           |
| error      | പിശക് നിലകൾ                   | പ്രവർത്തന പരാജയങ്ങൾ           |
| critical   | ഗൗരവമുള്ള പാളികൾ              | സിസ്റ്റം ഘടക പരാജയം            |
| alert      | ഉടൻ നടപടി സ്വീകരിക്കേണ്ടത്      | ഡാറ്റ കേടായതു കണ്ടെത്തൽ        |
| emergency  | സിസ്റ്റം ഉപയോഗിക്കാൻ പാടില്ല        | പൂർണ്ണ സിസ്റ്റം പരാജയം          |

## MCP-ൽ അറിയിപ്പുകൾ നടപ്പാക്കിയെടുക്കൽ

അറിയിപ്പുകൾ MCP-ൽ നടപ്പിലാക്കാൻ, സർവർ-ക്ലയന്റ് വശങ്ങൾ ഇരുവരും റിയൽ-ടൈം അപ്ഡേറ്റുകൾ കൈകാര്യം ചെയ്യാൻ ക്രമീകരിക്കണം. ഇത് നീണ്ട പ്രവർത്തനങ്ങളിൽ ഉപയോക്താക്കൾക്ക് ഉടൻ പ്രതികരണം നൽകാനുള്ള സൗകര്യം നൽകുന്നു.

### സർവർ വശം: അറിയിപ്പുകൾ അയയ്ക്കൽ

സർവർ വശം നിന്നു തുടങ്ങാം. MCP-യിൽ, അഭ്യർത്ഥന പ്രോസസ്സ് ചെയ്യുന്ന സമയത്ത് അറിയിപ്പുകൾ അയക്കാവുന്ന ആപ്ലിക്കേഷനായി ടൂളുകൾ നിർവ്വചിക്കുന്നു. സർവർ ക്യോൺടെക്സ്റ്റ് ഓબ્જക്ട് (സാധാരണ `ctx`) ഉപയോഗിച്ച് സന്ദേശങ്ങൾ ക്ലയന്റിൽ അയയ്ക്കുന്നു.

#### പൈതൺ

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

മുൻ ഉദാഹരണത്തിൽ `process_files` ടൂൾ ഓരോ ഫയൽ പ്രോസസ്സ് ചെയ്യുമ്പോൾ ക്ലയന്റിൽ മൂന്ന് അറിയിപ്പുകൾ അയക്കുന്നു. `ctx.info()` മെഥഡാണ് വിവരകുറിപ്പുകൾ അയയ്ക്കാൻ ഉപയോഗിക്കുന്നത്.

കൂടാതെ, അറിയിപ്പുകൾ ഉപയോഗിക്കാൻ, സർവർ ഒരു streaming transport (ഉദാ: `streamable-http`) ഉപയോഗിക്കണം, ക്ലയന്റ് ഒരു സന്ദേശ ഹാൻഡ്ലർ നടപ്പിലാക്കണം. `streamable-http` ട്രാൻസ്പോർട്ട് ഉപയോഗിക്കാൻ സർവറെ ക്രമീകരിക്കുന്ന വിധം:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ഈ .NET ഉദാഹരണത്തിൽ, `ProcessFiles` ടൂൾ Tool അറ്റ്രിബ്യൂട്ട് ഉപയോഗിച്ചാണ് പ്രവർത്തിക്കുന്നത്, ഓരോ ഫയൽ പ്രോസസ്സ് ചെയ്ത് മൂന്ന് അറിയിപ്പുകൾ ക്ലയന്റിൽ അയയ്ക്കുന്നു. `ctx.Info()` മെഥഡാണ് വിവരങ്ങൾ അയയ്ക്കാൻ.

.NET MCP സർവറിൽ അറിയിപ്പുകൾ പ്രവർത്തനക്ഷമമാക്കാൻ ട്രാൻസ്പോർട്ട് streaming ആക്കി ക്രമീകരിക്കുക:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### ക്ലയന്റ് വശം: അറിയിപ്പുകൾ സ്വീകരിക്കൽ

ക്ലയന്റ് ഒരു message handler നടപ്പിലാക്കി അറിയിപ്പുകൾ സ്വീകരിക്കുകയും പ്രദർശിപ്പിക്കുകയും വേണം.

#### പൈതൺ

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

മുകളിൽ കാണുന്ന കോഡിൽ, `message_handler` ഫംഗ്ഷൻ ഒരു വരവിലുള്ള സന്ദേശം അറിയിപ്പാണോ എന്ന് പരിശോധിക്കുന്നു. അതാവിൽ അറിയിപ്പ് പ്രിന്റ് ചെയ്യുന്നു; അല്ലാതായാൽ സാധാരണ സർവർ സന്ദേശമായി പ്രോസസ് ചെയ്യുന്നു. കൂടാതെ `ClientSession` initialization-ൽ `message_handler` അറിയിപ്പുകൾ കൈകാര്യം ചെയ്യാനായി ഉപയോഗിക്കുന്നു.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

.NET ഉദാഹരണത്തിൽ `MessageHandler` ഫംഗ്ഷൻ വരവിലെ സന്ദേശം അറിയിപ്പാണോ എന്ന് പരിശോധിക്കും. അതാവിൽ അറിയിപ്പ് പ്രിന്റ് ചെയ്യും; അല്ലെങ്കിൽ സാധാരണ സർവർ മസേജ് ആയി പ്രോസസ് ചെയ്യും. `ClientSessionOptions` ഉപയോഗിച്ച് message handler ക്ലയന്റ് സെഷനിൽ initialize ചെയ്യുന്നു.

അറിയിപ്പുകൾ പ്രവർത്തനക്ഷമമാക്കാൻ, സർവർ streamable transport (ഉദാ: `streamable-http`) ഉപയോഗിക്കുകയും ക്ലയന്റ് message handler നടപ്പിലാക്കുകയും വേണം.

## പ്രോഗ്രസ് അറിയിപ്പുകളും സാഹചരാരണങ്ങളും

ഈ വിഭാഗം MCP-ൽ പ്രോഗ്രസ് അറിയിപ്പുകളുടെ ആശയം, അവയെന്തിന് ആവശ്യമാണ്, Streamable HTTP ഉപയോഗിച്ച് എങ്ങനെ നടപ്പിലാക്കാം എന്നിവ വിശദീകരിക്കുന്നു. കൂടാതെ എങ്കിലും പഠനത്തിനായി പ്രായോഗിക അസൈൻമെന്റും നൽകുന്നു.

പ്രോഗ്രസ് അറിയിപ്പുകൾ നീണ്ടനേരം പ്രവർത്തനങ്ങളിൽ സർവറിൽ നിന്നു ക്ലയന്റിലേക്ക് റിയൽ-ടൈം സന്ദേശങ്ങളായി അയക്കുന്നവയാണ്. മുഴുവൻ പ്രക്രിയ പൂർത്തിയാകുന്നത് കാത്തിരിക്കാതെ, നിലവിലുള്ള സ്ഥിതിവിവരം ഉപയോക്താവിന് അറിയിക്കുന്നു. ഇത് സുതാര്യതയും ഉപയോക്തൃ അനുഭവവും മെച്ചപ്പെടുത്തിയതിനൊപ്പം ഡീബഗ്ഗിംഗും എളുപ്പമാക്കുന്നു.

**ഉദാഹരണം:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### പ്രോഗ്രസ് അറിയിപ്പുകൾ എന്തിന് ഉപയോഗിക്കണം?

പ്രോഗ്രസ് അറിയിപ്പുകൾക്ക് നിരവധി പ്രാധാന്യങ്ങൾ അടങ്ങിയിരിക്കുന്നു:

- **നല്ല ഉപയോക്തൃ അനുഭവം:** ജോലി പുരോഗമിക്കുമ്പോളും അപ്ഡേറ്റുകൾ കാണാൻ സാധിക്കുന്നു, അവസാനം മാത്രം അല്ല.
- **റിയൽ-ടൈം ഫീഡ്ബാക്ക്:** ക്ലയന്റുകൾ പ്രോഗ്രസ് ബാറുകൾ അല്ലെങ്കിൽ ലോഗുകൾ പ്രദർശിപ്പിക്കാനാകും, ആപ്പ് പ്രതികരിക്കുന്ന പോലെ തോന്നും.
- **ഡീബഗ്ഗിംഗ്, നിരീക്ഷണം എളുപ്പം:** ഡെവലപ്പർമാർക്കും ഉപയോക്താക്കൾക്കും പ്രക്രിയ എവിടെ മന്ദഗതിയാണ് എന്ന് കാണാം.

### പ്രോഗ്രസ് അറിയിപ്പുകൾ എങ്ങനെ നടപ്പിലാക്കാം

MCP-ൽ പ്രോഗ്രസ് അറിയിപ്പുകൾ നടപ്പിലാക്കുന്നത് ഇങ്ങനെ:

- **സർവറിൽ:** ഓരോ ഇനവും പ്രോസസ്സിംഗ് പൂർത്തിയാകുമ്പോളും `ctx.info()` അല്ലെങ്കിൽ `ctx.log()` ഉപയോഗിച്ച് അറിയിപ്പുകൾ അയയ്‌ക്കുക. ഇത് പ്രധാന ഫലം തയ്യാറാകുന്നതിന് മുൻപായി ക്ലയന്റിലേക്ക് സന്ദേശം അയയ്ക്കുന്നു.
- **ക്ലയന്റിൽ:** അറിയിപ്പുകൾ സ്വീകരിക്കുന്ന ഒരു message handler നടപ്പിലാക്കുക, അത arrival-ലെ അറിയിപ്പുകളെയും അന്തിമ ഫലത്തെയും വേർതിരിക്കുന്നു, പ്രദർശിപ്പിക്കുന്നു.

**സർവർ ഉദാഹരണം:**

#### പൈതൺ

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**ക്ലയന്റ് ഉദാഹരണം:**

#### പൈതൺ

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## സുരക്ഷാ പരിഗണനകൾ

HTTP അടിസ്ഥാനമുള്ള ട്രാൻസ്പോർട്ടുകൾ ഉപയോഗിച്ച് MCP സർവർകൾ നടപ്പിലാക്കുമ്പോൾ, സുരക്ഷ ഏറ്റവും പ്രധാനപ്പെട്ട ഒരു വിഷയം ആകുന്നു, പല ആക്രമണ വേളകളിലും സംരക്ഷണ സംവിധാനങ്ങളിലും ശ്രദ്ധ ആവശ്യമാണ്.

### അവലോകനം

HTTP മുഖേന MCP സെർവറുകൾ പ്രദർശിപ്പിക്കുന്നത് സുരക്ഷിതമാക്കേണ്ടതുണ്ട്. Streamable HTTP പുതിയ ആക്രമണ സാധ്യതകൾ തുറക്കുന്നു, അതിനാൽ സൂക്ഷ്മമായ കോൺഫിഗറേഷൻ ആവശ്യമാണ്.

### പ്രധാന കാര്യങ്ങൾ

- **Origin ഹെഡർ പരിശോധന:** DNS റിബൈൻഡിംഗ് ആക്രമണങ്ങൾ തടയാൻ ൽ `Origin` ഹെഡറെ എപ്പോഴും പരിശോധന ചെയ്യുക.
- **ലോക്കൽഹോസ്റ്റ് ബൈൻഡിംഗ്:** പ്രാദേശിക ഡെവലപ്‌മെന്റിനായി സർവർ എപ്പോഴും `localhost`-ലേക്ക് ബന്ധിപ്പിക്കുക, പൊതു ഇന്റർനെറ്റിൽ തുറക്കാൻ കഴിയാതെ.
- **പ്രാമാണീകരണം:** പ്രൊഡക്ഷൻ ഡിപ്ലോയ്മെന്റിനായി API കീകൾ, OAuth പോലുള്ള പ്രാമാണീകരണം നടപ്പിലാക്കുക.
- **CORS:** ക്രോസ്-ഓറിജിൻ റിസോഴ്‌സ് ഷെയറിംഗ് നയങ്ങൾ കൃത്യമായി ക്രമീകരിക്കുക, അനധികൃത പ്രവേശനം തടയാൻ.
- **HTTPS:** ട്രാഫിക് എൻക്രിപ്ഷൻ ഉറപ്പാക്കാൻ പ്രൊഡക്ഷനിൽ HTTPS ഉപയോഗിക്കുക.

### മികച്ച പ്രാക്ടിസുകൾ

- വരവിലെ അഭ്യർത്ഥനകളെ ഉറപ്പുള്ള പരിശോധനകൾക്ക് വിധേയമാക്കാതെ ഒരിക്കലും വിശ്വസിക്കരുത്.
- എല്ലാ പ്രവേശനങ്ങളും പിഴവുകളും രേഖപ്പെടുത്തുകയും നിരീക്ഷിക്കുകയും ചെയ്യുക.
- സുരക്ഷിതത്വം മെച്ചപ്പെടുത്താൻ ഡിപ്പൻഡൻസികൾregolari app ഡേറ്റകളും പാച്ചുകളും അപ്ഡേറ്റ് ചെയ്യുക.

### വെല്ലുവിളികൾ
- സുരക്ഷയും വികസന സുലഭതയും സമതുല്യമാക്കൽ
- വിവിധ ക്ലയന്റ് പരിസ്ഥിതികളുമായി പൊരുത്തം ഉറപ്പാക്കൽ

## SSE യിൽ നിന്നു Streamable HTTP ആയി അപ്ഗ്രേഡ് ചെയ്യുക

പ്രస్తుతం Server-Sent Events (SSE) ഉപയോഗിക്കുന്ന ആപ്ലിക്കേഷനുകൾക്ക് Streamable HTTP യിലേക്ക് മാറ്റം MCP ഉപയോഗങ്ങളുടെ വികസനം ഉറപ്പാക്കാനും കൂടുതൽ ശക്തമായ ശേഷികൾ നൽകാനും ഉത്തമമാണ്.

### അപ്ഗ്രേഡിന്റെ കാരണങ്ങൾ

SSE യിൽ നിന്ന് Streamable HTTP യിലേക്ക് അപ്ഗ്രേഡ് ചെയ്യേണ്ടത് രണ്ട് പ്രധാന കാരണം ഉണ്ട്:

- Streamable HTTP, SSE നൊപ്പം താരതമ്യപ്പെടുത്തുമ്പോൾ മികച്ച സ്കെയിലബിലിറ്റി, പൊരുത്തം, സമ്പുഷ്ടമായ അറിയിപ്പ് പിന്തുണ നൽകുന്നു.
- ഇത് പുതിയ MCP ആപ്ലിക്കേഷനുകള്ക്കുള്ള ശുപാർശ ചെയ്ത ട്രാൻസ്പോർട്ടാണ്.

### മൈഗ്രേഷൻ ഘട്ടങ്ങൾ

നിങ്ങളുടെ MCP ആപ്ലിക്കേഷനുകളിൽ SSE യിൽ നിന്ന് Streamable HTTP യിലേക്ക് മാറ്റം നടത്തേണ്ട വിധം:

- **സർവർ കോഡ് അപ്ഡേറ്റ് ചെയ്യുക** `mcp.run()`-ലിൽ `transport="streamable-http"` ഉപയോഗിക്കാൻ.
- **ക്ലയന്റ് കോഡ് അപ്ഡേറ്റ് ചെയ്യുക** SSE ക്ലയന്റിന് പകരം `streamablehttp_client` ഉപയോഗിക്കാൻ.
- **ക്ലയന്റിൽ മെസ്സജ് ഹാൻഡ്ലർ നടപ്പിലാക്കുക** അറിയിപ്പുകൾ പ്രക്രിയചെയ്യാൻ.
- **പഴയ ടൂളുകളും വർക്ക്‌ഫ്ലോകളുമായി പൊരുത്തം പരിശോധന ചെയ്യുക.**

### പൊരുത്തം നിലനിർത്തൽ

മൈഗ്രേഷൻ പ്രക്രിയയിൽ നിലവിലുള്ള SSE ക്ലയന്റുകളുമായി പൊരുത്തം നിലനിർത്തുക ശുപാർശ ചെയ്യപ്പെടുന്നു. ചില തന്ത്രങ്ങൾ:

- വ്യത്യസ്ത എൻഡ്പോയിന്‍റുകളിൽ രണ്ടു ട്രാൻസ്പോർട്ടുകളും നടപ്പാക്കികൊണ്ട് SSEയും Streamable HTTPയും ഒടുകളമായി പിന്തുണക്കാം.
- ക്ലയന്റുകൾ രൂപാന്തരം ചുരുക്കം പുതിയ ട്രാൻസ്പോർട്ടിലേക്ക് മറിക്കുന്നു.

### വെല്ലുവിളികൾ

മൈഗ്രേഷൻ സമയത്ത് താഴെ പറയുന്ന വെല്ലുവിളികൾ പരിഹരിക്കുക:

- എല്ലാ ക്ലയന്റുകളും അപ്ഡേറ്റ് ചെയ്യുക
- അറിയിപ്പ് ഡെലിവറി വ്യത്യാസങ്ങൾ കൈകാര്യം ചെയ്യുക

## സുരക്ഷാ പരിഗണനകൾ

ഏതൊരു സർവറും നടപ്പിലാക്കുമ്പോഴും, പ്രത്യേകിച്ച് MCP ലെ Streamable HTTP പോലുള്ള HTTP-ആധാരિત ട്രാൻസ്പോർട്ടുകൾ ഉപയോഗിക്കുന്നപ്പോൾ സുരക്ഷ പ്രധാനപ്പെട്ട പരിഗണനയാണ്.

MCP സർവർ HTTP വഴി അനാവരണം ചെയ്യുമ്പോൾ, സുരക്ഷ വിവിധ ആക്രമണ വഴികളും അവരോധന മാർഗ്ഗങ്ങളും വിശദമായി പ്രതിരോധിക്കേണ്ടതാണ്.

### അവലോകനം

MCP സർവർ ഓപ്പൺ ചെയ്യുന്നിടത്ത് സുരക്ഷ നിർണായകമാണ്. Streamable HTTP പുതിയ ആക്രമണ സാധ്യതകൾ കൊണ്ടുവരുന്നു, അതിനാൽ വ്യവസ്ഥാപിത രീതിയിലാണ് ക്രമീകരണം നിർബന്ധം.

പ്രധാന സുരക്ഷാ പരിഗണനകൾ:

- **Origin Header പരിശോധന**: DNS റീബൈൻഡിംഗ് ആക്രമണങ്ങൾ ഒഴിവാക്കാൻ എപ്പോഴും `Origin` ഹെഡർ പരിശോധിക്കുക.
- **Localhost ബൈൻഡിംഗ്**: ലോക്കൽ ഡെവലപ്പ്മെന്റിന് സർവറുകൾ `localhost`-ൽ ബൈൻഡ് ചെയ്യുക, പൊതു ഇന്റർനെറ്റിൽ തുറക്കാതിരിക്കാൻ.
- **അധികാരം ഉറപ്പിക്കൽ**: പ്രൊഡക്ഷൻ ഡിപ്പ്ലോയ്മെന്റുകൾക്ക് ഓതന്റിക്കേഷൻ (API കീകൾ, OAuth) നടപ്പിലാക്കുക.
- **CORS**: എക്സസ് നിയന്ത്രിക്കുന്ന Cors നയം ക്രമീകരിക്കുക.
- **HTTPS**: ട്രാഫിക് എൻക്രിപ്റ്റ് ചെയ്യാൻ പ്രൊഡക്ഷനിൽ HTTPS ഉപയോഗിക്കുക.

### മികച്ച രീതി

MCP സ്ട്രീമിംഗ് സർവറിൽ സുരക്ഷ നടപ്പിലാക്കുമ്പോൾ പാലിക്കേണ്ട മികച്ച പ്രകാരങ്ങൾ:

- വാക്കIncoming അഭ്യർത്ഥനകളെ തെറ്റായ വിശ്വാസം കാണിക്കരുത്.
- എല്ലാ ആക്സസുകളും പിശകുകൾക്കും ലോഗ് കൈവശപ്പെടുത്തുക ഒപ്പം നിരീക്ഷിക്കുക.
- സുരക്ഷാ ദുർബലതകൾ തടയുന്നതിന് ആശ്രിതങ്ങൾ നിതാന്തം അപ്ഡേറ്റ് ചെയ്യുക.

### വെല്ലുവിളികൾ

MCP സ്ട്രീമിംഗ് സർവറുകളിൽ സുരക്ഷ നടപ്പിലാക്കുമ്പോൾ നേരിടേണ്ട ചില വെല്ലുവിളികൾ:

- സുരക്ഷയും വികസന സൗകര്യവും ശരിയായ സമതുലനം കൈവരുത്തൽ
- വിവിധ ക്ലയന്റ് പരിസ്ഥിതികളുമായി പൊരുത്തം ഉറപ്പാക്കൽ

### അസൈൻമെന്റ്: നിങ്ങളുടെ സ്വന്തം സ്ട്രീമിംഗ് MCP ആപ്പ് നിർമ്മിക്കുക

**സീനാരിയോ:**
സർവർ ഈറ്റം (ഫയലുകൾ അല്ലെങ്കിൽ ഡോക്യുമെന്റുകൾ എന്നിവ) ലിസ്റ്റ് പ്രോസസ്സിങ് ചെയ്ത് ഓരോ ഐറ്റത്തിന്റെയും അറിയിപ്പുകൾ അയയ്ക്കുന്ന MCP സർവർ-ക്ലയന്റ് നിർമ്മിക്കുക. അറിയിപ്പുകൾ വരുമ്പോൾ ക്ലയന്റ് അവ ഡിസ്‌പ്ലേ ചെയ്യണം.

**ഘട്ടങ്ങൾ:**

1. ഒരു സർവർ ടൂൾ നടപ്പിലാക്കുക, ലിസ്റ്റ് പ്രോസസ്സിങ് ചെയ്ത് ഐറ്റങ്ങൾ prosess ചെയ്തതിന് അറിയിപ്പുകൾ അയയ്ക്കുക.
2. അറിയിപ്പുകൾ റിയൽ ടൈമിൽ കാണിക്കാൻ മെസ്സജ് ഹാൻഡ്ലർ ഉൾപ്പെട്ട ക്ലയന്റ് നടപ്പിലാക്കുക.
3. സർവർ-ക്ലയന്റ് രണ്ടു ഭാഗവും ഓടിച്ച് നിങ്ങളുടെ നടപ്പിലാക്കൽ പരീക്ഷിക്കുക, അറിയിപ്പുകൾ നിരീക്ഷിക്കുക.
