[Model Context Protocol (MCP)](https://github.com/modelcontextprotocol) — мощный стандартизированный фреймворк, оптимизирующий взаимодействие между большими языковыми моделями (LLM) и внешними инструментами, приложениями и источниками данных.  
В этом руководстве вы изучите основные концепции MCP. Вы узнаете о его архитектуре клиент-сервер, ключевых компонентах, механизмах коммуникации и лучших практиках реализации.

- **Явное согласие пользователя**: Все операции и доступ к данным требуют явного одобрения пользователя перед выполнением. Пользователь должен четко понимать, к каким данным будет осуществлен доступ и какие действия будут выполнены, с возможностью детального контроля разрешений и авторизаций.

- **Защита конфиденциальности данных**: Данные пользователя раскрываются только с явным согласием и должны быть защищены надежными средствами контроля доступа на протяжении всего взаимодействия. Реализации должны предотвращать несанкционированную передачу данных и строго соблюдать границы конфиденциальности.

- **Безопасность запуска инструментов**: Каждый вызов инструмента требует явного согласия пользователя с четким пониманием функций инструмента, параметров и возможного воздействия. Жесткие меры безопасности должны предотвращать непреднамеренное, небезопасное или вредоносное выполнение инструментов.

- **Безопасность транспортного уровня**: Все каналы связи должны использовать соответствующие механизмы шифрования и аутентификации. Удалённые соединения должны использовать защищённые протоколы передачи данных и надлежащий менеджмент учётных данных.

#### Рекомендации по реализации:

- **Управление разрешениями**: Реализуйте тонконастроенные системы разрешений, позволяющие пользователям контролировать, каким серверам, инструментам и ресурсам предоставляется доступ  
- **Аутентификация и авторизация**: Используйте безопасные методы аутентификации (OAuth, API-ключи) с правильным управлением токенами и сроками их действия  
- **Валидация ввода**: Проверяйте все параметры и входные данные по заданным схемам для предотвращения атак инъекций  
- **Ведение журналов аудита**: Поддерживайте полные логи всех операций для мониторинга безопасности и соблюдения нормативов

## Архитектура MCP: подробный взгляд

Экосистема MCP построена по модели клиент-сервер. Такая модульная структура позволяет ИИ-приложениям эффективно взаимодействовать с инструментами, базами данных, API и контекстуальными ресурсами. Разберем эту архитектуру на основные компоненты.

В основе MCP лежит архитектура клиент-сервер, в которой хост-приложение может подключаться к нескольким серверам:

```mermaid
flowchart LR
    subgraph "Ваш компьютер"
        Host["Хост с MCP (Visual Studio, VS Code, IDE, Инструменты)"]
        S1["Сервер MCP A"]
        S2["Сервер MCP B"]
        S3["Сервер MCP C"]
        Host <-->|"Протокол MCP"| S1
        Host <-->|"Протокол MCP"| S2
        Host <-->|"Протокол MCP"| S3
        S1 <--> D1[("Локальные\Данные Источник A")]
        S2 <--> D2[("Локальные\Данные Источник B")]
    end
    subgraph "Интернет"
        S3 <-->|"Веб API"| D3[("Удалённые\Сервисы")]
    end
```  
- **MCP Hosts (Хосты MCP)**: программы типа VSCode, Claude Desktop, IDE или ИИ-инструменты, желающие получать доступ к данным через MCP  
- **MCP Clients (Клиенты MCP)**: протокольные клиенты, поддерживающие 1:1 соединения с серверами  
- **MCP Servers (Серверы MCP)**: легковесные программы, предоставляющие конкретные возможности через стандартизованный Model Context Protocol  
- **Локальные источники данных**: файлы, базы данных и сервисы на вашем компьютере, к которым серверы MCP могут безопасно обращаться  
- **Удалённые сервисы**: внешние системы, доступные через интернет, с которыми серверы MCP могут взаимодействовать через API

Протокол MCP является развивающимся стандартом с версионированием по дате (формат ГГГГ-ММ-ДД). Текущая версия протокола — **2025-11-25**. Последние обновления спецификации можно посмотреть в [протокольной спецификации](https://modelcontextprotocol.io/specification/2025-11-25/)

### 1. Хосты

В Model Context Protocol (MCP) **Хосты** — это ИИ-приложения, которые служат основным интерфейсом для взаимодействия пользователей с протоколом. Хосты координируют и управляют подключениям к нескольким MCP серверам, создавая для каждого из них отдельного MCP клиента. Примеры Хостов:

- **ИИ-приложения**: Claude Desktop, Visual Studio Code, Claude Code  
- **Среды разработки**: IDE и редакторы кода с поддержкой MCP  
- **Кастомные приложения**: специализированные ИИ-агенты и инструменты

**Хосты** — это приложения, которые организуют взаимодействия с ИИ-моделями. Они:

- **Оркеструют ИИ-модели**: выполняют действия или взаимодействуют с LLM для генерации ответов и координации ИИ-процессов  
- **Управляют клиентскими подключениям**: создают и поддерживают по одному MCP клиенту для каждого сервера MCP  
- **Контролируют пользовательский интерфейс**: управляют ходом диалога, взаимодействиями с пользователем и отображением ответов  
- **Обеспечивают безопасность**: контролируют разрешения, ограничения безопасности и аутентификацию  
- **Обрабатывают согласие пользователя**: управляют пользовательским одобрением доступа к данным и выполнению инструментов

### 2. Клиенты

**Клиенты** — ключевые компоненты, поддерживающие выделенные соединения 1:1 между Хостами и MCP серверами. Каждый MCP клиент создаётся Хостом для подключения к конкретному MCP серверу, обеспечивая организованные и безопасные коммуникационные каналы. Наличие нескольких клиентов позволяет Хостам одновременно подключаться к множеству серверов.

**Клиенты** — это компоненты-коннекторы внутри хост-приложения. Они:

- **Обеспечивают протокольное общение**: отправляют JSON-RPC 2.0 запросы серверам с подсказками и инструкциями  
- **Ведут переговоры о возможностях**: согласовывают с серверами поддерживаемые функции и версии протокола при инициализации  
- **Управляют запуском инструментов**: обрабатывают запросы к инструментам от моделей и получают ответы  
- **Обрабатывают обновления в реальном времени**: принимают уведомления и обновления от серверов  
- **Форматируют ответы**: обрабатывают и форматируют ответы серверов для отображения пользователям

### 3. Серверы

**Серверы** — это программы, предоставляющие контекст, инструменты и возможности клиентам MCP. Они могут запускаться локально (на той же машине, что и Хост) или удалённо (на внешних платформах) и отвечают за обработку запросов клиентов и выдачу структурированных ответов. Серверы предоставляют специализированный функционал через стандартизованный Model Context Protocol.

**Серверы** — это сервисы, предоставляющие контекст и возможности. Они:

- **Регистрируют возможности**: регистрируют и открывают клиентам доступ к доступным примитивам (ресурсам, подсказкам, инструментам)  
- **Обрабатывают запросы**: принимают и выполняют вызовы инструментов, запросы ресурсов и подсказок от клиентов  
- **Предоставляют контекст**: дают контекстуальную информацию и данные для улучшения ответов моделей  
- **Управляют состоянием**: поддерживают состояние сессии и обеспечивают обработку с сохранением состояния при необходимости  
- **Отправляют уведомления в реальном времени**: сообщают клиентам дату об изменениях возможностей и обновлениях

Серверы могут разрабатываться любым желающим для расширения возможностей модели специализированным функционалом, и поддерживают как локальное, так и удалённое развёртывание.

### 4. Примитивы серверов

Серверы в Model Context Protocol (MCP) предоставляют три основных **примитива**, определяющих фундаментальные строительные блоки для богатых взаимодействий между клиентами, хостами и языковыми моделями. Эти примитивы задают типы контекстной информации и действий, доступных через протокол.

Серверы MCP могут предоставлять любое сочетание из следующих трёх основных примитивов:

#### Ресурсы 

**Ресурсы** — это источники данных, предоставляющие контекстную информацию для ИИ-приложений. Они представляют статический или динамический контент, который может улучшить понимание и принятие решений моделью:

- **Контекстные данные**: структурированная информация и контекст для потребления моделью ИИ  
- **Базы знаний**: репозитории документов, статьи, руководства и исследования  
- **Локальные источники данных**: файлы, базы данных и информация о локальной системе  
- **Внешние данные**: ответы API, веб-сервисы и удалённые данные систем  
- **Динамический контент**: данные в реальном времени, обновляющиеся в зависимости от внешних условий

Ресурсы идентифицируются URI и поддерживают обнаружение через методы `resources/list` и получение через `resources/read`:

```text
file://documents/project-spec.md
database://production/users/schema
api://weather/current
```
  
#### Подсказки

**Подсказки** — повторно используемые шаблоны, помогающие структурировать взаимодействия с языковыми моделями. Они предоставляют стандартизированные паттерны взаимодействия и шаблоны рабочих процессов:

- **Взаимодействия на основе шаблонов**: заранее структурированные сообщения и стартовые фразы для общения  
- **Шаблоны рабочих процессов**: стандартизированные последовательности для типовых задач и взаимодействий  
- **Примеры с несколькими образцами**: шаблоны на базе примеров для инструкций моделью  
- **Системные подсказки**: базовые подсказки, определяющие поведение и контекст модели  
- **Динамические шаблоны**: параметризованные подсказки, адаптирующиеся под конкретный контекст

Подсказки поддерживают подстановку переменных и обнаруживаются через `prompts/list`, а получаются через `prompts/get`:

```markdown
Generate a {{task_type}} for {{product}} targeting {{audience}} with the following requirements: {{requirements}}
```
  
#### Инструменты

**Инструменты** — исполняемые функции, которые языковые модели могут вызывать для выполнения конкретных действий. Они представляют "глаголы" экосистемы MCP, позволяя моделям взаимодействовать с внешними системами:

- **Исполняемые функции**: дискретные операции, вызываемые моделями с определёнными параметрами  
- **Интеграция с внешними системами**: вызовы API, запросы к базам данных, операции с файлами, вычисления  
- **Уникальная идентификация**: у каждого инструмента есть уникальное имя, описание и схема параметров  
- **Структурированный ввод/вывод**: инструменты принимают проверенные параметры и возвращают структурированные, типизированные ответы  
- **Возможности действий**: позволяют моделям выполнять реальные действия и получать актуальные данные

Инструменты описываются с помощью JSON Schema для валидации параметров и обнаруживаются через `tools/list`, вызываются через `tools/call`. Также инструменты могут содержать **иконки** как дополнительные метаданные для улучшения представления в интерфейсе пользователя.

**Аннотации инструментов**: инструменты поддерживают поведенческие аннотации (например, `readOnlyHint`, `destructiveHint`), которые описывают, является ли инструмент только для чтения или разрушительным, помогая клиентам принимать обоснованные решения о вызове инструмента.

Пример определения инструмента:

```typescript
server.tool(
  "search_products", 
  {
    query: z.string().describe("Search query for products"),
    category: z.string().optional().describe("Product category filter"),
    max_results: z.number().default(10).describe("Maximum results to return")
  }, 
  async (params) => {
    // Выполнить поиск и вернуть структурированные результаты
    return await productService.search(params);
  }
);
```
  
## Примитивы клиентов

В Model Context Protocol (MCP) **клиенты** могут предоставлять примитивы, которые позволяют серверам запрашивать дополнительные возможности у хост-приложения. Эти примитивы на стороне клиента обеспечивают более богатые и интерактивные реализации серверов с доступом к функциям ИИ моделей и взаимодействиям с пользователем.

### Сэмплирование

**Сэмплирование** позволяет серверам запрашивать генерацию дополнений от языковой модели клиента. Этот примитив дает серверам доступ к возможностям LLM без необходимости внедрять свои собственные зависимости моделей:

- **Независимый доступ к модели**: серверам не нужно включать SDK LLM или управлять доступом к модели  
- **Запуск ИИ с инициатором от сервера**: позволяет серверам автономно генерировать контент с помощью ИИ модели клиента  
- **Рекурсивные взаимодействия LLM**: поддерживает сложные сценарии, где серверам нужна помощь ИИ для обработки  
- **Генерация динамического контента**: позволяет создавать контекстные ответы с использованием модели хоста  
- **Поддержка вызова инструментов**: серверам можно передавать параметры `tools` и `toolChoice` для того, чтобы модель клиента могла вызывать инструменты во время сэмплирования

Сэмплирование вызывается через метод `sampling/complete`, где серверы отправляют клиентам запросы дополнений.

### Корневые каталоги

**Корни** обеспечивают стандартизированный способ для клиентов раскрывать серверам границы файловой системы, помогая серверам понимать, к каким директориям и файлам у них есть доступ:

- **Границы файловой системы**: определяют границы, в рамках которых серверы могут оперировать  
- **Контроль доступа**: помогают серверам понять, к каким директориям и файлам предоставлен доступ  
- **Динамические обновления**: клиенты сообщают серверам об изменениях в списке корней  
- **Идентификация на основе URI**: корни используют URI формата `file://` для определения доступных директорий и файлов

Корни обнаруживаются через метод `roots/list`, а клиенты уведомляют серверы через `notifications/roots/list_changed` при изменениях корней.

### Инициирование (Elicitation)  

**Инициирование** позволяет серверам запрашивать у пользователей дополнительную информацию или подтверждение через интерфейс клиента:

- **Запросы пользовательского ввода**: серверы могут просить дополнительную информацию для выполнения инструментов  
- **Диалоги подтверждения**: запрашивают одобрение пользователя для чувствительных или важных операций  
- **Интерактивные рабочие процессы**: позволяют создавать поэтапные взаимодействия с пользователем  
- **Динамический сбор параметров**: собирают недостающие или опциональные параметры во время выполнения инструмента

Запросы инициирования осуществляются с помощью метода `elicitation/request` для сбора пользовательского ввода через интерфейс клиента.

**Инициирование через URL**: серверы также могут запрашивать взаимодействие с пользователем через URL, направляя пользователя на внешние веб-страницы для аутентификации, подтверждения или ввода данных.

### Логирование

**Логирование** позволяет серверам отправлять структурированные сообщения журналов клиентам для отладки, мониторинга и операционной прозрачности:

- **Поддержка отладки**: позволяет серверам предоставлять подробные логи выполнения для устранения неполадок  
- **Оперативный мониторинг**: отправка статуса и метрик производительности клиентам  
- **Отчеты об ошибках**: предоставление детализированной информации об ошибках и диагностике  
- **Аудит**: создание полноценных журналов операций и принятых сервером решений

Сообщения логов отправляются клиентам для прозрачности работы серверов и поддержки отладки.

## Поток информации в MCP

Model Context Protocol (MCP) определяет структурированный поток информации между хостами, клиентами, серверами и моделями. Понимание этого потока помогает разобраться, как обрабатываются пользовательские запросы и как внешние инструменты и данные интегрируются в ответы модели.
- **Хост инициирует соединение**  
  Приложение-хост (например, IDE или интерфейс чата) устанавливает соединение с сервером MCP, как правило, через STDIO, WebSocket или другой поддерживаемый транспорт.

- **Переговоры возможностей**  
  Клиент (встроенный в хост) и сервер обмениваются информацией о поддерживаемых функциях, инструментах, ресурсах и версиях протокола. Это обеспечивает понимание обеими сторонами доступных возможностей для сессии.

- **Запрос пользователя**  
  Пользователь взаимодействует с хостом (например, вводит подсказку или команду). Хост собирает этот ввод и передает его клиенту для обработки.

- **Использование ресурса или инструмента**  
  - Клиент может запрашивать дополнительный контекст или ресурсы у сервера (например, файлы, записи в базе данных или статьи базы знаний) для обогащения понимания модели.  
  - Если модель определяет, что необходим инструмент (например, чтобы получить данные, выполнить вычисления или вызвать API), клиент отправляет серверу запрос на вызов инструмента с указанием имени инструмента и параметров.

- **Выполнение на сервере**  
  Сервер принимает запрос ресурса или инструмента, выполняет необходимые операции (например, вызов функции, запрос к базе данных или получение файла) и возвращает результаты клиенту в структурированном формате.

- **Генерация ответа**  
  Клиент интегрирует ответы сервера (данные ресурса, результаты инструмента и т.д.) в текущую взаимодействие с моделью. Модель использует эту информацию для генерации комплексного и контекстуально релевантного ответа.

- **Представление результата**  
  Хост получает финальный вывод от клиента и отображает его пользователю, зачастую включая сгенерированный моделью текст и любые результаты выполнения инструментов или обращения к ресурсам.

Этот процесс позволяет MCP поддерживать продвинутые, интерактивные и контекстно-зависимые AI-приложения, плавно связывая модели с внешними инструментами и источниками данных.

## Архитектура протокола и уровни

MCP состоит из двух отдельных архитектурных уровней, которые работают совместно, обеспечивая полноценную коммуникационную платформу:

### Уровень данных

**Уровень данных** реализует основной протокол MCP, используя **JSON-RPC 2.0** в качестве основы. Этот уровень определяет структуру сообщений, семантику и схемы взаимодействия:

#### Основные компоненты:

- **Протокол JSON-RPC 2.0**: Вся коммуникация использует стандартизированный формат сообщений JSON-RPC 2.0 для вызовов методов, ответов и уведомлений  
- **Управление жизненным циклом**: Обрабатывает инициализацию соединения, переговоры возможностей и завершение сессии между клиентами и серверами  
- **Примитивы сервера**: Позволяют серверам предоставлять базовый функционал через инструменты, ресурсы и подсказки  
- **Примитивы клиента**: Позволяют серверам запрашивать сэмплы у LLM, получать ввод от пользователя и отправлять лог-сообщения  
- **Уведомления в реальном времени**: Поддержка асинхронных уведомлений для динамических обновлений без опроса

#### Ключевые особенности:

- **Переговоры версий протокола**: Использует версионирование по дате (ГГГГ-ММ-ДД) для обеспечения совместимости  
- **Обнаружение возможностей**: Клиенты и серверы обмениваются информацией о поддерживаемых функциях при инициализации  
- **Сессионное состояние**: Сохраняет состояние соединения для нескольких взаимодействий для сохранения контекста

### Уровень транспорта

**Уровень транспорта** управляет каналами связи, формированием сообщений и аутентификацией между участниками MCP:

#### Поддерживаемые механизмы транспорта:

1. **Транспорт STDIO**:  
   - Использует стандартные потоки ввода/вывода для прямой коммуникации между процессами  
   - Оптимален для локальных процессов на одной машине без сетевых накладных расходов  
   - Часто используется для локальных реализаций серверов MCP

2. **Потоковый HTTP-транспорт**:  
   - Использует HTTP POST для сообщений от клиента к серверу  
   - Опционально поддерживает Server-Sent Events (SSE) для потоковой передачи от сервера к клиенту  
   - Позволяет коммуникацию с удалёнными серверами через сеть  
   - Поддерживает стандартную HTTP-аутентификацию (токены bearer, ключи API, кастомные заголовки)  
   - MCP рекомендует OAuth для безопасной аутентификации на основе токенов

#### Абстракция транспорта:

Уровень транспорта абстрагирует детали коммуникации от уровня данных, позволяя использовать единый формат сообщений JSON-RPC 2.0 на всех видах транспорта. Эта абстракция позволяет приложениям без проблем переключаться между локальными и удалёнными серверами.

### Вопросы безопасности

Реализации MCP должны придерживаться нескольких критических принципов безопасности, чтобы обеспечить безопасное, надежное и защищенное взаимодействие на всех операциях протокола:

- **Согласие и контроль пользователя**: Пользователи должны давать явное согласие перед доступом к данным или выполнением операций. Им должен быть предоставлен ясный контроль над тем, какие данные предоставляются и какие действия разрешены, при поддержке удобных интерфейсов для просмотра и одобрения действий.

- **Конфиденциальность данных**: Данные пользователя должны быть доступны только с его явного согласия и защищены соответствующими средствами контроля доступа. Реализации MCP должны предотвращать несанкционированную передачу данных и обеспечивать сохранность конфиденциальности во всех взаимодействиях.

- **Безопасность инструментов**: Перед вызовом любого инструмента требуется явное согласие пользователя. Пользователи должны чётко понимать назначение каждого инструмента, а также должны соблюдаться строгие меры безопасности, предотвращающие непреднамеренное или небезопасное выполнение инструментов.

Соблюдение этих принципов безопасности обеспечивает доверие пользователей, защиту конфиденциальности и безопасность при использовании MCP, сохраняя при этом мощные возможности интеграции AI.

### Пример на .NET: создание простого MCP-сервера с инструментами

Практический пример на .NET, демонстрирующий создание простого MCP-сервера с кастомными инструментами. Этот пример показывает, как определить и зарегистрировать инструменты, обрабатывать запросы и подключать сервер с помощью протокола Model Context Protocol.

```csharp
using System;
using System.Threading.Tasks;
using ModelContextProtocol.Server;
using ModelContextProtocol.Server.Transport;
using ModelContextProtocol.Server.Tools;

public class WeatherServer
{
    public static async Task Main(string[] args)
    {
        // Create an MCP server
        var server = new McpServer(
            name: "Weather MCP Server",
            version: "1.0.0"
        );
        
        // Register our custom weather tool
        server.AddTool<string, WeatherData>("weatherTool",

                return weatherData;
            });

        var transport = new StdioServerTransport();
        await server.ConnectAsync(transport);

        // Keep the server running until process is terminated
        await Task.Delay(-1);
    }
    
    private static async Task<WeatherData> GetWeatherDataAsync(string location)
    {
        // This would normally call a weather API
        // Simplified for demonstration
        await Task.Delay(100); // Simulate API call
        return new WeatherData { 
            Temperature = 72.5,
            Conditions = "Sunny",
            Location = location
        };
    }
}

public class WeatherData
{
    public double Temperature { get; set; }
    public string Conditions { get; set; }
```

### Пример на Java: компоненты MCP-сервера

В этом примере показана та же регистрация MCP-сервера и инструментов, что и в примере на .NET выше, но выполненная на Java.

```java
import io.modelcontextprotocol.server.McpServer;
import io.modelcontextprotocol.server.McpToolDefinition;
import io.modelcontextprotocol.server.transport.StdioServerTransport;
import io.modelcontextprotocol.server.tool.ToolExecutionContext;
import io.modelcontextprotocol.server.tool.ToolResponse;

public class WeatherMcpServer {
    public static void main(String[] args) throws Exception {
        // Создать сервер MCP
        McpServer server = McpServer.builder()
            .name("Weather MCP Server")
            .version("1.0.0")
            .build();
            
        // Зарегистрировать инструмент погоды
        server.registerTool(McpToolDefinition.builder("weatherTool")
            .description("Gets current weather for a location")
            .parameter("location", String.class)
            .execute((ToolExecutionContext ctx) -> {
                String location = ctx.getParameter("location", String.class);
                
                // Получить данные о погоде (упрощенно)
                WeatherData data = getWeatherData(location);
                
                // Вернуть отформатированный ответ
                return ToolResponse.content(
                    String.format("Temperature: %.1f°F, Conditions: %s, Location: %s", 
                    data.getTemperature(), 
                    data.getConditions(), 
                    data.getLocation())
                );
            })
            .build());
        
        // Подключить сервер с использованием транспорта stdio
        try (StdioServerTransport transport = new StdioServerTransport()) {
            server.connect(transport);
            System.out.println("Weather MCP Server started");
            // Держать сервер запущенным до завершения процесса
            Thread.currentThread().join();
        }
    }
    
    private static WeatherData getWeatherData(String location) {
        // Реализация должна вызывать API погоды
        // Упрощено для примера
        return new WeatherData(72.5, "Sunny", location);
    }
}

class WeatherData {
    private double temperature;
    private String conditions;
    private String location;
    
    public WeatherData(double temperature, String conditions, String location) {
        this.temperature = temperature;
        this.conditions = conditions;
        this.location = location;
    }
    
    public double getTemperature() {
        return temperature;
    }
    
    public String getConditions() {
        return conditions;
    }
    
    public String getLocation() {
        return location;
    }
}
```

### Пример на JavaScript: создание MCP-сервера

В этом примере показано создание MCP-сервера на JavaScript и регистрация двух инструментов, связанных с погодой.

```javascript
// Использование официального SDK Model Context Protocol
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod"; // Для проверки параметров

// Создать сервер MCP
const server = new McpServer({
  name: "Weather MCP Server",
  version: "1.0.0"
});

// Определить инструмент погоды
server.tool(
  "weatherTool",
  {
    location: z.string().describe("The location to get weather for")
  },
  async ({ location }) => {
    // Обычно это вызовет API погоды
    // Упрощено для демонстрации
    const weatherData = await getWeatherData(location);
    
    return {
      content: [
        { 
          type: "text", 
          text: `Temperature: ${weatherData.temperature}°F, Conditions: ${weatherData.conditions}, Location: ${weatherData.location}` 
        }
      ]
    };
  }
);

// Определить инструмент прогноза
server.tool(
  "forecastTool",
  {
    location: z.string(),
    days: z.number().default(3).describe("Number of days for forecast")
  },
  async ({ location, days }) => {
    // Обычно это вызовет API погоды
    // Упрощено для демонстрации
    const forecast = await getForecastData(location, days);
    
    return {
      content: [
        { 
          type: "text", 
          text: `${days}-day forecast for ${location}: ${JSON.stringify(forecast)}` 
        }
      ]
    };
  }
);

// Вспомогательные функции
async function getWeatherData(location) {
  // Симуляция вызова API
  return {
    temperature: 72.5,
    conditions: "Sunny",
    location: location
  };
}

async function getForecastData(location, days) {
  // Симуляция вызова API
  return Array.from({ length: days }, (_, i) => ({
    day: i + 1,
    temperature: 70 + Math.floor(Math.random() * 10),
    conditions: i % 2 === 0 ? "Sunny" : "Partly Cloudy"
  }));
}

// Подключить сервер с использованием транспорта stdio
const transport = new StdioServerTransport();
server.connect(transport).catch(console.error);

console.log("Weather MCP Server started");
```

Этот пример на JavaScript демонстрирует, как создать MCP-сервер с помощью Model Context Protocol SDK. Показано, как зарегистрировать два инструмента с именами `weatherTool` и `forecastTool` и сделать их доступными для клиентов MCP через `StdioServerTransport`.

## Безопасность и авторизация

MCP включает несколько встроенных концепций и механизмов управления безопасностью и авторизацией в протоколе:

1. **Контроль разрешений на использование инструментов**:  
   Клиенты могут указывать, какие инструменты модель может использовать в сессии. Это гарантирует, что доступны только явно разрешённые инструменты, снижая риск нежелательных или небезопасных операций. Права доступа могут настраиваться динамически на основе предпочтений пользователя, политик организации или контекста взаимодействия.

2. **Аутентификация**:  
   Сервера могут требовать аутентификацию перед предоставлением доступа к инструментам, ресурсам или чувствительным операциям. Это может включать ключи API, OAuth-токены или другие схемы аутентификации. Надлежащая аутентификация обеспечивает, что только доверенные клиенты и пользователи могут вызывать функции на серверной стороне.

3. **Валидация**:  
   Для всех вызовов инструментов осуществляется проверка параметров. Каждый инструмент определяет ожидаемые типы, форматы и ограничения для своих параметров, а сервер проверяет входящие запросы соответственно. Это предотвращает попадание некорректных или вредоносных входных данных в реализацию инструментов и помогает сохранять целостность операций.

4. **Ограничение частоты запросов (Rate Limiting)**:  
   Чтобы предотвратить злоупотребления и обеспечить справедливое использование ресурсов сервера, MCP-серверы могут внедрять ограничение частоты вызовов инструментов и доступа к ресурсам. Ограничения могут применяться к пользователю, сессии или глобально, помогая защититься от атак типа «отказ в обслуживании» или чрезмерного потребления ресурсов.

Комбинируя эти механизмы, MCP предоставляет безопасную основу для интеграции языковых моделей с внешними инструментами и источниками данных, при этом предоставляя пользователям и разработчикам детальный контроль доступа и использования.

## Сообщения протокола и поток коммуникации

Взаимодействие MCP использует структурированные сообщения **JSON-RPC 2.0** для обеспечения надежного и прозрачного общения между хостами, клиентами и серверами. Протокол определяет конкретные схемы сообщений для различных типов операций:

### Основные типы сообщений:

#### **Сообщения инициализации**
- Запрос `initialize`: устанавливает соединение и согласовывает версию протокола и возможности  
- Ответ `initialize`: подтверждает поддерживаемые функции и информацию о сервере  
- `notifications/initialized`: сигнализирует, что инициализация завершена и сессия готова

#### **Сообщения обнаружения**
- Запрос `tools/list`: получение списка доступных инструментов с сервера  
- Запрос `resources/list`: получение списка доступных ресурсов (источников данных)  
- Запрос `prompts/list`: получение доступных шаблонов подсказок

#### **Сообщения исполнения**  
- Запрос `tools/call`: выполнение конкретного инструмента с переданными параметрами  
- Запрос `resources/read`: получение содержимого конкретного ресурса  
- Запрос `prompts/get`: получение шаблона подсказки с опциональными параметрами

#### **Сообщения со стороны клиента**
- Запрос `sampling/complete`: сервер запрашивает дополнение LLM у клиента  
- `elicitation/request`: сервер запрашивает ввод пользователя через клиентский интерфейс  
- Логирующие сообщения: сервер отправляет структурированные сообщения лога клиенту

#### **Уведомления**
- `notifications/tools/list_changed`: сервер уведомляет клиента об изменениях в списке инструментов  
- `notifications/resources/list_changed`: сервер уведомляет клиента об изменениях в списке ресурсов  
- `notifications/prompts/list_changed`: сервер уведомляет клиента о изменениях в подсказках

### Структура сообщений:

Все сообщения MCP имеют формат JSON-RPC 2.0 с:  
- **Запросы**: включают `id`, `method` и необязательные `params`  
- **Ответы**: включают `id` и либо `result`, либо `error`  
- **Уведомления**: включают `method` и необязательные `params` (без `id`, ответ не ожидается)

Такая структурированная коммуникация гарантирует надежное, отслеживаемое и расширяемое взаимодействие, поддерживающее продвинутые сценарии, включая обновления в реальном времени, последовательное использование инструментов и устойчивую обработку ошибок.

### Задачи (экспериментально)

**Задачи** — это экспериментальная функция, обеспечивающая долговременную обёртку исполнения, позволяющую отложенное получение результатов и отслеживание статуса запросов MCP:

- **Долгие операции**: отслеживание дорогостоящих вычислений, автоматизации рабочих процессов и пакетной обработки  
- **Отложенные результаты**: опрос статуса задачи и получение результатов по завершении операции  
- **Отслеживание статуса**: наблюдение за ходом выполнения через определённые жизненные этапы  
- **Многоэтапные операции**: поддержка сложных рабочих процессов с несколькими взаимодействиями

Задачи оборачивают стандартные запросы MCP, позволяя выполнять асинхронные операции, которые не могут быть завершены немедленно.

## Ключевые выводы

- **Архитектура**: MCP использует клиент-серверную архитектуру, где хосты управляют несколькими клиентскими соединениями с серверами  
- **Участники**: экосистема включает хосты (AI-приложения), клиенты (коннекторы протокола) и серверы (поставщики возможностей)  
- **Механизмы транспорта**: коммуникация поддерживает STDIO (локально) и потоковый HTTP с опциональным SSE (удалённый)  
- **Основные примитивы**: серверы предоставляют инструменты (выполнимые функции), ресурсы (источники данных) и подсказки (шаблоны)  
- **Примитивы клиента**: серверы могут запрашивать сэмплы (LLM-завершения с поддержкой вызова инструментов), запрос ввода (включая режим URL), корни (границы файловой системы) и логи у клиентов  
- **Экспериментальные функции**: задачи предоставляют долговременную обёртку исполнения для долгих операций  
- **Основа протокола**: построен на JSON-RPC 2.0 с версионированием по дате (текущая: 2025-11-25)  
- **Возможности в реальном времени**: поддержка уведомлений для динамических обновлений и синхронизации в режиме реального времени  
- **Безопасность в приоритете**: явное согласие пользователя, защита конфиденциальности и безопасный транспорт являются ключевыми требованиями
