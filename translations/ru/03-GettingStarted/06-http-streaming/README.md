# HTTPS-потоковая передача с использованием протокола Model Context Protocol (MCP)

В этой главе представлен детальный гид по реализации безопасной, масштабируемой и в реальном времени потоковой передачи с использованием Model Context Protocol (MCP) поверх HTTPS. Рассматривается мотивация потоковой передачи, доступные транспортные механизмы, как реализовать потоковый HTTP в MCP, лучшие практики безопасности, миграция с SSE и практические советы по созданию собственных потоковых приложений MCP.

## Транспортные механизмы и потоковая передача в MCP

В этом разделе рассматриваются различные доступные транспортные механизмы в MCP и их роль в обеспечении возможностей потоковой передачи для реального времени общения между клиентами и серверами.

### Что такое транспортный механизм?

Транспортный механизм определяет, как данные обмениваются между клиентом и сервером. MCP поддерживает несколько типов транспорта для разных сред и требований:

- **stdio**: Стандартный ввод/вывод, подходит для локальных и CLI-инструментов. Простой, но не подходит для веба или облака.
- **SSE (Server-Sent Events)**: Позволяет серверам отправлять обновления в реальном времени клиентам через HTTP. Хорошо подходит для веб-интерфейсов, но ограничен по масштабируемости и гибкости. Согласно спецификации MCP 2025-06-18, отдельный транспорт SSE (Server-Sent Events) устарел и заменён транспортом "Streamable HTTP".
- **Streamable HTTP**: Современный транспорт потоковой передачи на основе HTTP, поддерживает уведомления и лучшую масштабируемость. Рекомендуется для большинства продакшн- и облачных сценариев.

### Таблица сравнения

Посмотрите таблицу сравнения ниже, чтобы понять различия между этими транспортными механизмами:

| Transport         | Обновления в реальном времени | Потоковая передача | Масштабируемость | Сценарий использования  |
|-------------------|-------------------------------|--------------------|------------------|-------------------------|
| stdio             | Нет                           | Нет                | Низкая           | Локальные CLI инструменты |
| SSE               | Да                            | Да                 | Средняя          | Веб, обновления в реальном времени |
| Streamable HTTP   | Да                            | Да                 | Высокая          | Облако, множественные клиенты |

> **Совет:** Выбор правильного транспорта влияет на производительность, масштабируемость и пользовательский опыт. Для современных, масштабируемых и облачных приложений рекомендуется использовать **Streamable HTTP**.

Обратите внимание на транспорт stdio и SSE, которые мы обсуждали в предыдущих главах, и то, что транспорт Streamable HTTP рассматривается в этой главе.

## Потоковая передача: концепции и мотивация

Понимание фундаментальных концепций и мотиваций потоковой передачи важно для реализации эффективных систем общения в реальном времени.

**Потоковая передача** — это метод в сетевом программировании, позволяющий отправлять и получать данные небольшими, управляемыми порциями или в виде последовательности событий, вместо того чтобы ждать готовности всего ответа целиком. Это особенно полезно для:

- Больших файлов или наборов данных.
- Обновлений в реальном времени (например, чат, индикаторы прогресса).
- Длительно выполняемых вычислений, когда нужно информировать пользователя о ходе.

Основное, что нужно знать о потоковой передаче на высоком уровне:

- Данные доставляются постепенно, а не сразу полностью.
- Клиент может обрабатывать данные по мере их поступления.
- Снижает воспринимаемую задержку и улучшает пользовательский опыт.

### Зачем использовать потоковую передачу?

Причины использования потоковой передачи таковы:

- Пользователь мгновенно получает обратную связь, а не только по завершении.
- Позволяет создавать приложения в реальном времени и отзывчивые UI.
- Более эффективное использование сетевых и вычислительных ресурсов.

### Простой пример: HTTP потоковый сервер и клиент

Ниже пример простой реализации потоковой передачи:

#### Python

**Сервер (Python, с использованием FastAPI и StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**Клиент (Python, с использованием requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

Этот пример демонстрирует сервер, который отправляет серию сообщений клиенту по мере их готовности, а не ждет, пока все сообщения будут готовы сразу.

**Как это работает:**

- Сервер отдаёт каждое сообщение по мере готовности.
- Клиент получает и выводит каждый фрагмент по мере поступления.

**Требования:**

- Для сервера необходим потоковый ответ (например, `StreamingResponse` в FastAPI).
- Клиент должен обрабатывать ответ как поток (`stream=True` в requests).
- Обычно Content-Type — `text/event-stream` или `application/octet-stream`.

#### Java

**Сервер (Java, с использованием Spring Boot и Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**Клиент (Java, с использованием Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Особенности реализации на Java:**

- Использует реактивный стек Spring Boot с `Flux` для потоковой передачи
- `ServerSentEvent` обеспечивает структурированную потоковую передачу с типами событий
- `WebClient` с `bodyToFlux()` позволяет реактивно обрабатывать потоковые данные
- `delayElements()` симулирует время обработки между событиями
- События могут иметь типы (`info`, `result`) для удобной обработки на клиенте

### Сравнение: классическая потоковая передача против MCP потоковой передачи

Различия между классическим способом потоковой передачи и тем, как это реализовано в MCP, представлены в таблице:

| Функция                | Классическая HTTP потоковая передача | Потоковая передача MCP (уведомления) |
|------------------------|-------------------------------------|--------------------------------------|
| Основной ответ         | Отправляется по чанкам               | Один ответ, в конце                   |
| Обновления прогресса    | Отправляются как чанки данных        | Отправляются как уведомления          |
| Требования к клиенту    | Должен обрабатывать поток            | Должен реализовать обработчик сообщений |
| Сценарии использования | Большие файлы, потоки AI токенов      | Прогресс, логи, обратная связь в реальном времени |

### Ключевые отличия

Дополнительно, ключевые отличия таковы:

- **Шаблон общения:**
  - Классическая HTTP потоковая передача: Использует простое chunked кодирование для отправки данных
  - Потоковая передача MCP: Использует структурированную систему уведомлений с протоколом JSON-RPC

- **Формат сообщений:**
  - Классический HTTP: простой текст с разделением по строкам
  - MCP: Структурированные объекты LoggingMessageNotification с метаданными

- **Реализация клиента:**
  - Классический HTTP: простой клиент, обрабатывающий потоковые ответы
  - MCP: Более сложный клиент с обработчиком сообщений для разного типа сообщений

- **Обновления прогресса:**
  - Классический HTTP: прогресс в основном потоке ответа
  - MCP: прогресс отправляется отдельными уведомлениями, основной результат приходит в конце

### Рекомендации

При выборе между классической потоковой передачей (например, эндпоинт, показанный ранее как `/stream`) и MCP потоковой передачей рекомендуем следующее:

- **Для простых потребностей потоковой передачи:** классический HTTP поток проще реализовать и достаточно для базовых задач.
- **Для сложных, интерактивных приложений:** MCP потоковая передача предоставляет более структурированный подход с богатыми метаданными и разделением уведомлений и окончательных результатов.
- **Для AI-приложений:** система уведомлений MCP особенно полезна для длительных AI-задач, когда важно держать пользователя в курсе прогресса.

## Потоковая передача в MCP

Итак, вы уже видели рекомендации и сравнения касательно классической и MCP потоковой передачи. Теперь мы подробно рассмотрим, как именно использовать потоковую передачу в MCP.

Понимание работы потоковой передачи в рамках MCP важно для построения отзывчивых приложений, предоставляющих обратную связь в реальном времени пользователям во время длительных операций.

В MCP потоковая передача — это не отправка главного ответа по частям, а отправка **уведомлений** клиенту в процессе обработки запроса. Эти уведомления могут содержать обновления прогресса, логи или другие события.

### Как это работает

Главный результат по-прежнему отправляется одним ответом. Однако уведомления могут передаваться отдельными сообщениями в процессе обработки и тем самым обновлять клиента в реальном времени. Клиент должен уметь обрабатывать и отображать эти уведомления.

## Что такое уведомление?

Мы сказали "уведомление", что это значит в контексте MCP?

Уведомление — это сообщение, отправляемое сервером клиенту для информирования о ходе, статусе или других событиях во время длительной операции. Уведомления повышают прозрачность и улучшают пользовательский опыт.

Например, клиент должен отправить уведомление после установления начального рукопожатия с сервером.

Уведомление выглядит так в формате JSON:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

Уведомления относятся к теме в MCP, называемой ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

Для работы логирования сервер должен включить его как функцию/возможность следующим образом:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> В зависимости от используемого SDK логирование может быть включено по умолчанию, или его нужно явно активировать в конфигурации сервера.

Существует несколько типов уведомлений:

| Уровень   | Описание                        | Пример использования            |
|-----------|--------------------------------|--------------------------------|
| debug     | Подробная отладочная информация | Точки входа/выхода функций      |
| info      | Общие информационные сообщения  | Обновления прогресса операции   |
| notice    | Нормальные, но значимые события | Изменения конфигурации          |
| warning   | Предупредительные состояния     | Использование устаревших функций|
| error     | Ошибки                         | Неудачи операций                |
| critical  | Критические состояния           | Отказы компонентов системы     |
| alert     | Требуется немедленное действие  | Обнаружено повреждение данных  |
| emergency | Система неработоспособна        | Полный системный сбой          |

## Реализация уведомлений в MCP

Для реализации уведомлений в MCP необходимо настроить серверную и клиентскую части для обработки обновлений в реальном времени. Это позволит вашему приложению обеспечивать мгновенную обратную связь пользователям во время длительных операций.

### На сервере: отправка уведомлений

Начнём с серверной части. В MCP вы определяете инструменты, которые могут отправлять уведомления во время обработки запросов. Сервер использует объект контекста (обычно `ctx`) для отправки сообщений клиенту.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

В приведенном примере инструмент `process_files` отправляет три уведомления клиенту по мере обработки каждого файла. Метод `ctx.info()` используется для отправки информационных сообщений.

Кроме того, чтобы включить уведомления, убедитесь, что сервер использует потоковый транспорт (например, `streamable-http`), а клиент реализует обработчик сообщений для обработки уведомлений. Вот как можно настроить сервер для использования транспорта `streamable-http`:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

В этом примере на .NET инструмент `ProcessFiles` помечен атрибутом `Tool` и отправляет три уведомления клиенту по мере обработки каждого файла. Метод `ctx.Info()` используется для отправки информационных сообщений.

Для включения уведомлений в сервере MCP на .NET убедитесь, что используется потоковый транспорт:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### На клиенте: получение уведомлений

Клиент должен реализовать обработчик сообщений, чтобы обрабатывать и отображать уведомления по мере их поступления.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

В приведённом коде функция `message_handler` проверяет, является ли входящее сообщение уведомлением. Если это уведомление, оно выводится; иначе обрабатывается как обычное сообщение сервера. Также обратите внимание, что `ClientSession` инициализируется с `message_handler` для обработки входящих уведомлений.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

В этом примере на .NET функция `MessageHandler` проверяет, является ли входящее сообщение уведомлением. Если да, она выводит уведомление, иначе обрабатывает как обычное сообщение сервера. `ClientSession` инициализируется с обработчиком сообщений через `ClientSessionOptions`.

Для включения уведомлений убедитесь, что сервер использует потоковый транспорт (например, `streamable-http`) и клиент реализует обработчик сообщений для обработки уведомлений.

## Уведомления о прогрессе и сценарии

В этом разделе объясняется концепция уведомлений о прогрессе в MCP, почему они важны и как их реализовать с использованием Streamable HTTP. Кроме того, приведено практическое задание для закрепления изученного материала.

Уведомления о прогрессе — это сообщения в реальном времени, отправляемые сервером клиенту во время длительных операций. Вместо того чтобы ждать окончания процесса, сервер постоянно информирует клиента о текущем статусе. Это повышает прозрачность, улучшает опыт пользователя и облегчает отладку.

**Пример:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### Зачем нужны уведомления о прогрессе?

Уведомления о прогрессе важны по нескольким причинам:

- **Лучший пользовательский опыт:** пользователи видят обновления по мере выполнения работы, а не только по окончании.
- **Обратная связь в реальном времени:** клиенты могут отображать индикаторы прогресса или логи, придавая приложению отзывчивость.
- **Проще отладка и мониторинг:** разработчики и пользователи видят, где процесс может тормозить или зависать.

### Как реализовать уведомления о прогрессе

Вот как можно реализовать уведомления о прогрессе в MCP:

- **На сервере:** используйте `ctx.info()` или `ctx.log()` для отправки уведомлений по мере обработки каждого элемента. Это отправляет сообщение клиенту до готовности основного результата.
- **На клиенте:** реализуйте обработчик сообщений, который слушает и отображает уведомления по мере их поступления. Этот обработчик отличает уведомления от окончательного результата.

**Пример сервера:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**Пример клиента:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## Вопросы безопасности

При реализации MCP-серверов с транспортами на базе HTTP безопасность становится первостепенной задачей, требующей внимательного отношения к многочисленным вектором атак и механизмам защиты.

### Обзор

Безопасность критична при открытии MCP-серверов через HTTP. Streamable HTTP вводит новые поверхности для атак и требует тщательной настройки.

### Основные моменты

- **Валидация заголовка Origin:** всегда проверяйте заголовок `Origin`, чтобы предотвратить атаки DNS rebinding.

## Обновление с SSE на Streamable HTTP

Для приложений, которые в настоящее время используют Server-Sent Events (SSE), переход на Streamable HTTP предоставляет расширенные возможности и обеспечивает лучшую долгосрочную устойчивость ваших реализаций MCP.

### Почему обновляться?

Существует две веские причины для перехода с SSE на Streamable HTTP:

- Streamable HTTP предлагает лучшую масштабируемость, совместимость и более богатую поддержку уведомлений, чем SSE.
- Это рекомендуемый транспорт для новых приложений MCP.

### Шаги миграции

Вот как вы можете перейти с SSE на Streamable HTTP в ваших приложениях MCP:

- **Обновите серверный код**, чтобы использовать `transport="streamable-http"` в `mcp.run()`.
- **Обновите клиентский код**, чтобы использовать `streamablehttp_client` вместо SSE клиента.
- **Реализуйте обработчик сообщений** на клиенте для обработки уведомлений.
- **Проверьте совместимость** с существующими инструментами и рабочими процессами.
