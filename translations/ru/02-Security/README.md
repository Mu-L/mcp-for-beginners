# MCP Security: Комплексная защита AI-систем

[![MCP Security Best Practices](../../../translated_images/ru/03.175aed6dedae133f.webp)](https://youtu.be/88No8pw706o)

Безопасность является фундаментальной в проектировании AI-систем, поэтому мы уделяем ей приоритетное внимание во втором разделе. Это соответствует принципу Microsoft **Secure by Design** из [Secure Future Initiative](https://www.microsoft.com/security/blog/2025/04/17/microsofts-secure-by-design-journey-one-year-of-success/).

Протокол контекста модели (MCP) открывает новые мощные возможности для приложений с использованием AI, при этом вводя уникальные проблемы безопасности, выходящие за рамки традиционных рисков программного обеспечения. Системы MCP сталкиваются как с проверенными вопросами безопасности (безопасное кодирование, наименьшие привилегии, безопасность цепочки поставок), так и с новыми угрозами, специфичными для AI, такими как инъекция подсказок, отравление инструментов, перехват сессии, атаки «смущенного заместителя», уязвимости пропуска токенов и динамическое изменение возможностей.

В этом уроке рассматриваются наиболее критичные риски безопасности в реализации MCP — аутентификация, авторизация, чрезмерные разрешения, косвенная инъекция подсказок, безопасность сессий, проблемы «смущенного заместителя», управление токенами и уязвимости цепочки поставок. Вы узнаете практические меры и лучшие методы для снижения этих рисков с использованием решений Microsoft, таких как Prompt Shields, Azure Content Safety и GitHub Advanced Security для повышения надежности развертывания MCP.

## Учебные цели

К концу этого урока вы сможете:

- **Определять угрозы, специфичные для MCP**: Распознавать уникальные риски безопасности систем MCP, включая инъекцию подсказок, отравление инструментов, чрезмерные разрешения, перехват сессии, проблемы «смущенного заместителя», уязвимости пропуска токенов и риски цепочки поставок
- **Применять меры безопасности**: Внедрять эффективные меры защиты, включая надежную аутентификацию, доступ с наименьшими привилегиями, безопасное управление токенами, контроль безопасности сессий и проверку цепочки поставок
- **Использовать решения безопасности Microsoft**: Понимать и использовать Microsoft Prompt Shields, Azure Content Safety и GitHub Advanced Security для защиты рабочих нагрузок MCP
- **Проверять безопасность инструментов**: Понимать важность валидации метаданных инструментов, мониторинга динамических изменений и защиты от косвенных атак инъекции подсказок
- **Интегрировать лучшие практики**: Совмещать проверенные основы безопасности (безопасное кодирование, укрепление серверов, модель нулевого доверия) с мерами, специфичными для MCP, для комплексной защиты

Быстро развивающийся стандарт MCP продолжает совершенствовать меры безопасности, обеспечивая лучшее взаимодействие с корпоративными архитектурами безопасности и установленными лучшими практиками.

Наиболее эффективная стратегия защиты сочетает базовые практики безопасности с мерами, специфичными для MCP — проверенные базовые меры остаются самым мощным способом снижения общего уровня риска.

### Эволюция аутентификации MCP

Спецификация MCP значительно развилась в способах аутентификации и авторизации:

- **Изначальный подход**: Ранние спецификации требовали от разработчиков внедрения собственных серверов аутентификации, где MCP-серверы выступали как OAuth 2.0 Authorization Server, управляя аутентификацией пользователей напрямую
- **Текущий стандарт (2025-11-25)**: Обновленная спецификация позволяет MCP-серверам делегировать аутентификацию внешним провайдерам идентичности (например, Microsoft Entra ID), что улучшает безопасность и упрощает реализацию
- **Безопасность транспортного уровня**: Расширенная поддержка безопасных транспортных механизмов с корректными паттернами аутентификации как для локальных (STDIO), так и удалённых (Streamable HTTP) подключений

## Безопасность аутентификации и авторизации

### Текущие проблемы безопасности

Современные реализации MCP сталкиваются с рядом проблем в области аутентификации и авторизации:

### Риски и векторы атак

- **Неправильная логика авторизации**: Ошибки в реализации авторизации на MCP-серверах могут привести к раскрытию данных и неправильному применению контроля доступа
- **Компрометация OAuth токенов**: Кража токенов локального MCP-сервера позволяет злоумышленникам выдавать себя за сервер и получать доступ к сервисам
- **Уязвимости пропуска токенов**: Некорректное обращение с токенами создает обходы мер безопасности и проблемы с подотчетностью
- **Чрезмерные разрешения**: MCP-серверы с избыточными правами нарушают принцип наименьших привилегий и расширяют поверхность атаки

#### Пропуск токена: критическая анти-практика

**Пропуск токена категорически запрещён** в текущей спецификации авторизации MCP из-за тяжелых последствий для безопасности:

##### Обход мер безопасности
- MCP-серверы и downstream API реализуют ключевые меры безопасности (лимитирование запросов, валидация, мониторинг трафика), зависящие от корректной проверки токенов
- Использование токенов напрямую от клиента к API обходит эти важные защиты, подрывая архитектуру безопасности

##### Проблемы подотчетности и аудита
- MCP-серверы не могут различать клиентов, использующих токены, выданные upstream, что ломает цепочки аудита
- Логи downstream ресурсов показывают ложное происхождение запросов вместо фактических посредников — MCP-серверов
- Расследования инцидентов и проверки соответствия усложняются

##### Риски утечки данных
- Непроверенные утверждения токенов дают злоумышленникам с украденными токенами возможность использовать MCP-серверы в качестве прокси для вывода данных
- Нарушения границы доверия позволяют обходить меры безопасности, получая несанкционированный доступ

##### Векторы многосервисных атак
- Компрометация токенов, принятых несколькими сервисами, позволяет перемещаться по связанным системам
- Нарушение предположений доверия возникает, когда невозможно проверить происхождение токена

### Меры контроля и смягчения

**Критические требования к безопасности:**

> **ОБЯЗАТЕЛЬНО**: MCP-серверы **НЕ ДОЛЖНЫ** принимать токены, которые явно не выданы для данного MCP-сервера

#### Контроль аутентификации и авторизации

- **Тщательный аудит авторизации**: Проводите всестороннюю проверку логики авторизации MCP-серверов, чтобы гарантировать доступ только для назначенных пользователей и клиентов
  - **Руководство по внедрению**: [Azure API Management как шлюз аутентификации для MCP-серверов](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690)
  - **Интеграция идентичности**: [Использование Microsoft Entra ID для аутентификации MCP-серверов](https://den.dev/blog/mcp-server-auth-entra-id-session/)

- **Безопасное управление токенами**: Используйте [лучшие практики Microsoft по валидации и жизненному циклу токенов](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens)
  - Проверяйте утверждения аудитории токена на соответствие идентичности MCP-сервера
  - Реализуйте политику ротации и истечения срока токенов
  - Предотвращайте повторное использование токенов и несанкционированное применение

- **Защищенное хранение токенов**: Обеспечивайте шифрование токенов в покое и при передаче
  - **Лучшие практики**: [Руководство по безопасному хранению и шифрованию токенов](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)

#### Реализация контроля доступа

- **Принцип наименьших привилегий**: Предоставляйте MCP-серверам только минимально необходимые разрешения для выполнения задач
  - Регулярно проверяйте и обновляйте права для предотвращения расширения привилегий
  - **Документация Microsoft**: [Безопасный доступ с минимальными привилегиями](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)

- **Управление доступом на основе ролей (RBAC)**: Внедряйте тонкое распределение ролей
  - Ограничивайте роли конкретными ресурсами и операциями
  - Избегайте избыточных или ненужных разрешений, расширяющих поверхность атаки

- **Непрерывный мониторинг разрешений**: Реализуйте постоянный аудит и мониторинг доступа
  - Следите за аномалиями в использовании прав
  - Оперативно устраняйте избыточные или неиспользуемые привилегии

## Специфичные для AI угрозы безопасности

### Атаки инъекций подсказок и манипулирование инструментами

Современные реализации MCP сталкиваются со сложными специфичными AI векторами атак, которые не всегда могут быть полностью устранены традиционными мерами безопасности:

#### **Косвенная инъекция подсказок (Cross-Domain Prompt Injection)**

**Косвенная инъекция подсказок** — одна из самых критичных уязвимостей в AI-системах с MCP. Злоумышленники встраивают вредоносные инструкции в внешний контент: документы, веб-страницы, письма или источники данных, которые AI-система воспринимает как легитимные команды.

**Сценарии атаки:**
- **Инъекция через документы**: Вредоносные инструкции в обрабатываемых документах, вызывающие непреднамеренные действия AI
- **Эксплуатация веб-контента**: Скомпрометированные веб-страницы с встроенными подсказками, влияющими на поведение AI при их парсинге
- **Атаки через электронную почту**: Вредоносные подсказки в письмах, заставляющие AI-ассистентов раскрывать информацию или выполнять несанкционированные действия
- **Загрязнение источников данных**: Скомпрометированные базы данных или API, поставляющие заражённый контент AI-системам

**Реальные последствия**: такие атаки могут привести к утечкам данных, нарушениям приватности, генерации вредоносного контента и манипуляциям взаимодействием с пользователем. Подробнее — [Prompt Injection в MCP (Simon Willison)](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/).

![Диаграмма атаки инъекции подсказок](../../../translated_images/ru/prompt-injection.ed9fbfde297ca877.webp)

#### **Атаки отравления инструментов**

**Отравление инструментов** направлено на метаданные, которые определяют инструменты MCP, эксплуатируя то, как большие языковые модели интерпретируют описания и параметры инструментов для принятия решений о выполнении.

**Механизмы атаки:**
- **Манипулирование метаданными**: Злоумышленники внедряют вредоносные инструкции в описания инструментов, определения параметров или примеры использования
- **Невидимые инструкции**: Скрытые подсказки в метаданных инструментов, которые обрабатываются AI, оставаясь незаметными для человека
- **Динамическое изменение инструментов («Руг пулы»)**: Инструменты, изначально одобренные пользователем, позже модифицируются для выполнения вредоносных действий без уведомления пользователя
- **Инъекция параметров**: Вредоносное содержимое, встроенное в схемы параметров инструментов, влияющее на поведение модели

**Риски для хостинг-серверов**: Удалённые MCP-серверы представляют повышенный риск, так как определения инструментов могут обновляться после одобрения пользователем, создавая ситуации, когда ранее безопасные инструменты становятся вредоносными. Подробный анализ — [Tool Poisoning Attacks (Invariant Labs)](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks).

![Диаграмма атаки отравления инструментов](../../../translated_images/ru/tool-injection.3b0b4a6b24de6bef.webp)

#### **Дополнительные векторы атак AI**

- **Кросс-доменные инъекции подсказок (XPIA)**: Сложные атаки, использующие контент из нескольких доменов для обхода мер безопасности
- **Динамическое изменение возможностей**: Изменения возможностей инструментов в реальном времени, обходящие начальные оценки безопасности  
- **Отравление контекстного окна**: Атаки, манипулирующие большими контекстными окнами для скрытия вредоносных инструкций  
- **Атаки путаницы модели**: Использование ограничений модели для создания непредсказуемого или небезопасного поведения  


### Влияние рисков безопасности ИИ

**Последствия с высоким уровнем влияния:**  
- **Экспфильтрация данных**: Несанкционированный доступ и кража конфиденциальных данных предприятия или личных данных  
- **Нарушения конфиденциальности**: Разглашение персональных данных (PII) и конфиденциальной бизнес-информации  
- **Манипуляции системой**: Непреднамеренные изменения критических систем и рабочих процессов  
- **Кража учетных данных**: Компрометация токенов аутентификации и сервисных учетных данных  
- **Латеральное движение**: Использование скомпрометированных ИИ-систем в качестве точек опоры для более широкой атаки на сеть  

### Решения Microsoft для безопасности ИИ

#### **AI Prompt Shields: Продвинутая защита от атак инъекций в подсказки**

Microsoft **AI Prompt Shields** обеспечивают всеобъемлющую защиту от прямых и косвенных атак инъекций в подсказки посредством нескольких уровней безопасности:

##### **Основные механизмы защиты:**

1. **Продвинутое обнаружение и фильтрация**  
   - Алгоритмы машинного обучения и методы обработки естественного языка (NLP) обнаруживают вредоносные инструкции во внешнем контенте  
   - Анализ в реальном времени документов, веб-страниц, электронных писем и источников данных на наличие встроенных угроз

## Продвинутые угрозы безопасности MCP

### Уязвимости перехвата сессии

**Перехват сессии** представляет собой критический вектор атаки в stateful-реализациях MCP, когда неавторизованные лица получают и используют легитимные идентификаторы сессий для имитации клиентов и выполнения несанкционированных действий.

#### **Сценарии атак и риски**

- **Инъекция подсказок с перехваченными сессиями**: Атакующие со украденными ID сессий внедряют вредоносные события на серверах, разделяющих состояние сессии, что может вызывать вредоносные действия или доступ к конфиденциальным данным  
- **Прямая имитация**: Украденные ID сессий позволяют выполнять прямые вызовы MCP сервера, обходящие аутентификацию и воспринимающие атакующего как легитимного пользователя  
- **Скомпрометированные возобновляемые потоки**: Атакующие могут преждевременно завершать запросы, заставляя легитимных клиентов продолжать с потенциально вредоносным содержимым  

#### **Контроль безопасности управления сессиями**

**Критические требования:**
- **Проверка авторизации**: MCP серверы, реализующие авторизацию, **ДОЛЖНЫ** проверять ВСЕ входящие запросы и **НЕ ДОЛЖНЫ** полагаться на сессии для аутентификации  
- **Безопасная генерация сессий**: Использовать криптографически защищённые, недетерминированные ID сессий, созданные с помощью генераторов случайных чисел  
- **Привязка к пользователю**: Привязывать ID сессий к информации о пользователе в формате `<user_id>:<session_id>`, чтобы предотвращать межпользовательское злоупотребление сессиями  
- **Управление жизненным циклом сессии**: Реализовать правильное истечение срока, ротацию и инвалидизацию для ограничения временных окон уязвимости  
- **Безопасность транспорта**: Обязательное использование HTTPS для всей коммуникации с целью предотвращения перехвата ID сессий  

### Проблема «Запутавшийся заместитель» (Confused Deputy)

Проблема **confused deputy** возникает, когда MCP серверы выступают в роли прокси-аутентификации между клиентами и сторонними сервисами, создавая возможности обхода авторизации путём эксплуатации статических client ID.

#### **Механика атак и риски**

- **Обход согласия на основе cookie**: Предыдущая аутентификация пользователя создаёт cookie с согласием, которые злоумышленники эксплуатируют через вредоносные запросы авторизации с поддельными URI перенаправления  
- **Кража кода авторизации**: Существующие cookie с согласием могут заставлять серверы авторизации пропускать экраны согласия и перенаправлять коды на контрольируемые атакующим конечные точки  
- **Несанкционированный доступ к API**: Украденные коды авторизации позволяют обмениваться токенами и имитировать пользователей без явного разрешения  

#### **Стратегии смягчения**

**Обязательные меры:**
- **Требование явного согласия**: Прокси MCP серверы со статическими client ID **ДОЛЖНЫ** получать согласие пользователя для каждого динамически зарегистрированного клиента  
- **Реализация безопасности OAuth 2.1**: Следовать современным стандартам безопасности OAuth, включая PKCE (Proof Key for Code Exchange) для всех запросов авторизации  
- **Строгая проверка клиента**: Реализовать строгую валидацию URI перенаправления и идентификаторов клиентов для предотвращения эксплуатации  

### Уязвимости передачи токенов  

**Передача токенов** — это явно нежелательная практика, при которой MCP серверы принимают клиентские токены без надлежащей проверки и пересылают их к нижележащим API, нарушая спецификации авторизации MCP.

#### **Последствия для безопасности**

- **Обход контроля**: Прямое использование клиентских токенов для API обходит важные механизмы ограничения скорости, валидации и мониторинга  
- **Порча аудита**: Токены, выпущенные на верхнем уровне, делают невозможной идентификацию клиента, нарушая возможности расследования инцидентов  
- **Экспфильтрация данных через прокси**: Непроверенные токены позволяют злоумышленникам использовать серверы как прокси для несанкционированного доступа к данным  
- **Нарушение границ доверия**: Предположения о доверии нижележащих сервисов нарушаются, когда происхождение токенов не может быть подтверждено  
- **Расширение атак по нескольким сервисам**: Принятие скомпрометированных токенов на нескольких сервисах способствует латеральному движению  

#### **Необходимые меры безопасности**

**Обязательные требования:**
- **Проверка токенов**: MCP серверы **НЕ ДОЛЖНЫ** принимать токены, не выданные явно для MCP сервера  
- **Проверка аудитории**: Всегда проверять, что аудитория токена соответствует идентичности MCP сервера  
- **Срок действия токенов**: Использовать краткоживущие access токены с безопасными процедурами ротации  


## Безопасность цепочки поставок AI-систем

Безопасность цепочки поставок вышла за рамки традиционных зависимостей ПО и охватывает всю экосистему ИИ. Современные реализации MCP должны тщательно проверять и контролировать все AI-компоненты, так как каждый из них представляет потенциальную уязвимость, способную подорвать целостность системы.

### Расширенные компоненты цепочки поставок ИИ

**Традиционные программные зависимости:**  
- Библиотеки и фреймворки с открытым исходным кодом  
- Образы контейнеров и базовые системы  
- Инструменты разработки и конвейеры сборки  
- Инфраструктурные компоненты и сервисы  

**Специфические для AI элементы цепочки поставок:**  
- **Фундаментальные модели**: Предобученные модели от разных поставщиков с необходимостью проверки происхождения  
- **Сервисы эмбеддингов**: Внешние сервисы векторизации и семантического поиска  
- **Поставщики контекста**: Источники данных, базы знаний и хранилища документов  
- **Сторонние API**: Внешние AI-сервисы, ML-конвейеры и конечные точки обработки данных  
- **Артефакты моделей**: Веса, конфигурации и варианты тонкой настройки моделей  
- **Источники данных для обучения**: Наборы данных для обучения и дообучения моделей  

### Полноценная стратегия безопасности цепочки поставок

#### **Проверка компонентов и доверие**  
- **Проверка происхождения**: Подтверждать источник, лицензионные условия и целостность всех AI-компонентов перед интеграцией  
- **Оценка безопасности**: Проводить сканирование уязвимостей и ревизии безопасности моделей, источников данных и AI-сервисов  
- **Анализ репутации**: Оценивать репутацию и практики обеспечения безопасности поставщиков AI-сервисов  
- **Проверка соответствия**: Убедиться, что все компоненты соответствуют требованиям организации по безопасности и регулированию  

#### **Безопасные конвейеры развертывания**  
- **Автоматизированная безопасность CI/CD**: Интегрировать сканирование безопасности во все этапы автоматизированного развертывания  
- **Целостность артефактов**: Внедрять криптографическую проверку всех развертываемых артефактов (код, модели, конфигурации)  
- **Пошаговое развертывание**: Использовать прогрессивные стратегии с проверкой безопасности на каждом этапе  
- **Доверенные репозитории артефактов**: Развертывать только из проверенных, защищённых реестров и хранилищ  

#### **Непрерывный мониторинг и реагирование**  
- **Сканирование зависимостей**: Постоянный мониторинг уязвимостей для всех зависимостей программного обеспечения и AI-компонентов  
- **Мониторинг моделей**: Непрерывная оценка поведения моделей, дрейфа производительности и аномалий безопасности  
- **Отслеживание здоровья сервисов**: Контроль доступности, инцидентов безопасности и изменений политики внешних AI-сервисов  
- **Интеграция разведки угроз**: Включение потоков разведданных, специфичных для рисков безопасности AI и ML  

#### **Контроль доступа и минимальные привилегии**  
- **Права на уровне компонента**: Ограничивать доступ к моделям, данным и сервисам в зависимости от бизнес-необходимости  
- **Управление сервисными аккаунтами**: Использовать выделенные сервисные аккаунты с минимально необходимыми правами  
- **Сегментация сети**: Изолировать AI-компоненты и ограничить сетевой доступ между сервисами  
- **Контроль через API-шлюзы**: Использовать централизованные API-шлюзы для контроля и мониторинга доступа к внешним AI-сервисам  

#### **Реагирование на инциденты и восстановление**  
- **Процедуры быстрого реагирования**: Налаженные процессы для патчинга или замены скомпрометированных AI-компонентов  
- **Ротация учетных данных**: Автоматизированные системы для смены секретов, API-ключей и сервисных учетных данных  
- **Возможности отката**: Способность быстро возвращать предыдущие известные стабильные версии AI-компонентов  
- **Восстановление после нарушения цепочки поставок**: Специфические процедуры реагирования на компрометацию вышестоящих AI-сервисов

## Лучшие практики безопасности на базовом уровне

Реализации MCP наследуют и строят над существующим уровнем безопасности вашей организации. Усиление базовых практик безопасности существенно повышает общую защиту AI-систем и развертываний MCP.

### Основы безопасности

#### **Безопасные практики разработки**  
- **Соответствие OWASP**: Защита от [OWASP Top 10](https://owasp.org/www-project-top-ten/) уязвимостей веб-приложений  
- **Специфические защиты ИИ**: Внедрение контролей для [OWASP Top 10 для LLM](https://genai.owasp.org/download/43299/?tmstv=1731900559)  
- **Безопасное управление секретами**: Использование выделенных хранилищ для токенов, API-ключей и конфиденциальных настроек  
- **Конечное шифрование**: Обеспечение защищённой связи во всех компонентах приложения и потоках данных  
- **Валидация вводимых данных**: Строгая проверка всех пользовательских вводов, параметров API и источников данных  

#### **Укрепление инфраструктуры**  
- **Многофакторная аутентификация**: Обязательное MFA для всех административных и сервисных аккаунтов  
- **Управление обновлениями**: Автоматизированное и своевременное патчинг ОС, фреймворков и зависимостей  
- **Интеграция провайдера идентичности**: Централизованное управление идентичностями через корпоративные провайдеры (Microsoft Entra ID, Active Directory)  
- **Сегментация сети**: Логическая изоляция компонентов MCP для ограничения латерального движения  
- **Принцип минимальных привилегий**: Минимально необходимые права для всех системных компонентов и аккаунтов  

#### **Мониторинг и обнаружение угроз**  
- **Всестороннее логирование**: Детальный аудит активности AI-приложений, включая взаимодействия клиента и сервера MCP  
- **Интеграция SIEM**: Централизованные системы управления безопасностью и событиями для обнаружения аномалий  
- **Аналитика поведения**: Мониторинг с использованием ИИ для выявления необычных моделей поведения систем и пользователей  
- **Разведка угроз**: Интеграция внешних потоков угроз и индикаторов компрометации (IOC)  
- **Реагирование на инциденты**: Чётко определённые процедуры обнаружения, реагирования и восстановления безопасности  

#### **Архитектура нулевого доверия**  
- **Никогда не доверять, всегда проверять**: Непрерывная проверка пользователей, устройств и сетевых подключений  
- **Микросегментация**: Гранулярный сетевой контроль, изолирующий отдельные рабочие нагрузки и сервисы  
- **Безопасность на основе идентичности**: Политики безопасности, основанные на проверенных личностях, а не на сетевом расположении  
- **Непрерывная оценка рисков**: Динамическая оценка уровня безопасности исходя из текущего контекста и поведения  
- **Условный доступ**: Контроль доступа, адаптирующийся в зависимости от факторов риска, местоположения и доверия к устройству  

### Паттерны корпоративной интеграции

#### **Интеграция в экосистему безопасности Microsoft**  
- **Microsoft Defender for Cloud**: Комплексное управление безопасностью облака  
- **Azure Sentinel**: Облачные SIEM и SOAR-возможности для защиты AI-нагрузок  
- **Microsoft Entra ID**: Управление корпоративной идентичностью и доступом с условным доступом  
- **Azure Key Vault**: Централизованное управление секретами с поддержкой аппаратных модулей безопасности (HSM)  
- **Microsoft Purview**: Управление данными и соответствие требованиям для источников данных и рабочих процессов AI  

#### **Соответствие и управление**  
- **Соответствие нормативным требованиям**: Обеспечение выполнения отраслевых стандартов (GDPR, HIPAA, SOC 2) в реализации MCP  
- **Классификация данных**: Правильная категоризация и обращение с конфиденциальными данными, обрабатываемыми AI-системами  
- **Аудиторские трассировки**: Комплексное логирование для соответствия требованиям и судебного анализа  
- **Контроль конфиденциальности**: Внедрение принципов privacy-by-design в архитектуру AI-систем  
- **Управление изменениями**: Формальные процессы проверки безопасности при изменениях AI-систем  

Эти базовые практики создают надёжную основу безопасности, усиливающую эффективность специфичных для MCP мер и обеспечивающую комплексную защиту AI-приложений.

## Главное по безопасности
- **Многоуровневый подход к безопасности**: Сочетайте базовые практики безопасности (безопасное программирование, принцип минимальных привилегий, проверка цепочки поставок, непрерывный мониторинг) с контролями, специфичными для ИИ, для комплексной защиты

- **Специфические угрозы ИИ**: Системы MCP сталкиваются с уникальными рисками, включая внедрение запросов (prompt injection), отравление инструментов, перехват сессий, проблемы «исполнителя по ошибке» (confused deputy), уязвимости перенаправления токенов и чрезмерные разрешения, требующие специализированных мер защиты

- **Отличная аутентификация и авторизация**: Внедрите надежную аутентификацию с использованием внешних провайдеров идентификации (Microsoft Entra ID), обеспечьте правильную проверку токенов и никогда не принимайте токены, явно не выданные вашему MCP серверу

- **Предотвращение атак на ИИ**: Разверните Microsoft Prompt Shields и Azure Content Safety для защиты от косвенного внедрения запросов и отравления инструментов, проверяйте метаданные инструментов и следите за динамическими изменениями

- **Безопасность сессий и транспорта**: Используйте криптографически защищенные, недетерминированные идентификаторы сессий, связанные с идентификацией пользователей, реализуйте правильное управление жизненным циклом сессий и никогда не используйте сессии для аутентификации

- **Лучшие практики безопасности OAuth**: Предотвращайте атаки типа «исполнитель по ошибке» через явное согласие пользователя для динамически регистрируемых клиентов, правильную реализацию OAuth 2.1 с PKCE и строгую проверку redirect URI

- **Принципы безопасности токенов**: Избегайте антипаттернов перенаправления токенов, проверяйте claims аудитории токенов, внедряйте краткоживущие токены с безопасной ротацией и поддерживайте четкие границы доверия

- **Комплексная безопасность цепочки поставок**: Обращайтесь со всеми компонентами экосистемы ИИ (модели, эмбеддинги, поставщики контекста, внешние API) с тем же уровнем безопасности, что и с традиционными зависимостями программного обеспечения

- **Непрерывное развитие**: Следите за быстрыми изменениями спецификаций MCP, вносите вклад в стандарты сообщества по безопасности и поддерживайте адаптивную позицию безопасности по мере развития протокола

- **Интеграция безопасности Microsoft**: Используйте комплексную экосистему безопасности Microsoft (Prompt Shields, Azure Content Safety, GitHub Advanced Security, Entra ID) для повышения защиты развертываний MCP
