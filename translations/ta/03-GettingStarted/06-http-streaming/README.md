# Model Context Protocol (MCP) உடன் HTTPS ஸ்ட்ரீமிங்

இந்த அத்தியாயம் HTTPS பயன்படுத்தி Model Context Protocol (MCP) மூலம் பாதுகாப்பான, அளவிடக்கூடிய மற்றும் நேரடி ஸ்ட்ரீமிங்கை செயல்படுத்துவதற்கான விரிவான வழிகாட்டுதலை வழங்குகிறது. ஸ்ட்ரீமிங்கிற்கான முனைப்புகள், கிடைக்கும் போக்குவரத்து முறைமைகள், MCP இல் ஸ்ட்ரீமிங் HTTP-ஐ செயல்படுத்துவது, பாதுகாப்பு சிறந்த நடைமுறைகள், SSE-இல் இருந்து இடமாற்றம் மற்றும் உங்கள் சொந்த ஸ்ட்ரீமிங் MCP செயலிகளை கட்டியெழுப்பும் நடைமுறைகள் ஆகியவற்றை கையாள்கிறது.

## MCP இல் போக்குவரத்து முறைமைகள் மற்றும் ஸ்ட்ரீமிங்

இந்த பகுதியில் MCP இல் கிடைக்கும் வெவ்வேறு போக்குவரத்து முறைமைகள் மற்றும் அவை கிளையன்ட்கள் மற்றும் சர்வர்களுக்கு இடையேயான நேரடி தொடர்புக்கு ஸ்ட்ரீமிங் திறன்களை இயற்றுவதில் என்ன பங்கு வகிக்கின்றன என்று ஆராய்கிறது.

### போக்குவரத்து முறைமை எது?

ஒரு போக்குவரத்து முறைமை என்பது கிளையன்ட் மற்றும் சர்வர் இடையே தரவை எப்படி பரிமாறுகின்றது என்று வரையறுக்கிறது. MCP பல்வேறு சூழல்கள் மற்றும் தேவைகளுக்கு பொருந்துமாறு பல போக்குவரத்து வகைகளை ஆதரிக்கிறது:

- **stdio**: நிலையான உள்ளீடு/வெளியீடு, உள்ளரங்க மற்றும் CLI அடிப்படையிலான கருவிகளுக்கு பொருந்தும். எளிமையானது ஆனால் வலை அல்லது கிளவுட்-க்கு பொருத்தமற்றது.
- **SSE (Server-Sent Events)**: சர்வர்கள் HTTP மூலமாக நேரடி புதுப்பிப்புகளை கிளையன்ட்களுக்கு தள்ள அனுமதிக்கிறது. வலை UI-களுக்கு சிறந்தது, ஆனால் அளவிடுதல் மற்றும் நெகிழ்வுத்தன்மையில் வரம்புகள் உள்ளன. MCP குறிப்புப்படுத்தல் 2025-06-18 ஆம் தேதியிடை, தனித்துத்தான் SSE போக்குவரத்துக்கான ஆதரவு நிறுத்தப்பட்டது மற்றும் அதற்கு பதிலாக "Streamable HTTP" போக்குவரத்து வந்துள்ளது.
- **Streamable HTTP**: நவீன HTTP அடிப்படையிலான ஸ்ட்ரீமிங் போக்குவரத்து, அறிவிப்புகள் மற்றும் சிறந்த அளவிடுதலை ஆதரிக்கிறது. பெரும்பாலான உற்பத்தி மற்றும் கிளவுட் சூழல்களுக்கு பரிந்துரைக்கப்படுகிறது.

### ஒப்பீட்டு அட்டவணை

இந்த போக்குவரத்து முறைமைகளின் வித்தியாசங்களைப் புரிந்து கொள்ள கீழ்காணும் ஒப்பீட்டு அட்டவணையை பாருங்கள்:

| போக்குவரத்து       | நேரடி புதுப்பிப்புகள் | ஸ்ட்ரீமிங் | அளவிடுதல்   | பயன்பாட்டு நிலை        |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | இல்லை            | இல்லை    | குறைவு      | உள்ளூர் CLI கருவிகள்     |
| SSE               | ஆம்              | ஆம்      | நடுத்தர     | வலை, நேரடி புதுப்பிப்புகள் |
| Streamable HTTP   | ஆம்              | ஆம்      | உயர்       | கிளவுட், பல கிளையன்ட்  |

> **கருத்து:** சரியான போக்குவரத்தை தேர்வு செய்வது செயல்திறன், அளவிடுதல் மற்றும் பயனர் அனுபவத்தில் தாக்கம் அளிக்கும். **Streamable HTTP** நவீன, அளவிடக்கூடிய மற்றும் கிளவுட்-தயார் செயலிகளுக்கு பரிந்துரைக்கப்படுகிறது.

முந்தைய அத்தியாயங்களில் உங்களுக்கு காட்டப்பட்ட stdio மற்றும் SSE போக்குவரத்துகளையும், இத்தகைய ஸ்ட்ரீமிங் HTTP போக்குவரத்துக்களையும் கவனியுங்கள்.

## ஸ்ட்ரீமிங்: கருத்துக்கள் மற்றும் முனைப்புகள்

ஸ்ட்ரீமிங் என்பதன் அடிப்படைக் கருத்துகளை மற்றும் முனைப்புகளை புரிந்து கொள்ளுதல், பயனுள்ள நேரடி தொடர்பு அமைப்புகளை உருவாக்குவதற்கு அவசியம்.

**ஸ்ட்ரீமிங்** என்பது நெட்வொர்க் நிரலாக்கத்தில் பயன்படுத்தப்படும் ஒரு தொழில்நுட்பம்; இது முழு பதிலை உருவாக்கும் வரை காத்திருக்காமல், சிறிய, கையாளக்கூடிய துண்டுகள் அல்லது நிகழ்வுகளின் தொடராக தரவை அனுப்பவும் பெறவும் அனுமதிக்கிறது. குறிப்பாக கீழ்காணும் சூழலுக்கு உதவியாக இருக்கும்:

- பெரிய கோப்புகள் அல்லது தரவுத்தொகுப்புகள்.
- நேரடி புதுப்பிப்புகள் (எ.கா., சந்திப்பு, முன்னேற்ற பட்டிகள்).
- நீண்டநேரம் ஓடும் கணினிசார் கணக்கீடுகள், இதில் பயனருக்கு தகவல் வழங்க வேண்டியது.

மேல்முறை ஸ்ட்ரீமிங் பற்றி தெரிந்து கொள்வதற்கு முக்கிய விவரங்கள்:

- தரவு ஒரு முறை முழுதாக அல்லாமல் தொடர்ச்சியாக வழங்கப்படுகிறது.
- கிளையன்ட் வந்து சேரும் தரவை உடனவே செயலாக்கலாம்.
- உணரப்படும் தாமதத்தை குறைத்து, பயனர் அனுபவத்தை மேம்படுத்துகிறது.

### ஏன் ஸ்ட்ரீமிங் பயன்படுத்த வேண்டும்?

ஸ்ட்ரீமிங் பயன்படுத்துவதற்கான காரணங்கள் கீழ்வருமாறு:

- பயனர்கள் உடனடி பின்னூட்டம் பெறுவர், முடிவடையும்போது மட்டும் அல்ல.
- நேரடி செயலிகள் மற்றும் பதிலளிக்கும் UI-களை இயக்குகின்றது.
- நெட்வொர்க்கும் கணினி வளங்களின் திறனுள்ள பயன்பாடு.

### எளிய எடுத்துக்காட்டு: HTTP ஸ்ட்ரீமிங் சர்வர் மற்றும் கிளையன்ட்

ஸ்ட்ரீமிங் எவ்வாறு செயல்படுகிறதென எளிமையான ஒரு எடுத்துக்காட்டு:

#### Python

**சர்வர் (Python, FastAPI மற்றும் StreamingResponse பயன்படுத்தி):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**கிளையன்ட் (Python, requests பயன்படுத்தி):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

இந்த எடுத்துக்காட்டு, எல்லா செய்திகளும் தயாராக காத்திருக்காமல், சர்வர் கிளையண்டிற்கு ஒரு தொடர் செய்திகளை அனுப்புகிறதென காட்டுகிறது.

**எப்படி செயல்படுகிறது:**

- சர்வர் ஒவ்வொரு செய்தியையும் தயாராக இருக்கும் பொழுதே உற்பத்தி செய்கிறது.
- கிளையன்ட் வரும் ஒவ்வொரு துண்டையும் பெற்றதும் அச்சிடுகிறது.

**தேவைகள்:**

- சர்வர் ஸ்ட்ரீமிங் பதிலாக (எ.கா., `StreamingResponse` FastAPI-இல்) பயன்படுத்த வேண்டும்.
- கிளையன்ட் பதிலை ஸ்ட்ரீமாக (`stream=True` requests-இல்) செயலாக்க வேண்டும்.
- Content-Type பொதுவாக `text/event-stream` அல்லது `application/octet-stream`.

#### Java

**சர்வர் (Java, Spring Boot மற்றும் Server-Sent Events பயன்படுத்தி):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**கிளையன்ட் (Java, Spring WebFlux WebClient பயன்படுத்தி):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java செயலாக்க குறிப்புகள்:**

- ஸ்ப்ரிங் பூட்டின் எதிரொலி கூறுகளுடன் `Flux` மூலம் ஸ்ட்ரீமிங் செய்கிறது
- `ServerSentEvent` நிகழ்வுகளுடன் கட்டமைக்கப்பட்ட ஸ்ட்ரீமிங் தருகிறது
- `WebClient` `bodyToFlux()` மூலம் எதிரொலி ஸ்ட்ரீமிங் வசதியை வழங்குகிறது
- `delayElements()` நிகழ்வுகளுக்கு இடையே செயலாக்க நேரத்தை உருவாக்குகிறது
- நிகழ்வுகளுக்கு வகைகள் (`info`, `result`) கொடுத்துள்ளன கிளையன்ட் மேலாண்மை மேம்படுத்தப்படுகிறது

### ஒப்பீடு: பாரம்பரிய ஸ்ட்ரீமிங் மற்றும் MCP ஸ்ட்ரீமிங்

ஒரு "பாரம்பரிய" முறையில் ஸ்ட்ரீமிங் மற்றும் MCP இல் ஸ்ட்ரீமிங் எப்படி வேறுபடுகிறது என்பதைக் குறைந்த சித்திரத்தில்:

| அம்சம்               | பாரம்பரிய HTTP ஸ்ட்ரீமிங்       | MCP ஸ்ட்ரீமிங் (அறிவிப்புகள்)           |
|-----------------------|--------------------------------|-------------------------------------|
| பிரதான பதில்          | துண்டுகளாக                     | ஒருங்கிணைந்த, கடைசியில்                 |
| முன்னேற்ற புதுப்பிப்புகள் | தரவு துண்டுகளாக அனுப்பப்படுகிறது | அறிவிப்புகளாக அனுப்பப்படுகிறது           |
| கிளையன்ட் தேவைகள்      | ஸ்ட்ரீம் செயலாக்க வேண்டும்       | செய்தி செயல்படுத்தும் நிரல் வேண்டும்           |
| பயன்பாடு             | பெரிய கோப்புகள், AI டோக்கன் ஸ்ட்ரீமுகள் | முன்னேற்றம், குறிகள், நேரடி பின்னூட்டம்       |

### முக்கிய வேறுபாடுகள்

மேலும் சில முக்கிய வேறுபாடுகள்:

- **தொடர்பு முறை:**
  - பாரம்பரிய HTTP ஸ்ட்ரீமிங்: எளிதான துண்டாக்கப்பட்ட தகவல் பரிமாற்ற உணர்வு
  - MCP ஸ்ட்ரீமிங்: கட்டமைக்கப்பட்ட அறிவிப்பு அமைப்பு JSON-RPC நெறிமுறை மூலம்

- **செய்தி வடிவம்:**
  - பாரம்பரிய HTTP: புதிய வரிகளுடன் இனிய வாக்கிய துண்டுகள்
  - MCP: கட்டமைக்கப்பட்ட LoggingMessageNotification பொருட்கள் உருவாக்கப்பட்டுள்ளன

- **கிளையன்ட் செயலாக்கம்:**
  - பாரம்பரிய HTTP: எளிய ஸ்ட்ரீமிங் பதில்களை செயலாக்கும் கிளையன்ட்
  - MCP: பலவகையான செய்திகளுக்கு செயல்படுத்தும் திறன் உடைய மேம்பட்ட கிளையன்ட்

- **முன்னேற்ற புதுப்பிப்புகள்:**
  - பாரம்பரிய HTTP: பிரதான பதில் ஸ்ட்ரீமின் பகுதியாக
  - MCP: முன்னேற்றம் தனித்தனி அறிவிப்புகளாக அனுப்பப்படுகிறது, பிரதான பதில் கடைசியில் வருகிறது

### பரிந்துரைகள்

பாரம்பரிய ஸ்ட்ரீமிங் (மேலே `/stream` என்கிய உதாரணத்தைக் காட்டிய) மற்றும் MCP வழியாக ஸ்ட்ரீமிங் செய்வதில் தேர்வு செய்யும்போது பரிந்துரைகள்:

- **எளிய ஸ்ட்ரீமிங் தேவைகளுக்கு:** பாரம்பரிய HTTP ஸ்ட்ரீமிங் எளிதாக செயலாக்கக்கூடியது மற்றும் அடிப்படையான தேவைகளுக்கு போதுமானது.

- **சிக்கலான, இடையூறான செயலிகளுக்கு:** MCP ஸ்ட்ரீமிங் வளமான மேட்டா தரவு மற்றும் அறிவிப்புகள் மற்றும் இறுதி முடிவுக்களின் வெளிநிலை பிரித்தலை வழங்கும் குறியீடு.

- **AI செயலிகளுக்காக:** MCP அறிவிப்பு அமைப்பு நீண்டநேரம் ஓடும் AI பணி முன்னேற்றத்தை பயனர்களுக்கு தெரிவிக்க சிறப்பாக உதவுகிறது.

## MCP இல் ஸ்ட்ரீமிங்

நீங்கள் இப்போது பாரம்பரிய ஸ்ட்ரீமிங் மற்றும் MCP இல் ஸ்ட்ரீமிங் இடையேயான பரிந்துரைகள் மற்றும் ஒப்பீடுகள் பற்றி சிலவற்றைக் காண்ந்துள்ளீர்கள். இப்போது MCP இல் ஸ்ட்ரீமிங் எப்படி செயல்படுகிறது என்பதனை விவரமாகப் பார்ப்போம்.

MCP கட்டமைப்பில் ஸ்ட்ரீமிங் எப்படி இயங்குகின்றது என்பதை அறிந்து கொள்ளுதல், நீண்டநேரம் ஓடும் செயலிகளுக்கு பயனர்களுக்கு நேரடி பாதுகாப்பான பின்னூட்டத்துடன் பதிலளிக்கும் உறுதியான செயலிகளை உருவாக்க அவசியம்.

MCP இல், பிரதான பதிலை துண்டுகளாக அனுப்புவது அல்ல, ஒரு கருவி ஒரு கோரிக்கையை செயல்படுத்தும் போது **அறிவிப்புகள்** கிளையன்டுக்கு அனுப்பப்படுகின்றன. இவற்றில் முன்னேற்ற புதுப்பிப்புகள், குறிகள் அல்லது பிற நிகழ்வுகள் அடங்கலாம்.

### அது எப்படி செயல்படுகிறது

பிரதான முடிவு இன்னும் ஒரு ஒரே பதிலாக அனுப்பப்படுகிறது. இருப்பினும், செயல்முறை நடப்பிலிருந்தே தனித்தனி செய்திகளாக அறிவிப்புகள் அனுப்பப்படலாம், இதனால் கிளையன்ட் நேரடி தகவலை பெறுகிறது. கிளையன்டு இவ்வாறு அனுப்பப்படும் அறிவிப்புகளை கையாளவும் காட்சி அளிக்கவும் முடிவு செய்ய வேண்டும்.

## அறிவிப்பு என்பதென்ன?

நாம் "அறிவிப்பு" என்று சொன்னோம், MCP சூழலில் அதின் பொருள் என்ன?

அறிவிப்பு என்பது நீண்டநேரம் ஓடும் செயல்முறையின் போது, முன்னேற்றம், நிலை அல்லது பிற நிகழ்வுகள் பற்றி சர்வர் கிளையண்டுக்கு அனுப்பும் செய்தி ஆகும். அறிவிப்புகள் தெளிவுத்தன்மையை மேம்படுத்தி பயனர் அனுபவத்தை செம்மைப்படுத்துகின்றன.

உதாரணமாக, கிளையன்ட் ஆரம்ப கைமாற்றத்தை சரிவர செய்துவிட்டதும் அறிவிப்பைக் அனுப்ப வேண்டும்.

அறிவிப்பு ஒரு JSON செய்தியாக காட்சியளிக்கிறது:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

அறிவிப்புகள் MCP இல் ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) எனும் தலைப்பிற்கு சேர்ந்தது.

Logging செயல்படுத்த சர்வர் இதைப் போல ஒரு அம்சமாக (feature/capability) செயல்படுத்த வேண்டும்:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> பயன்படுத்திய SDK இல் இருந்து, logging இயல்பாக இயங்கலாம் அல்லது அதற்கு உங்கள் சர்வர் இயல்பின்மையில் கட்டமைக்கப்பட வேண்டும்.

வித்தியாசமான வகை அறிவிப்புகள் உள்ளன:

| நிலை       | விளக்கம்                      | உதாரண பயன்பாடு             |
|-----------|-------------------------------|-----------------------------|
| debug     | விரிவான பிழைத்திருத்த தகவல்  | செயல்பாடு நுழைவு/வெளியேற்ற புள்ளிகள் |
| info      | பொது அறிவிப்பு செய்திகள்      | செயல்முறை முன்னேற்றம் புதுப்பிப்புகள்  |
| notice    | சாதாரண ஆனால் முக்கிய நிகழ்வுகள் | கட்டமைப்பு மாற்றங்கள்          |
| warning   | எச்சரிக்கை நிலைகள்           | பழைய அம்ச பயன்பாடு           |
| error     | பிழை நிலைகள்                | செயல்முறை தோல்விகள்           |
| critical  | கடுமையான நிலைகள்            | கணினிச் கூறு தோல்விகள்        |
| alert     | உடனடி நடவடிக்கை தேவை       | தரவு சேதம் கண்டறிதல்        |
| emergency | கணினி பயன்படுத்த முடியவில்லை    | முழுமையான கணினி தோல்வி       |

## MCP இல் அறிவிப்புகளை செயல்படுத்துதல்

MCP இல் அறிவிப்புகளை செயல்படுத்த, நீங்கள் சர்வரும் கிளையண்டும் நேரடி புதுப்பிப்புகளை கையாள வேண்டும். இதன் மூலம் நீண்டநேரம் ஓடும் செயல்பாடுகளின் போது பயன்பாட்டை உடனடி பின்னூட்டம் வழங்கும் வகையில் அமைக்க முடியும்.

### சர்வர் பக்கம்: அறிவிப்புகள் அனுப்புதல்

சர்வர் பக்கம் தொடங்குவோம். MCP இல், கருவிகள் கோரிக்கையை செயல்படுத்தும் போது அறிவிப்புகளை அனுப்ப முடியும். சர்வர் பொதுவாக `ctx` எனும் சந்தர்ப்ப பொருள் மூலம் செய்திகளை கிளையண்டுக்கு அனுப்புகிறது.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

மேலே உள்ள எடுத்துக்காட்டில், `process_files` கருவி ஒவ்வொரு கோப்பை செயல்படுத்தும் போது மூன்று அறிவிப்புகளை கிளையண்டிற்கு அனுப்புகிறது. `ctx.info()` முறை தகவல் செய்திகளை அனுப்ப பயன்படுத்தப்படுகிறது.

மேலும், அறிவிப்புகளை செயல்படுத்த, உங்கள் சர்வர் ஸ்ட்ரீமிங் போக்குவரத்தை (உதா., `streamable-http`) பயன்படுத்த வேண்டும் மற்றும் உங்கள் கிளையன்ட் அறிவிப்புகளை செயலாக்க ஒரு செய்தி நிரலை அமல்படுத்த வேண்டும். சர்வரை `streamable-http` போக்குவரத்து பயன்படுத்தச் செயல்படுத்த கீழே உள்ளது:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

இந்த .NET எடுத்துக்காட்டில் `ProcessFiles` கருவி `Tool` பண்புச் செயலியில் அலங்கரிக்கப்பட்டு ஒவ்வொரு கோப்பையும் செயல்படுத்தும் போது மூன்று அறிவிப்புகளையும் கிளையண்டிற்கு அனுப்புகிறது. `ctx.Info()` தகவல் விடுபடுத்த பயன்படுத்தப்படுகிறது.

.NET MCP சர்வரில் அறிவிப்புகளை இயக்க, ஸ்ட்ரீமிங் போக்குவரத்தைப் பயன்படுத்திச் செயற்படுவதை உறுதி செய்யுங்கள்:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### கிளையன்ட் பக்கம்: அறிவிப்புகளை பெறுதல்

அறிவிப்புகளை செயலாக்கி காட்சியளிக்கும் வகையில் செய்தி நிரலை கிளையண்டில் அமல்படுத்த வேண்டும்.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

மேல்குறிப்பிட்ட நிரலில், `message_handler` செயலி வரும் செய்தி அறிவிப்பா என்பதை சோதிக்கிறது. அப்படின், அறிவிப்பை அச்சிடுகிறது; இல்லாவிட்டால் சாதாரண சர்வர் செய்தியாக செயலாக்கிறது. மேலும் `ClientSession`-ஐ `message_handler` உடன் ஆரம்பித்துள்ள இதனால் வரும் அறிவிப்புகளை கையாள்கிறது.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

இந்த .NET எடுத்துக்காட்டில், `MessageHandler` செயலி வரும் செய்தி அறிவிப்பா என்பதை பரிசோதிக்கிறது. அது எனில் அறிவிப்பை அச்சிடுகிறது; இல்லெனில் சாதாரண சர்வர் செய்தியாக செயல்படுத்துகிறது. `ClientSession`-ஐ `ClientSessionOptions` மூலம் செய்தி நிரலை உடன் ஆரம்பிக்கிறார்.

அறிவிப்புகளை செயலில் வைத்திருக்க, சர்வர் `streamable-http` போன்ற ஸ்ட்ரீமிங் போக்குவரத்தைப் பயன்படுத்தி செயல்பட வேண்டும் மேலும் கிளையன்ட் அறிவிப்புகளை செயலாக்கும் செய்தி நிரலை உருவாக்க வேண்டும்.

## முன்னேற்ற அறிவிப்புகள் & சூழல்கள்

இந்த பகுதிகள் MCP இல் முன்னேற்ற அறிவிப்புகள் என்ற கருத்தை விளக்குகின்றன, அவை ஏன் முக்கியம், மற்றும் Streamable HTTP பயன்படுத்தி அவைகளைக் கிடைக்கும் வகையில் உருவாக்குவது பற்றியும் விளக்குகிறது. உங்கள் புரிந்துகொள்ளுதலை உறுதிப்படுத்த ஒரு நடைமுறை பணிகளும் உள்ளன.

முன்னேற்ற அறிவிப்புகள் நீண்டநேரம் ஓடும் செயல்முறைகளின் போது சர்வரிலிருந்து கிளையண்டுக்கு நேரடி செய்திகளாக அனுப்பப்படும். முழு செயல்முறை முடியும் வரை காத்திருக்காமல், சர்வர் தற்போதைய நிலையை கிளையண்டுக்கு தெளிவுபடுத்தியிருக்கிறது. இது தெளிவுத்தன்மையை, பயனர் அனுபவத்தை மேம்படுத்தி பிழைத்திருத்தத்தை எளிதாக்கும்.

**எடுத்துக்காட்டு:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ஏன் முன்னேற்ற அறிவிப்புகளை பயன்படுத்த வேண்டும்?

முன்னேற்ற அறிவிப்புகள் பின்வரை காரணங்களுக்காக முக்கியம்:

- **மேம்பட்ட பயனர் அனுபவம்:** பயனர்கள் வேலை முன்னேறுது என்று உடனே காண்கிறார்கள், முடிவில் மட்டும் அல்ல.
- **நேரடி பின்னூட்டம்:** கிளையண்டுகள் முன்னேற்ற பட்டிகள் அல்லது குறிகள் காட்டி செயலியை பதிலளிக்கப்போல் உணரச் செய்கிறது.
- **பிழைத்திருத்தம் மற்றும் கண்காணிப்பு எளிதாகும்:** டெவலப்பர்கள் மற்றும் பயனர்களுக்கு செயல்முறை எங்கு மெதுவாகவோ அல்லது திமிர்ச்சலோ இருப்பது தெரிகிறது.

### முன்னேற்ற அறிவிப்புகளை எப்படி செயல்படுத்துவது

MCP இல் முன்னேற்ற அறிவிப்புகளை செயல்படுத்துவது பின்வருமாறு:

- **சர்வரில்:** `ctx.info()` அல்லது `ctx.log()` மூலம், ஒவ்வொரு அங்கமானது செயலாக்கப்படும்போது அறிவிப்புகளை அனுப்புங்கள். இதன்மூலம் பிரதான முடிவு தயார் இல்லாமல் முன்னர் கிளையண்டிற்கு மெசேஜ் அனுப்பப்படுகிறது.
- **கிளையண்டில்:** வரும் அறிவிப்புகளை கேட்டு காட்சிப்படுத்த ஒரு செய்தி நிரலை செயல்படுத்துங்கள். இது அறிவிப்புகளையும் இறுதி முடிவையும் வேறுபடுத்த கையாளும்.

**சர்வர் எடுத்துக்காட்டு:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**கிளையண்ட் எடுத்துக்காட்டு:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## பாதுகாப்பு பரிசீலனைகள்

HTTP அடிப்படையிலான போக்குவரத்துகளுடன் MCP சர்வர்களை செயல்படுத்தும் போது, பல்வேறு தாக்குதல்களை தடுக்கவும் பாதுகாப்பை நல்லதாய்க் கவனிக்க வேண்டும்.

### பார்வை

HTTP மூலம் MCP சர்வர்களை வெளிப்படுத்தும் போது பாதுகாப்பு மிகவும் அவசியம். Streamable HTTP புதிய தாக்குதல் முகப்புகளை அறிமுகப்படுத்துவதால், துல்லியமான கட்டமைப்பும் தேவை.

### முக்கிய அம்சங்கள்

- **Origin தலைப்பு சரிபார்ப்பு**: DNS rebinding தாக்குதலைத் தடுக்கும் வகையில் எப்போதும் `Origin` தலைப்பை சரிபார்க்கவும்.
- **Localhost பைண்டிங்**: உள்ளக உருவாக்கத்துக்கு சர்வரை `localhost`-இல் பைண்ட் செய்யவும், பொதுப்பணியில் வெளிப்படுத்துவதைத் தவிர்க்க.
- **அங்கீகாரம்**: தயாரிப்பு வெளியீடுகளில் அங்கீகாரம் (API விசைகள், OAuth) செயல்படுத்தவும்.
- **CORS**: அக்கிரம அணுகலை கட்டுப்படுத்த குறுக்கு-மூலம் பகிர்வு கொள்கைகளை அமைக்கவும்.
- **HTTPS**: உற்பத்தியில் HTTPS பயன்படுத்தி வர்த்தகத்தை குறியாக்கவும்.

### சிறந்த நடைமுறைகள்

- எந்த கோரிக்கையையும் அங்கீகாரம் இல்லாமல் நம்ப வேண்டாம்.
- எல்லா அணுகல் மற்றும் பிழைகளையும் பதிவு செய்து கண்காணிக்கவும்.
- பாதுகாப்பு லாபங்களை மேம்படுத்த தவறாமல் அடிப்படைக் கூறுகளை புதுப்பிக்கவும்.

### சவால்கள்
- மேம்பாடு எளிமையுடன் பாதுகாப்பை சமநிலைப்படுத்தல்
- பலவித கிளையன்ட் சூழல்களுடன் இணக்கமூட்டல் உறுதி செய்தல்

## SSE இல் இருந்து Streamable HTTP க்கு மேம்படுத்தல்

இப்போது Server-Sent Events (SSE) பயன்படுத்தும் பயன்பாடுகளுக்கான MCP அமலாக்கங்களுக்கு மேம்பட்ட திறன்கள் மற்றும் நீண்டகால நிலைத்தன்மையை வழங்க Streamable HTTP க்கு மாற்றுதல் நல்லது.

### ஏன் மேம்படுத்த வேண்டும்?

SSE இல் இருந்து Streamable HTTP க்கு மாற்றுவதற்கான இரு முக்கிய காரணங்கள் உள்ளன:

- Streamable HTTP SSE ஐவிட சிறந்த அளவிட தன்மை, இணக்கமுடிவு மற்றும் விருத்தமான அறிவிப்பு ஆதரவை வழங்குகிறது.
- புதிய MCP பயன்பாடுகளுக்கான பரிந்துரைக்கப்பட்ட பரிமாற்ற முறையாகும்.

### மாற்று படிகள்

உங்கள் MCP பயன்பாடுகளில் SSE இருந்து Streamable HTTP க்கு எப்படி மாற்றலாம் என்பதை இங்கே காணலாம்:

- **சேவரின் குறியீட்டை புதுப்பிக்கவும்** `mcp.run()` இல் `transport="streamable-http"` உபயோகிக்க.
- **கிளையன்ட் குறியீட்டை புதுப்பிக்கவும்** SSE கிளையன்டின் இடத்தில் `streamablehttp_client` பயன்படுத்தவும்.
- **கிளையன்டில் செய்தி கைையாளரை செயல்படுத்தவும்** அறிவிப்புகளை செயலாக்க.
- **இணக்கமுடிவை சோதிக்கவும்** உள்ளடக்க கருவிகள் மற்றும் வேலைவழக்கங்களுடன்.

### இணக்கமுடிவை பராமரித்தல்

மாற்று செயல்முறை முழுவதும் அப்பாவையுள்ள SSE கிளையன்ட்களுடன் இணக்கமுண்டாக்க பரிந்துரைக்கப்படுகிறது. சில யுக்திகள் இங்கே:

- SSE மற்றும் Streamable HTTP இரண்டையும் வேறு முனைகளில் இயக்கி இரு பரிமாற்றங்களையும் ஆதரிக்கலாம்.
- கிளையன்ட்களை புதிய பரிமாற்றத்திற்கு படிப்படியாக மாற்றுங்கள்.

### சவால்கள்

மாற்று நடவடிக்கையின் போது பின்வரும் சவால்களைத் தீர்க்க வேண்டும்:

- அனைத்து கிளையன்ட்களும் புதுப்பிக்கப்பட்டுள்ளதை உறுதிப்படுத்தல்
- அறிவிப்பு விநியோகத்தில் மாறுபாடுகளை கையாளுதல்

## பாதுகாப்பு கருத்துக்கள்

ஏதாவது சர்வரை செயல்படுத்தும்போது, குறிப்பாக MCP இல் Streamable HTTP போன்ற HTTP அடிப்படையிலான பரிமாற்றங்களை பயன்படுத்தும்போது பாதுகாப்பு உச்ச முன்னுரிமையாக இருக்க வேண்டும்.

HTTP அடிப்படையிலான பரிமாற்றங்களுடன் MCP சர்வர்களை செயல்படுத்தும் போது, பல தாக்குதல் வழிகள் மற்றும் பாதுகாப்பு முறைகள் மீது கவனமாக இருக்க வேண்டும்.

### மேற்பார்வை

HTTP வழியாக MCP சர்வர்களை வெளிப்படுத்தும்போது பாதுகாப்பு மிகவும் முக்கியம். Streamable HTTP புதிய தாக்குதல் முகவரிகள் மற்றும் கவனமாக உள்ளமைவு தேவை.

பின்வரும் முக்கிய பாதுகாப்பு கருத்துக்கள் உள்ளன:

- **Origin Header சரிபார்ப்பு**: DNS மறுசார்பாடு தாக்குதல்களைத் தடுப்பதற்காக `Origin` தலைப்பை எப்போதும் சரிபார்க்கவும்.
- **Localhost பைண்டிங்**: உள்ளூர் மேம்பாட்டிற்காக சர்வர்கள் `localhost` க்கு பைன் செய்யப்படும், பொது இணையத்தை வெளிப்படுத்தாமல் இருக்க.
- **அங்கீகாரம்**: உற்பத்தியிற்கான அமலானங்களில் அங்கீகாரம் (API விசைகள், OAuth) செயல்படுத்தவும்.
- **CORS**: அணுகலை வரையறுக்கும் கடந்து-மேல்நாட்டார்வ பகிர்வு (CORS) கொள்கைகளை அமைக்கவும்.
- **HTTPS**: உற்பத்தியில HTTPS பயன்படுத்தி போக்குவரத்தை குறியாக்கம் செய்யவும்.

### சிறந்த நடைமுறைகள்

MCP ஸ்ட்ரீமிங் சர்வரில் பாதுகாப்பை செயல்படுத்தும் போது பின்வரும் சிறந்த நடைமுறைகளைக் கடைபிடிக்கவும்:

- வந்த கோரிக்கைகளை எப்போதும் சரிபார்த்தல் இல்லாமல் நம்ப வேண்டாம்.
- அனைத்து அணுகல்கள் மற்றும் பிழைகளை பதிவுசெய்தும் கண்காணித்தும் இருங்கள்.
- பாதுகாப்பு குறைகள் அடையாளம் காணும் வகையில் பொறுப்பேற்கும் நூலகங்களை அடிக்கடி புதுப்பித்தல்.

### சவால்கள்

MCP ஸ்ட்ரீமிங் சர்வரில் பாதுகாப்பை செயல்படுத்தும்போது நீங்கள் சில சவால்களை எதிர்கொள்வீர்கள்:

- மேம்பாட்டின் எளிமையுடன் பாதுகாப்பை சமநிலைப்படுத்தல்
- பலவித கிளையன்ட் சூழல்களுடன் இணக்கமுற அனுபவப்படுத்தல்

### பணிகள்: உங்கள் சொந்த ஸ்ட்ரீமிங் MCP பயன்பாட்டை கட்டியெழுது

**நிலைமை:**
ஒரு MCP சர்வர் மற்றும் கிளையன்டை கட்டியெழுதுங்கள், அங்கு சர்வர் ஒரு பொருட்களின் பட்டியலை (கோப்புகள் அல்லது ஆவணங்கள் போன்றவை) செயலாக்கி ஒவ்வொரு பொருளுக்கும் அறிவிப்பை அனுப்பும். கிளையன்ட் அந்த அறிவிப்புகளை வந்து சேர்ந்தவுடன் காட்சிப்படுத்த வேண்டும்.

**படிகள்:**

1. ஒரு சர்வர் கருவி செயற்படுத்தி பட்டியலை செயலாக்கி ஒவ்வொரு பொருளுக்குமான அறிவிப்புகளை அனுப்புங்கள்.
2. அறிவிப்புகளை நேரடியாக காட்சிப்படுத்த ஒரு செய்தி கைையாளரை கொண்ட கிளையன்டை செயல்படுத்தவும்.
3. உங்கள் செயல்பாட்டை சர்வரும் கிளையன்டும் இயக்கி சோதனை செய்யவும், அறிவிப்புகளை கவனிக்கவும்.

## கூடுதல் வாசிப்பு மற்றும் அடுத்த படி?

MCP ஸ்ட்ரீமிங் பயணத்தை தொடரவும் உங்கள் அறிவைப் பரப்பவும், இந்த பகுதி கூடுதல் வளங்கள் மற்றும் முன்னேற்ற பயன்பாடுகளை கட்டியெழுதுவதற்கான அடுத்த படிகள் வழங்குகிறது.
