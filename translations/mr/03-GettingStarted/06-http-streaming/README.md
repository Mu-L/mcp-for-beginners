# मॉडेल कॉन्टेक्स्ट प्रोटोकॉल (MCP) सह HTTPS स्ट्रीमिंग

हा विभाग सुरक्षित, प्रमाणबद्ध आणि रिअल-टाइम स्ट्रीमिंगसाठी मॉडेल कॉन्टेक्स्ट प्रोटोकॉल (MCP) चा वापर करून HTTPS सह स्ट्रीमिंग कशी अंमलात आणायची याचे सविस्तर मार्गदर्शन प्रदान करतो. यात स्ट्रीमिंगसाठी प्रेरणा, उपलब्ध ट्रान्सपोर्ट यंत्रणा, MCP मध्ये स्ट्रीमेबल HTTP कसे अंमलात आणायचे, सुरक्षा सर्वोत्तम पद्धती, SSE मधून स्थलांतर आणि स्वतःच्या स्ट्रीमिंग MCP अनुप्रयोग निर्मितीसाठी व्यावहारिक मार्गदर्शन यांचा समावेश आहे.

## MCP मध्ये ट्रान्सपोर्ट यंत्रणा आणि स्ट्रीमिंग

हा विभाग MCP मध्ये उपलब्ध विविध ट्रान्सपोर्ट यंत्रणा आणि क्लायंट आणि सर्व्हरमधील रिअल-टाइम संवादासाठी स्ट्रीमिंग क्षमता सक्षम करण्यात त्यांची भूमिका याचा शोध घेतो.

### ट्रान्सपोर्ट यंत्रणा म्हणजे काय?

ट्रान्सपोर्ट यंत्रणा हा क्लायंट आणि सर्व्हर दरम्यान डेटा कसा विनिमय होतो हे ठरवते. MCP विविध वातावरणे आणि गरजांनुसार विविध ट्रान्सपोर्ट प्रकार समर्थन करते:

- **stdio**: मानक इनपुट/आउटपुट, स्थानिक आणि CLI-आधारित साधनांसाठी योग्य. सोपे पण वेब किंवा क्लाउडसाठी योग्य नाही.
- **SSE (सर्व्हर-सेंट इव्हेंट्स)**: सर्व्हर HTTP द्वारे क्लायंटना रिअल-टाइम अपडेट पाठवू शकतात. वेब UI साठी चांगले, पण प्रमाणबद्धता आणि लवचिकतेत मर्यादित. MCP स्पेसिफिकेशन 2025-06-18 नुसार, स्वतंत्र SSE ट्रान्सपोर्ट दुर्द्शीत समजले गेले असून त्याऐवजी "Streamable HTTP" ट्रान्सपोर्टने त्याला बदलले आहे.
- **Streamable HTTP**: आधुनिक HTTP-आधारित स्ट्रीमिंग ट्रान्सपोर्ट, सूचना आणि उच्च प्रमाणबद्धता समर्थित. बहुतांश उत्पादन आणि क्लाउड परिस्थितीसाठी शिफारस केली जाते.

### तुलना सारणी

खालील तुलना सारणीकडे पहा ज्यातून या ट्रान्सपोर्ट यंत्रणांतील फरक समजेल:

| ट्रान्सपोर्ट         | रिअल-टाइम अपडेट्स | स्ट्रीमिंग | प्रमाणबद्धता | वापर प्रकरण                |
|---------------------|--------------------|------------|--------------|----------------------------|
| stdio               | नाही               | नाही       | कमी          | स्थानिक CLI साधने         |
| SSE                 | होय                | होय        | मध्यम       | वेब, रिअल-टाइम अपडेट्स   |
| Streamable HTTP     | होय                | होय        | उच्च         | क्लाउड, बहु-क्लायंट       |

> **टीप:** योग्य ट्रान्सपोर्ट निवडल्याने कार्यक्षमता, प्रमाणबद्धता आणि वापरकर्ता अनुभवावर परिणाम होतो. आधुनिक, प्रमाणबद्ध आणि क्लाउड-तयार अनुप्रयोगांसाठी **Streamable HTTP** शिफारसीय आहे.

पूर्वीच्या विभागांमध्ये आपण stdio आणि SSE हे ट्रान्सपोर्ट पाहिले होते आणि या विभागात Streamable HTTP ट्रान्सपोर्टने कसे काम होते ते समजले.

## स्ट्रीमिंग: संकल्पना आणि प्रेरणा

स्ट्रीमिंगपाठीमागील मूलभूत संकल्पना आणि प्रेरणा समजणे प्रभावी रिअल-टाइम संवाद प्रणाली अंमलात आणण्यासाठी आवश्यक आहे.

**स्ट्रीमिंग** ही नेटवर्क प्रोग्रामिंगमधील एक तंत्र आहे जिला मोठ्या प्रमाणात डेटा कळवल्या जातो आणि प्राप्त केला जातो तो लहान, व्यवस्थापनयोग्य तुकड्यांमध्ये किंवा घटनांच्या क्रमाने पाठवला जातो, संपूर्ण प्रतिसाद तयार होईपर्यंत थांबण्याऐवजी. हे विशेषतः उपयोगी आहे:

- मोठ्या फायली किंवा डेटा संचसाठी.
- रिअल-टाइम अपडेट्ससाठी (उदा. चॅट, प्रगती पट्ट्या).
- दीर्घकालीन संगणकीय प्रगती दरम्यान वापरकर्त्याला माहिती देण्यासाठी.

स्ट्रीमिंगची उच्च पातळीवरील मुख्य बाबी:

- डेटा प्रगतीशीलपणे वितरित होतो, सगळा एकदाच नाही.
- क्लायंट येताना डेटा प्रक्रिया करू शकतो.
- भासमान विलंब कमी होतो आणि वापरकर्ता अनुभव सुधारतो.

### स्ट्रीमिंग का वापरावे?

स्ट्रीमिंगच्या वापरासाठी खालील कारणे आहेत:

- वापरकर्त्याला लगेच अभिप्राय मिळतो, फक्त शेवटी नाही.
- रिअल-टाइम अनुप्रयोग व प्रतिसादात्मक UI सक्षम करते.
- नेटवर्क आणि संगणकीय संसाधनांचे अधिक कार्यक्षम वापर.

### सोपा उदाहरण: HTTP स्ट्रीमिंग सर्व्हर आणि क्लायंट

हे कसे स्ट्रीमिंग अंमलात आणू शकता याचे एक सोपे उदाहरण:

#### Python

**सर्व्हर (Python, FastAPI आणि StreamingResponse वापरून):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**क्लायंट (Python, requests वापरून):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

हा उदाहरण दर्शवतो की सर्व्हर ग्राहकाला संदेशांची मालिका तत्काळ पाठवतो, सर्व संदेश तयार होईपर्यंत थांबत नाही.

**हे कसे कार्य करते:**

- सर्व्हर प्रत्येक संदेश तयार होताच तो निर्माण करतो.
- क्लायंट प्रत्येक तुकडा येताच प्राप्त करून तो प्रिंट करतो.

**आवश्यकता:**

- सर्व्हरसाठी स्ट्रीमिंग प्रतिसाद वापरणे आवश्यक (उदा. FastAPI मध्ये `StreamingResponse`).
- क्लायंटने प्रतिसाद स्ट्रीम म्हणून प्रक्रिया करणे (`stream=True` requests मध्ये).
- `Content-Type` सामान्यतः `text/event-stream` किंवा `application/octet-stream`.

#### Java

**सर्व्हर (Java, Spring Boot आणि Server-Sent Events वापरून):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**क्लायंट (Java, Spring WebFlux WebClient वापरून):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java अंमलबजावणी टिपा:**

- Spring Boot च्या प्रतिक्रियाशील स्टॅकचा `Flux` वापरून स्ट्रीमिंग
- `ServerSentEvent` घटनांचे प्रकारांसह संरचित इव्हेंट स्ट्रीमिंग प्रदान करतो
- `WebClient` च्या `bodyToFlux()` सह प्रतिक्रियाशील स्ट्रीमिंग वापर
- `delayElements()` इव्हेंट्स दरम्यान प्रक्रिया वेळ सिम्युलेट करते
- इव्हेंट्सना त्यांच्या प्रकारांनुसार (`info`, `result`) वेगळ्या प्रक्रियेसाठी मदत आहे

### तुलना: क्लासिक स्ट्रीमिंग विरुद्ध MCP स्ट्रीमिंग

क्लासिक पद्धतीने स्ट्रीमिंग कसे कार्य करते आणि MCP मध्ये जर स्ट्रीमिंग कसे कार्य करते यातील फरक खालीलप्रमाणे दर्शवता येतो:

| वैशिष्ट्य                 | क्लासिक HTTP स्ट्रीमिंग          | MCP स्ट्रीमिंग (सूचनांसाठी)            |
|--------------------------|---------------------------------|----------------------------------------|
| मुख्य प्रतिसाद            | तुकड्यांमध्ये                    | एकदाच, शेवटी                          |
| प्रगती अपडेट्स           | डेटा तुकड्यांसारखे पाठवलेले    | सूचना म्हणून पाठवलेले                   |
| क्लायंटची आवश्यकता       | स्ट्रीम प्रक्रिया करू शकणे आवश्यक| संदेश हँडलर अंमलात आणणे आवश्यक         |
| वापर प्रकरण              | मोठ्या फायली, AI टोकन स्ट्रीम   | प्रगती, लॉग, रिअल-टाइम अभिप्राय        |

### प्रमुख फरक

त्याचप्रमाणे काही प्रमुख फरक:

- **संवाद नमुना:**
  - क्लासिक HTTP स्ट्रीमिंग: साधे तुकडा ट्रान्सफर कोडिंग वापरून डेटा पाठवतो
  - MCP स्ट्रीमिंग: JSON-RPC प्रोटोकॉलसह संरचित सूचना प्रणाली वापरतो

- **संदेश स्वरूप:**
  - क्लासिक HTTP: साध्या मजकूर तुकड्यांसह न्यूलाईन्स
  - MCP: मेटाडेटासह संरचित LoggingMessageNotification ऑब्जेक्ट्स

- **क्लायंट अंमलबजावणी:**
  - क्लासिक HTTP: साधा क्लायंट जो स्ट्रीमिंग प्रतिसाद प्रक्रिया करतो
  - MCP: अधिक जटिल क्लायंट जो वेगवेगळ्या प्रकारच्या संदेशांसाठी संदेश हँडलर ठेवतो

- **प्रगती अपडेट्स:**
  - क्लासिक HTTP: प्रगती मुख्य प्रतिसाद स्ट्रीमचा भाग
  - MCP: मुख्य प्रतिसाद शेवटी येतो, प्रगती वेगळ्या सूचना संदेशांद्वारे पाठवली जाते

### शिफारसी

क्लासिक स्ट्रीमिंग (जसे की `/stream` वापरून) विरुद्ध MCP द्वारे स्ट्रीमिंग निवडताना काही गोष्टी आम्ही शिफारस करतो:

- **सोप्या स्ट्रीमिंग गरजांसाठी:** क्लासिक HTTP स्ट्रीमिंग अंमलबजावणीस सोपे आणि मूलभूत गरजांसाठी पुरेसे आहे.

- **जटिल, संवादात्मक अनुप्रयोगांसाठी:** MCP स्ट्रीमिंग अधिक संरचित दृष्टिकोन देते ज्यामध्ये समृद्ध मेटाडेटा आणि सूचनांमधील व अंतिम निकालांमधील विभाजन आहे.

- **AI अनुप्रयोगांसाठी:** MCP ची सूचना प्रणाली दीर्घकालीन AI कार्यांसाठी विशेष उपयुक्त आहे जिथे प्रगतीविषयी वापरकर्त्यांना नियमित माहिती देणे आवश्यक असते.

## MCP मध्ये स्ट्रीमिंग

ठीक आहे, आपण इथेपर्यंत क्लासिक स्ट्रीमिंग आणि MCP मधील स्ट्रीमिंगमधील तुलना आणि शिफारसी पाहिल्या आहेत. आता पाहूया की MCP मध्ये स्ट्रीमिंग आपण कसे वापरू शकतो.

MCP फ्रेमवर्कमध्ये स्ट्रीमिंग कसे काम करते हे समजणे महत्वाचे आहे, जेणेकरून आपण प्रतिक्रियाशील अनुप्रयोग तयार करू शकता जे दीर्घकालीन ऑपरेशन्स दरम्यान वापरकर्त्यांना रिअल-टाइम अभिप्राय देतात.

MCP मध्ये, मुख्य प्रतिसाद तुकड्यांमध्ये पाठवायचा नाही, तर टूल विनंतीच्या प्रक्रिया दरम्यान क्लायंटला **सूचना** पाठवायच्या असतात. या सूचनांमध्ये प्रगती अपडेट्स, लॉग्स किंवा इतर घटना असू शकतात.

### हे कसे काम करते

मुख्य निकाल अजूनही एकदाच प्रतिसाद म्हणून पाठवला जातो. मात्र, प्रक्रिया दरम्यान सूचना स्वतंत्र संदेश म्हणून पाठवून क्लायंटला रिअल-टाइम अपडेट दिला जातो. क्लायंटने या सूचनांना हाताळण्याची आणि प्रदर्शित करण्याची क्षमता असली पाहिजे.

## सूचना म्हणजे काय?

आम्ही "सूचना" म्हटले, तर MCP च्या संदर्भात त्याचा अर्थ काय?

सूचना म्हणजे सर्व्हरकडून क्लायंटकडे पाठवलेला संदेश जो दीर्घकालीन ऑपरेशन दरम्यान प्रगती, स्थिती किंवा इतर घटनांविषयी माहिती देतो. सूचना पारदर्शकता वाढवतात आणि वापरकर्त्याचा अनुभव सुधारतात.

उदाहरणार्थ, क्लायंटने सर्व्हरसह प्राथमिक हैंडशेक पूर्ण केल्यानंतर एक सूचना पाठवायला हवी.

सूचना JSON संदेश म्हणून खालीलप्रमाणे दिसते:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

सूचना MCP मधील ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) या विषयाशी संबंधित आहेत.

लॉगिंग कार्यान्वित करण्यासाठी सर्व्हरने हे फीचर/क्षमता प्रमाणे सक्षम करावे लागते:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> वापरलेल्या SDK नुसार, लॉगिंग प्राथमिकरित्या सक्षम असू शकते, किंवा आपल्याला सर्व्हर कॉन्फिगरेशनमध्ये ते स्पष्टपणे सक्षम करावे लागेल.

## MCP मध्ये सूचना अंमलात आणणे

MCP मध्ये सूचना अंमलात आणण्यासाठी, आपण सर्व्हर आणि क्लायंट दोन्ही बाजू तयार करायला हव्यात जेणेकरून रिअल-टाइम अपडेट्स हाताळता येतील. यामुळे आपला अनुप्रयोग दीर्घकालीन ऑपरेशन्स दरम्यान वापरकर्त्यांना तत्काळ अभिप्राय देऊ शकतो.

### सर्व्हर बाजू: सूचना पाठवणे

सर्व्हर बाजूपासून सुरू करूया. MCP मध्ये, आपण असे टूल्स परिभाषित करता जे विनंत्या प्रक्रियेदरम्यान सूचना पाठवू शकतात. सर्व्हर संदर्भ ऑब्जेक्ट (साधारणपणे `ctx`) वापरून क्लायंटकडे संदेश पाठवतो.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

पूर्वीच्या उदाहरणात, `process_files` टूल प्रत्येक फाइल प्रक्रिया करताना क्लायंटकडे तीन सूचना पाठवते. `ctx.info()` पद्धत माहिती संदेश पाठवण्यासाठी वापरली जाते.

तसेच, सूचना सक्षम करण्यासाठी, आपला सर्व्हर स्ट्रीमिंग ट्रान्सपोर्ट (जसे `streamable-http`) वापरतो आणि क्लायंट संदेश हँडलर वापरून सूचना प्रक्रिया करतो याची खात्री करा. खाली कसे `streamable-http` ट्रान्सपोर्ट वापरून सर्व्हर सेटअप करायचा त्याचे उदाहरण आहे:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

या .NET उदाहरणात, `ProcessFiles` टूल `Tool` अ‍ॅट्रिब्यूटने सुसज्ज असून प्रत्येक फाइल प्रक्रिया करताना क्लायंटकडे तीन सूचना पाठवते. `ctx.Info()` पद्धत माहिती संदेश पाठवण्यासाठी वापरली जाते.

आपल्या .NET MCP सर्व्हरमध्ये सूचना सक्षम करण्यासाठी, स्ट्रीमिंग ट्रान्सपोर्ट वापरता याची खात्री करा:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### क्लायंट बाजू: सूचना प्राप्त करणे

क्लायंटने संदेश हँडलर अंमलात आणला पाहिजे जो येणाऱ्या सूचनांना प्रक्रिया करून प्रदर्शित करेल.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

पूर्वीच्या कोड मध्ये, `message_handler` फंक्शन तपासते की येणारा संदेश सूचना आहे का. असल्यास, सूचना छापतो; अन्यथा तो नियमित सर्व्हर संदेशाप्रमाणे प्रक्रिया करतो. तसेच `ClientSession` हे `message_handler` सह इनिशियलाईझ करण्यात आले की ज्याने येणाऱ्या सूचनांचा हाताळणी केली जाते.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

या .NET उदाहरणात, `MessageHandler` फंक्शन तपासते की येणारा संदेश सूचना आहे का. असल्यास, सूचना छापतो; अन्यथा तो नियमित सर्व्हर संदेशाप्रमाणे प्रक्रिया करतो. `ClientSession` संदेश हँडलरसह `ClientSessionOptions` द्वारे इनिशियलाईझ केला आहे.

सूचना सक्षम करण्यासाठी, आपला सर्व्हर स्ट्रीमिंग ट्रान्सपोर्ट (जसे `streamable-http`) वापरतो आणि आपला क्लायंट सूचना प्रक्रिया करणार्‍या संदेश हँडलर अंमलात आणतो याची खात्री करा.

## प्रगती सूचना व परिस्थिती

हा विभाग MCP मधील प्रगती सूचना काय आहेत, त्यांचे महत्त्व का आहे आणि Streamable HTTP वापरून त्यांची अंमलबजावणी कशी करायची याबद्दल स्पष्ट करतो. येथे एक व्यावहारिक कार्य सुद्धा आहे जे आपली समज अधिक मजबूत करेल.

प्रगती सूचना म्हणजे दीर्घकालीन ऑपरेशन्स दरम्यान सर्व्हरकडून क्लायंटकडे पाठवण्यात येणारे रिअल-टाइम संदेश. संपूर्ण प्रक्रिया पूर्ण होईपर्यंत वाट पाहण्याऐवजी, सर्व्हर वर्तमान स्थितीबद्दल क्लायंटला सतत अपडेट देतो. यामुळे पारदर्शकता, वापरकर्ता अनुभव सुधारतो आणि डीबगिंग सोपे होते.

**उदाहरण:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### प्रगती सूचना का वापराव्यात?

प्रगती सूचना काही कारणांसाठी अत्यंत आवश्यक आहेत:

- **सुधारित वापरकर्ता अनुभव:** वापरकर्त्याला कार्य प्रगतीदरम्यान सतत अपडेट दिसतात, फक्त शेवटी नाही.
- **रिअल-टाइम अभिप्राय:** क्लायंट प्रगती पट्ट्या किंवा लॉग्स दाखवू शकतो, ज्यामुळे अ‍ॅप प्रतिसादात्मक वाटते.
- **डिबगिंग आणि मॉनिटरिंग सोपे:** विकसक आणि वापरकर्ते पाहू शकतात की प्रक्रिया कुठे मंदीला कारणीभूत आहे किंवा अडचणीत आहे.

### प्रगती सूचना कशा अंमलात आणायच्या

MCP मध्ये प्रगती सूचना कशा अंमलात आणायच्या हे खालीलप्रमाणे:

- **सर्व्हरवर:** प्रत्येक आयटम प्रक्रिया करताना `ctx.info()` किंवा `ctx.log()` वापरुन सूचना पाठवा. हे मुख्य निकाल तयार होण्याआधीच क्लायंटला संदेश पाठवते.
- **क्लायंटवर:** असा संदेश हँडलर अंमलात आणा जो येणाऱ्या सूचनांना ऐकतो आणि प्रदर्शित करतो. हा हँडलर सूचना आणि अंतिम निकाल वेगळा करतो.

**सर्व्हर उदाहरण:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**क्लायंट उदाहरण:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## सुरक्षा विचार

जेव्हा MCP सर्व्हर HTTP-आधारित ट्रान्सपोर्टसह अंमलात आणले जातात, तेव्हा सुरक्षा अत्यंत महत्त्वाची असते आणि अनेक हल्ल्यांच्या दृष्टीकोनांच्या आणि संरक्षण यंत्रणांच्या काळजीपूर्वक तपासणीची गरज असते.

### आढावा

HTTP वर MCP सर्व्हर उघडल्यावर सुरक्षा अत्यंत आवश्यक आहे. Streamable HTTP नवीन हल्ल्यांचे धोक्यांची शक्यता निर्माण करते आणि काळजीपूर्वक कॉन्फिगरेशन आवश्यक आहे.

### प्रमुख मुद्दे

- **Origin हेडर सत्यापन:** DNS रिबाइंडिंग हल्ल्यांपासून संरक्षणासाठी नेहमी `Origin` हेडरची चाचणी करा.
- **Localhost बाइंडिंग:** स्थानिक विकासासाठी सर्व्हर `localhost` वर बाइंड करा, जेणेकरून ते सार्वजनिक इंटरनेटसाठी उघडे राहू नयेत.
- **प्रमाणीकरण:** उत्पादनासाठी API कीज, OAuth सारखे प्रमाणीकरण अमलात आणा.
- **CORS:** प्रवेश मर्यादित करण्यासाठी Cross-Origin Resource Sharing (CORS) धोरणे कॉन्फिगर करा.
- **HTTPS:** उत्पादनात ट्रॅफिक एनक्रिप्ट करण्यासाठी HTTPS वापरा.

### सर्वोत्तम पद्धती

- येणाऱ्या विनंत्यांवर कधीही तपासणी न करता पूर्ण विश्वास ठेवू नका.
- सर्व प्रवेश आणि त्रुटींचे लॉगिंग व निरीक्षण करा.
- सुरक्षा त्रुटींचे दुरुस्ती करण्यासाठी अवलंबित्वे नियमितपणे अपडेट करा.

### आव्हाने
- सुरक्षा आणि विकास सुलभतेमध्ये समतोल साधणे
- विविध क्लायंट पर्यावरणांसह सुसंगतता सुनिश्चित करणे

## SSE मधून Streamable HTTP वर अपग्रेड करणे

सध्या Server-Sent Events (SSE) वापरणाऱ्या अनुप्रयोगांसाठी, Streamable HTTP कडे स्थलांतर केल्याने तुमच्या MCP अंमलबजावणीस अधिक क्षमता आणि दीर्घकालीन टिकाऊपणा मिळतो.

### का अपग्रेड करावे?

SSE मधून Streamable HTTP कडे अपग्रेड करण्यासाठी दोन मुख्य कारणे आहेत:

- Streamable HTTP SSE पेक्षा चांगली स्केलेबिलिटी, सुसंगतता आणि अधिक समृद्ध सूचना समर्थन प्रदान करते.
- नवीन MCP अनुप्रयोगांसाठी याला शिफारस केलेले ट्रान्सपोर्ट आहे.

### स्थलांतर टप्पे

तुमच्या MCP अनुप्रयोगांमध्ये कसे SSE मधून Streamable HTTP कडे स्थलांतर कराल ते येथे दिले आहे:

- **सरवर कोड अपडेट करा** जेणेकरून `mcp.run()` मध्ये `transport="streamable-http"` वापरले जाईल.
- **क्लायंट कोड अपडेट करा** जेणेकरून SSE क्लायंट ऐवजी `streamablehttp_client` वापरले जाईल.
- **क्लायंटमध्ये संदेश हँडलर अमलात आणा** जेणेकरून सूचना प्रक्रिया करता येतील.
- **विद्यमान साधने आणि कार्यप्रवाहांसह सुसंगतता तपासा.**

### सुसंगतता राखणे

स्थलांतर प्रक्रियेदरम्यान विद्यमान SSE क्लायंट्ससह सुसंगतता राखणे शिफारसीय आहे. येथे काही धोरणे दिली आहेत:

- तुम्ही दोन्ही SSE आणि Streamable HTTP दोन्ही समर्थित करू शकता, वेगवेगळ्या एंडपॉइंटवर दोन्ही ट्रान्सपोर्ट्स चालवून.
- हळूहळू क्लायंट्सना नवीन ट्रान्सपोर्टवर स्थलांतर करा.

### आव्हाने

स्थलांतर दरम्यान खालील आव्हाने हाताळण्याची खात्री करा:

- सर्व क्लायंट्स अपडेट झाले आहेत याची खात्री करणे
- नोटिफिकेशन वितरणातील फरक हाताळणे

## सुरक्षा विचार

कोणत्याही सर्व्हरच्या अंमलबजावणीत सुरक्षा ही सर्वोच्च प्राधान्याची बाब असावी, विशेषतः MCP मध्ये Streamable HTTP सारख्या HTTP आधारित ट्रान्सपोर्ट्स वापरताना.

HTTP आधारित ट्रान्सपोर्टसह MCP सर्व्हर अंमलबजावणी करताना सुरक्षा हा एक अत्यंत महत्त्वाचा प्रश्न असतो ज्यासाठी अनेक हल्ल्यांच्या मार्गांवर लक्ष देणे आणि संरक्षण यंत्रणा लागू करणे आवश्यक असते.

### आढावा

MCP सर्व्हर HTTP द्वारे उघडताना सुरक्षा अतिशय महत्त्वाची आहे. Streamable HTTP नवीन हल्ला पृष्ठभाग सादर करते आणि काळजीपूर्वक संरचना आवश्यक आहे.

येथे काही मुख्य सुरक्षा विचार दिले आहेत:

- **Origin Header वैधता तपासा**: DNS रीबाइंडिंग हल्ल्यांपासून रोखण्यासाठी नेहमी `Origin` हेडर वैधता तपासा.
- **Localhost बाइंडिंग**: स्थानिक विकासासाठी, सर्व्हर `localhost` वर बाइंड करा जेणेकरून तो सार्वजनिक इंटरनेटवर उघडणार नाही.
- **प्रमाणीकरण**: उत्पादन डिप्लॉयमेंटसाठी प्रमाणीकरण (उदाहरणार्थ API कीज, OAuth) लागू करा.
- **CORS**: प्रवेश मर्यादित करण्यासाठी Cross-Origin Resource Sharing (CORS) धोरणे कॉन्फिगर करा.
- **HTTPS**: प्रवाह एन्क्रिप्ट करण्यासाठी उत्पादनात HTTPS वापरा.

### सर्वोत्तम पद्धती

तुमच्या MCP स्ट्रीमिंग सर्व्हरची सुरक्षा अंमलबजावणी करताना खालील सर्वोत्तम पद्धती अंमलात आणा:

- येणाऱ्या विनंत्यांवर पडताळणी न करता कधीही विश्वास ठेवू नका.
- सर्व प्रवेश आणि त्रुटी नोंदवा आणि निरीक्षण करा.
- सुरक्षा कमकुवतता दूर करण्यासाठी नियमितपणे अवलंबित्वे अद्ययावत करा.

### आव्हाने

MCP स्ट्रीमिंग सर्व्हरमध्ये सुरक्षा अंमलबजावणी करताना तुम्हाला काही आव्हानांना सामोरे जावे लागेल:

- विकास सुलभतेवर सुरक्षा कशी समतोल साधायची
- विविध क्लायंट पर्यावरणांसह सुसंगतता सुनिश्चित करणे

### कार्य: तुमचा स्वतःचा स्ट्रीमिंग MCP अॅप तयार करा

**परिस्थिती:**
एक MCP सर्व्हर आणि क्लायंट तयार करा जिथे सर्व्हर आयटम (उदाहरणार्थ फाइल्स किंवा दस्तऐवज) ची यादी प्रक्रिया करतो आणि प्रत्येक प्रक्रियाकृत आयटमसाठी सूचना पाठवतो. क्लायंट प्रत्येक सूचना पोहोचल्यावर ती प्रदर्शित करेल.

**टप्पे:**

1. एक सर्व्हर टूल तयार करा जे यादी प्रक्रिया करते आणि प्रत्येक आयटमसाठी सूचना पाठवते.
2. संदेश हँडलर असलेला क्लायंट तयार करा जो सूचना रिअल टाइम मध्ये दाखवेल.
3. दोन्ही सर्व्हर आणि क्लायंट चालवून तुमच्या अंमलबजावणीची चाचणी करा आणि सूचना निरीक्षण करा.
