# HTTPS Стриминг са Протоколом за Контекст Модела (MCP)

Ово поглавље пружа свеобухватан водич за имплементацију безбедног, скалабилног и реално-временског стриминга са Протоколом за Контекст Модела (MCP) користећи HTTPS. Обухвата мотивацију за стриминг, доступне транспортне механизме, како имплементирати стримовани HTTP у MCP, најбоље безбедносне праксе, миграцију са SSE и практичне савете за изградњу ваших сопствених MCP апликација са стримингом.

## Транспортни Механизми и Стриминг у MCP

Овај одељак истражује различите доступне транспортне механизме у MCP и њихову улогу у омогућавању могућности стриминга за реално-временску комуникацију између клијената и сервера.

### Шта је транспортни механизам?

Транспортни механизам дефинише како се подаци размењују између клијента и сервера. MCP подржава више типова транспорта да би одговарао различитим окружењима и захтевима:

- **stdio**: Стандардни улаз/излаз, погодан за локалне и ЦЛИ алате. Једноставан али није прикладан за веб или облак.
- **SSE (Server-Sent Events)**: Омогућава серверима да шаљу реално-временска ажурирања клијентима преко HTTP. Добро за веб корисничке интерфејсе, али ограничено скалабилношћу и флексибилношћу. Од MCP Спецификације 2025-06-18, самостални SSE (Server-Sent Events) транспорт је застарео и замијењен је транспортом „Streamable HTTP“.
- **Streamable HTTP**: Модерни HTTP-базирани транспорт за стриминг, који подржава нотификације и бољу скалабилност. Препоручује се за већину продукцијских и облачних сценарија.

### Табела поређења

Погледајте доњу табелу поређења да бисте разумели разлике између ових транспортних механизама:

| Transport         | Real-time Updates | Streaming | Scalability | Use Case                |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | Не               | Не        | Ниска       | Локални ЦЛИ алати       |
| SSE               | Да               | Да        | Средња      | Веб, реално-временска ажурирања |
| Streamable HTTP   | Да               | Да        | Висока      | Облак, више клијената   |

> **Савет:** Избор правог транспорта утиче на перформансе, скалабилност и корисничко искуство. **Streamable HTTP** је препоручен за модерне, скалабилне и облаку прилагођене апликације.

Обратите пажњу на транспортe stdio и SSE које сте видели у претходним поглављима и како је Streamable HTTP транспорт који се покрива у овом поглављу.

## Стриминг: Концепти и Мотивација

Разумевање основних појмова и мотивације иза стриминга је неопходно за имплементацију ефикасних система за реално-временску комуникацију.

**Стриминг** је техника у мрежном програмирању која омогућава слање и примање података у малим, управљивим деловима или као низ догађаја, уместо да се чека цео одговор да буде спреман. Ово је посебно корисно за:

- Велике датотеке или скупова података.
- Реално-временска ажурирања (нпр. чет, траке напретка).
- Дуготрајне израчуне када желите да корисник буде информисан.

Ево шта треба да знате о стримингу на високом нивоу:

- Подаци се испоручују прогресивно, не сви одједном.
- Клијент може да обрађује податке како стижу.
- Смањује перципирано кашњење и побољшава корисничко искуство.

### Зашто користити стриминг?

Разлози за коришћење стриминга су следећи:

- Корисници добијају повратну информацију одмах, а не само на крају
- Омогућава реално-временске апликације и одзивне корисничке интерфејсе
- Ефикаснија употреба мрежних и рачунарских ресурса

### Једноставан пример: HTTP стриминг сервер и клијент

Ево једноставног примера како се стриминг може имплементирати:

#### Python

**Сервер (Python, користећи FastAPI и StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**Клијент (Python, користећи requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

Овај пример приказује сервер који шаље низ порука клијенту како постају доступне, уместо да чека да све поруке буду спремне.

**Како функционише:**

- Сервер издаје сваку поруку како је спремна.
- Клијент прима и штампа сваки део како стиже.

**Захтеви:**

- Сервер мора користити стриминг одговор (нпр. `StreamingResponse` у FastAPI).
- Клијент мора обрађивати одговор као ток (`stream=True` у requests).
- Content-Type је обично `text/event-stream` или `application/octet-stream`.

#### Java

**Сервер (Java, користећи Spring Boot и Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**Клијент (Java, користећи Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Напомене о имплементацији у Јави:**

- Користи реактивни стек Spring Boot-а са `Flux` за стриминг
- `ServerSentEvent` пружа структуриран стриминг догађаја са типовима догађаја
- `WebClient` са `bodyToFlux()` омогућава реактивну конзумцију стриминга
- `delayElements()` симулира време обраде између догађаја
- Догађаји могу имати типове (`info`, `result`) за боље руковање на клијенту

### Поређење: Класични стриминг против MCP стриминга

Разлике између класичног стриминга и MCP стриминга могу се илустровати овако:

| Карактеристика          | Класични HTTP стриминг         | MCP стриминг (Нотификације)      |
|-------------------------|--------------------------------|----------------------------------|
| Главни одговор          | Подељен на делове               | Један, на крају                   |
| Напредак ажурирања      | Слање као делове података       | Слање као нотификације           |
| Захтеви клијента        | Мора да обрађује стрим          | Мора да имплементира обраду порука |
| Примена                | Велике датотеке, AI токен стримови | Напредак, логови, реално-временска повратна информација |

### Кључне разлике

Поред тога, ево неких кључних разлика:

- **Обрасци комуникације:**
  - Класични HTTP стриминг: Користи једноставно преношење података у деловима
  - MCP стриминг: Користи структуриран систем нотификација са JSON-RPC протоколом

- **Формат поруке:**
  - Класични HTTP: Обични текстуални делови са новим редовима
  - MCP: Структурирани LoggingMessageNotification објекти са метаподацима

- **Имплементација клијента:**
  - Класични HTTP: Једноставан клијент који обрађује стриминг одговоре
  - MCP: Сложенији клијент са обрадом порука за обраду различитих типова порука

- **Напредак ажурирања:**
  - Класични HTTP: Напредак је део главног стрима одговора
  - MCP: Напредак се шаље посебним нотификацијама док главни одговор стиже на крају

### Препоруке

Постоје неке ствари које препоручујемо када треба изабрати између имплементације класичног стриминга (као endpoint који смо вам показали горе коришћењем `/stream`) и стриминга преко MCP.

- **За једноставне потребе стриминга:** Класични HTTP стриминг је једноставнији за имплементацију и довољан за основне потребе стриминга.

- **За сложене, интерактивне апликације:** MCP стриминг пружа структуиранији приступ са богатим метаподацима и раздвајање између нотификација и коначних резултата.

- **За AI апликације:** MCP систем нотификација је посебно користан за дуготрајне AI задатке где желите да кориснике обавештавате о напретку.

## Стриминг у MCP

Добро, видели сте неке препоруке и поређења до сада у вези разлике између класичног стриминга и стриминга у MCP. Хајде да детаљније видимо како тачно можете искористити стриминг у MCP.

Разумевање како стриминг функционише у оквиру MCP је од суштинског значаја за изградњу одзивних апликација које пружају повратне информације у реалном времену корисницима током дуготрајних операција.

У MCP-у, стриминг није слање главног одговора у деловима, већ слање **нотификација** клијенту док алат обрађује захтев. Ове нотификације могу укључивати ажурирања напретка, логове или друге догађаје.

### Како функционише

Главни резултат се и даље шаље као један одговор. Међутим, нотификације могу бити послате као одвојене поруке током обраде и тиме ажурирати клијента у реалном времену. Клијент мора моћи да обрађује и приказује ове нотификације.

## Шта је Нотификација?

Рекли смо „Нотификација“, шта то значи у контексту MCP?

Нотификација је порука послата са сервера клијенту да би га обавестила о напретку, статусу или другим догађајима током дуготрајне операције. Нотификације побољшавају транспарентност и корисничко искуство.

На пример, клијент треба да пошаље нотификацију када је успостављен почетни рукопостављање са сервером.

Нотификација изгледа овако као JSON порука:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

Нотификације припадају теми у MCP познатој као ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

Да би логовање радило, сервер мора омогућити ову функцију/способност на овако:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> У зависности од коришћеног SDK-а, логовање може бити подразумевано омогућено, или ћете можда морати експлицитно да га омогућите у конфигурацији сервера.

Постоје различите врсте нотификација:

| Ниво      | Опис                           | Пример употребе               |
|-----------|--------------------------------|------------------------------|
| debug     | Детаљне дијагностичке информације | Точки уласка/изласка функције |
| info      | Опште информативне поруке       | Ажурирања напретка операције |
| notice    | Нормални али значајни догађаји  | Промене конфигурације         |
| warning   | Упозорења                      | Коришћење застарелих функција |
| error     | Услов грешке                   | Неуспеси операција            |
| critical  | Критични услови                | Квара системских компоненти   |
| alert     | Мора се одмах реаговати        | Откривена корупција података |
| emergency | Систем је неисправан           | Потпуни квар система         |

## Имплементација нотификација у MCP

Да бисте имплементирали нотификације у MCP, потребно је да подесите и серверску и клијентску страну да обрађују реално-временска ажурирања. Ово омогућава вашој апликацији да корисницима пружи одмах повратне информације током дуготрајних операција.

### Серверска страна: Слање нотификација

Почнимо са серверском страном. У MCP-у дефинишете алате који могу слати нотификације током обраде захтева. Сервер користи контекстни објекат (обично `ctx`) за слање порука клијенту.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

У претходном примеру, алат `process_files` шаље три нотификације клијенту како обрађује сваки фајл. Метод `ctx.info()` се користи за слање информативних порука.

Такође, да бисте омогућили нотификације, уверите се да ваш сервер користи стриминг транспорт (нпр. `streamable-http`), а клијент имплементира обраду порука за процесирање нотификација. Ево како да подесите сервер да користи `streamable-http` транспорт:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

У овом .NET примеру, алат `ProcessFiles` је означен атрибутом `Tool` и шаље три нотификације клијенту док обрађује сваки фајл. Метод `ctx.Info()` се користи за слање информативних порука.

Да бисте омогућили нотификације у вашем .NET MCP серверу, уверите се да користите стриминг транспорт:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### Клијентска страна: Пријем нотификација

Клијент мора имплементирати обраду порука да би процесирао и приказивао нотификације како стижу.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

У претходном коду, функција `message_handler` проверава да ли је долазна порука нотификација. Ако јесте, штампа нотификацију; иначе, обрађује је као обичну серверску поруку. Такође, обратите пажњу како је `ClientSession` иницијализован са `message_handler` за обраду долазних нотификација.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

У овом .NET примеру, функција `MessageHandler` проверава да ли је долазна порука нотификација. Ако јесте, штампа нотификацију; иначе, обрађује је као обичну серверску поруку. `ClientSession` је иницијализован са обрадом порука преко `ClientSessionOptions`.

Да бисте омогућили нотификације, уверите се да ваш сервер користи стриминг транспорт (нпр. `streamable-http`) и да ваш клијент имплементира обраду порука за нотификације.

## Прогрес нотификације & сценарији

Овај одељак објашњава концепт прогрес нотификација у MCP, зашто су важне и како их имплементирати користећи Streamable HTTP. Такође ћете наћи и практичан задатак за учвршћивање разумевања.

Прогрес нотификације су реално-временске поруке послате са сервера клијенту током дуготрајних операција. Уместо да се чека да цео процес заврши, сервер стално обавештава клијента о тренутном статусу. Ово побољшава транспарентност, корисничко искуство и олакшава дијагностику.

**Пример:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### Зашто користити прогрес нотификације?

Прогрес нотификације су важне из више разлога:

- **Боље корисничко искуство:** Корисници виде ажурирања како рад напредује, а не само на крају.
- **Повратна информација у реалном времену:** Клијенти могу приказивати траке напретка или логове, чинећи апликацију одзивнијом.
- **Лакша дијагностика и праћење:** Програмери и корисници могу видети где процес може бити спор или заглављен.

### Како имплементирати прогрес нотификације

Ево како можете имплементирати прогрес нотификације у MCP:

- **На серверу:** Користите `ctx.info()` или `ctx.log()` да шаљете нотификације како се сваки предмет обрађује. Ово шаље поруку клијенту пре него што главни резултат буде спреман.
- **На клијенту:** Имплементирајте обраду порука која слуша и приказује нотификације како стижу. Тај обработач разликује између нотификација и коначног резултата.

**Пример сервера:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**Пример клијента:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## Безбедносни аспекти

Када имплементирате MCP сервере са HTTP-базираним транспортом, безбедност постаје приоритетни аспект који захтева пажљиво посматрање више нападачких вектора и механизама заштите.

### Преглед

Безбедност је критична када се MCP сервери излажу преко HTTP-а. Streamable HTTP уводи нове површине напада и захтева пажљиву конфигурацију.

### Кључне тачке

- **Валидација Origin заглавља:** Увек валидирајте `Origin` заглавље да бисте спречили DNS rebinding нападе.
- **Повеезивање на localhost:** За локални развој везујте сервере за `localhost` да их не бисте изложили јавном интернету.
- **Аутентификација:** Имплементирајте аутентификацију (нпр. API кључеве, OAuth) за продукцијска окружења.
- **CORS:** Конфигуришите политике Cross-Origin Resource Sharing (CORS) да ограничите приступ.
- **HTTPS:** Користите HTTPS у продукцији за шифровање саобраћаја.

### Најбоље праксе

- Никада не верујте долазним захтевима без валидирања.
- Логовање и праћење сваког приступа и грешака.
- Редовно ажурирајте зависности да бисте закрпили безбедносне рањивости.

### Изазови
- Балансирање безбедности са лакоћом развоја
- Обезбеђивање компатибилности са различитим окружењима клијената

## Надоградња са SSE на Streamable HTTP

За апликације које тренутно користе Server-Sent Events (SSE), миграција на Streamable HTTP пружа побољшане могућности и бољу дугорочну одрживост ваших MCP имплементација.

### Зашто надоградити?

Постоје два убедљива разлога за надоградњу са SSE на Streamable HTTP:

- Streamable HTTP нуди бољу скалабилност, компатибилност и богатију подршку за нотификације него SSE.
- Препоручени је транспорт за нове MCP апликације.

### Кораци миграције

Ево како можете мигрирати са SSE на Streamable HTTP у вашим MCP апликацијама:

- **Ажурирајте код сервера** да користи `transport="streamable-http"` у `mcp.run()`.
- **Ажурирајте код клијента** да користи `streamablehttp_client` уместо SSE клијента.
- **Имплементирајте обраду порука** на клијенту за обраду нотификација.
- **Тестирајте компатибилност** са постојећим алатима и радним токовима.

### Одржавање компатибилности

Препоручује се одржавање компатибилности са постојећим SSE клијентима током процеса миграције. Ево неких стратегија:

- Можете подржати и SSE и Streamable HTTP тако што ћете покретати оба транспорта на различитим крајњим тачкама.
- Постепено мигрирајте клијенте на нови транспорт.

### Изазови

Обезбедите да решите следеће изазове током миграције:

- Обезбеђивање да су сви клијенти ажурирани
- Руковање разликама у испоруци нотификација

## Безбедносне разматрања

Безбедност треба да буде највиши приоритет када имплементирате било који сервер, посебно када користите HTTP базиране транспорте као што је Streamable HTTP у MCP.

Када имплементирате MCP сервере са HTTP базираним транспортерима, безбедност постаје кључни аспект који захтева пажљиву пажњу према више нападачких вектора и заштитних механизама.

### Преглед

Безбедност је критична када изложите MCP сервере преко HTTP. Streamable HTTP уводи нове површине за нападе и захтева пажљиву конфигурацију.

Ево неких кључних безбедносних разматрања:

- **Валидација Origin заглавља**: Увек проверите `Origin` заглавље како бисте спречили DNS rebinding нападе.
- **Повезивање на localhost**: За локални развој, везујте сервере за `localhost` како бисте избегли излагање јавном интернету.
- **Аутентификација**: Имплементирајте аутентификацију (нпр. API кључеве, OAuth) за продукцијске имплементације.
- **CORS**: Конфигуришите CORS политике да бисте ограничили приступ.
- **HTTPS**: Користите HTTPS у продукцији за шифровање саобраћаја.

### Најбоље праксе

Додатно, ево неких најбољих пракси које треба пратити приликом имплементације безбедности у вашем MCP стриминг серверу:

- Никада не верујте долазним захтевима без валидације.
- Логирајте и пратите сав приступ и грешке.
- Редовно ажурирајте зависности како бисте закрпили безбедносне рањивости.

### Изазови

Суочићете се са неколико изазова приликом имплементације безбедности у MCP стриминг серверима:

- Балансирање безбедности са лакоћом развоја
- Обезбеђивање компатибилности са различитим окружењима клијената
