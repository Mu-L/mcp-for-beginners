# HTTPS Поточно предаване с протокол Model Context (MCP)

Тази глава предоставя изчерпателно ръководство за внедряване на сигурно, мащабируемо и поточно предаване в реално време с протокола Model Context Protocol (MCP) чрез HTTPS. Тя обхваща мотивацията за поточно предаване, наличните транспортни механизми, как да се реализира поточно HTTP в MCP, най-добри практики за сигурност, миграция от SSE и практическо ръководство за изграждане на собствени поточни MCP приложения.

## Транспортни механизми и поточно предаване в MCP

В този раздел се разглеждат различните налични транспортни механизми в MCP и тяхната роля за осигуряване на възможности за поточно предаване при комуникация в реално време между клиенти и сървъри.

### Какво е транспортен механизъм?

Транспортният механизъм определя как се обменят данни между клиента и сървъра. MCP поддържа няколко типа транспорт, за да отговори на различни среди и изисквания:

- **stdio**: Стандартен вход/изход, подходящ за локални инструменти и CLI. Прост, но не е подходящ за уеб или облак.
- **SSE (Server-Sent Events)**: Позволява на сървърите да изпращат актуализации в реално време към клиентите през HTTP. Подходящ за уеб интерфейси, но има ограничения по отношение на мащабируемостта и гъвкавостта. От MCP Спецификация 2025-06-18, автономният транспорт SSE е декларирано остарял и заменен с "Streamable HTTP" транспорт.
- **Streamable HTTP**: Модерен HTTP-базиран поточен транспорт, който поддържа уведомления и по-добра мащабируемост. Препоръчва се за повечето продукционни и облачни среди.

### Таблица за сравнение

Вижте таблицата за сравнение по-долу, за да разберете разликите между тези транспортни механизми:

| Транспорт        | Актуализации в реално време | Поточно предаване | Мащабируемост | Примерно приложение      |
|------------------|-----------------------------|-------------------|---------------|-------------------------|
| stdio            | Не                          | Не                | Ниска         | Локални CLI инструменти |
| SSE              | Да                          | Да                | Средна        | Уеб, актуализации в реално време |
| Streamable HTTP  | Да                          | Да                | Висока        | Облак, многоклиентски   |

> **Съвет:** Изборът на правилен транспорт повлиява на производителността, мащабируемостта и потребителското изживяване. **Streamable HTTP** се препоръчва за съвременни, мащабируеми и готови за облак приложения.

Обърнете внимание на транспорти stdio и SSE, които видяхте в предходните глави, както и как Streamable HTTP е транспортът, разгледан в тази глава.

## Поточно предаване: концепции и мотивация

Разбирането на основните концепции и мотивации зад поточното предаване е от съществено значение за ефективната реализация на системи за комуникация в реално време.

**Поточно предаване** е техника в мрежовото програмиране, която позволява данните да се изпращат и получават на малки, управляеми части или като последователност от събития, вместо да се чака цялостен отговор да бъде готов. Това е особено полезно за:

- Големи файлове или набори от данни.
- Актуализации в реално време (например чат, ленти за напредък).
- Продължителни изчисления, където искате да държите потребителя информиран.

Ето какво трябва да знаете за поточното предаване на високо ниво:

- Данните се предоставят постепенно, не наведнъж.
- Клиентът може да обработва данните, докато пристигат.
- Намалява възприеманата латентност и подобрява потребителското изживяване.

### Защо да използваме поточно предаване?

Причините за използване на поточно предаване са следните:

- Потребителите получават обратна връзка незабавно, а не само в края
- Позволява реалновременни приложения и отзивчиви интерфейси
- По-ефективно използване на мрежови и изчислителни ресурси

### Прост пример: Поточен HTTP сървър и клиент

Ето един прост пример как може да се реализира поточно предаване:

#### Python

**Сървър (Python, използващ FastAPI и StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**Клиент (Python, използващ requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

Този пример демонстрира сървър, който изпраща серия от съобщения към клиента, докато те стават налични, вместо да чака всички съобщения да са готови.

**Как работи:**

- Сървърът изпраща всяко съобщение веднага щом е готово.
- Клиентът получава и отпечатва всяка част, докато пристига.

**Изисквания:**

- Сървърът трябва да използва поточен отговор (например `StreamingResponse` във FastAPI).
- Клиентът трябва да обработва отговора като поток (`stream=True` в requests).
- Content-Type обикновено е `text/event-stream` или `application/octet-stream`.

#### Java

**Сървър (Java, използващ Spring Boot и Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**Клиент (Java, използващ Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Бележки за имплементацията на Java:**

- Използва реактивния стек на Spring Boot с `Flux` за поточно предаване
- `ServerSentEvent` предоставя структурирано поточно предаване със зададени типове събития
- `WebClient` с `bodyToFlux()` позволява реактивна консумация на потока
- `delayElements()` симулира време за обработка между събития
- Събитията могат да имат типове (`info`, `result`) за по-добра обработка от клиента

### Сравнение: Класическо поточно предаване срещу MCP Поточно предаване

Разликите между работата на поточното предаване по "класически" начин и как то работи в MCP могат да се илюстрират така:

| Функция                | Класическо HTTP поточно предаване | MCP Поточно предаване (уведомления) |
|------------------------|------------------------------------|-------------------------------------|
| Основен отговор         | На части (chunked)                 | Единствен, в края                   |
| Актуализации за напредък| Изпращат се като части от данни    | Изпращат се като уведомления         |
| Изисквания към клиента | Трябва да обработва потока          | Трябва да имплементира обработчик на съобщения |
| Приложение             | Големи файлове, AI токен потоци    | Напредък, логове, обратна връзка в реално време |

### Основни наблюдавани разлики

Освен това, ето някои ключови разлики:

- **Комуникационен модел:**
  - Класическо HTTP поточно предаване: Използва просто chunked transfer encoding за изпращане на данни на части
  - MCP поточно предаване: Използва структурирана система за уведомления с JSON-RPC протокол

- **Формат на съобщенията:**
  - Класическо HTTP: Обикновен текст с нови редове
  - MCP: Структурирани обекти LoggingMessageNotification с метаданни

- **Имплементация на клиента:**
  - Класическо HTTP: Прост клиент, който обработва поточни отговори
  - MCP: По-усъвършенстван клиент с обработчик на съобщения за различни типове съобщения

- **Актуализации на напредъка:**
  - Класическо HTTP: Напредъкът е част от основния поток
  - MCP: Напредъкът се изпраща чрез отделни съобщения уведомления, докато основният отговор идва накрая

### Препоръки

Има някои препоръки при избора между реализиране на класическо поточно предаване (като крайна точка показана по-горе с `/stream`) и избор на стрийминг чрез MCP.

- **За прости нужди за поточно предаване:** Класическото HTTP поточно предаване е по-просто за имплементиране и е достатъчно за основни нужди.

- **За сложни, интерактивни приложения:** MCP поточно предаване предоставя по-структуриран подход с по-богати метаданни и отделяне на уведомленията от крайния резултат.

- **За AI приложения:** Системата за уведомления на MCP е особено полезна за дълготрайни AI задачи, когато искате да държите потребителите информирани за напредъка.

## Поточно предаване в MCP

Добре, вече видяхте някои препоръки и сравнения за разликата между класическо поточно предаване и това в MCP. Нека разгледаме подробно точно как можете да използвате поточно предаване в MCP.

Разбирането как работи поточното предаване в рамките на MCP е от съществено значение за изграждането на отзивчиви приложения, които предоставят обратна връзка в реално време на потребителите по време на дълготрайни операции.

В MCP поточното предаване не се състои в изпращане на основния отговор на части, а в изпращане на **уведомления** към клиента докато инструментът обработва заявка. Тези уведомления могат да включват актуализации на напредъка, логове или други събития.

### Как работи

Основният резултат все още се изпраща като един отговор. Въпреки това, уведомленията се изпращат като отделни съобщения по време на обработка и по този начин актуализират клиента в реално време. Клиентът трябва да може да ги обработва и показва.

## Какво е Уведомление?

Казахме „Уведомление“, какво означава това в контекста на MCP?

Уведомлението е съобщение от сървъра към клиента, което информира за напредъка, състоянието или други събития по време на продължителна операция. Уведомленията подобряват прозрачността и потребителското изживяване.

Например, клиент трябва да изпрати уведомление веднага след като първоначалното свързване със сървъра е осъществено.

Уведомление изглежда така като JSON съобщение:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

Уведомленията принадлежат към тема в MCP, наречена ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

За да заработи логването, сървърът трябва да го активира като функция/възможност, както следва:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> В зависимост от използвания SDK, логването може да е активирано по подразбиране или може да се наложи изрична активация в конфигурацията на сървъра.

Има различни типове уведомления:

| Ниво      | Описание                      | Примерно приложение            |
|-----------|-------------------------------|--------------------------------|
| debug     | Подробна отстраняваща грешки информация | Вход/изход от функции         |
| info      | Общи информационни съобщения | Актуализации на напредъка     |
| notice    | Нормални, но значими събития | Промени в конфигурацията       |
| warning   | Предупредителни състояния     | Използване на остаряла функционалност |
| error     | Грешки                      | Провали при операцията         |
| critical  | Критични състояния           | Неизправности на системни компоненти |
| alert     | Незабавни действия необходими | Открито повреждане на данни   |
| emergency | Системата е неизползваема    | Пълна системна повреда        |

## Имплементиране на уведомления в MCP

За да имплементирате уведомления в MCP, трябва да настроите както сървърната, така и клиентската страна да обработват актуализации в реално време. Това позволява на вашето приложение да предоставя незабавна обратна връзка на потребителите по време на дълготрайни операции.

### От страна на сървъра: Изпращане на уведомления

Нека започнем със сървърната страна. В MCP дефинирате инструменти, които могат да изпращат уведомления докато обработват заявки. Сървърът използва контекстния обект (обикновено `ctx`), за да изпраща съобщения към клиента.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

В предходния пример инструментът `process_files` изпраща три уведомления към клиента, докато обработва всеки файл. Методът `ctx.info()` се използва за изпращане на информационни съобщения.

Освен това, за да активирате уведомленията, уверете се, че вашият сървър използва потоков транспорт (като `streamable-http`) и клиентът ви имплементира обработчик на съобщения за обработка на уведомления. Ето как може да настроите сървъра да използва `streamable-http` транспорт:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

В този .NET пример, инструментът `ProcessFiles` е декориран с атрибута `Tool` и изпраща три уведомления към клиента, докато обработва всеки файл. Методът `ctx.Info()` се използва за изпращане на информационни съобщения.

За да активирате уведомленията в .NET MCP сървъра си, уверете се, че използвате потоков транспорт:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### От страна на клиента: Получаване на уведомления

Клиентът трябва да имплементира обработчик на съобщения, който обработва и показва уведомленията при пристигането им.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

В предходния код функцията `message_handler` проверява дали полученото съобщение е уведомление. Ако е, го отпечатва; ако не е, го обработва като обикновено съобщение от сървъра. Обърнете внимание как `ClientSession` е инициализиран с `message_handler`, за да обработва входящите уведомления.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

В този .NET пример функцията `MessageHandler` проверява дали входящото съобщение е уведомление. Ако е, го отпечатва; ако не е, го обработва като обикновено съобщение от сървъра. `ClientSession` е инициализиран с обработчика на съобщения чрез `ClientSessionOptions`.

За да активирате уведомленията, уверете се, че сървърът използва потоков транспорт (като `streamable-http`) и клиентът имплементира обработчик на съобщения.

## Уведомления за напредъка и сценарии

Този раздел обяснява концепцията за уведомления за напредъка в MCP, защо са важни и как да ги имплементирате с помощта на Streamable HTTP. Ще намерите и практическо задание за затвърждаване на знанията.

Уведомленията за напредъка са съобщения в реално време, изпращани от сървъра към клиента по време на дълготрайни операции. Вместо да се чака процесът да приключи, сървърът поддържа клиента информиран за текущия статус. Това подобрява прозрачността, потребителското изживяване и улеснява отстраняването на грешки.

**Пример:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### Защо да използваме уведомления за напредъка?

Уведомленията за напредъка са важни по няколко причини:

- **По-добро потребителско изживяване:** Потребителите виждат актуализации докато работата напредва, а не само в края.
- **Обратна връзка в реално време:** Клиентите могат да показват ленти на напредъка или логове, правейки приложението да изглежда отзивчиво.
- **По-лесно отстраняване на грешки и следене:** Разработчици и потребители могат да видят къде процесът може да е бавен или блокиран.

### Как да имплементираме уведомления за напредъка

Ето как може да имплементирате уведомления за напредъка в MCP:

- **На сървъра:** Използвайте `ctx.info()` или `ctx.log()`, за да изпращате уведомления докато се обработва всеки артикул. Това изпраща съобщение към клиента преди основния резултат да е готов.
- **На клиента:** Имплементирайте обработчик на съобщения, който слуша и показва уведомления докато пристигат. Този обработчик различава уведомленията от крайния резултат.

**Пример на сървър:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

## Актуализация от SSE към Streamable HTTP

За приложения, които в момента използват Server-Sent Events (SSE), миграцията към Streamable HTTP осигурява разширени възможности и по-добра дългосрочна устойчивост за вашите MCP имплементации.

### Защо да актуализираме?

Има две основателни причини да преминете от SSE към Streamable HTTP:

- Streamable HTTP предлага по-добра скалируемост, съвместимост и по-богата поддръжка на уведомления в сравнение със SSE.
- Това е препоръчителният транспорт за нови MCP приложения.

### Стъпки за миграция

Ето как можете да мигрирате от SSE към Streamable HTTP в вашите MCP приложения:

- **Актуализирайте сървърния код**, за да използва `transport="streamable-http"` в `mcp.run()`.
- **Актуализирайте клиентския код**, за да използва `streamablehttp_client` вместо SSE клиента.
- **Имплементирайте обработчик на съобщения** в клиента за обработка на уведомления.
- **Тествайте съвместимостта** с наличните инструменти и работни потоци.

### Поддържане на съвместимост

- Можете да поддържате и SSE, и Streamable HTTP като пуснете и двата транспорта на различни крайни точки.
- Постепенно мигрирайте клиентите към новия транспорт.

### Предизвикателства

- Обновяване на всички клиенти
- Обработка на разлики в доставянето на уведомления
