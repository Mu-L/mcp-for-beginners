# پخش جریانی HTTPS با پروتکل زمینه مدل (MCP)

این فصل راهنمای جامعی برای پیاده‌سازی پخش جریانی امن، مقیاس‌پذیر و در زمان واقعی با پروتکل زمینه مدل (MCP) با استفاده از HTTPS ارائه می‌دهد. این فصل انگیزه پخش جریانی، مکانیزم‌های انتقال موجود، نحوه پیاده‌سازی HTTP قابل پخش در MCP، بهترین شیوه‌های امنیتی، مهاجرت از SSE و راهنمای عملی برای ساخت برنامه‌های پخش جریانی MCP را پوشش می‌دهد.

## مکانیزم‌های انتقال و پخش جریانی در MCP

این بخش مکانیزم‌های انتقال مختلف موجود در MCP و نقش آن‌ها در فعال‌سازی قابلیت‌های پخش جریانی برای ارتباط در زمان واقعی بین کلاینت‌ها و سرورها را بررسی می‌کند.

### مکانیزم انتقال چیست؟

یک مکانیزم انتقال مشخص می‌کند که داده چگونه بین کلاینت و سرور تبادل می‌شود. MCP از انواع انتقال مختلف برای سازگاری با محیط‌ها و نیازهای گوناگون پشتیبانی می‌کند:

- **stdio**: ورودی/خروجی استاندارد، مناسب برای ابزارهای محلی و مبتنی بر CLI. ساده اما مناسب وب یا فضای ابری نیست.
- **SSE (Server-Sent Events)**: اجازه می‌دهد سرورها به‌روزرسانی‌های در زمان واقعی را به کلاینت‌ها از طریق HTTP ارسال کنند. برای رابط‌های وب خوب است، اما در مقیاس‌پذیری و انعطاف محدود است. از تاریخ ۱۸-۰۶-۲۰۲۵ مشخصات MCP، انتقال مستقل SSE (Server-Sent Events) منسوخ شده و جایگزین آن انتقال "HTTP قابل پخش" شده است.
- **HTTP قابل پخش**: انتقال پخش جریانی مبتنی بر HTTP مدرن، با پشتیبانی از اعلان‌ها و مقیاس‌پذیری بهتر. برای بیشتر سناریوهای تولید و ابری توصیه می‌شود.

### جدول مقایسه

برای درک تفاوت‌های بین این مکانیزم‌های انتقال، به جدول مقایسه زیر نگاه کنید:

| مکانیزم انتقال     | به‌روزرسانی در زمان واقعی | پخش جریانی | مقیاس‌پذیری | مورد استفاده           |
|-------------------|--------------------------|-----------|-------------|-----------------------|
| stdio             | خیر                      | خیر        | کم          | ابزارهای محلی CLI     |
| SSE               | بله                      | بله        | متوسط       | وب، به‌روزرسانی‌های در زمان واقعی |
| HTTP قابل پخش     | بله                      | بله        | بالا        | فضای ابری، چند کلاینت  |

> **نکته:** انتخاب مکانیزم انتقال مناسب بر عملکرد، مقیاس‌پذیری و تجربه کاربری تأثیر می‌گذارد. برای برنامه‌های مدرن، مقیاس‌پذیر و آماده ابر، **HTTP قابل پخش** توصیه می‌شود.

به مکانیزم‌های انتقال stdio و SSE که در فصل‌های قبلی به شما نشان داده شد و اینکه HTTP قابل پخش همان مکانیزمی است که در این فصل پوشش داده می‌شود توجه کنید.

## پخش جریانی: مفاهیم و انگیزه‌ها

درک مفاهیم و انگیزه‌های بنیادی پشت پخش جریانی برای پیاده‌سازی سیستم‌های ارتباطی در زمان واقعی مؤثر ضروری است.

**پخش جریانی** تکنیکی در برنامه‌نویسی شبکه است که اجازه می‌دهد داده‌ها به صورت بخش‌های کوچک و قابل مدیریت یا به عنوان دنباله‌ای از رویدادها ارسال و دریافت شوند، به جای اینکه منتظر آماده شدن کل پاسخ باشیم. این روش به خصوص برای موارد زیر مفید است:

- فایل‌ها یا مجموعه داده‌های بزرگ.
- به‌روزرسانی‌های در زمان واقعی (مثلاً چت، نوار پیشرفت).
- محاسبات طولانی که می‌خواهید کاربر را در جریان نگه دارید.

در اینجا نکات کلیدی پخش جریانی به صورت خلاصه آمده است:

- داده به تدریج تحویل داده می‌شود، نه همه به‌یکباره.
- کلاینت می‌تواند داده را به محض دریافت پردازش کند.
- کاهش تأخیر درک‌شده و بهبود تجربه کاربری.

### چرا از پخش جریانی استفاده کنیم؟

دلایل استفاده از پخش جریانی عبارتند از:

- کاربران بلافاصله بازخورد دریافت می‌کنند، نه فقط در انتها
- امکان برنامه‌های در زمان واقعی و رابط‌های پاسخگو را فراهم می‌کند
- استفاده بهینه‌تر از منابع شبکه و محاسبات

### مثال ساده: سرور و کلاینت پخش جریانی HTTP

در اینجا یک مثال ساده از پیاده‌سازی پخش جریانی ارائه شده است:

#### پایتون

**سرور (پایتون، با استفاده از FastAPI و StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**کلاینت (پایتون، با استفاده از requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

این مثال نشان می‌دهد که سرور چگونه به جای انتظار برای آماده شدن همه پیام‌ها، یک سری پیام را به محض آماده شدن به کلاینت ارسال می‌کند.

**نحوه کار:**

- سرور هر پیام را به محض آماده شدن ارسال می‌کند.
- کلاینت هر بخش را هنگام دریافت چاپ می‌کند.

**نیازمندی‌ها:**

- سرور باید از پاسخ پخش جریانی استفاده کند (مثلاً `StreamingResponse` در FastAPI).
- کلاینت باید پاسخ را به صورت جریان پردازش کند (`stream=True` در requests).
- Content-Type معمولاً `text/event-stream` یا `application/octet-stream` است.

#### جاوا

**سرور (جاوا، با استفاده از Spring Boot و Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**کلاینت (جاوا، با استفاده از Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**یادداشت‌های پیاده‌سازی در جاوا:**

- استفاده از پشته واکنشی Spring Boot با `Flux` برای پخش جریانی
- `ServerSentEvent` برای پخش جریانی ساختارمند با انواع رویدادها
- `WebClient` با `bodyToFlux()` امکان مصرف جریان واکنشی را می‌دهد
- `delayElements()` زمان پردازش بین رویدادها را شبیه‌سازی می‌کند
- رویدادها می‌توانند نوع داشته باشند (`info`، `result`) برای مدیریت بهتر در کلاینت

### مقایسه: پخش جریانی کلاسیک در مقابل پخش جریانی MCP

تفاوت‌های کارکرد پخش جریانی به سبک "کلاسیک" در برابر عملکرد آن در MCP را می‌توان به صورت زیر نشان داد:

| ویژگی                | پخش جریانی کلاسیک HTTP            | پخش جریانی MCP (اعلان‌ها)           |
|----------------------|----------------------------------|-------------------------------------|
| پاسخ اصلی            | بخش‌بندی شده                     | یک پاسخ واحد در پایان               |
| به‌روزرسانی پیشرفت   | به صورت بخش‌های داده ارسال می‌شود | به صورت اعلان‌ها ارسال می‌شود       |
| نیازمندی‌های کلاینت  | باید جریان را پردازش کند          | باید هندلر پیام برای مدیریت پیام‌ها پیاده‌سازی شود |
| مورد استفاده         | فایل‌های بزرگ، جریان توکن‌های AI  | پیشرفت، لاگ‌ها، بازخورد در زمان واقعی |

### تفاوت‌های کلیدی مشاهده شده

همچنین، برخی تفاوت‌های کلیدی عبارتند از:

- **الگوی ارتباط:**
  - پخش جریانی کلاسیک HTTP: از رمزگذاری انتقال بخش‌بندی شده استفاده می‌کند تا داده‌ها را به بخش‌ها ارسال کند.
  - پخش جریانی MCP: از سیستم اعلان ساختارمند با پروتکل JSON-RPC استفاده می‌کند.

- **فرمت پیام:**
  - HTTP کلاسیک: بخش‌های متن ساده با خطوط جدید
  - MCP: اشیاء ساختارمند LoggingMessageNotification با فراداده

- **پیاده‌سازی کلاینت:**
  - HTTP کلاسیک: کلاینت ساده که پاسخ‌های پخش جریانی را پردازش می‌کند
  - MCP: کلاینت پیشرفته‌تر با هندلر پیام برای پردازش انواع مختلف پیام‌ها

- **به‌روزرسانی پیشرفت:**
  - HTTP کلاسیک: پیشرفت بخشی از جریان پاسخ اصلی است
  - MCP: پیشرفت از طریق پیام‌های اعلان جداگانه ارسال می‌شود در حالی که پاسخ اصلی در پایان داده می‌شود

### توصیه‌ها

مواردی هست که در انتخاب بین پیاده‌سازی پخش جریانی کلاسیک (به عنوان یک نقطه انتهایی که بالاتر با `/stream` نشان دادیم) و انتخاب پخش جریانی از طریق MCP توصیه می‌کنیم:

- **برای نیازهای پخش جریانی ساده:** پخش جریانی کلاسیک HTTP ساده‌تر است و برای نیازهای پایه کافی است.

- **برای برنامه‌های تعاملی و پیچیده:** پخش جریانی MCP روش ساختاریافته‌تر با فراداده غنی‌تر و جداسازی بین اعلان‌ها و نتایج نهایی را ارائه می‌دهد.

- **برای برنامه‌های AI:** سیستم اعلان MCP برای وظایف طولانی AI که می‌خواهید کاربران را از پیشرفت مطلع نگه دارید بسیار مفید است.

## پخش جریانی در MCP

خب، تا اینجا توصیه‌ها و مقایسه‌هایی درباره تفاوت بین پخش جریانی کلاسیک و پخش جریانی در MCP دیده‌اید. حالا بیایید دقیقاً بررسی کنیم چگونه می‌توانید از پخش جریانی در MCP بهره‌مند شوید.

درک نحوه کار پخش جریانی در چارچوب MCP برای ساخت برنامه‌های پاسخگو که بازخورد در زمان واقعی را در طول عملیات طولانی به کاربر ارائه می‌دهند، ضروری است.

در MCP، پخش جریانی به معنی ارسال پاسخ اصلی به بخش‌های مختلف نیست، بلکه ارسال **اعلان‌ها** به کلاینت در حالی است که یک ابزار در حال پردازش درخواست است. این اعلان‌ها می‌توانند شامل به‌روزرسانی پیشرفت، لاگ‌ها یا رویدادهای دیگر باشند.

### نحوه کار

نتیجه اصلی هنوز به صورت یک پاسخ واحد ارسال می‌شود. اما اعلان‌ها می‌توانند به صورت پیام‌های جداگانه در طول پردازش ارسال شده و بدین ترتیب کلاینت را در زمان واقعی به‌روزرسانی کنند. کلاینت باید بتواند این اعلان‌ها را پردازش و نمایش دهد.

## اعلان چیست؟

ما "اعلان" گفتیم، این در زمینه MCP چه مفهومی دارد؟

اعلان پیامی است که از سرور به کلاینت ارسال می‌شود تا درباره پیشرفت، وضعیت یا رویدادهای دیگر در طول یک عملیات طولانی اطلاع‌رسانی کند. اعلان‌ها شفافیت و تجربه کاربری را بهبود می‌بخشند.

مثلاً کلاینت باید پس از انجام handshake اولیه با سرور یک اعلان ارسال کند.

یک اعلان به صورت یک پیام JSON به این شکل است:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

اعلان‌ها به یک موضوع در MCP به نام ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) تعلق دارند.

برای فعال کردن لاگ‌گیری، سرور باید آن را به عنوان یک ویژگی/قابلیت به این شکل فعال کند:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> بسته به SDK استفاده‌شده، لاگ‌گیری ممکن است به طور پیش‌فرض فعال باشد یا لازم باشد به صورت صریح در تنظیمات سرور فعال شود.

انواع مختلفی از اعلان‌ها وجود دارد:

| سطح       | توضیح                        | نمونه مورد استفاده           |
|-----------|-----------------------------|-----------------------------|
| debug     | اطلاعات دقیق اشکال‌زدایی      | نقاط ورود/خروج تابع         |
| info      | پیام‌های اطلاعات عمومی        | به‌روزرسانی پیشرفت عملیات   |
| notice    | رویدادهای طبیعی اما مهم      | تغییرات پیکربندی            |
| warning   | شرایط هشدار                  | استفاده از ویژگی منسوخ شده  |
| error     | شرایط خطا                   | شکست‌های عملیات             |
| critical  | شرایط بحرانی                | خرابی اجزای سیستم           |
| alert     | نیاز به اقدام فوری           | کشف فساد داده                |
| emergency | سیستم غیرقابل استفاده       | شکست کامل سیستم             |

## پیاده‌سازی اعلان‌ها در MCP

برای پیاده‌سازی اعلان‌ها در MCP، باید هر دو طرف سرور و کلاینت را برای رسیدگی به به‌روزرسانی‌های در زمان واقعی آماده کنید. این امکان را می‌دهد که برنامه شما بازخورد فوری به کاربران در طول عملیات طولانی ارائه دهد.

### سمت سرور: ارسال اعلان‌ها

بیایید با سمت سرور شروع کنیم. در MCP ابزارهایی تعریف می‌کنید که هنگام پردازش درخواست‌ها می‌توانند اعلان ارسال کنند. سرور از شی زمینه (معمولاً `ctx`) برای ارسال پیام به کلاینت استفاده می‌کند.

#### پایتون

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

در مثال قبلی، ابزار `process_files` هنگام پردازش هر فایل سه اعلان به کلاینت ارسال می‌کند. متد `ctx.info()` برای ارسال پیام‌های اطلاع‌رسانی استفاده شده است.

علاوه بر این، برای فعال کردن اعلان‌ها، مطمئن شوید که سرور شما از انتقال پخش جریانی (مانند `streamable-http`) استفاده می‌کند و کلاینت شما هندلر پیام برای پردازش اعلان‌ها پیاده‌سازی کرده است. در اینجا نحوه تنظیم سرور برای استفاده از انتقال `streamable-http` آمده است:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

در این مثال .NET، ابزار `ProcessFiles` با صفت `Tool` مشخص شده و هنگام پردازش هر فایل، سه اعلان به کلاینت ارسال می‌کند. متد `ctx.Info()` برای ارسال پیام‌های اطلاع‌رسانی استفاده می‌شود.

برای فعال کردن اعلان‌ها در سرور MCP خود در دات‌نت، مطمئن شوید که از انتقال پخش جریانی استفاده می‌کنید:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### سمت کلاینت: دریافت اعلان‌ها

کلاینت باید هندلر پیام برای پردازش و نمایش اعلان‌ها به محض دریافت پیاده‌سازی کند.

#### پایتون

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

در کد بالا، تابع `message_handler` بررسی می‌کند که آیا پیام دریافتی یک اعلان است یا خیر. اگر هست، اعلان چاپ می‌شود؛ در غیر این صورت به عنوان پیام عادی سرور پردازش می‌شود. همچنین مشاهده می‌کنید که `ClientSession` با `message_handler` برای دریافت اعلان‌ها مقداردهی شده است.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

در این مثال دات‌نت، تابع `MessageHandler` بررسی می‌کند که آیا پیام دریافتی یک اعلان است یا نه. اگر چنین باشد، اعلان چاپ می‌شود؛ در غیر این صورت به عنوان پیام عادی سرور پردازش می‌شود. `ClientSession` با هندلر پیام از طریق `ClientSessionOptions` مقداردهی اولیه می‌شود.

برای فعال کردن اعلان‌ها، مطمئن شوید که سرور شما از انتقال پخش جریانی (مثل `streamable-http`) استفاده می‌کند و کلاینت شما هندلر پیام برای پردازش اعلان‌ها را پیاده‌سازی کرده است.

## اعلان‌های پیشرفت و سناریوها

این بخش مفهوم اعلان‌های پیشرفت در MCP، اهمیت آن‌ها و نحوه پیاده‌سازی آن‌ها با استفاده از HTTP قابل پخش را شرح می‌دهد. همچنین یک تمرین عملی برای تقویت فهم ارائه شده است.

اعلان‌های پیشرفت پیام‌های در زمان واقعی هستند که از سرور به کلاینت طی عملیات طولانی ارسال می‌شوند. به جای اینکه منتظر پایان کامل فرایند باشید، سرور وضعیت جاری را به کلاینت اطلاع می‌دهد. این کار شفافیت، تجربه کاربری و دیباگ را ساده‌تر می‌کند.

**مثال:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### چرا از اعلان‌های پیشرفت استفاده کنیم؟

اعلان‌های پیشرفت دلایل مهمی دارند:

- **تجربه کاربری بهتر:** کاربران به محض پیشرفت کار، به‌روزرسانی‌ها را می‌بینند، نه فقط در انتها.
- **بازخورد در زمان واقعی:** کلاینت‌ها می‌توانند نوار پیشرفت یا لاگ‌ها را نمایش دهند و برنامه پاسخگو به نظر برسد.
- **دیباگ و مانیتورینگ آسان‌تر:** توسعه‌دهندگان و کاربران می‌توانند ببینند فرایند کجا ممکن است کند یا متوقف شود.

### چگونه اعلان‌های پیشرفت را پیاده کنیم

در ادامه نحوه پیاده‌سازی اعلان‌های پیشرفت در MCP آمده است:

- **در سرور:** از `ctx.info()` یا `ctx.log()` برای ارسال اعلان به محض پردازش هر آیتم استفاده کنید. این پیام قبل از آماده شدن نتیجه اصلی به کلاینت ارسال می‌شود.
- **در کلاینت:** هندلر پیام پیاده کنید که اعلان‌ها را دریافت و نمایش می‌دهد. این هندلر تفاوت بین اعلان‌ها و نتیجه نهایی را تشخیص می‌دهد.

**مثال سرور:**

#### پایتون

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**مثال کلاینت:**

#### پایتون

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## ملاحظات امنیتی

هنگام پیاده‌سازی سرورهای MCP با انتقال‌های مبتنی بر HTTP، امنیت به یک نگرانی اساسی تبدیل می‌شود که نیازمند توجه دقیق به بردارهای حمله مختلف و مکانیزم‌های محافظتی است.

### مرور کلی

امنیت در مواجهه با سرورهای MCP از طریق HTTP حیاتی است. HTTP قابل پخش سطوح حمله جدیدی را معرفی می‌کند و نیازمند پیکربندی دقیق است.

### نکات کلیدی

- **اعتبارسنجی هدر Origin**: همیشه هدر `Origin` را اعتبارسنجی کنید تا از حملات بازبایند DNS جلوگیری شود.
- **بستن به localhost**: برای توسعه محلی، سرورها را به `localhost` محدود کنید تا در معرض اینترنت عمومی قرار نگیرند.
- **احراز هویت**: برای استقرار در محیط تولید، احراز هویت (مانند کلید API، OAuth) را پیاده‌سازی کنید.
- **CORS**: سیاست‌های اشتراک منابع بین مبدأها را تنظیم کنید تا دسترسی محدود شود.
- **HTTPS**: در محیط تولید از HTTPS برای رمزگذاری ترافیک استفاده کنید.

### بهترین شیوه‌ها

- به درخواست‌های ورودی بدون اعتبارسنجی هرگز اعتماد نکنید.
- همه دسترسی‌ها و خطاها را لاگ و مانیتور کنید.
- به طور منظم وابستگی‌ها را به‌روزرسانی کنید تا آسیب‌پذیری‌های امنیتی رفع شود.

### چالش‌ها
- متعادل ساختن امنیت با سهولت توسعه
- اطمینان از سازگاری با محیط‌های مختلف کلاینت

## ارتقا از SSE به Streamable HTTP

برای برنامه‌هایی که در حال حاضر از Server-Sent Events (SSE) استفاده می‌کنند، مهاجرت به Streamable HTTP قابلیت‌های پیشرفته‌تر و پایداری بهتر در بلندمدت برای پیاده‌سازی‌های MCP شما فراهم می‌کند.

### چرا ارتقا دهیم؟

دلایل قانع‌کننده‌ای برای ارتقا از SSE به Streamable HTTP وجود دارد:

- Streamable HTTP مقیاس‌پذیری، سازگاری و پشتیبانی غنی‌تر از اعلان‌ها نسبت به SSE ارائه می‌دهد.
- این روش حمل‌ونقل برای برنامه‌های جدید MCP توصیه می‌شود.

### مراحل مهاجرت

در اینجا نحوه مهاجرت از SSE به Streamable HTTP در برنامه‌های MCP شما آمده است:

- **کد سرور را به‌روزرسانی کنید** تا از `transport="streamable-http"` در `mcp.run()` استفاده کند.
- **کد کلاینت را به‌روزرسانی کنید** تا به جای کلاینت SSE از `streamablehttp_client` استفاده شود.
- **یک هندلر پیام پیاده‌سازی کنید** در کلاینت برای پردازش اعلان‌ها.
- **سازگاری با ابزارها و جریان‌های کاری موجود را تست کنید**.

### حفظ سازگاری

توصیه می‌شود در طول فرایند مهاجرت، سازگاری با کلاینت‌های SSE موجود را حفظ کنید. در اینجا برخی استراتژی‌ها آمده است:

- می‌توانید هم SSE و هم Streamable HTTP را با اجرای هر دو روش حمل‌ونقل در نقاط انتهایی مختلف پشتیبانی کنید.
- به تدریج کلاینت‌ها را به حمل‌ونقل جدید مهاجرت دهید.

### چالش‌ها

مطمئن شوید که در طول مهاجرت به چالش‌های زیر پرداخته‌اید:

- اطمینان از به‌روزرسانی همه کلاینت‌ها
- مدیریت تفاوت‌ها در تحویل اعلان‌ها

## ملاحظات امنیتی

امنیت باید در هنگام پیاده‌سازی هر سروری، به‌ویژه هنگام استفاده از حمل‌ونقل‌های مبتنی بر HTTP مانند Streamable HTTP در MCP، اولویت بالایی داشته باشد.

هنگام پیاده‌سازی سرورهای MCP با حمل‌ونقل‌های مبتنی بر HTTP، امنیت یک نگرانی اساسی است که نیازمند توجه دقیق به مجموعه‌ای از بردارهای حمله و مکانیزم‌های محافظتی است.

### مرور کلی

امنیت در معرض قرار دادن سرورهای MCP روی HTTP حیاتی است. Streamable HTTP سطوح حمله جدیدی ایجاد می‌کند و نیاز به پیکربندی دقیق دارد.

در اینجا برخی ملاحظات کلیدی امنیتی آمده است:

- **اعتبارسنجی هدر Origin**: همیشه هدر `Origin` را برای جلوگیری از حملات DNS rebinding اعتبارسنجی کنید.
- **بایند به لوکال‌هاست**: برای توسعه محلی، سرورها را به `localhost` متصل کنید تا از دسترسی از طریق اینترنت عمومی جلوگیری شود.
- **احراز هویت**: برای استقرار در محیط تولید، احراز هویت (مثلاً کلیدهای API، OAuth) را پیاده‌سازی کنید.
- **CORS**: سیاست‌های Cross-Origin Resource Sharing (CORS) را برای محدود کردن دسترسی پیکربندی کنید.
- **HTTPS**: در محیط تولید از HTTPS برای رمزنگاری ترافیک استفاده کنید.

### بهترین تمرین‌ها

علاوه بر این، در اینجا برخی بهترین تمرین‌ها برای پیاده‌سازی امنیت در سرور استریمینگ MCP شما آمده است:

- هرگز به درخواست‌های ورودی بدون اعتبارسنجی اعتماد نکنید.
- تمام دسترسی‌ها و خطاها را لاگ و نظارت کنید.
- به طور منظم وابستگی‌ها را برای رفع آسیب‌پذیری‌های امنیتی به‌روزرسانی کنید.

### چالش‌ها

در پیاده‌سازی امنیت در سرورهای استریمینگ MCP با برخی چالش‌ها مواجه خواهید شد:

- متعادل ساختن امنیت با سهولت توسعه
- اطمینان از سازگاری با محیط‌های مختلف کلاینت

### تمرین: ساخت اپلیکیشن استریمینگ MCP خودتان

**سناریو:**
یک سرور و کلاینت MCP بسازید که سرور یک فهرست از آیتم‌ها (مثلاً فایل‌ها یا اسناد) را پردازش کند و برای هر آیتم پردازش شده یک اعلان ارسال نماید. کلاینت باید هر اعلان را هنگام رسیدن نمایش دهد.

**مراحل:**

1. پیاده‌سازی یک ابزار سرور که فهرستی را پردازش کند و برای هر آیتم اعلان ارسال کند.
2. پیاده‌سازی کلاینت با یک هندلر پیام برای نمایش اعلان‌ها در زمان واقعی.
3. پیاده‌سازی خود را با اجرای همزمان سرور و کلاینت تست کنید و اعلان‌ها را مشاهده نمایید.
