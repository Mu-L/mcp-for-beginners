# HTTPS స్ట్రిమింగ్ మోడల్ కాంటెక్స్ట్ ప్రోటోకాల్ (MCP) తో

ఈ అధ్యాయం HTTPS ఉపయోగించి మోడల్ కాంటెక్స్ట్ ప్రోటోకాల్ (MCP) తో సురక్షిత, స్కేలబుల్ మరియు రియల్‌టైమ్ స్ట్రిమింగ్ ను అమలు చేయడానికి సమగ్ర మార్గదర్శకాన్ని అందిస్తుంది. ఇది స్ట్రిమింగ్ కోసం ప్రేరణ, అందుబాటులో ఉన్న ట్రాన్స్‌పోర్ట్ మెకానిజమ్‌లు, MCP లో స్ట్రిమింగ్ HTTP ఎలా అమలు చేయాలో, భద్రతా ఉత్తమ పద్ధతులు, SSE నుండి మైగ్రేషన్ మరియు మీ స్వంత స్ట్రీమింగ్ MCP అనువర్తనాలను నిర్మించడానికి ప్రాయోగిక మార్గదర్శకత్వాన్ని కవర్ చేస్తుంది.

## MCP లో ట్రాన్స్‌పోర్ట్ మెకానిజమ్‌లు మరియు స్ట్రిమింగ్

ఈ విభాగం MCP లో అందుబాటులో ఉన్న వివిధ ట్రాన్స్‌పోర్ట్ మెకానిజమ్‌లను మరియు క్లయింట్లు మరియు సర్వర్ల మధ్య రియల్‌టైమ్ కమ్యూనికేషన్ కోసం స్ట్రీమింగ్ సామర్థ్యాలను అందించే వాటి పాత్రను పరిశీలిస్తుంది.

### ట్రాన్స్‌పోర్ట్ మెకానిజమ్ అంటే ఏమిటి?

ట్రాన్స్‌పోర్ట్ మెకానిజమ్ అనేది డేటా క్లయింట్ మరియు సర్వర్ మధ్య ఎలా మార్పిడి అవుతుందో నిర్వచిస్తుంది. MCP వివిధ వాతావరణాలు మరియు అవసరాలకు అనుకూలంగా గల బహుళ ట్రాన్స్‌పోర్ట్ రకాలను మద్దతు ఇస్తుంది:

- **stdio**: ప్రామాణిక ఇన్‌పుట్/అవుట్‌పుట్, స్థానిక మరియు CLI-ఆధారిత టూల్‌లకు అనుకూలంగా ఉంది. సింపుల్, అయితే వెబ్ లేదా క్లౌడ్ కు అనుకూలం కాదు.
- **SSE (సర్వర్-సెంట్స్ ఈవెంట్స్)**: సర్వర్లు HTTP మీద క్లయింట్లకు రియల్‌టైమ్ అప్డేట్స్ పంపడానికి అనుమతిస్తుంది. వెబ్ UI లకు మంచిది, కానీ స్కేలబిలిటీ మరియు అనుకూలత పరిమితంగా ఉంటుంది. MCP స్పెసిఫికేషన్ 2025-06-18 నుండి, స్వతంత్ర SSE ట్రాన్స్‌పోర్ట్‌ను తీసివేయడం జరిగింది మరియు దీని స్థానంలో "స్ట్రీమబుల్ HTTP" ట్రాన్స్‌పోర్ట్ ప్రవేశపెట్టబడింది.
- **స్ట్రీమబుల్ HTTP**: ఆధునిక HTTP ఆధారిత స్ట్రీమింగ్ ట్రాన్స్‌పోర్ట్, నోటిఫికేషన్స్ మరియు మెరుగైన స్కేలబిలిటీ మద్దతు ఇస్తుంది. ఎక్కువగా ఉత్పత్తి మరియు క్లౌడ్ సన్నివేశాల కోసం సిఫార్సు చేయబడింది.

### పోలిక పట్టిక

ఈ ట్రాన్స్‌పోర్ట్ మెకానిజమ్‌ల మధ్య తేడాలు అర్థం చేసుకోవడానికి దిగువ పోలిక పట్టికను చూడండి:

| ట్రాన్స్‌పోర్ట్      | రియల్-టైమ్ అప్డేట్స్ | స్ట్రీమింగ్ | స్కేలబిలిటీ | ఉపయోగం              |
|--------------------|----------------------|------------|-------------|---------------------|
| stdio              | లేదు                 | లేదు       | తక్కువ      | స్థానిక CLI టూల్‌లు |
| SSE                | ఉన్నాయి              | ఉన్నాయి     | మధ్యస్థం    | వెబ్, రియల్-టైమ్ అప్డేట్స్ |
| స్ట్రీమబుల్ HTTP      | ఉన్నాయి              | ఉన్నాయి     | అధికం      | క్లౌడ్, బహుళ క్లయింట్లు |

> **సూచన:** సరైన ట్రాన్స్‌పోర్ట్ ఎంపిక పనితీరు, స్కేలబిలిటీ మరియు ఉపయోగదారుడు అనుభవంపై ప్రభావం చూపుతుంది. ఆధునిక, స్కేలబుల్ మరియు క్లౌడ్-పరిమితి అప్లికేషన్లకు **స్ట్రీమబుల్ HTTP**ను సిఫార్సు చేయబడుతుంది.

గత అధ్యాయాలలో మీరు చూసిన stdio మరియు SSE ట్రాన్స్‌పోర్ట్లను గమనించండి మరియు ఈ అధ్యాయం పరిచయం చేస్తోన్న స్ట్రీమబుల్ HTTP ట్రాన్స్‌పోర్ట్ గురించి తెలుసుకోండి.

## స్ట్రీమింగ్: సంకల్పాలు మరియు ప్రేరణ

స్ట్రీమింగ్ వెనుక ఉన్న ప్రాథమిక భావనలు మరియు ప్రేరణలను అర్థం చేసుకోవడం రియల్‌టైమ్ కమ్యూనికేషన్ సిస్టమ్లను సమర్థవంతంగా అమలు చేయడానికి కీలకం.

**స్ట్రీమింగ్** అనేది నెట్‌వర్క్ ప్రోగ్రామింగ్ లో ఒక సాంకేతిక లక్షణం, ఇది మొత్తం ప్రతిస్పందన సిద్ధంగా ఉండటానికి ఎదురు చూస్తుంది కాకుండా, చిన్న, నిర్వహించగల భాగాలుగా లేదా ఘటనా శ్రేణులుగా డేటాను పంపడం మరియు స్వీకరించడం అనుమతిస్తుంది. ఇది ముఖ్యంగా ఉపయోగకరంగా ఉంటుంది:

- పెద్ద ఫైల్స్ లేదా డేటాసెట్స్ కోసం.
- రియల్‌టైమ్ అప్డేట్స్ కోసం (ఉదాహరణకు, చాట్, ప్రగతి బార్లు).
- యూజర్‌కు సమాచారం అందిస్తూ నడిచే పెద్ద లెక్కలు.

స్ట్రీమింగ్ యొక్క ప్రాథమిక అంశాలు:

- డేటా పదేపదే పంపబడుతుంది, ఒకేసారి అన్ని కాదు.
- క్లయింట్ ఆ డేటాను అందిన వెంటనే ప్రాసెస్ చేయగలదు.
- గమనించిన ఆలస్యం తగ్గించి ఉపయోగదారుని అనుభవాన్ని మెరుగుపరుస్తుంది.

### ఎందుకు స్ట్రీమింగ్ ఉపయోగించాలి?

స్ట్రీమింగ్ ఉపయోగించడానికి కారణాలు:

- యూజర్లు వెంటనే అభిప్రాయం పొందుతారు, కేవలం చివర్లో కాదు
- రియల్‌టైమ్ అనువర్తనాలు మరియు స్పందనాత్మక UI లను సాధించడానికి సహాయపడుతుంది
- నెట్‌వర్క్ మరియు కంప్యూట్ వనరుల మరింత సమర్థవంతమైన వినియోగం

### సాదా ఉదాహరణ: HTTP స్ట్రీమింగ్ సర్వర్ & క్లయింట్

స్ట్రీమింగ్‌ను ఎలా అమలు చేయగలమో సులభ ఉదాహరణ ఇక్కడ ఉంది:

#### Python

**సర్వర్ (Python, FastAPI మరియు StreamingResponse ఉపయోగించి):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**క్లయింట్ (Python, requests ఉపయోగించి):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ఈ ఉదాహరణలో సర్వర్ అందుబాటులోకి వచ్చిన సందేశాలను వేయమెలా క్లయింట్‌కి పంపుతూనే ఉంటుంది, మొత్తం సందేశాలు సిద్ధంగా ఉండటానికి ఎదురు చూడలేదు.

**పనితీరు:**

- సర్వర్ ప్రతి సందేశాన్ని అందుబాటులో ఉన్న వెంటనే ప్రేరేపిస్తుంది.
- క్లయింట్ అందిన ప్రతి భాగాన్ని స్వీకరించి ప్రింట్ చేస్తుంది.

**అవసరాలు:**

- సర్వర్ ఒక స్ట్రీమింగ్ ప్రతిస్పందన ఉపయోగించాలి (ఉదాహరణకు, FastAPI లో `StreamingResponse`).
- క్లయింట్ ప్రతిస్పందనను స్ట్రీమ్‌గా (requests లో `stream=True`) ప్రాసెస్ చేయాలి.
- Content-Type సాధారణంగా `text/event-stream` లేదా `application/octet-stream` ఉంటుంది.

#### Java

**సర్వర్ (Java, Spring Boot మరియు Server-Sent Events ఉపయోగించి):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**క్లయింట్ (Java, Spring WebFlux WebClient ఉపయోగించి):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java అమలు వివరాలు:**

- స్ట్రీమింగ్ కోసం Spring Boot యొక్క రియాక్టివ్ స్టాక్ `Flux` ఉపయోగిస్తోంది
- `ServerSentEvent` ఈవెంట్ రకాలతో సాంఘికీకృత ఈవెంట్ స్ట్రీమింగ్‌ను అందిస్తుంది
- `WebClient` లో `bodyToFlux()` రియాక్టివ్ స్ట్రీమింగ్ గ్రహణాన్ని అనుమతిస్తుంది
- `delayElements()` ఈవెంట్ల మధ్య ప్రాసెసింగ్ సమయాన్ని అనుకరిస్తుంది
- ఈవెంట్లు మెరుగైన క్లయింట్ హ్యాండ్లింగ్ కోసం రకాలను (`info`, `result`) కలిగి ఉండవచ్చు

### పోలిక: క్లాసిక్ స్ట్రీమింగ్ వర్సెస్ MCP స్ట్రీమింగ్

స్ట్రీమింగ్ "సాంప్రదాయ" రీతిలో ఎలా పనిచేస్తుంది మరియు MCP లో ఎలా పనిచేస్తుందో మధ్య తేడాలను ఈ విధంగా చిత్రించవచ్చు:

| లక్షణం                   | క్లాసిక్ HTTP స్ట్రీమింగ్         | MCP స్ట్రీమింగ్ (నోటిఫికేషన్స్)     |
|--------------------------|---------------------------------|-----------------------------------|
| ప్రధాన ప్రతిస్పందనం       | భాగాలుగా                      | ఒక్కటి, చివరిలో                  |
| ప్రగతి అప్డేట్స్           | డేటా భాగాలుగా పంపబడతాయి     | నోటిఫికేషన్లుగా పంపబడతాయి      |
| క్లయింట్ అవసరాలు          | స్ట్రీమ్ ప్రాసెస్ చేయాలి        | సందేశ హ్యాండ్లర్ ఏర్పాటు చేయాలి   |
| ఉపయోగం                   | పెద్ద ఫైల్స్, AI టోకెన్ స్ట్రీమ్స్ | ప్రగతి, లాగ్లు, రియల్-టైమ్ అభిప్రాయం |

### ముఖ్యమైన తేడాలు

కూడా, కొన్ని ముఖ్యమైన తేడాలు:

- **కమ్యూనికేషన్ ప్యాటర్న్:**
  - క్లాసిక్ HTTP స్ట్రీమింగ్: డేటాను భాగాలుగా పంపడానికి సాదారణ chunked ట్రాన్స్‌ఫర్ కోడింగ్ ఉపయోగిస్తుంది
  - MCP స్ట్రీమింగ్: JSON-RPC ప్రోటోకాల్ తో నిర్మించబడిన నోటిఫికేషన్ వ్యవస్థను ఉపయోగిస్తుంది

- **సंदేశ ఫార్మాట్:**
  - క్లాసిక్ HTTP: కొత్త లైన్లతో సాధారణ టెక్స్ట్ భాగాలు
  - MCP: మెటాడేటాతో నిర్మిత LoggingMessageNotification ఆబ్జెక్టులు

- **క్లయింట్ అమలు:**
  - క్లాసిక్ HTTP: స్ట్రీమింగ్ ప్రతిస్పందనలను ప్రాసెస్ చేసే సాదా క్లయింట్
  - MCP: వివిధ రకాల సందేశాలను ప్రాసెస్ చేయడానికి సందేశ హ్యాండ్లర్ కలిగిన క్లిష్టమైన క్లయింట్

- **ప్రగతి అప్డేట్స్:**
  - క్లాసిక్ HTTP: ప్రగతి ప్రధాన ప్రతిస్పందన స్ట్రీమ్ లో భాగం
  - MCP: ప్రగతి వేర్వేరు నోటిఫికేషన్ సందేశాల ద్వారా, మరియు ప్రధాన ప్రతిస్పందన చివర అందుతుంది

### సిఫార్సులు

క్లాసిక్ స్ట్రీమింగ్ (ముందు చూపించిన `/stream` ఎండ్‌పాయింట్ వలె) అమలు చేయడమో, MCP ద్వారా స్ట్రీమింగ్ ఎంచుకోవడమో మధ్య ఎంపికలో కొన్ని సిఫార్సులు:

- **సింపుల్ స్ట్రీమింగ్ అవసరాలకు:** క్లాసిక్ HTTP స్ట్రీమింగ్ అమలు సులభం మరియు ప్రాథమిక అవసరాలకు సరిపోతుంది.

- **స复杂, ఇంటరాక్టివ్ అప్లికేషన్లకోసం:** MCP స్ట్రీమింగ్ సమృద్ధిగా మెటాడేటాతో కూడిన నిర్మిత దృక్పథాన్ని అందిస్తుంది, నోటిఫికేషన్లు మరియు తుది ఫలితాలు వేరు చేయబడతాయి.

- **AI అప్లికేషన్లకోసం:** MCP యొక్క నోటిఫికేషన్ వ్యవస్థ యూజర్లకు ప్రోగ్రెస్ తెలియజేయడానికి ఎక్కువ ప్రయోజనకరం, ముఖ్యంగా దీర్ఘకాలిక AI పనుల విషయంలో.

## MCP లో స్ట్రీమింగ్

ఇప్పుడు, మీరు క్లాసిక్ స్ట్రీమింగ్ మరియు MCP స్ట్రీమింగ్ మధ్య తేడాలపై కొన్ని సిఫార్సులు మరియు పోలికలు చూసిన తర్వాత, MCP లో స్ట్రీమింగ్‌ను ఎలా ఉపయోగించాలో వివరంగా చూద్దాం.

MCP ఫ్రేమ్‌వర్క్ లో స్ట్రీమింగ్ ఎలా పనిచేస్తుందో అర్థం చేసుకోవడం, దీర్ఘకాలిక ఆపరేషన్‌ల సమయంలో వాడుకదారులకు రియల్‌టైమ్ అభిప్రాయం అందించే స్పందనాత్మక అనువర్తనాలు తయారు చేయాలనుకునే వారికి ముఖ్యం.

MCP లో, ప్రధాన ప్రతిస్పందనను భాగాలుగా పంచడం కాకుండా, టూల్ ఆ పని చేస్తున్న సమయంలో క్లయింట్‌కు **నోటిఫికేషన్లు** పంపడం స్ట్రీమింగ్. ఈ నోటిఫికేషన్లు ప్రగతి అప్డేట్లు, లాగ్లు లేదా ఇతర ఈవెంట్లుగా ఉండవచ్చు.

### ఇది ఎలా పనిచేస్తుంది

ప్రధాన ఫలితం మాత్రం ఒక్కటిగా పంపబడుతుంది. కానీ ప్రాసెసింగ్ సమయంలో, నోటిఫికేషన్లు వేరు సందేశాలుగా పంపబడతాయి, తద్వారా క్లయింట్‌ను తక్షణ అప్డేట్ చేస్తాయి. క్లయింట్ ఈ నోటిఫికేషన్లను నిర్వహించి, ప్రదర్శించగలగాలి.

## నోటిఫికేషన్ అంటే ఏమిటి?

మనం "నోటిఫికేషన్" అనగానే MCP సందర్భంలో దాని అర్థం ఏమిటి?

నోటిఫికేషన్ అనేది సర్వర్ నుండి క్లయింట్ కు పంపే సందేశం, అది దీర్ఘకాలిక ఆపరేషన్ సమయంలో ప్రగతి, స్థితి లేదా ఇతర ఈవెంట్ల గురించి సమాచారం ఇస్తుంది. ఇది పారదర్శకత మరియు ఉపయోగదారుని అనుభవాన్ని మెరుగుపరుస్తుంది.

ఉదాహరణకు, క్లయింట్ మొదటి హ్యాండ్షేక్ పూర్తయిన తరువాత ఒక నోటిఫికేషన్ పంపాలి.

నోటిఫికేషన్ JSON సందేశం ఈ విధంగా ఉంటుంది:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

నోటిఫికేషన్లు MCP లో ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) అనే టాపిక్ కి చెందినవి.

లాగింగ్ పనిచేయడానికి, సర్వర్ దీనిని ఫీచర్/సామర్థ్యంగా ఇలా ఎనేబుల్ చేయాలి:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ఉపయోగిస్తున్న SDK పై ఆధారపడి, లాగింగ్ డిఫాల్ట్ గా ఎనేబుల్ అయి ఉండవచ్చు లేదా సర్వర్ కాన్ఫిగరేషన్ లో స్పష్టంగా ఎనేబుల్ చేయాల్సి ఉండవచ్చు.

పలుస రకాల నోటిఫికేషన్లు ఉంటాయి:

| స్థాయి      | వివరణ                        | ఉదాహరణ ఉపయోగం               |
|------------|------------------------------|------------------------------|
| debug      | విశదమైన డీబగ్గింగ్ సమాచారం   | ఫంక్షన్ ఎంట్రీ/ఎగ్జిట్ పాయింట్లు |
| info       | సాధారణ సమాచార సందేశాలు      | ఆపరేషన్ ప్రగతి అప్డేట్స్       |
| notice     | సాధారణ కాని ముఖ్య ఈవెంట్స్     | కాన్ఫిగరేషన్ మార్పులు          |
| warning    | హెచ్చరిక స్థితులు             | పాత ఫీచర్ వినియోగం          |
| error      | లోప స్థితులు                | ఆపరేషన్ వైఫల్యాలు           |
| critical   | క్రిటికల్ స్థితులు            | సిస్టమ్ భాగాలు వైఫల్యం      |
| alert      | వెంటనే చర్య తీసుకోవాలి       | డేటా భंगమయ్యింది గుర్తింపు  |
| emergency  | సిస్టమ్ ఉపయోగించలేని స్థితి | సంపూర్ణ సిస్టమ్ వైఫల్యం     |

## MCP లో నోటిఫికేషన్లు అమలు

నోటిఫికేషన్లను MCP లో అమలు చేయాలంటే, సర్వర్ మరియు క్లయింట్ రెండింటిని రియల్-టైమ్ అప్డేట్లను నిర్వహించడానికి సెట్ చేయాలి. ఇది మీ అప్లికేషన్ దీర్ఘకాలిక ఆపరేషన్‌ల సమయంలో యూజర్లకు తక్షణ అభిప్రాయం ఇవ్వగలదు.

### సర్వర్-సైడ్: నోటిఫికేషన్లు పంపడం

మనం మొదలు పెడదాం సర్వర్ వైపు నుండి. MCP లో, మీరు అభ్యర్థనలు ప్రాసెస్ చేస్తున్నప్పుడు నోటిఫికేషన్లు పంపగల టూల్స్‌ను నిర్వచిస్తారు. సర్వర్ కాంటెక్స్ట్ ఆబ్జెక్ట్ (సాధారణంగా `ctx`) ఉపయోగించి క్లయింట్‌కు సందేశాలు పంపుతుంది.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

పూర్వపు ఉదాహరణలో, `process_files` టూల్ ప్రతి ఫైల్ ప్రాసెస్ చేస్తున్నప్పుడు క్లయింట్ కు మూడు నోటిఫికేషన్లు పంపుతోంది. `ctx.info()` మెథడ్ సమాచార సందేశాలను పంపడానికి ఉపయోగించబడుతుంది.

అదనంగా, నోటిఫికేషన్లు ఎనేబుల్ చేయడానికి, మీ సర్వర్ స్ట్రీమింగ్ ట్రాన్స్‌పోర్ట్ (ఇలా `streamable-http`) ఉపయోగించాలి మరియు క్లయింట్ నోటిఫికేషన్లు ప్రాసెస్ చేయడానికి సందేశ హ్యాండ్లర్ అమలు చేయాలి. మీ సర్వర్ `streamable-http` టTransport ఉపయోగించే విధంగా ఇక్కడ సెట్ చేయవచ్చు:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ఈ .NET ఉదాహరణలో, `ProcessFiles` టూల్ `Tool` అట్రిబ్యూట్‌తో అలంకరించబడింది మరియు ప్రతి ఫైల్ ప్రాసెస్ చేసిన వెంటనే క్లయింట్ కు మూడు నోటిఫికేషన్లు పంపుతోంది. `ctx.Info()` మెథడ్ సమాచార సందేశాలను పంపడానికి వాడబడుతుంది.

మీ .NET MCP సర్వర్ లో నోటిఫికేషన్లు ఎనేబుల్ చేయడానికి, స్ట్రీమింగ్ ట్రాన్స్‌పోర్ట్ ఉపయోగిస్తున్నారని నిర్ధారించుకోండి:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### క్లయింట్-సైడ్: నోటిఫికేషన్లు స్వీకరణ

క్లయింట్ ఒక సందేశ హ్యాండ్లర్ అమలు చేయాలి, అది నోటిఫికేషన్లు వచ్చినప్పుడు వాటిని ప్రాసెస్ చేసి ప్రదర్శిస్తుంది.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

పూర్వపు కోడ్ లో, `message_handler` ఫంక్షన్ వచ్చిన సందేశం నోటిఫికేషన్ కు చెందింది లేదా కాదో తనిఖీ చేస్తుంది. ఇది అయితే నోటిఫికేషన్ ను ప్రింట్ చేస్తుంది, లేదా సాధారణ సర్వర్ సందేశం మాదిరిగా ప్రాసెస్ చేస్తుంది. అలాగే `ClientSession`ని `message_handler`తో సెట్ చేసి నోటిఫికేషన్లు స్వీకరించగలుగుతుంది.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ఈ .NET ఉదాహరణలో, `MessageHandler` ఫంక్షన్ వచ్చిన సందేశం నోటిఫికేషన్ కాదా అని తనిఖీ చేస్తుంది. ఇది అయితే నోటిఫికేషన్ ను ప్రింట్ చేస్తుంది, లేనట్టయితే సాధారణ సర్వర్ సందేశం మాదిరిగా ప్రాసెస్ చేస్తుంది. `ClientSession` `ClientSessionOptions` ద్వారా సందేశ హ్యాండ్లర్‌తో ప్రారంభించబడుతుంది.

నోటిఫికేషన్లు ఎనేబుల్ చేయడానికి, మీ సర్వర్ స్ట్రీమింగ్ ట్రాన్స్‌పోర్ట్ (ఉదా: `streamable-http`) ఉపయోగించాలి మరియు క్లయింట్ నోటిఫికేషన్లు ప్రాసెస్ చేయడానికి సందేశ హ్యాండ్లర్ అమలు చేయాలి.

## ప్రగతి నోటిఫికేషన్లు & సన్నివేశాలు

ఈ విభాగం MCP లో ప్రగతి నోటిఫికేషన్ల భావన, వాటి ప్రాముఖ్యత, మరియు Streamable HTTP ఉపయోగించి వాటిని ఎలా అమలు చేయాలో వివరిస్తుంది. మీరు ఇది మరింత అర్థం చేసుకోవడానికి ఒక ప్రాయోగిక వ్యాయామం కూడా కనుగొంటారు.

ప్రగతి నోటిఫికేషన్లు సర్వర్ నుండి క్లయింట్ కు దీర్ఘకాలిక ఆపరేషన్ సమయంలో పంపే రియల్‌టైమ్ సందేశాలు. మొత్తం ప్రక్రియ ముగియడానికి ఎదురు చూడకుండా, సర్వర్ ప్రస్తుత స్థితిని క్లయింట్ కు తెలియజేస్తూ ఉంటుంది. ఇది పారదర్శకత, ఉపయోగదారుల అనుభవం మెరుగుపరచడం, మరియు డీబగ్గింగ్ సులభతరం చేస్తుంది.

**ఉదాహరణ:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ఎందుకు ప్రగతి నోటిఫికేషన్లు ఉపయోగించాలి?

ప్రగతి నోటిఫికేషన్ల అవసరాలు ఏహెచ్చుకోండి:

- **మంచి ఉపయోగదారు అనుభవం:** పని కొనసాగుతున్నట్లు యూజర్లు అప్డేట్స్‌ను చూస్తారు, కేవలం చివరి సమయంలో కాదు.
- **రియల్-టైమ్ ఫీడ్‌బ్యాక్:** క్లయింట్లు ప్రగతి బార్లు లేదా లాగ్లను ప్రదర్శించి అప్‌కి యాప్‌ను స్పందనాత్మకంగా తెప్పిస్తాయి.
- **డీబగ్గింగ్ మరియు మానిటరింగ్ సులభతరం:** డెవలపర్లు మరియు యూజర్లు ఒక ప్రక్రియ ఎక్కడ నెమ్మదిగా లేదా నిలిచిపోయిందో చూడగలుగుతారు.

### ప్రగతి నోటిఫికేషన్లు ఎలా అమలు చేయాలి

ప్రగతి నోటిఫికేషన్లు MCP లో క్రింద చూపిన విధంగా అమలు చేస్తారు:

- **సర్వర్ పక్షం:** ప్రతి అంశం ప్రాసెస్ అయినప్పుడు `ctx.info()` లేదా `ctx.log()` ద్వారా నోటిఫికేషన్లు పంపండి. ఇది ప్రధాన ఫలితం సిద్ధం కావడానికి ముందు క్లయింట్‌కి సందేశం పంపుతుంది.
- **క్లయింట్ పక్షం:** వచ్చిన నోటిఫికేషన్లను వినిపించి ప్రదర్శించే సందేశ హ్యాండ్లర్ అమలు చేయండి. ఈ హ్యాండ్లర్ నోటిఫికేషన్లు మరియు తుది ఫలితాన్ని విడగొడుతుంది.

**సర్వర్ ఉదాహరణ:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**క్లయింట్ ఉదాహరణ:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## భద్రతా పరామర్శలు

HTTP ఆధారిత ట్రాన్స్‌పోర్ట్లతో MCP సర్వర్‌లను అమలు చేసే సమయంలో, భద్రత అత్యంత ముఖ్యమైన అంశమైంది. ఇది అనేక దాడి మార్గాలపై జాగ్రత్తలు తీసుకోవడం అవసరం.

### అవలోకనం

HTTP ద్వారా MCP సర్వర్‌లను ప్రదర్శిస్తే భద్రత ముఖ్యమవుతుంది. స్ట్రీమబుల్ HTTP కొత్త దాడి ముంచు పాయింట్లను పరిచయం చేస్తుంది మరియు జాగ్రత్తగా సెట్టింగ్స్ అవసరం.

### ముఖ్య అంశాలు

- **Origin హెడర్ ధృవీకరణ**: డిఎన్ఎస్ రీబైండింగ్ దాడులను నివారించడానికి `Origin` హెడర్‌ను ఎప్పటికప్పుడు ధృవీకరించండి.
- **లోకలహోస్ట్ బైండింగ్**: స్థానిక అభివృద్ధి కోసం సర్వర్లను `localhost` కు బంధించండి, ప్రజలకు అందుబాటులో ఉండకూడదు.
- **ఆథెంటికేషన్**: ఉత్పత్తి వాతావరణాలకు ఆథెంటికేషన్ (ఉదా: API కీలు, OAuth) అమలు చేయండి.
- **CORS**: ఆక్రమణలు పరిమితి చేయడానికి క్రాస్-ఒరిజిన్ రిసోర్స్ షేరింగ్ విధానం సెట్ చేయండి.
- **HTTPS**: ట్రాఫిక్ ఎంక్రిప్ట్ చేయడానికి ఉత్పత్తిలో HTTPS ఉపయోగించండి.

### ఉత్తమ పద్ధతులు

- ధృవీకరణ లేకుండా వచ్చే అభ్యర్థనలను ఎప్పుడూ నమ్మకండి.
- అన్ని యాక్సెస్ మరియు లోపాలను లాగ్ చేసి, పర్యవేక్షించండి.
- భద్రతా లోపాలను సరిచేయడానికి రెగ్యులర్‌గా డిపెండెన్సీలను అప్‌డేట్ చేయండి.

### సవాళ్లు
- అభివృద్ధి సౌలభ్యతతో భద్రతను సంతులితం చేసుకోవడం
- వివిధ క్లయింట్ పరిసరాలతో అనుకూలతను నిర్ధారించడం

## SSE నుండి Streamable HTTP కి అప్గ్రేడ్ చేయడం

ప్రస్తుతం Server-Sent Events (SSE) ఉపయోగిస్తున్న అనువర్తనాల కోసం, Streamable HTTP కి మారడం మీ MCP అమలులకు మెరుగైన సామర్థ్యాలు మరియు దీర్ఘకాలిక స్థిరత్వాన్ని అందిస్తుంది.

### ఎందుకు అప్గ్రేడ్ చేయాలి?

SSE నుండి Streamable HTTP కి రెండు ప్రధాన కారణాలు ఉన్నాయి:

- Streamable HTTP SSE కంటే మెరుగైన స్కేలబిలిటీ, అనుకూలత మరియు సమృద్ధిగా నోటిఫికేషన్ మద్దతును అందిస్తుంది.
- ఇది కొత్త MCP అనువర్తనాల కోసం సిఫార్సు చేయబడిన ట్రాన్స్‌పోర్ట్.

### మార్పిడి దశలు

MCP అనువర్తనాలలో SSE నుండి Streamable HTTP కి ఎలా మారాలో ఇక్కడ ఉంది:

- **సెర్వర్ కోడ్‌ని అప్డేట్ చేయండి** `mcp.run()` లో `transport="streamable-http"` ఉపయోగించడానికి.
- **క్లయింట్ కోడ్‌ని అప్డేట్ చేయండి** SSE క్లయింట్ బదులగా `streamablehttp_client` ఉపయోగించడానికి.
- **క్లయింట్‌లో సందేశ హ్యాండ్లర్‌ను అమలు చేయండి** నోటిఫికేషన్లను ప్రాసెస్ చేయడానికి.
- **ఉన్న టూల్స్ మరియు వర్క్‌ఫ్లోలతో అనుకూలత కోసం పరీక్షించండి**.

### అనుకూలత నిలుపుకోవడం

మార్పిడి సమయంలో ప్రస్తుతం ఉన్న SSE క్లయింట్లతో అనుకూలతను నిలుపుకోవడం సిఫార్సు చేయబడింది. ఇక్కడ కొన్ని వ్యూహాలు ఉన్నాయి:

- మీరు వేర్వేరు ఎండ్పాయింట్ల వద్ద SSE మరియు Streamable HTTP రెండింటినీ ఒకేసారి మద్దతు అందించవచ్చు.
- క్రమంగా క్లయింట్లను కొత్త ట్రాన్స్‌పోర్ట్‌కు మార్చండి.

### సవాళ్లు

మార్చడం సమయంలో క్రింది సవాళ్లను చర్చించుకోవాలి:

- అన్ని క్లయింట్లు అప్‌డేట్ చేయబడ్డారంటే ఖచ్చితంగా చూసుకోవడం
- నోటిఫికేషన్ డెలివరీలో తేడాలను నిర్వహించడం

## భద్రతా సూచనలు

సర్వర్ అమలులో ముఖ్యంగా MCPలో Streamable HTTP వంటి HTTP ఆధారిత ట్రాన్స్‌పోర్ట్లు ఉపయోగిస్తే భద్రతా ఇబ్బందులను గొప్ప ప్రాధాన్యతగా పరిగణించాలి.

HTTP ఆధారిత ట్రాన్స్‌పోర్ట్లతో MCP సర్వర్‌లను అమలు చేయాలంటే, అనేక దాడి దారులు మరియు రక్షణ విధానాలు మీద జాగ్రత్తగా దృష్టి సారించడం అవసరం.

### అవలోకనం

MCP సర్వర్‌లను HTTP ద్వారా పబ్లిక్‌గా అందించేటప్పుడు భద్రత అత్యంత ముఖ్యం. Streamable HTTP కొత్త దాడి ఉపరితలాలను పరిచయం చేస్తుంది మరియు జాగ్రత్తగా కాంఫిగర్ చేయాల్సి ఉంటుంది.

ముఖ్య భద్రతా సూచనలు:

- **Origin హెడర్ ధృవీకరణ**: DNS రీబైన్డింగ్ దాడులను నివారించడానికి `Origin` హెడర్‌ను ఎప్పుడు సరిచూసుకోండి.
- **লোকలహోస్ట్ బైండింగ్**: లోకల్ అభివృద్ధి కోసం సర్వర్‌లను `localhost` కు బైండ్ చేయండి, పబ్లిక్ ఇంటర్నెట్ నుండి రక్షించడానికి.
- **ఆథెంటికేషన్**: ప్రొడక్షన్ డిప్లాయ్‌మెంట్‌లకు ఆథెంటికేషన్ అమలు చేయండి (ఉదా: API కీలు, OAuth).
- **CORS**: యాక్సెస్‌ను పరిమితం చేయడానికి Cross-Origin Resource Sharing (CORS) పాలసీలు అమలు చేయండి.
- **HTTPS**: ట్రాఫిక్‌ను ఎన్‌క్రిప్ట్ చేయడానికి ప్రొడక్షన్‌లో HTTPS ఉపయోగించండి.

### ఉత్తమ ఆచారాలు

MCP స్ట్రీమింగ్ సర్వర్‌లో భద్రతను అమలు చేసేటప్పుడు పాటించాల్సిన కొన్ని ఉత్తమ ఆచారాలు:

- ధృవీకరణ లేకుండా ఏ అభ్యర్థనను విశ్వసించవద్దు.
- అన్ని యాక్సెస్ మరియు పొరపాట్లను లాగ్ చేసి పరిశీలించండి.
- భద్రతా లోపాలను ప్యాచ్ చేయడానికి డిపెండెన్సీలను సకాలంలో అప్డేట్ చేసుకోండి.

### సవాళ్లు

MCP స్ట్రీమింగ్ సర్వర్‌లలో భద్రతను అమలు చేసేటప్పుడు మీరు ఎదుర్కొనే సవాళ్లు:

- అభివృద్ధి సౌలభ్యతతో భద్రతను సంతులితం చేసుకోవడం
- వివిధ క్లయింట్ పరిసరాలతో అనుకూలతను నిర్ధారించడం

### అసైన్మెంట్: మీ స్వంత స్ట్రీమింగ్ MCP అనువర్తనం రూపొందించండి

**పరిస్థితి:**
సర్వర్ ఒక వస్తు జాబితాను (ఉదా: ఫైళ్లు లేదా డాక్యుమెంట్స్) ప్రాసెస్ చేసి, ప్రతి వస్తువు ప్రాసెస్ అయిన తర్వాత ఒక నోటిఫికేషన్ పంపుతుంది. క్లయింట్ ప్రతి వచ్చిన నోటిఫికేషన్‌ను చూపించాలి.

**దశలు:**

1. జాబితాను ప్రాసెస్ చేసి, ప్రతి అంశానికి నోటిఫికేషన్ పంపే సర్వర్ టూల్‌ను అమలు చేయండి.
2. రియల్‌టైంలో నోటిఫికేషన్లను ప్రదర్శించడానికి సందేశ హ్యాండ్లర్ ఉన్న క్లయింట్‌ను అమలు చేయండి.
3. సర్వర్ మరియు క్లయింట్ రెండింటినీ চালించి, నోటిఫికేషన్లను పరిశీలించండి.

## మరింత చదవడం & తర్వాత ఏమి చేయాలి?

MCP స్ట్రీమింగ్ ప్రయాణాన్ని కొనసాగించడానికి మరియు మీ జ్ఞానాన్ని విస్తరించడానికి, ఈ విభాగంలో అదనపు వనరులు మరియు తదుపరి సూచనలు ఉన్నాయి.

- [Microsoft: HTTP Streaming పరిచయం](https://learn.microsoft.com/aspnet/core/fundamentals/http-requests?view=aspnetcore-8.0&WT.mc_id=%3Fwt.mc_id%3DMVP_452430#streaming)

- [Microsoft: ASP.NET Core లో CORS](https://learn.microsoft.com/aspnet/core/security/cors?view=aspnetcore-8.0&WT.mc_id=%3Fwt.mc_id%3DMVP_452430)
- [Python requests: Streaming Requests](https://requests.readthedocs.io/en/latest/user/advanced/#streaming-requests)

### తర్వాత ఏమి చేయాలి?

- రియల్‌టైమ్ అనలిటిక్స్, చాట్ లేదా సహకారంగా సవరించడానికి స్ట్రీమింగ్ ఉపయోగించే మరింత కాంప్లెక్స్ MCP టూల్స్ రూపొందించండి.
- ప్రత్యక్ష UI నవీకరణల కోసం MCP స్ట్రీమింగ్‌ను ఫ్రంట్ఎండ్ ఫ్రేమ్‌వర్క్స్ (React, Vue, మొదలైనవి) తో సమన్వయపరచడం అన్వేషించండి.
